# 27 · 标准库地图与数据·AI 生态 ⭐

> 你在 Java 里靠 JDK 标准库 + Maven 上海量第三方库吃饭。Python 也一样有"batteries included"的庞大标准库，但分类和 JDK 不完全对得上；更关键的是，Python 在**数据/AI**领域几乎垄断，靠的不是语言本身（它慢，见 [05 CPython内部](05_CPython内部.md)），而是 numpy/pandas/PyTorch 这套把计算下沉到 C/CUDA 的生态。本章给你两张地图：**标准库速查地图**（哪个需求用哪个模块）和**数据/AI 生态选型地图**（numpy/pandas/matplotlib/scikit-learn/PyTorch 各管什么）。其中 numpy 部分会挖到内存布局——理解 `ndarray` 为什么快，是看懂整个数据生态的钥匙。

## ② 设计理念（为什么 Python 这么设计）

### 2.1 "batteries included"——标准库自带电池

Python 的口号是 **"batteries included"**：装好解释器，标准库就附带了正则、HTTP 服务器、JSON、SQLite、单元测试、异步框架、CSV、压缩、加密哈希……一大堆开箱即用的东西。设计动机是**让脚本和小工具零依赖就能跑**——这正是 Python 作为"胶水语言"和运维/数据脚本语言的根基。

对照 Java：JDK 也很大，但实践中你几乎必然要引第三方——日志要 SLF4J/Logback，JSON 要 Jackson，工具类要 Guava，HTTP 客户端要 OkHttp。Python 里这些**很多标准库就够用**（`logging`、`json`、`http.client`/标准库够脚本用，正式项目才上 `requests`/`httpx`）。

但"自带电池"有代价：标准库里沉淀了不少**老旧 API**（`urllib`、`os.path`），社区后来用更好的第三方或新模块替代（`requests`、`pathlib`）。所以本章地图会标注"标准库够用"还是"实践上换第三方"。

### 2.2 Python 是数据/AI 的"胶水"，计算下沉到 C/CUDA

[05 CPython内部](05_CPython内部.md) 讲过：CPython 纯解释、无 JIT、一切皆堆对象、还有 GIL，纯 Python 数值循环比 Java 慢 10~100 倍。那为什么数据/AI 圈反而**人手一个 Python**？

因为**真正的计算根本不在 Python 层跑**。numpy 的数组运算在连续内存的 C 代码里完成，PyTorch 的张量运算在 CUDA kernel 里完成。Python 只负责**表达逻辑、编排流程、调度这些 C/GPU 库**——也就是当"胶水"。这套分工的好处：

- **写起来像伪代码**：科研人员、分析师不必懂 C，就能调用世界级的数值实现。
- **算起来是 C 速度**：热点全在原生层，解释器开销被摊薄到可忽略。
- **交互式探索**：配合 Jupyter，边写边看，契合数据分析"试错"的工作方式。

> 心智模型：**Python 在数据/AI 里的角色 = 遥控器，numpy/PyTorch 才是发动机。** 你按按钮（写几行 Python），发动机（C/CUDA）干重活。

### 2.3 生态以 `ndarray` 为"公共数据底座"

整个 PyData 生态能无缝协作，靠的是一个共识：**大家都用 numpy 的 `ndarray` 作为内存里数据的通用表示**。

- pandas 的 `DataFrame` 列，底层就是 numpy 数组。
- scikit-learn 的输入输出是 `ndarray`。
- matplotlib 直接画 `ndarray`。
- PyTorch/TensorFlow 的张量能和 `ndarray` 零拷贝互转。

这就像 Java 世界大家都认 `Collection`/`Stream` 接口一样——`ndarray` 是 PyData 的**通用数据契约**。理解 `ndarray` 的内存布局（下一段），等于理解了整个生态的地基。

## ③ 实现细节（深挖 numpy）★本书重点★

### 3.1 `ndarray` 的真身：一块连续 C 内存 + 元数据

Python `list` 慢的根因（[05 CPython内部](05_CPython内部.md) 2.3 讲过）：它存的是**一排指针**，每个元素是堆上独立的 `PyObject`（一个 `int` 28 字节），数据散落各处、访问要解引用、缓存命中差。

`numpy.ndarray` 把这一切反过来。一个 `ndarray` 在 C 层是这样一个结构（简化自 `PyArrayObject`）：

```
ndarray 对象（元数据，很小）          连续数据缓冲区（在别处，很大）
┌───────────────────────────┐       ┌──┬──┬──┬──┬──┬──┬──┬──┬──┐
│ data    ─────────────────────────▶ │ 1│ 2│ 3│ 4│ 5│ 6│ 7│ 8│ ...
│ dtype   = int64 (8 字节/元素)  │   └──┴──┴──┴──┴──┴──┴──┴──┴──┘
│ ndim    = 2                   │     裸值紧密排列，无对象头、无指针
│ shape   = (2, 4)              │
│ strides = (32, 8)  字节       │
└───────────────────────────┘
```

四个关键字段：

| 字段 | 含义 | 对照 Java |
|------|------|-----------|
| `data` | 指向一整块**连续**内存的指针 | 像 `double[]` 的内部数组 |
| `dtype` | 元素类型与字节宽度（`int64`/`float64`/…） | 像数组的元素类型，但运行时可查 |
| `shape` | 各维长度，如 `(2, 4)` | 多维数组的维度 |
| `strides` | **每个维度走一步要跳多少字节** | JVM 没暴露，是 numpy 的灵魂 |

关键认知：**数据是裸值紧密排列的**（8 个 `int64` = 64 字节连续内存），没有对象头、没有指针、没有 28 字节/元素的开销。一个一千万元素的 `int64` 数组 ≈ 80 MB；同样数据的 Python `list` 要存一千万个指针 + 一千万个 int 对象，轻松 400 MB+。

```python
import numpy as np

a = np.arange(8, dtype=np.int64).reshape(2, 4)
print(a.shape)      # → (2, 4)
print(a.strides)    # → (32, 8)   行间跳 32 字节(=4个int64)，列间跳 8 字节
print(a.dtype)      # → int64
print(a.itemsize)   # → 8         每个元素 8 字节
print(a.nbytes)     # → 64        总共 64 字节，纯数据
```

### 3.2 `strides`：多维、转置、广播全靠它

`strides` 回答一个问题：**逻辑坐标 `(i, j)` 对应内存里第几个字节？** 答案是 `i*strides[0] + j*strides[1]`。这一个公式撑起了 numpy 几乎所有"看起来在搬数据、其实没搬"的魔法。

**转置不复制数据，只换 shape/strides：**

```python
a = np.arange(8).reshape(2, 4)   # strides=(32, 8)
t = a.T                          # shape=(4, 2), strides=(8, 32)  ← 只是把两个数交换
print(t.strides)                 # → (8, 32)
print(np.shares_memory(a, t))    # → True   转置和原数组共享同一块内存！
```

转置后 `(i, j)` 的寻址公式变成 `i*8 + j*32`，正好等于原数组 `(j, i)`。**没动一个字节**，只改了元数据。这和 Java 里转置一个 `int[][]` 必须逐元素拷贝形成鲜明对比。

### 3.3 向量化：在 C 层整块循环，且释放 GIL

"向量化"（vectorization）= 用**一次 numpy 调用**处理整个数组，把循环下沉到 C。对照纯 Python 循环：

```python
import numpy as np, time

n = 10_000_000
xs = list(range(n))
arr = np.arange(n, dtype=np.int64)

# 纯 Python：解释器逐元素循环，每步都拆装 int 对象、类型分派
t = time.perf_counter()
s1 = sum(x * x for x in xs)
print("纯Python:", round(time.perf_counter() - t, 3), "秒")   # → 纯Python: ~0.8 秒

# 向量化：整个平方+求和在 C 层连续内存上做一遍
t = time.perf_counter()
s2 = (arr * arr).sum()
print("向量化 :", round(time.perf_counter() - t, 3), "秒")     # → 向量化 : ~0.02 秒
print(s1 == s2)                                                # → True  (注意溢出风险，见 ⑤)
```

向量化快**几十倍**，原因正是 [05 CPython内部](05_CPython内部.md) 的三条慢因被一次性绕开：

1. **没有解释器循环**：一千万次迭代在编译好的 C 代码里跑，不经过 `ceval` 字节码分派。
2. **没有对象拆装**：直接对连续的 `int64` 裸值做运算，不碰 `PyObject`。
3. **缓存友好**：数据连续，CPU 预取和 SIMD 都能发力。

还有第四条，呼应 [20 并发与GIL](20_并发与GIL.md)：**numpy 在做大块纯计算时会主动释放 GIL**。所以 numpy 计算期间，别的线程（包括另一个 numpy 运算）能真并行——这是少数"多线程能加速 CPU 任务"的场景，根源就是 C 扩展释放了 GIL。

### 3.4 广播（broadcasting）：用 stride=0 假装维度对齐

广播让**形状不同**的数组也能逐元素运算，无需手动复制。经典例子：给矩阵每行加一个向量。

```python
m = np.arange(6).reshape(2, 3)   # shape (2, 3)
v = np.array([10, 20, 30])       # shape (3,)
print(m + v)
# → [[10 21 32]
#    [13 24 35]]   v 被"广播"到每一行
```

规则：从尾部维度对齐，长度相等或其一为 1 即可广播。**底层实现优雅得惊人**：把长度为 1（或缺失）的维度的 `stride 设为 0`。stride=0 意味着"这个维度走一步内存地址不变"——于是同一份数据被**反复读取**，制造出"被复制成多行"的假象，**实际一个字节都没多占**。

```python
v2 = v[np.newaxis, :]            # shape (1, 3), strides=(0 概念上, 8)
# 加法时第 0 维 stride=0 → 两行都读同一份 v 数据，无真实复制
```

这就是为什么广播既省内存又快：没有物化出中间的大数组，C 循环里靠 stride 算法直接寻址。Java 里你只能老老实实写双重循环或手动 tile。

### 3.5 view vs copy：共享内存还是新建

因为有 strides 这层间接，numpy 的很多操作返回的是**视图（view）**——和原数组共享同一块数据缓冲区，只是元数据不同。**切片返回视图**（不像 Python `list` 切片是复制！）：

```python
a = np.arange(10)
s = a[2:5]              # 视图，共享内存
s[0] = 999
print(a)               # → [  0   1 999   3   4   5   6   7   8   9]  原数组被改了！
print(np.shares_memory(a, s))   # → True

b = a[2:5].copy()      # 显式复制，独立内存
b[0] = -1
print(a[2])            # → 999  不受影响
```

| 操作 | 返回 | 改了会不会影响原数组 |
|------|------|--------------------|
| 基本切片 `a[2:5]`、`a.T`、`reshape` | **视图** | 会 |
| 花式索引 `a[[0,2,4]]`、布尔索引 `a[a>3]` | **复制** | 不会 |
| `a.copy()`、`a.astype()` | **复制** | 不会 |

> 这是 numpy 的头号陷阱（见 ⑤）：Java 里 `subList`/切片语义各异，但 numpy 切片**默认共享**，习惯性以为是拷贝就会埋雷。拿不准用 `np.shares_memory(a, b)` 验证，或显式 `.copy()`。

### 3.6 pandas DataFrame：建在 numpy 之上的列式表

`DataFrame` 是带行列标签的二维表，可类比"单机版 Spark DataFrame"或"内存里的一张 SQL 表"。它的底层是**列式存储**：每一列（`Series`）背后通常是一个 numpy 一维数组（同 dtype），多列拼成表。

```python
import pandas as pd
df = pd.DataFrame({"city": ["BJ", "SH", "BJ"], "sales": [100, 200, 150]})
print(df["sales"].to_numpy())   # → [100 200 150]   列底层就是 ndarray
print(df["sales"].dtype)        # → int64
```

列式布局的后果（和 Java 按行存 `List<Pojo>` 正相反）：

- **按列运算极快**：`df["sales"] * 1.1` 直接对底层 numpy 数组向量化，享受 3.3 的全部加速。
- **分组聚合高效**：`groupby` 在 C 层做哈希分组 + 向量化聚合。
- **按行遍历慢**：`for _, row in df.iterrows()` 每行造一个 `Series` 对象，是反模式（Java 思维最容易踩，见 ⑤）。

> 一句话：**DataFrame = 一束 numpy 列 + 行列标签 + 一堆 SQL/Excel 风格的操作**。性能逻辑和 numpy 完全一致——能向量化/分组就别按行循环。

### 4.2 numpy 基础速成

```python
import numpy as np

# —— 创建 ——
a = np.array([1, 2, 3])              # 从 list
z = np.zeros((2, 3))                 # 全 0
r = np.arange(0, 10, 2)              # → [0 2 4 6 8]
lin = np.linspace(0, 1, 5)           # → [0. 0.25 0.5 0.75 1.]
rng = np.random.default_rng(42)      # 新式随机数生成器（推荐）
data = rng.normal(0, 1, size=(3, 4)) # 3x4 标准正态

# —— 索引与切片（切片是视图！）——
m = np.arange(12).reshape(3, 4)
print(m[1, 2])        # → 6        单元素，逗号分隔多维
print(m[:, 1])        # → [1 5 9]  第 1 列
print(m[m > 5])       # → [ 6  7  8  9 10 11]  布尔索引（返回复制）

# —— 向量化运算（无需循环）——
print(a * 10)         # → [10 20 30]
print(a + a)          # → [2 4 6]
print(np.sqrt(a))     # → [1. 1.414 1.732]

# —— 广播 ——
print(m + np.array([0, 100, 200, 300]))   # 每行加同一向量

# —— 聚合（可指定轴）——
print(m.sum())        # → 66       全体求和
print(m.sum(axis=0))  # → [12 15 18 21]  按列求和
print(m.mean(axis=1)) # → [1.5 5.5 9.5]  按行均值
```

## ⑤ ⚠️ 易错点（Java 思维翻车）

| Java 习惯 | 翻车现场 | 正确做法 / 认知 |
|----------|---------|---------------|
| 用 `for` 循环逐元素处理大数组 | 纯 Python 循环慢几十倍（[05 CPython内部](05_CPython内部.md)） | 用 numpy **向量化**，把循环下沉到 C |
| 以为切片是复制（像 `List.subList` 的某些语义） | 改 numpy 切片**意外改了原数组** | 切片是**视图**；要独立数据显式 `.copy()`，拿不准 `np.shares_memory` |
| 以为 `int` 永不溢出（Python int 确实不溢出） | numpy `int64` 默认类型，**会静默溢出**回绕 | numpy 是定宽整数；大数用 `dtype=object`/`float64`/检查范围 |
| 在 pandas 上 `iterrows` 按行循环 | 慢到离谱，且常触发 `SettingWithCopyWarning` | 用**向量化列运算**或 `groupby`，别按行遍历 |
| 链式索引后赋值 `df[df.a>0]["b"]=1` | `SettingWithCopyWarning`，改的是临时副本，原表没变 | 用 `df.loc[df.a>0, "b"] = 1` 一次定位 |
| 什么都自己写工具方法 | 重复造轮子，还没标准库稳 | 先查标准库（`itertools`/`collections`/`pathlib`） |
| 把数值算法用纯 Python 撸 | 性能不可用 | 数据/AI 一律下沉到 numpy/pandas/PyTorch |

**dtype 溢出实锤**（这是 Java 老手最反直觉的一个，呼应 [08 数字与算术语义](08_数字与算术语义.md)）：

```python
import numpy as np

# Python 原生 int：任意精度，不溢出
print(2 ** 63)                          # → 9223372036854775808  没问题

# numpy int64：定宽，溢出回绕（和 Java long 一样会 wrap，但 Python int 不会）
a = np.array([2**62], dtype=np.int64)
print(a * 4)                            # → [0]  溢出！静默回绕，无异常
print((a * 4)[0])                       # → 0    不是报错，是错值

# 防御：用更宽类型或浮点，或在运算前检查范围
b = a.astype(np.float64)
print(b * 4)                            # → [1.844e+19]  浮点不回绕（但损失整数精度）
```

> 关键反差：**Python 的 `int` 不溢出，但你一进 numpy 就回到了 Java 定宽整数的世界，`int64` 溢出静默回绕、不抛异常。** 处理累加、计数、大整数列时务必留意 dtype。

**SettingWithCopyWarning 完整示例：**

```python
import pandas as pd
df = pd.DataFrame({"a": [1, -2, 3], "b": [10, 20, 30]})

# ❌ 链式索引：df[mask] 可能返回副本，赋值落空 + 警告
sub = df[df["a"] > 0]
# sub["b"] = 0          # SettingWithCopyWarning，且不保证改到 df

# ✅ 用 .loc 一步到位
df.loc[df["a"] > 0, "b"] = 0
print(df)
# →    a   b
#   0  1   0
#   1 -2  20
#   2  3   0
```

### 参考答案

**1.** Python `list` 存的是一排**指针**，每个元素是堆上独立的 `PyObject`（一个 int 约 28 字节），数据散落、访问要解引用。numpy `int64` 数组是**一块连续 C 内存**里紧密排列的裸值（8 字节/元素），由 `data`（数据指针）+ `dtype`（类型宽度）+ `strides`（寻址步长）描述。快几十倍的三个原因：① 向量化运算在编译好的 C 代码里整块循环，不经过解释器逐条字节码分派；② 直接对裸值运算，不做 `PyObject` 拆装与运行时类型分派；③ 大块计算时 numpy **主动释放 GIL**，且连续内存对 CPU 缓存/SIMD 友好。（呼应 [05 CPython内部](05_CPython内部.md)、[20 并发与GIL](20_并发与GIL.md)）

**2.** (a) `collections.Counter`（`Counter(text.split())`）；(b) `heapq`（堆即优先队列，`heappush`/`heappop`）；(c) `pathlib.Path`（`Path(a) / b`、`.exists()`，跨平台且面向对象，优于老的 `os.path`）。

**3.** 根因：用**纯 Python 循环遍历 numpy 数组**，把 numpy 最大的优势（向量化）彻底浪费了——每次迭代都把 numpy 标量拆成 Python 对象、解释器逐条执行、`list.append` 再装箱，比直接用 list 还慢。`sum(arr)` 也是 Python 内置 sum 而非 `arr.sum()`。向量化改法：

```python
def normalize(arr):
    return arr - arr.mean()      # 一行，整块在 C 层完成；广播自动把标量减到每个元素
```

`arr.mean()` 在 C 层求均值，`arr - mean` 通过广播向量化逐元素相减，无 Python 循环、无中间 list。

**4.** dtype 溢出。`int32` 最大约 21.4 亿（2^31-1 ≈ 2_147_483_647），而三项之和 27 亿超出范围，求和过程中**静默溢出回绕**成负数，不抛异常。修法：用更宽的 dtype。

```python
monthly = np.array([800_000_000, 900_000_000, 1_000_000_000], dtype=np.int64)
print(monthly.sum())            # → 2700000000  正确
# 或让 sum 用更宽的累加类型：
# monthly.sum(dtype=np.int64)
```

教训：numpy 是定宽整数，和 Java 的 `int`/`long` 一样会溢出回绕，**不像 Python 原生 `int` 那样自动扩展**（呼应 [08 数字与算术语义](08_数字与算术语义.md)）。

**5.**

```python
import pandas as pd

data = {
    "order_id": [1, 2, 3, 4, 5, 6],
    "city":     ["BJ", "SH", "BJ", "GZ", "SH", "BJ"],
    "amount":   [120,  80,   200,  150,  300,  90],
}
df = pd.DataFrame(data)

# ② 筛选
big = df[df["amount"] > 100]

# ③ 分组聚合
stats = big.groupby("city")["amount"].agg(["sum", "mean", "count"])
print(stats)
# →       sum   mean  count
#   city
#   BJ    320  160.0      2     (120, 200)
#   GZ    150  150.0      1
#   SH    300  300.0      1

# ④ 总额最高的城市
top_city = stats["sum"].idxmax()
print("总额最高:", top_city)        # → 总额最高: BJ
```

要点：`groupby(...).agg([...])` 一次算出多个聚合，是 pandas 替代 SQL `GROUP BY` 的核心；`idxmax()` 返回最大值对应的索引标签（城市名）。全程向量化，没有一行按行循环。
