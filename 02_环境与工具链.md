# 02 · 环境与工具链

> Java 的依赖与构建体系（JDK 多版本、Maven/Gradle、`pom.xml`、classpath 隔离、`~/.m2` 缓存）你已经烂熟。本章把这套心智模型一一映射到 Python，并深挖一个 Java 老手最容易当成"黑箱"接受的东西：**虚拟环境到底是什么、`sys.path` 怎么决定 import 找哪里、wheel 是什么格式、uv 为什么快**。看懂 ③，你就不会再把 venv 当成"某种魔法目录"，而是知道它只是改了几行配置和一条 `PATH`。

---

## ① Java 对照表

| 你在 Java 里的 | Python 对应物 | 关键差异 |
|----------------|--------------|---------|
| JDK 多版本（sdkman / jenv） | 解释器多版本（`pyenv` / `uv python`） | Java 一个 JDK 可跑所有项目；Python 常每项目一套解释器+依赖 |
| `pom.xml` / `build.gradle` | `pyproject.toml` | 项目元数据 + 依赖声明 + 工具配置，集中一处 |
| Maven Central | PyPI（pypi.org） | 公共包仓库 |
| `mvn install` / `gradle build` | `pip install` / `uv add` | 装依赖 |
| classpath 隔离（依赖在项目本地） | **虚拟环境（venv）** | ⭐ 核心差异：Python 默认装全局，要靠 venv 才隔离 |
| `~/.m2/repository` 缓存 | `~/.cache/pip`、`~/.cache/uv` wheel 缓存 | 下载缓存；uv 还做硬链接复用 |
| `jar`（含 `.class` 字节码） | `wheel`（`.whl`，含 `.py` + 预编译产物） | 都是 zip；wheel 是"装好即用"的二进制分发 |
| 源码 jar / `mvn source:jar` | `sdist`（`.tar.gz` 源码分发） | 需在目标机现场构建 |
| Gradle 锁文件 / `package-lock.json` | `uv.lock` / `poetry.lock` | 锁定全部传递依赖的精确版本+哈希 |
| `java -jar app.jar` | `python app.py` / `python -m pkg` | 运行入口 |
| `System.getProperty` / `Thread.currentThread().getContextClassLoader()` | `sys` 模块（`sys.path`、`sys.executable`） | 运行期环境内省 |

## ② 设计理念（为什么 Python 这么设计）

### 2.1 依赖装在哪：全局 site-packages vs 项目本地 classpath

这是 Java 老手必须重建的第一个直觉。

**Java 的模型是"项目本地"的**：Maven 把 jar 下到 `~/.m2`，但**运行时靠 classpath 把本项目要的那几个 jar 拼成一条路径**喂给 JVM。项目 A 用 `guava-30`、项目 B 用 `guava-32`，各自的 classpath 互不相干，你几乎感受不到"全局污染"。隔离是**运行时按需组装**出来的。

**Python 的模型默认是"全局共享"的**：`pip install requests` 会把包解压进解释器的 `site-packages/` 目录，`import requests` 时解释器扫 `sys.path` 找到它。问题来了——如果项目 A 要 `requests==2.20`、项目 B 要 `requests==2.31`，全局 `site-packages` **只能装下一个版本**，第二个项目直接冲突。Python 没有 classpath 这种"每次启动现拼依赖列表"的机制，import 永远去固定的几个目录找。

> 一句话：**Java 的隔离是运行时组装（classpath），Python 的隔离要靠物理上换一套 `site-packages`（venv）**。虚拟环境就是为了把"全局共享"重新变回"项目本地"。

### 2.2 为什么需要虚拟环境

既然全局只能有一个版本，解药就是：**给每个项目一份独立的 `site-packages`**。这就是 venv 的全部动机——它不是什么沙箱或容器，只是"换一个装包和找包的目录"。这等价于强行把 Java 的"每项目独立依赖"模型搬到 Python 上。

为什么 Java 不需要这层？因为 classpath 天生就是每项目独立的，JVM 启动时你已经显式告诉它用哪些 jar 了。Python 的 import 没有"启动参数级别的依赖清单"，只有一套全局目录，所以必须靠 venv 在文件系统层面做隔离。

### 2.3 从 pip+requirements 到 pyproject+lock+uv 的演进

Python 打包工具的历史是一部"逐步补齐 Maven 早就有的能力"的历史：

1. **`pip` + `requirements.txt`**（远古）：一行一个包。问题是它**只记你直接写的包，不锁传递依赖**。`flask` 会拉一堆间接依赖，今天装到的版本和半年后装到的可能不同 → "在我机器能跑"。相当于 Maven 只有 `<dependencies>` 却没有依赖解析锁定。
2. **`setup.py` / `setup.cfg`**：能打包发布，但用可执行的 Python 脚本描述元数据，既不安全也难解析。
3. **`pyproject.toml`**（PEP 518/621，现代标准）：声明式的项目描述文件，**这才是真正对标 `pom.xml` 的东西**——元数据、依赖、构建后端、各种工具配置全集中在一个 TOML 里。
4. **锁文件 + uv**（2024 起）：`uv.lock` 锁定**整棵依赖树**的精确版本与哈希，配合 Rust 写的 `uv` 把"解释器管理 + venv + 解析 + 安装 + 打包"收敛成一个飞快的工具。

> 贯穿这条演进线的理念只有一个词：**可复现性（reproducibility）**。Maven 靠 Central 的不可变坐标 + 你锁死的版本就能复现构建；Python 早年因为"不锁传递依赖"长期做不到，`pyproject.toml` + lock 文件 + uv 就是把这块短板补齐。Java 老手要把"提交锁文件、CI 从锁文件还原"当成和提交 `pom.xml` 一样天经地义。

## ③ 实现细节（底层怎么实现的）★本章重点★

这一段把 venv、`sys.path`、wheel、uv 全部拆开看里面。

### 3.1 虚拟环境到底是什么：一个目录 + 一个配置文件 + 一条 PATH

很多人以为 venv 是某种容器或沙箱。**不是**。它就是一个普通目录，标准库 `venv` 模块（`python -m venv .venv`）生成。我们直接看它的结构：

```
.venv/
├── pyvenv.cfg              # ★关键：告诉解释器"我是个 venv"
├── bin/                    # Windows 上是 Scripts/
│   ├── python              # → 指向真正解释器的 symlink（或副本）
│   ├── python3             # → 同上
│   ├── pip
│   └── activate            # 激活脚本（改 PATH 用）
├── lib/
│   └── python3.12/
│       └── site-packages/  # ★这个 venv 独享的包目录，pip 装这里
└── include/
```

**核心机关只有三个：**

**(1) `pyvenv.cfg`** —— venv 的身份证。打开看看：

```ini
home = /Users/you/.local/share/uv/python/cpython-3.12.7/bin
include-system-site-packages = false
version = 3.12.7
```

启动时，`.venv/bin/python` 会向上找到同级目录的 `pyvenv.cfg`。**只要这个文件存在，解释器就进入"虚拟环境模式"**：它把 `site-packages` 定位到 `.venv/lib/.../site-packages`，而不是全局那套。`home` 指向真正的解释器在哪，`include-system-site-packages = false` 决定要不要把全局包也纳入（默认不纳入，所以你的 venv 是干净的）。

**(2) `bin/python`** —— 不是一份完整的 Python 拷贝，而是一个 **symlink 指回真正的解释器**（Windows 上或某些配置下是小副本 + redirector）。venv 不复制几十 MB 的解释器，它复用宿主解释器的标准库，只是用 `pyvenv.cfg` 把"包目录"重定向到自己这份。所以建 venv 几乎是瞬间的（对照：它根本没拷贝 runtime）。

**(3) `activate`** —— 激活的本质就一行：**改 `PATH`**。看脚本核心：

```bash
# activate 里实际干的事（简化）
export VIRTUAL_ENV="/path/to/.venv"
export PATH="$VIRTUAL_ENV/bin:$PATH"     # ★把 .venv/bin 插到 PATH 最前
```

激活后你敲 `python`，shell 按 `PATH` 从左到右找，**先命中 `.venv/bin/python`**，于是用的就是这个 venv。`deactivate` 只是把 `PATH` 还原。**所谓"激活"没有任何魔法，就是临时把 venv 的 bin 目录排到 PATH 最前面。**

> 推论：你完全可以不激活，直接 `./.venv/bin/python app.py`——效果一样。`uv run` 正是这么做的：它绕过激活，直接用 venv 里的解释器路径执行，所以 `uv run` 从来不需要你 `source activate`。

### 3.2 `sys.path`：import 到底去哪儿找（呼应第 18 章）

解释器启动后，`import X` 按什么顺序找 `X`？答案是扫 **`sys.path`**——一个目录字符串组成的列表，相当于 Python 版的 classpath，但**是在解释器启动时动态算出来的**。在 venv 里打印它：

```python
import sys
for p in sys.path:
    print(p)
# →
# ''                                              # 当前脚本所在目录（或 '' 表 cwd）
# /Users/.../.local/.../python3.12.zip            # 标准库的 zip 形式
# /Users/.../python3.12                           # 标准库源码目录
# /Users/.../python3.12/lib-dynload               # 标准库 C 扩展(.so)
# /path/to/.venv/lib/python3.12/site-packages     # ★venv 的第三方包目录
```

构成顺序（也就是 import 查找优先级）：

1. **脚本目录 / `''`（当前工作目录）** —— 这就是为什么你建个 `random.py` 会 shadow 掉标准库 `random`（经典翻车，见 ⑤）。
2. **标准库目录**（zip + 源码 + C 扩展）。
3. **`site-packages`**（venv 里就是 venv 那份，全局就是全局那份）—— 第三方包都在这。

谁决定第 4 项指向 venv 还是全局？正是 3.1 的 `pyvenv.cfg`。启动时 `site` 模块读它，把 `site-packages` 计算成 venv 那份。**所以"换 venv"在底层 = "换 `sys.path` 里 site-packages 那一项"**。

import 机制本身（finder/loader/`sys.meta_path`、`.pyc` 缓存、`importlib`）属于 [18 章 模块包与导入](18_模块包与导入.md) 的范畴，这里你只需记住：**venv 通过改写 `sys.path` 来实现依赖隔离**。

```python
import sys
print(sys.executable)   # 当前用的是哪个 python → 一眼看出在不在 venv 里
print(sys.prefix)       # venv 根目录（在 venv 里就是 .venv/）；对比 sys.base_prefix（宿主）
# sys.prefix != sys.base_prefix  →  说明你正处在一个虚拟环境里
```

### 3.3 wheel(.whl) vs sdist：两种分发格式

`pip install requests` 时，pip 从 PyPI 下的可能是两种东西之一：

| | **wheel（`.whl`）** | **sdist（`.tar.gz`）** |
|---|---|---|
| 本质 | 预编译的 **zip** 包 | 源码归档 |
| 装的时候 | **直接解压**进 site-packages | 要在你机器上**现场 build** |
| 类比 Java | `jar`（已编译，开箱即用） | `mvn source:jar`（需本地构建） |
| 含 C 扩展时 | 已为某平台编好 `.so`/`.pyd` | 要本地有编译器现编 |
| 速度 | 快 | 慢，且可能因缺编译器/头文件失败 |

wheel 文件名编码了兼容性信息（PEP 427），这点很像 Java 的 classifier：

```
numpy-2.1.0-cp312-cp312-macosx_11_0_arm64.whl
      └版本─ └Py标签┘ └ABI┘  └─── 平台标签 ───┘
```

`cp312` = CPython 3.12，`arm64` = Apple 芯片。pip 会**挑一个和你解释器+平台匹配的 wheel** 下载。如果 PyPI 上没有匹配的 wheel，只有 sdist，pip 就得现场编译——这正是新手装 `numpy`/`lxml`/`psycopg` 时"突然要装一堆编译工具还失败"的根源。`.whl` 解压开就是 `包名/` 目录 + `包名-版本.dist-info/`（元数据），跟 jar 解压是一堆 `.class` + `META-INF/` 完全同构。

### 3.4 pip 的依赖解析 + `.pyc` 缓存

`pip install` 背后两步：

1. **依赖解析**：pip 用一个**回溯式解析器**（backtracking resolver），从你要的包出发，递归读每个包的元数据里声明的依赖区间，求一组同时满足所有约束的版本。这和 Maven 的依赖调解类似，但有个有名的差异——pip 历史上对"同一个包被两条路径要求不同版本"没有 Maven 的"最近优先"那种确定规则，2020 年换成回溯解析器后才严格起来。复杂依赖树解析慢正是因为回溯可能反复试错。
2. **安装 + 编译字节码**：解压 wheel 进 site-packages 后，pip 会把 `.py` 预编译成 `.pyc` 放进 `__pycache__/`（字节码缓存，对照 Java 的 `.class`，详见 [05 CPython 内部](05_CPython内部.md)）。这样首次 import 不必现编。

### 3.5 uv 为什么快：Rust + 全局硬链接缓存

`uv` 比 `pip` 快 10～100 倍，不是玄学，是几个具体工程决定：

1. **Rust 实现，无解释器启动开销**：pip 本身是 Python 写的，每次跑都要起解释器、import 一堆模块；uv 是编译好的原生二进制，启动即干活。
2. **全局缓存 + 硬链接（hard link）安装**：uv 维护一个全局缓存（`~/.cache/uv`），里面存解压好的包内容。给某个项目装包时，它**不复制文件，而是在 venv 的 site-packages 里建硬链接指向缓存中的同一份 inode**。结果是：同一个 `numpy` 在你 50 个项目里只占一份磁盘空间，且"安装"几乎只是建链接，瞬间完成。对照 Maven 的 `~/.m2` 是"共享一份 jar、classpath 引用它"——uv 用硬链接在文件系统层达到了类似的"装一次到处用"，但每个 venv 看起来仍是独立的目录。
3. **并行下载 + 并行解析**：充分用多核同时拉包、同时解析。
4. **更快的解析算法 + 缓存的元数据**：避免重复网络往返。

> 心智模型：**pip 每次都"复制安装"，uv 是"全局缓存 + 硬链接共享"**。这就是为什么你第一次 `uv add` 某大包要下载，之后任何项目再要它都几乎零成本——和 Maven 第二次构建不再重新下 jar 是一个道理。

## ④ 使用方法与最佳实践

### 4.1 解释器版本：别用系统自带的

macOS 自带的可能是 **Python 3.9.6**（2021 年旧版）。**铁律：永远不要往系统 Python 装包、也别拿它开发**——OS 工具依赖它，弄乱会影响系统。这等价于你不会拿系统预装的某个老 JDK 跑生产。

```bash
# 安装 uv（官方一键脚本）
curl -LsSf https://astral.sh/uv/install.sh | sh

# 用 uv 装独立解释器，完全不碰系统 Python
uv python install 3.12
uv python list
```

### 4.2 uv 完整工作流（推荐主线）

```bash
uv init weather          # 建项目骨架：pyproject.toml + .python-version + .git + .gitignore
cd weather
uv add httpx             # 解析+装依赖；自动建 .venv、写进 pyproject.toml、生成 uv.lock
uv add --dev pytest ruff # 开发依赖（对照 Maven test scope）
uv run python main.py    # 在项目环境里跑，无需 activate（见 3.1 为什么不用激活）
uv lock                  # 显式重算锁文件
uv sync                  # ★从 uv.lock 精确还原环境——CI、新同事 clone 后第一条命令
uv remove httpx          # 删依赖并更新 lock
```

`uv sync` 是可复现性的关键：它**只信 `uv.lock`**，在任何机器上还原出逐个哈希一致的环境，对标 `mvn` 在锁定版本下的确定性构建。

### 4.3 手动 venv（理解原理用，对照 3.1）

```bash
python3.12 -m venv .venv          # 生成 3.1 描述的目录结构
source .venv/bin/activate          # 改 PATH（Windows: .venv\Scripts\activate）
# 提示符出现 (.venv)，此时 which python → .venv/bin/python
python -c "import sys; print(sys.prefix != sys.base_prefix)"  # True，确认在 venv 里
pip install requests               # 只进这个 venv 的 site-packages
deactivate                         # 还原 PATH
```

### 4.4 pip 基本命令（接手老项目时要会）

```bash
pip install "requests>=2.31"       # 装并约束版本
pip install -r requirements.txt    # 从清单批量装
pip list                           # 已装列表
pip show requests                  # 看某包元数据（版本、位置、依赖）
pip freeze > requirements.txt      # 冻结当前精确版本（穷人版锁，不含哈希）
pip uninstall requests
```

### 4.5 pyproject.toml 结构

```toml
[project]
name = "weather"
version = "0.1.0"
requires-python = ">=3.12"
dependencies = [                    # 运行时依赖（对照 pom 的 compile scope）
    "httpx>=0.27",
    "fastapi>=0.110",
]

[project.optional-dependencies]
dev = ["pytest>=8.0", "ruff", "mypy"]   # extras，按需安装：uv sync --extra dev

[build-system]                      # 构建后端（对照 Maven 的打包插件）
requires = ["hatchling"]
build-backend = "hatchling.build"

# 工具配置集中在 [tool.*]（对照 Gradle 里配各种插件）
[tool.ruff]
line-length = 100

[tool.mypy]
strict = true
```

锁文件（`uv.lock`）记录**全部传递依赖的精确版本与哈希**——**提交进 git**，和提交 `pom.xml` 一个道理。

### 4.6 IDE 选对解释器

让 IDE 指向项目的 `.venv`，否则它用全局解释器，找不到你装的包，到处标红。

- **PyCharm**：Settings → Project → Python Interpreter → 选 `.venv/bin/python`。
- **VS Code**：命令面板 `Python: Select Interpreter` → 选 `.venv`。

这是新手第一号"环境没配对"问题，本质就是 IDE 用错了 3.2 那条 `sys.path`。

### 4.7 REPL：随手验证神器

Java 9 才有 jshell，Python 一直有。验证语法、查对象有什么方法，比写测试快：

```python
>>> "user_42".upper()
'USER_42'
>>> dir(str)            # str 有哪些方法
>>> help(str.split)     # 看某方法文档
>>> import sys; sys.executable   # 当前哪个解释器
```

### 4.8 `.gitignore` 掉 `.venv`

`.venv/` 是本地产物（含平台相关的 `.so` 和 symlink），**绝不提交**，和你不提交 `target/` 一样。只提交 `pyproject.toml` + `uv.lock`。

```gitignore
.venv/
__pycache__/
*.pyc
```

## ⑤ ⚠️ 易错点（Java 思维翻车）

| Java 习惯 | 翻车现场 | 正确做法 / 认知 |
|----------|---------|----------------|
| 依赖天然项目隔离，没 venv 概念 | 直接全局 `pip install`，项目 A 升级 `requests` 把项目 B 搞挂（全局只有一个版本） | 每项目一个 `.venv`，`uv init`/`uv add` 自动管 |
| 用系统/IDE 默认 Python 开发 | 给系统 Python 装包搞坏 OS 工具，或版本太旧（3.9）跑不了新语法 | 用 `uv python install` 装独立解释器，永不碰系统 Python |
| 以为装完包到处能 import | 没激活 venv（或 IDE 选错解释器），`ModuleNotFoundError: No module named 'httpx'` | 确认 `sys.executable` 指向 `.venv`；用 `uv run` 免激活；IDE 选对解释器 |
| 把文件随便命名（Java 靠包名隔离） | 建个 `random.py` 或 `email.py`，结果 `import random` 拿到自己的文件（`sys.path` 第一项是脚本目录） | 别用标准库/三方库同名做文件名；冲突时看 `module.__file__` |
| 只写 `requirements.txt` 不锁传递依赖 | "我机器能跑"，CI 装到不同版本的间接依赖直接报错 | 用 `uv.lock` 锁全树，CI `uv sync` 还原 |
| 觉得装包慢/占空间是常态 | 每个项目重新下载、复制一份大包，磁盘爆炸 | 用 uv 全局缓存 + 硬链接，装一次到处复用（见 3.5） |
| 把 `.venv/` / `__pycache__/` 提交 git | 仓库膨胀、含平台相关 symlink/.so，换机器直接坏 | `.gitignore` 掉，只提交 `pyproject.toml` + 锁文件 |

## ⑥ 练习题

### 基础理解

1. 从**依赖安装位置**的角度，解释为什么 Python 需要虚拟环境而 Java 项目"天然"不太需要。提示：全局 `site-packages` vs 运行时 classpath。
2. `.venv/pyvenv.cfg` 和 `activate` 脚本各自起什么作用？为什么 `uv run` 不需要先 `source activate`？

### 找坑改错

3. 一个 Java 同事在 `~/projects/mailer/` 下写了文件 `email.py`，内容里 `import smtplib` 一切正常，但他在同目录另写 `send.py`：

```python
# send.py
import email.mime.text   # 想用标准库的 email 包
msg = email.mime.text.MIMEText("hi")
```

运行 `python send.py` 报 `ModuleNotFoundError: No module named 'email.mime'`。从本章 `sys.path` 的构成解释为什么，并给出修复办法。

4. 下面是某项目交接给你的全部"依赖管理"，指出它的可复现性问题，并写出迁移到现代工具链的命令序列：

```
# requirements.txt
flask
requests
pandas
```

### 综合实战

5. 你要给团队搭一个新的后端服务 `orders-api`，要求：用 Python 3.12、依赖 `fastapi` 和 `httpx`、开发期需要 `pytest` 和 `ruff`、CI 能逐字节复现环境、仓库干净不含本地产物。写出从零到 CI 可复现的完整命令序列与需要提交/忽略的文件清单。

---

### 参考答案

**1.** Java 把依赖下到 `~/.m2`，但运行时靠 **classpath 在启动时现拼**出本项目要的 jar 列表，每项目各拼各的，天然隔离。Python 的 `pip install` 默认把包装进解释器**全局**的 `site-packages`，`import` 永远扫固定的 `sys.path` 目录，全局对同一个包**只能存一个版本**，多项目需要不同版本就冲突。虚拟环境给每个项目一份独立的 `site-packages`（通过 `pyvenv.cfg` 重定向 `sys.path`），把 Java 那种"项目本地依赖"在文件系统层面还原出来。

**2.** `pyvenv.cfg` 是 venv 的"身份证"：解释器启动时读它，据此把 `site-packages` 定位到 `.venv` 那份而非全局，并记录宿主解释器位置（`home`）。`activate` 脚本本质只是把 `.venv/bin` 插到 `PATH` 最前，让你敲 `python` 时先命中 venv 里的解释器——纯粹是改环境变量，没有魔法。`uv run` 直接用 `.venv/bin/python` 的**绝对路径**执行，绕过了"靠 PATH 查找"这一步，所以根本不需要激活。

**3.** `sys.path` 的第一项是**脚本所在目录**（这里是 `~/projects/mailer/`）。运行 `send.py` 时，该目录排在标准库前面，`import email` 先命中同目录下那个自定义的 `email.py`，于是 `email` 这个名字被**遮蔽（shadow）**成了用户文件，它里面当然没有 `mime` 子模块 → `No module named 'email.mime'`。修复：把 `email.py` 改名（如 `mailer_email.py`），删掉对应的 `__pycache__/email.*.pyc`。教训：不要用标准库/三方库同名做文件名；排查时 `import email; print(email.__file__)` 一眼看出加载的是哪个。

**4.** 问题：三个包都**无版本号、且不锁传递依赖**。今天装到 `flask 3.0 + pandas 2.2`，半年后重装可能变成完全不同的版本，连它们拉的间接依赖也漂移 → "我机器能跑、CI/同事报错"，无法复现历史构建。迁移：

```bash
uv init                      # 在项目里生成 pyproject.toml
uv add flask requests pandas # 解析、装、写进 pyproject、生成 uv.lock（锁全树+哈希）
# 删掉旧 requirements.txt
git add pyproject.toml uv.lock
echo ".venv/" >> .gitignore
git add .gitignore && git commit -m "迁移到 uv + 锁文件"
# 此后任何机器/CI：uv sync  → 逐哈希还原一致环境
```

**5.** 完整序列：

```bash
uv python install 3.12            # 装独立解释器，不碰系统 Python
uv init orders-api                # 生成 pyproject.toml + .python-version + .gitignore
cd orders-api
uv add fastapi httpx              # 运行时依赖 → pyproject [project.dependencies] + uv.lock
uv add --dev pytest ruff          # 开发依赖 → [dependency-groups]/optional-dependencies
uv run pytest                     # 验证环境（免激活，直接用 .venv 解释器）
git add pyproject.toml uv.lock .python-version .gitignore
git commit -m "init orders-api"
```

CI 里只需 `uv sync`（从 `uv.lock` 逐哈希还原），再 `uv run pytest`。

- **提交**：`pyproject.toml`、`uv.lock`、`.python-version`、源码。
- **忽略**（`.gitignore`）：`.venv/`、`__pycache__/`、`*.pyc`。
