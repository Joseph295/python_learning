# 18 · 模块、包与导入 ⭐

> 在 Java 里 `import` 只是个编译期的"名字别名"——真正的加载是 JVM 在第一次用到某个类时才惰性触发的，`package` 必须对应目录、`classpath` 决定去哪找 `.class`。Python 的 `import` 长得几乎一样，神却完全不同：**`import` 是一条运行时执行的语句，它会从头到尾跑一遍目标模块的顶层代码**，把结果缓存成一个单例对象。理解这条"导入即执行"的模型，你才会懂循环导入为什么炸、为什么 `if __name__ == "__main__"` 是必需的、为什么自己写个 `random.py` 会让标准库瞬间失灵。本章把 CPython 的 import 系统（finder / loader / `sys.modules` / `sys.meta_path`）整个拆开给你看。

---

## ① Java 对照表

| 概念 | Java | Python | 关键差异 |
|------|------|--------|---------|
| 编译单元 / 模块 | `.class`（一个类一个文件） | **模块 = 一个 `.py` 文件** | Python 一个文件可装任意多个类/函数/变量 |
| 名字空间组织 | `package`（必须匹配目录） | **包 = 含代码的目录** | 包目录可有 `__init__.py`，也可没有（命名空间包） |
| 引入声明 | `import com.x.Foo;` | `import x` / `from x import Foo` | Java 是编译期符号；Python 是**运行时执行语句** |
| 查找路径 | `classpath`（`-cp`/`CLASSPATH`） | **`sys.path`**（呼应 [02 环境与工具链](02_环境与工具链.md)） | 一个 list，运行时可改 |
| 加载器 | `ClassLoader`（双亲委派） | **finder + loader**（`sys.meta_path`） | 可插拔，纯 Python 可自定义 |
| 加载产物缓存 | 方法区 / 已加载类表 | **`sys.modules`（dict）** | 模块是**单例**，第二次 import 直接返回缓存 |
| 字节码缓存 | 无（`.class` 即字节码） | **`.pyc`（`__pycache__/`）**（呼应 [05 CPython内部](05_CPython内部.md)） | 编译产物缓存，源码没改就复用 |
| 加载时机 | **首次主动使用时惰性触发** | **import 语句执行那一刻，整个模块顶层全跑一遍** | Python 更"急"，且会执行副作用 |
| 静态初始化 | `static {}` 块 | **模块顶层代码**（import 时执行一次） | 顶层任意语句都是"静态初始化块" |
| 资源文件 | jar 里 `getResource()` | `importlib.resources`（包数据） | 见 [22 文件IO与系统交互](22_文件IO与系统交互.md) |
| 可执行入口 | `public static void main` | **`if __name__ == "__main__"`** | 无编译期入口概念，靠运行时变量判断 |
| 相对引用 | 无（包名总是绝对的） | **显式相对导入 `from . import x`** | Java 没有对应物 |

> 一句话记牢：**Java 的 `import` 是"告诉编译器这个短名指向哪个全名"，零运行成本；Python 的 `import` 是"现在就去把那个模块从头执行一遍并绑定到一个名字"。** 后者带副作用、有顺序、可能失败——这正是本章所有坑的源头。

## ② 设计理念（为什么 Python 这么设计）

### 2.1 模块即对象，即命名空间

在 [01 设计哲学与心智模型](01_设计哲学与心智模型.md) 里你已经知道"命名空间本质是 dict"。模块把这个理念贯彻到底：

**一个模块就是一个普通对象**（类型是 `module`），它的全局变量表就是它的 `__dict__`。`import json` 之后，`json` 是一个你能传递、`type()`、`dir()`、甚至塞进容器的一等对象。

```python
import json
print(type(json))                 # 输出: <class 'module'>
print(json.dumps is json.__dict__["dumps"])  # 输出: True —— 属性访问就是查 __dict__
```

对比 Java：`com.fasterxml.jackson` 这个 package 不是运行时对象，你不能把"一个 package"赋值给变量。Python 里 `m = json` 完全合法，因为模块只是对象。这种统一性让 `importlib`、依赖注入、插件系统都能用普通对象操作来实现，不需要反射那一套特殊 API。

### 2.2 包用目录组织，`__init__.py` 是"包的模块体"

Java 的 package 名必须和目录结构严格对应（`com.x.Foo` 必在 `com/x/Foo.class`）。Python 类似——**包就是目录**——但多了一个角色：目录里的 `__init__.py` 文件**就是这个包"作为模块"时执行的代码**。

```
shop/                 # 包（目录）
├── __init__.py       # import shop 时执行的就是它；shop.__dict__ 来自它
├── models.py         # 子模块 shop.models
└── cli.py            # 子模块 shop.cli
```

`import shop` 执行 `shop/__init__.py`；`import shop.models` 执行 `shop/models.py`。`__init__.py` 的存在让"包"既是命名空间容器，又能有自己的初始化逻辑和对外 API——这是 Java package 没有的能力（Java 靠 `package-info.java` 也只放注解/文档，不能跑初始化代码）。

### 2.3 "导入即执行"——和 Java 类的惰性加载根本不同

这是 Java 老手最大的认知缺口，必须单独强调。

- **Java**：`import` 不加载任何东西，只是给编译器的别名。类的真正加载（解析、验证、`static{}` 初始化）由 JVM 在**第一次主动使用**（new、调静态方法、读静态字段）时**惰性**触发。
- **Python**：`import shop.models` 这条语句**当场**就会把 `shop/__init__.py` 和 `shop/models.py` 的**顶层代码从第一行执行到最后一行**。模块里写的 `print(...)`、`CONNECT_DB()`、注册逻辑——全在 import 那一刻跑掉。

```python
# 文件 config.py 顶层有：
# print("config 模块被加载了！")
# DB_URL = os.environ["DB_URL"]
import config     # ← 就在这一行，上面两句立刻执行
```

所以 Python 的模块顶层 ≈ Java 的 `static{}` 静态初始化块，但触发时机是**import 语句执行时**而非"首次使用类时"。这带来两个后果：模块顶层别写重活（拖慢 import，见易错点），以及——顺序很重要，于是有了循环导入问题（③ 详解）。

### 2.4 为什么坚持"显式相对导入"

Python 2 曾有"隐式相对导入"（`import models` 在包内会先找同包的 `models`），结果是包内代码和顶层脚本行为不一致、易和标准库撞名。Python 3 的取舍是**显式胜于隐式**（The Zen）：包内引用同包模块必须写成 `from . import models` 或 `from .models import Order`，那个点号明确表示"从当前包找"。绝对导入则永远从 `sys.path` 的根开始找。规则清晰，代价是直接运行包内脚本时相对导入会失效（这是设计的逻辑后果，⑤ 详解）。

## ③ 实现细节（import 系统底层全过程）★本书重点★

### 3.1 一次 `import foo` 到底发生了什么

CPython 的 import 机制由 `importlib`（大部分用 Python 写，引导部分冻结进解释器）实现。一条 `import foo` 的完整流程：

```
import foo
   │
   ① 查 sys.modules[ 'foo' ] —— 缓存命中？
   │       命中 → 直接返回那个模块对象（不再执行！）  ← 模块单例的根基
   │       未命中 ↓
   ② 遍历 sys.meta_path 里的每个 finder，调 find_spec('foo', ...)
   │       谁先返回非 None 的 ModuleSpec，就用谁
   │       全返回 None → ModuleNotFoundError
   │       拿到 spec ↓
   ③ 用 spec.loader 创建空模块对象 module
   │       **先把 module 放进 sys.modules['foo']**（注意：执行前就放！）
   │       ↓
   ④ loader.exec_module(module) —— 执行模块顶层代码，填充它的 __dict__
   │       （执行期间若再 import foo，第①步就能从 sys.modules 拿到这个半成品）
   │       ↓
   ⑤ 在当前命名空间把名字 'foo' 绑定到该模块对象
```

第 ③ 步"执行前就把模块放进 `sys.modules`"是循环导入能"部分工作"也能"炸"的关键，记住它。

### 3.2 `sys.modules`：模块单例缓存（对照 JVM 已加载类表）

`sys.modules` 是一个 `{模块名: 模块对象}` 的全局 dict，相当于 JVM 每个 ClassLoader 维护的"已加载类表"。**第二次 import 同一个模块，第 ① 步就命中缓存、直接返回，顶层代码不会再跑**：

```python
import sys
import json
print(sys.modules["json"] is json)   # 输出: True
import json as json2                  # 第二次 import：命中缓存，json.__init__ 不再执行
print(json2 is json)                  # 输出: True —— 同一个对象，模块是单例
```

这解释了一个常见困惑：**你在 REPL 里改了模块文件，再 `import` 却没生效**——因为缓存命中，根本没重新读文件（解决见 4.7 `importlib.reload`）。

你甚至能手动往 `sys.modules` 塞东西来打桩测试（mock 整个模块），这是 Java 改 ClassLoader 才能干的事。

### 3.3 `sys.meta_path`：可插拔的查找链（对照双亲委派）

第 ② 步遍历的 `sys.meta_path` 是一个 finder 列表。默认三个：

```python
import sys
for f in sys.meta_path:
    print(f)
# 输出:
# <class '_frozen_importlib.BuiltinImporter'>     内置模块（sys、_io 等编译进解释器的）
# <class '_frozen_importlib.FrozenImporter'>      冻结模块（importlib 自身等）
# <class '_frozen_importlib_external.PathFinder'>  ★最常用：基于 sys.path 找文件★
```

这条链类比 JVM 的双亲委派：先问内置、再问冻结、最后才让 `PathFinder` 去 `sys.path` 找磁盘文件。**`PathFinder` 才是把 `sys.path`（呼应 [02 环境与工具链](02_环境与工具链.md)）和具体加载器连起来的那一环**：它遍历 `sys.path` 的每个目录，用 path hooks 找到能处理该位置的 loader。

**finder 与 loader 分工**（对照 Java 的 `ClassLoader`，Python 把它拆成两半）：

| 角色 | 职责 | Java 类比 |
|------|------|----------|
| **finder** | 回答"这个模块在哪、用什么 loader 加载"，产出一个 `ModuleSpec` | `ClassLoader.findClass` 的"定位"部分 |
| **loader** | 真正读取源码/字节码、创建并执行模块对象 | `ClassLoader.defineClass` 的"加载"部分 |
| **ModuleSpec** | 描述模块的元数据（名字、loader、origin 路径、是否包） | 无直接对应 |

看一个真实 spec：

```python
import json
print(json.__spec__)
# ModuleSpec(name='json',
#            loader=<...SourceFileLoader object...>,         ← 普通 .py 用源码加载器
#            origin='.../python3.12/json/__init__.py',       ← 它是个包（有 __init__.py）
#            submodule_search_locations=['.../json'])         ← 包才有这个，子模块去这找
```

因为整条链是纯 Python 可插拔的，所以"从 zip 导入"（`zipimport`）、"从网络导入"、import hook 工具都能实现——这是比 JVM ClassLoader 更开放的扩展点。

### 3.4 `.pyc` 字节码缓存（呼应 05 章）

`SourceFileLoader` 加载 `.py` 时：先看 `__pycache__/foo.cpython-312.pyc` 是否存在且新鲜（比对源文件 mtime + size，或哈希），新鲜就直接读字节码跳过编译；否则把 `.py` 编译成字节码、执行、并把字节码写回 `.pyc` 供下次复用。

```
shop/
├── models.py
└── __pycache__/
    └── models.cpython-312.pyc   # 字节码缓存，按解释器版本命名
```

这呼应 [05 CPython内部](05_CPython内部.md)：`.pyc` 就是 Python 版的"`.class`"，只是它是 import 时按需生成的缓存而非提前编译的产物。删掉无害（下次自动重建），不该提交 git。注意它**只缓存编译结果，不缓存执行结果**——每次进程启动后第一次 import 仍要执行顶层代码。

### 3.5 ★循环导入为什么炸★（用最小例子复现）

把 3.1 的流程和 2.3 的"导入即执行"合起来，就能精确解释循环导入。

**场景**：`a.py` 顶层 `import b`，`b.py` 顶层又用到 `a` 的某个名字。

```python
# a.py
print("a: 开始执行")
import b                     # ← 执行到这里，转去执行 b
print("a: import b 完成")
X = "a-value"               # ← X 在 import b 之后才定义！
def get_x():
    return X

# b.py
print("b: 开始执行")
import a                     # a 已在 sys.modules（半成品），命中缓存、不重新执行
print("b: 拿到的 a.X =", a.X)   # ← 此刻 a 只执行到 import b，X 还不存在 → 炸
Y = "b-value"
```

运行 `python -m a`（或 `import a`）的真实输出：

```
a: 开始执行
b: 开始执行
AttributeError: partially initialized module 'a' has no attribute 'X'
                (most likely due to a circular import)
```

**逐帧拆解**（对着 3.1 的流程看）：

1. import `a` → `sys.modules['a']` = 空壳 `a`（**执行前就放进去了**）→ 开始执行 `a.py`。
2. `a.py` 第 2 行 `import b` → `sys.modules['b']` = 空壳 `b` → 开始执行 `b.py`。
3. `b.py` 第 2 行 `import a` → 查 `sys.modules['a']`，**命中**！但拿到的是步骤 1 那个**只执行了一半、`X` 还没定义**的半成品 `a`。import 语句本身不报错（拿到对象了）。
4. `b.py` 第 3 行 `a.X` → 半成品 `a` 的 `__dict__` 里还没有 `X` → `AttributeError: partially initialized module`。

**核心结论**：循环导入炸不炸，取决于**"用到对方名字"的时机是否早于"对方定义该名字"的时机**。`a` 还没执行到 `X = ...`，`b` 就急着读 `a.X`，于是拿到半成品。CPython 3.5+ 特意把错误信息写成 "partially initialized module ... most likely due to a circular import" 来直接点破。

**`from` 形式更脆**。`from a import main` 不仅要求 `a` 存在，还要求**此刻 `a` 已经定义了 `main`**，否则立刻 `ImportError`（连半成品对象都拿不到名字）：

```python
# a.py: from b import helper  ;  b.py: from a import main
# 报错: ImportError: cannot import name 'main' from partially initialized module 'a'
#       (most likely due to a circular import)
```

> 对照 Java：JVM 的类循环依赖大多没事，因为类是**惰性、按需**初始化的，A 用到 B 时才触发 B 的 `static{}`，B 反过来用 A 时 A 通常已初始化完成；即便撞上也由 JVM 的类初始化状态机协调。Python 没有这层惰性——import 是顺序执行的语句，谁先跑到谁，半成品就这么暴露出来。解决方案见 4.6。

### 3.6 `__name__` 机制：`"__main__"` 还是模块名

每个模块的 `__dict__` 里都有一个 `__name__`。它的值由**这个模块是怎么被进入的**决定：

- 被 `import` 进来 → `__name__` = 模块的**全限定名**（如 `"shop.models"`）。
- 作为**顶层脚本直接运行**（`python greet.py`）或 `python -m greet` → 该模块的 `__name__` = **`"__main__"`**。

```python
# greet.py
print(f"greet 顶层执行, __name__ = {__name__!r}")
def hello(): return "hi"
if __name__ == "__main__":
    print("作为脚本直接运行")
```

三种进入方式的输出：

```
$ python greet.py            → greet 顶层执行, __name__ = '__main__'   +  作为脚本直接运行
$ python -c "import greet"   → greet 顶层执行, __name__ = 'greet'      （if 块不执行）
$ python -m greet            → greet 顶层执行, __name__ = '__main__'   +  作为脚本直接运行
```

机制：解释器启动时，把"入口模块"的 `__name__` 设成字符串 `"__main__"`（入口模块对象在 `sys.modules['__main__']`）。所以 `if __name__ == "__main__":` 这行的真实含义是**"仅当本文件是被直接运行的入口、而非被别人 import 时，才执行下面的代码"**。这正是 Java 那个编译期约定的 `public static void main` 在 Python 里的运行时等价物——但 Python 没有"特殊入口方法"概念，它就是一次普通的字符串比较。

### 3.7 命名空间包（无 `__init__.py`）

Python 3.3+（PEP 420）支持**没有 `__init__.py`** 的包，叫命名空间包。它的设计目的：**让同一个顶层包名跨多个目录/分发包拼装**（类似 Java 的 split package，但是官方支持的）。

```
path1/acme/util/__init__.py    # 子包 acme.util
path2/acme/data/__init__.py    # 子包 acme.data
```

当 `path1` 和 `path2` 都在 `sys.path` 上时：

```python
# PYTHONPATH=path1:path2
import acme.util, acme.data
print(list(acme.__path__))
# ['.../path1/acme', '.../path2/acme']   ← 两个目录合并成同一个 acme 包
print(acme.__file__)                      # None —— 命名空间包没有 __init__.py，所以无 __file__
```

实现上，`PathFinder` 发现某目录名匹配但里面没有 `__init__.py` 时，不立刻放弃，而是把它记为一个"命名空间路径片段"；扫完 `sys.path` 后把所有同名片段合并，造出一个 `__path__` 是多目录列表、`__file__` 为 `None` 的包对象。**普通包（有 `__init__.py`）则是单目录、有 `__file__`，且 `__init__.py` 一执行就定型。** 实战中插件体系（多个 pip 包都往 `mycorp.plugins` 命名空间下塞东西）会用到它；普通项目用常规包即可，别无意中漏写 `__init__.py` 而踩到命名空间包的微妙行为（见 ⑤）。

## ④ 使用方法与最佳实践

### 4.1 包目录结构与 `__init__.py` 暴露 API

`__init__.py` 的惯用法：把子模块里的"对外公开"符号上提到包顶层，让使用方 `from shop import Order` 而不必关心它实际定义在 `shop/models.py`：

```python
# shop/__init__.py —— 充当包的"门面（facade）"
from .models import Order, Customer
from .service import place_order

__all__ = ["Order", "Customer", "place_order"]   # 见 4.2
__version__ = "1.2.0"
```

这样 `shop` 包对外是一个干净的 API 面，内部模块怎么拆分、改名都不影响调用方——相当于 Java 里用一个 facade 类或精心设计的 public 包边界。**注意权衡**：上提会让 `import shop` 连带执行所有被 import 的子模块（导入即执行），子模块多/重时会拖慢首次导入，按需取舍。

### 4.2 `__all__`：控制 `from pkg import *` 与公共 API 声明

`__all__` 是一个字符串列表，作用有二：

1. **决定 `from module import *` 导出哪些名字**（不设则导出所有不以下划线开头的顶层名字）。
2. 充当**"这是本模块公共 API"的文档声明**，工具（linter、文档生成）和读者都据此判断什么是稳定接口。

```python
# shop/__init__.py 里设了 __all__ = ["Order", ...]
from shop import *
print(Order)        # 输出: <class 'shop.models.Order'>  —— 只有 __all__ 里的被导出
# place_order 之外没列进 __all__ 的内部名字不会被 * 引入
```

> `import *` 在生产代码里基本是反模式（污染命名空间、来源不清），但库作者**仍应认真维护 `__all__`** 作为公共 API 契约。

### 4.3 绝对导入优先，相对导入用于包内

```python
# ✅ 绝对导入：从 sys.path 根开始，清晰、可移动文件位置时不易错
from shop.models import Order
import shop.service

# ✅ 显式相对导入：仅在包内部引用兄弟模块时用，重构包名时不用改
# shop/service.py 内：
from .models import Order          # 同包的 models
from .util.format import to_json   # 子包 util 下的 format
from ..common import settings      # 上一级包的 common（.. 表示父包）
```

官方风格（PEP 8）：**优先绝对导入**；包内部模块互相引用时，显式相对导入也可接受且有优点（包改名时无需改这些 import）。**永远不用隐式相对导入**（Python 3 已移除）。

### 4.4 `if __name__ == "__main__"` 写可执行入口

让一个文件既能被 import 复用、又能直接运行：

```python
# tools/report.py
def build_report(date):
    ...

def main():
    import sys
    build_report(sys.argv[1] if len(sys.argv) > 1 else "today")

if __name__ == "__main__":
    main()       # 仅直接运行时执行；被 import 时 build_report 可复用、main 不触发
```

惯例是把入口逻辑封进 `main()`，`if __name__` 块里只调 `main()`，便于测试时单独调用 `main()`。

### 4.5 `python -m pkg` vs 直接运行脚本：`sys.path` 的关键差异

这是部署/运行环节最容易踩的坑，必须讲透。两种启动方式对 `sys.path[0]`（决定"从哪开始找模块"）的设置**不同**：

| 启动方式 | `sys.path[0]` 是什么 | 含义 |
|---------|---------------------|------|
| `python sub/run.py`（直接运行脚本） | **脚本所在目录** `sub/` | 找模块从 `sub/` 开始，看不到 `sub` 的父目录 → 包结构错乱 |
| `python -m sub.run`（按模块运行） | **当前工作目录**（`''`，即启动时的 CWD） | 找模块从项目根开始，`sub` 被当作真正的包 |

实测：

```
$ python sub/run.py          → sys.path[0] = '.../pathdemo/sub'    （只看得到 sub 内部）
$ python -m sub.run          → sys.path[0] = '.../pathdemo'        （看得到整个项目）
```

**后果**：包内带相对导入的模块，**直接 `python shop/cli.py` 会失败，`python -m shop.cli` 才成功**：

```
$ python shop/cli.py
ImportError: attempted relative import with no known parent package
$ python -m shop.cli
创建: 1          # ✅ 正常
```

原因：直接运行脚本时，该脚本的 `__package__` 是空、`__name__` 是 `"__main__"`，它"不知道自己属于哪个包"，于是 `from .models import ...` 的那个"当前包"无从确定 → 报错。`-m` 方式则按 `shop.cli` 的全名进入，`__package__` 正确设为 `"shop"`，相对导入有据可依。

> 规则：**项目内带包结构的代码，一律用 `python -m 包.模块` 运行**，别用 `python 路径/脚本.py`。前者让解释器把它当包成员、`sys.path` 指向项目根；后者把脚本目录塞进 `sys.path[0]`、丢掉包上下文。

### 4.6 解决循环导入的三招

按优先级：

**① 重构（最优）**：把双方都依赖的东西抽到第三个"更底层"的模块，打破环。`a→b、b→a` 改成 `a→c、b→c`。循环导入往往是分层不清的信号。

**② 延迟导入（函数内 import）**：把 import 从模块顶层挪进用到它的函数体内——执行到函数时对方模块通常已初始化完成：

```python
# service.py
def place_order(...):
    from shop.notify import send_email   # 延迟到调用时才 import，此刻 notify 已就绪
    send_email(...)
```

代价：每次调用都查一次 `sys.modules`（命中缓存，开销很小），但顶层依赖关系变得不直观，是务实的妥协。

**③ 类型注解用 `TYPE_CHECKING`**：很多循环导入只是为了**写类型注解**而 import（运行时根本不需要那个类的实体）。用 `typing.TYPE_CHECKING`——它运行时恒为 `False`、只对类型检查器（[14 类型系统与注解](14_类型系统与注解.md)）为真——把这类 import 隔离掉：

```python
from __future__ import annotations   # 让注解延迟求值（字符串化），不在运行时执行
from typing import TYPE_CHECKING

if TYPE_CHECKING:
    from shop.models import Order     # 仅类型检查时存在，运行时不 import → 不成环

def total(order: "Order") -> int:     # 注解里用名字，运行时不需要真的 Order
    ...
```

### 4.7 `importlib`：动态导入与重新导入

字符串名动态导入（写插件系统、按配置加载）用 `importlib.import_module`，而非内置 `__import__`（后者底层、不好用）：

```python
import importlib
name = "json"                                  # 名字可来自配置文件
mod = importlib.import_module(name)
print(mod.dumps({"a": 1}))                     # 输出: {"a": 1}

# 按全名加载子模块
models = importlib.import_module("shop.models")
```

**重新加载**改过的模块（解决 3.2 缓存命中导致改了没生效）：

```python
import importlib, conf
print(conf.VALUE)            # 输出: 1
# ……此时在外部把 conf.py 改成 VALUE = 2……
import conf                  # 再 import 无效：sys.modules 缓存命中，顶层不重跑
print(conf.VALUE)            # 输出: 1（还是旧值！）
importlib.reload(conf)       # 强制重新执行 conf 的顶层代码
print(conf.VALUE)            # 输出: 2
```

> `reload` 在 REPL/Jupyter 调试时有用，**但生产代码别依赖它**：已经绑定的旧名字（别的模块里 `from conf import VALUE` 拿到的旧值）不会被更新，容易出现新旧对象并存的诡异状态。要"改代码生效"，正道是重启进程。

### 4.8 src 布局（工程化推荐）

现代 Python 项目推荐把包放在 `src/` 子目录下，而非项目根：

```
myproj/
├── pyproject.toml
├── src/
│   └── shop/              # 真正的包
│       ├── __init__.py
│       └── models.py
└── tests/
    └── test_models.py
```

好处：**强制你以"已安装"的方式（`pip install -e .`）来 import 包**，而不是依赖"项目根恰好在 `sys.path` 上"这种巧合。这样测试跑的就是真实安装后的导入路径，避免"本地能跑、打包后 import 失败"的经典事故。这是 [29 工程化最佳实践](29_工程化最佳实践.md) 的话题，这里先建立认知。

## ⑤ ⚠️ 易错点（Java 思维翻车）

| Java 习惯 | 翻车现场 | 正确做法 / 认知 |
|----------|---------|----------------|
| 以为 import 零成本、循环依赖随便 | A、B 顶层互相 import 并用对方的名字 → `AttributeError: partially initialized module` | import 是顺序执行语句；重构分层 / 延迟导入 / `TYPE_CHECKING`（4.6） |
| 随手给脚本起名 `random.py`、`json.py`、`queue.py` | 自己的文件 shadow 了标准库，`import random` 拿到你的文件，标准库瞬间"消失"、报莫名其妙的 `AttributeError` | 别和标准库/第三方库重名；用 `python -c "import random; print(random.__file__)"` 验明正身 |
| 直接 `python shop/cli.py` 运行包内脚本 | `ImportError: attempted relative import with no known parent package`，或导入路径全乱 | 用 `python -m shop.cli`；脚本目录会被塞进 `sys.path[0]` 破坏包结构（4.5） |
| 以为改了模块再 import 就生效（像热部署） | REPL/服务里改了 `.py`，再 `import` 还是旧行为 | `sys.modules` 缓存命中不重跑；调试用 `importlib.reload`，生产重启进程（4.7） |
| 把 `__init__.py` 当"必须放满初始化代码"的地方 | 在 `__init__.py` 里连数据库、加载大模型、扫描磁盘 → 任何 `import 该包` 都卡几秒、还可能在 import 期抛异常 | `__init__.py` 只做轻量的 API 上提；重活放函数里惰性触发 |
| 漏写 `__init__.py` 却以为是普通包 | 目录成了命名空间包，行为微妙（`__file__` 为 None、跨目录合并），相对导入/打包出怪问题 | 普通项目每个包目录都放 `__init__.py`（哪怕空文件），命名空间包是有意为之才用 |
| 把 `__pycache__/`、`.pyc` 当源码或提交 git | 仓库脏、换 Python 版本后旧 `.pyc` 干扰 | 它是字节码缓存，`.gitignore` 掉；删了无害（3.4） |
| 用 `from pkg import *` 图省事（当成 Java 的 `import x.*`） | 命名空间被一堆来源不明的名字污染，看不出某个名字哪来的 | 显式导入具体名字；`*` 仅在 REPL 或经 `__all__` 严控的场景用（4.2） |
| 以为 `import a.b.c` 只加载 `c` | 实际把 `a`、`a.b`、`a.b.c` 三层的 `__init__.py` 全执行了一遍 | 导入子模块会沿途执行所有父包的初始化代码（3.1/4.1） |

**shadow 标准库的真实演示**（这个坑太常见，单独给个复现）：

```python
# 当前目录有个你自己写的 random.py，里面是 def shuffle_deck(): ...
import random
random.randint(1, 6)
# AttributeError: module 'random' has no attribute 'randint'
# ← import 找到的是你的 random.py（CWD 在 sys.path 靠前），不是标准库！
# 排查: print(random.__file__) 会指向你的文件而非 .../lib/python3.12/random.py
```

## ⑥ 练习题

### 基础理解

1. 用"导入即执行 + `sys.modules` 缓存"两个概念，解释为什么连续两次 `import json`，`json` 模块顶层代码只执行一次。第二次 import 在 3.1 流程的哪一步就返回了？

2. 解释 `if __name__ == "__main__":` 这行的运行时含义。`python foo.py`、`python -m foo`、`import foo` 三种方式下，`foo` 模块的 `__name__` 分别是什么？哪些会执行 `if` 块里的代码？

### 找坑改错

3. **循环导入复现与修复**。下面两个模块，运行 `python -m order` 会报错。先**说出报错类型和原因**（对照 3.5 的逐帧拆解），再给出**两种**不同的修复方案。

```python
# order.py
import inventory
DEFAULT_TAX = 0.1
def make_order(item):

    return {"item": item, "tax": DEFAULT_TAX}

# inventory.py
import order
PRICE_TABLE = {"book": 30}
print("库存模块加载，默认税率 =", order.DEFAULT_TAX)   # ← 这里出事
def reserve(item):
    print("预留:", item)

if __name__ == "__main__":
    pass
```

4. 一位 Java 同事写了下面的脚本放在 `app/` 目录里，命名为 `app/email.py`，直接 `python app/email.py` 运行，报 `ImportError: cannot import name 'message_from_string' from 'email'` 之类的怪错。指出**两个**问题并说明原因。

```python
# app/email.py
from .utils import clean        # 想引用同目录的 utils.py
import email                    # 想用标准库 email 解析邮件
def parse(raw):
```

### 综合实战

5. 你要为一个后端服务设计一个名为 `payment` 的包，需求：
   - 对外只暴露 `PaymentClient` 类和 `charge()` 函数，调用方写 `from payment import PaymentClient, charge`。
   - 内部拆成 `client.py`（客户端）、`models.py`（数据模型）、`gateways/`（含 `stripe.py`、`alipay.py` 两个子模块）。
   - 网关模块要能按配置字符串动态加载（如配置 `"stripe"` 就加载 `payment.gateways.stripe`）。
   - 提供 `python -m payment` 直接跑一个自检（self-check）。

   请给出**目录结构**、各 `__init__.py` 的关键内容、动态加载网关的代码，并说明为什么用 `python -m payment` 而不是 `python payment/__main__.py`。

### 参考答案

**1.** 第一次 `import json`：`sys.modules` 无 `json`（流程 ① 未命中）→ finder 找到 spec → loader 创建模块并**执行顶层代码一次** → 存入 `sys.modules['json']` → 绑定名字。第二次 `import json`：在 **流程第 ① 步**查 `sys.modules['json']` 就命中，**直接返回那个已存在的模块对象，根本不进入 finder/loader、不再执行顶层代码**。所以顶层只执行一次，模块是单例（`import json as j; j is json` 为 `True`）。

**2.** 含义：`__name__` 是模块自己的"身份名"；解释器把**直接运行的入口模块**的 `__name__` 设为字符串 `"__main__"`。所以这行等价于"仅当本文件是被直接运行的入口时才执行下面的代码"。三种方式：

| 方式 | `foo` 的 `__name__` | 执行 if 块？ |
|------|--------------------|------------|
| `python foo.py` | `"__main__"` | 是 |
| `python -m foo` | `"__main__"` | 是 |
| `import foo` | `"foo"` | 否 |

**3.** 报错：`AttributeError: partially initialized module 'order' has no attribute 'DEFAULT_TAX' (most likely due to a circular import)`。

逐帧：`python -m order` → `order` 进 `sys.modules`（空壳）→ 执行 `order.py` 第 1 行 `import inventory` → `inventory` 进 `sys.modules` → 执行 `inventory.py` 第 1 行 `import order`（命中 `sys.modules` 里的**半成品 order**，此刻它只执行到第 1 行，`DEFAULT_TAX` 尚未定义）→ `inventory.py` 第 3 行 `order.DEFAULT_TAX` → 半成品 `order` 没有该属性 → 炸。根因：`inventory` 在模块顶层（import 阶段）就读 `order.DEFAULT_TAX`，时机早于 `order` 定义它。

修复方案（两种，任选其二）：

- **延迟读取**：把 `inventory.py` 顶层那句 `print(... order.DEFAULT_TAX)` 删掉，或挪进函数体内（用到时 `order` 已初始化完）。如果确实要 import，也把 `import order` 移进 `reserve` 函数里。
- **重构打破环**：把 `DEFAULT_TAX`、`PRICE_TABLE` 这类共享常量抽到第三个底层模块 `constants.py`，让 `order` 和 `inventory` 都只 `from constants import ...`，二者不再互相 import。
- （也可接受）**调整顺序**：把 `order.py` 的 `DEFAULT_TAX = 0.1` 移到 `import inventory` **之前**——这样 `inventory` 读 `order.DEFAULT_TAX` 时它已存在。但这种"靠语句顺序续命"很脆弱，不如前两种。

**4.** 两个问题：

- **文件名 shadow 标准库**：脚本自己叫 `email.py`，而它又 `import email` 想用标准库。直接运行脚本时脚本所在目录在 `sys.path` 最前（4.5），`import email` 找到的是**它自己**（一个还在执行中的半成品模块），而非标准库 `email`，所以 `email.message_from_string` 不存在 → 怪错。改名（如 `mail_handler.py`）即可。
- **直接运行脚本时相对导入失效**：`from .utils import clean` 是显式相对导入，但 `python app/email.py` 直接运行时该模块 `__name__` 是 `"__main__"`、`__package__` 为空，"当前包"无从确定 → `ImportError: attempted relative import with no known parent package`。应改用 `python -m app.email`（并先确保 `app/` 是包、在项目根下运行），或在能用绝对导入时写 `from app.utils import clean`。

**5.** 目录结构：

```
payment/
├── __init__.py          # 门面：上提公共 API
├── __main__.py          # python -m payment 的入口
├── client.py            # PaymentClient
├── models.py            # 数据模型
└── gateways/
    ├── __init__.py
    ├── stripe.py        # 含 class StripeGateway / 函数
    └── alipay.py
```

`payment/__init__.py`（门面 + 公共 API 声明）：

```python
from .client import PaymentClient
from .service import charge          # 或直接定义在某模块再上提

__all__ = ["PaymentClient", "charge"]
__version__ = "1.0.0"
```

动态加载网关（用 `importlib.import_module`，4.7）：

```python
# client.py
import importlib

def load_gateway(name: str):
    # name 来自配置，如 "stripe" → 加载 payment.gateways.stripe
    module = importlib.import_module(f"payment.gateways.{name}")
    return module.Gateway()          # 约定每个网关模块都暴露 Gateway
```

`payment/__main__.py`（自检入口）：

```python
from payment import PaymentClient

def self_check():
    client = PaymentClient(gateway="stripe")
    print("自检通过:", client.ping())

if __name__ == "__main__":
    self_check()
```

为什么用 `python -m payment` 而非 `python payment/__main__.py`：

- `-m payment` 让解释器把 `payment` 当作**真正的包**来运行（`sys.path[0]` 指向项目根，`__package__` 正确设为 `"payment"`），于是 `__main__.py` 里的 `from payment import ...` 和动态加载 `payment.gateways.*` 这些**绝对导入和相对导入都能正常解析**。
- 直接 `python payment/__main__.py` 会把 `payment/` 目录本身塞进 `sys.path[0]`，`payment` 这个包名反而**找不到了**（它的父目录才在路径上），包内导入全部错乱——正是 4.5 讲的陷阱。`python -m` 是运行"包里的入口"的唯一正道。
