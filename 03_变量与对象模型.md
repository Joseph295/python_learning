# 03 · 变量与对象模型 ⭐

> 全书最关键的一章。Java 老手 80% 的 Python 坑——意外修改、`is`/`==` 混淆、默认参数陷阱、深浅拷贝、可变对象当 key——全部源于一个根本误解：**把 Python 变量当成 Java 变量**。Java 的变量是"带类型的盒子"，Python 的变量是"贴在对象上的名字标签"。这一个差异渗透到赋值、传参、比较、拷贝的每个角落。读透本章，后面很多坑会自动消失；读不透，后面每一章都会反复被它咬。

## ① Java 对照表

| 概念 | Java | Python | 一句话差异 |
|------|------|--------|-----------|
| 变量本质 | 有类型的"盒子"，值/引用放进盒子 | **无类型的名字标签**，绑定到对象 | 盒子 vs 标签 |
| 赋值 `=` | 把值拷进盒子（基本类型）或把引用拷进盒子 | 让名字**指向**一个对象，**永远是绑定，从不拷贝** | Python 没有"赋值即拷贝"这回事 |
| 基本类型 | `int/double/boolean` 是栈上裸值 | 没有基本类型，`1` 也是堆上对象 | Python 一切皆对象 |
| `==` | 基本类型比值；对象比引用 | 比**值**（调 `__eq__`） | `==` 在 Python 永远比值 |
| 比引用相等 | `==`（对象时） | `is` | `is` 才是"同一个对象" |
| 空值 | `null`（无类型的空引用） | `None`（一个真实存在的单例对象） | None 是对象，有类型 `NoneType` |
| 不可变 | `final`（限制**变量**不能重新赋值） | 取决于**对象类型**：`int/str/tuple` 不可变 | final 管名字，immutable 管对象 |
| 传参 | 值传递（对象传引用的副本） | pass-by-assignment（形参=实参的一次绑定） | 行为等价，但 Python 更统一（无基本类型特例） |
| 拷贝副本 | `clone()` / 拷贝构造 | `copy.copy` / `copy.deepcopy` | 默认从不拷贝，要副本得显式要 |

> 5 秒记忆法：**Java 变量是盒子，盒子里装值或地址；Python 变量是便利贴，贴在堆上的对象身上。** 一切差异都从这里推导出来。

## ② 设计理念（为什么 Python 这么设计）

### 2.1 为什么是"名字绑定对象"，而不是"变量是带类型的盒子"

Java 的模型里，`int x = 5` 在编译期就决定了：`x` 是一个能装 `int` 的 4 字节槽位，栈上分配，类型钉死，里面直接躺着 `5`。`Object o` 则是一个能装引用的槽位，里面躺着指向堆对象的地址。**变量自带类型，是"先有盒子后有值"。**

Python 反过来：**先有对象，名字只是事后贴上去的标签。** `x = 5` 的含义是"在堆上有一个 `int` 对象 `5`，让名字 `x` 指向它"。名字本身不带类型——同一个 `x` 这一行指向 `int`，下一行可以指向 `str`，因为类型是**对象**的属性，不是名字的属性。

为什么这样设计？因为 Python 是**动态类型**语言，它把"类型属于值而非变量"这条原则贯彻到底：

- 既然类型在运行期才确定，把类型钉在变量上就没意义了——干脆让变量只做"引用"这一件事。
- 既然万物都是对象（连函数、类、模块都是对象），那就不需要"基本类型 vs 引用类型"的二分。**只有一种东西：对象；只有一种操作：把名字绑定到对象。** 模型反而更小、更一致。

代价是性能（每个 `5` 都是堆对象，见 [05 CPython 内部](05_CPython内部.md) 2.3），收益是统一性和灵活性。这是 Python "宁可慢、要简单一致"哲学的典型体现。

### 2.2 可变 vs 不可变：一个被刻意引入的二分

Java 没有"对象内容能不能变"这个语言级概念——`final` 管的是**变量能不能重新指向别处**，跟对象内部状态能不能改是两码事（`final List` 照样能 `add`）。

Python 引入了一个 Java 没有的核心分类：**对象本身是 mutable（可变）还是 immutable（不可变）**。

| 不可变（immutable） | 可变（mutable） |
|---------------------|-----------------|
| `int`, `float`, `bool`, `complex` | `list` |
| `str` | `dict` |
| `tuple`（但其元素可能是可变对象！） | `set` |
| `frozenset` | `bytearray` |
| `bytes` | 自定义类的实例（默认可变） |
| `None` | |

这个二分是**故意的设计**，背后有三层意图：

1. **可哈希性 = 可不可变。** 字典的 key、集合的元素必须可哈希，而"可哈希"要求对象生命周期内 `hash()` 稳定。只有不可变对象能保证这点。所以 `dict`/`set` 的 key 只能是不可变对象——这不是限制，是为了让哈希表的不变式成立（详见 [06 内置数据结构](06_内置数据结构.md)）。
2. **共享安全。** 不可变对象可以被随意共享、缓存、驻留（小整数缓存、字符串驻留，见 ③ 段），不怕被任何一方偷偷改坏。Python 大量内部优化建立在"不可变对象可安全共享"之上。
3. **意图表达。** 用 `tuple` 而非 `list`、`frozenset` 而非 `set`，等于在代码里声明"这组数据不该被修改"，从源头杜绝意外写入。

> 心智模型：**不可变对象 ≈ Java 里"事实不可变"的 `String`/`Integer`**——你拿它没办法改内容，只能造一个新的。区别是 Python 把这个概念系统化、扩展到 `tuple`/`frozenset`，并和哈希、共享、传参语义全部打通。

### 2.3 为什么 `==` 比值、`is` 比身份要分成两个

Java 把这两件事都压在 `==` 上，靠"操作数是基本类型还是对象"来区分语义——这正是 Java 新手 `"a" == "a"` 翻车的根源（有时 True 有时 False，取决于字符串常量池）。Python 认为这种"同一个符号、两种语义、靠类型隐式切换"是设计缺陷，于是**拆成两个明确的运算符**：

- `==`：**值相等**，问"内容一样吗"，调用 `__eq__`，对应 Java 的 `equals()`。
- `is`：**身份相等**，问"是同一个对象吗"，比较 `id()`，对应 Java 对象的 `==`。

拆开之后语义零歧义：业务比较一律 `==`，判单例（主要是 `None`）一律 `is`。你再也不用纠结"这里该用哪种相等"——问题本身被语法消灭了。

### 2.4 为什么 None 是对象，不是 null

Java 的 `null` 是一个"无类型的空引用"——它不是对象，没有方法，`null.toString()` 直接 NPE。Python 的 `None` 是一个**真实存在的单例对象**，类型是 `NoneType`，全进程只有一个实例。

这个选择和"一切皆对象"一脉相承：既然连"空"也要表达，那就让它也是个正经对象，而不是语言里的特例空洞。好处是 `None` 可以有行为、可以被 `is` 高效比较（全局唯一，比地址即可）、可以安全地放进容器和当默认值。判空写 `x is None` 不仅地道，而且比 `==` 更快更安全（见 ④ 段）。

### 2.5 pass-by-assignment：没有基本类型，反而更一致

Java 课堂上永远在吵"传值还是传引用"，标准答案是"Java 永远传值，对象传的是引用的副本"——但因为有基本类型和对象类型两套规则，新手总是绕晕。

Python 的传参规则只有一句话：**形参 = 实参，就是一次普通的名字绑定。** 官方叫 **pass-by-assignment**（按赋值传递）或 **call-by-sharing**（按共享传递）。实参和形参绑定到**同一个对象**，没有任何拷贝。

因为 Python 没有基本类型这个特例，所有参数——数字、字符串、列表、对象——全走同一套规则，**反而比 Java 更统一**。函数里能不能影响外部，不取决于"基本类型还是对象"，而**只取决于对象可不可变**：

- 对可变对象做**原地修改**（`append`/`update`/下标赋值）→ 调用方看得见。
- 对形参**重新绑定**（`x = ...`）→ 只是让本地标签改指，调用方看不见。

把"形参 = 实参"当成一次赋值来读，所有传参行为都能推导出来，不需要背特例。

## ③ 实现细节（底层怎么实现的）★本章重点★

前面讲了 WHY，这一段挖 HOW：名字绑定在 CPython 里到底是什么、对象在内存里长什么样、"不可变"和"缓存"在字节码和 `id()` 层面如何实证。

### 3.1 对象三要素：identity / type / value

CPython 里每个对象在 C 层都是一个 `PyObject`，它决定了对象的三个不可分割的属性（呼应 [05 CPython 内部](05_CPython内部.md) 2.3）：

| 属性 | 含义 | 怎么查 | 谁来比较 | 能否改变 |
|------|------|--------|----------|----------|
| **identity（身份）** | 对象在内存中的唯一标识（CPython 里就是地址） | `id(obj)` | `is` | 创建后终生不变 |
| **type（类型）** | 对象属于哪个类，决定它的行为 | `type(obj)` | `isinstance` | 创建后基本不变 |
| **value（值）** | 对象的内容 | 直接看 / `==` | `==`（调 `__eq__`） | 可变对象能改，不可变对象不能 |

```python
a = [1, 2, 3]
print(id(a))        # 4309xxxxxx  身份（这一进程内唯一）
print(type(a))      # <class 'list'>  类型
print(a)            # [1, 2, 3]  值
```

`is` 比的就是 `id()`，`==` 比的就是 value。把这张表刻进脑子，后面所有 `is`/`==` 的诡异行为都能解释。

### 3.2 名字绑定在字节码层：STORE_NAME / STORE_FAST

"变量是名字标签"不是比喻，它在字节码里是实打实的指令。CPython 把名字存在**命名空间字典**里，赋值就是往里写一个"名字 → 对象指针"的条目。

模块/类作用域的名字走 `STORE_NAME`（写进模块的 `__dict__`），函数局部名字走 `STORE_FAST`（写进栈帧的快速局部数组，按下标访问，更快——见 [05 CPython 内部](05_CPython内部.md) 2.4）。

```python
import dis
dis.dis(compile("x = 256\ny = x\n", "<m>", "exec"))
```

输出（Python 3.12）：
```
  1   RESUME           0
      LOAD_CONST       0 (256)    # 把对象 256 压栈
      STORE_NAME       0 (x)      # 把栈顶对象的指针绑到名字 x

  2   LOAD_NAME        0 (x)      # 取出 x 指向的对象，压栈
      STORE_NAME       1 (y)      # 同一个对象的指针又绑到名字 y
      RETURN_CONST     1 (None)
```

注意第 2 行：`y = x` 编译成 `LOAD_NAME x` + `STORE_NAME y`，**全程没有任何拷贝指令**。它做的事就是"把 x 现在指向的那个对象，让 y 也指过去"。两个名字、一个对象——这就是 `b = a` 不拷贝副本的字节码铁证。

### 3.3 引用计数视角：`a = b` 只是多一个引用

从内存管理角度看（呼应 [04 运行时与内存模型](04_运行时与内存模型.md) 2.1），每个对象头部有个引用计数 `ob_refcnt`。`b = a` 让多一个名字指向同一对象，引用计数 +1，并没有产生新对象：

```python
import sys

a = [10, 20, 30]
print(sys.getrefcount(a))   # 2（a 一个 + getrefcount 形参临时一个）

b = a                       # 不是拷贝！只是又一个标签指过来
print(sys.getrefcount(a))   # 3
print(a is b)               # True  同一个对象，id 相同

b.append(40)                # 通过 b 原地修改这唯一的 list
print(a)                    # [10, 20, 30, 40]  ← a 当然"也"变了，因为根本是同一个
```

> 一图胜千言：`a` 和 `b` 是两根指向**同一个堆对象**的箭头。`b.append(40)` 改的是箭头指向的那个对象，`a` 这根箭头看到的自然是改后的结果。

### 3.4 不可变对象"修改即新建"的字节码体现

不可变对象没有任何能改内容的操作。所有看起来像"修改"的写法，本质都是**造一个新对象 + 把名字重新绑过去**。这在 `id()` 和字节码上都看得见：

```python
s = "hello"
print(id(s))        # 地址 A
s = s + " world"    # 不是原地拼接！造了个新 str，再把 s 重绑过去
print(id(s))        # 地址 B（≠ A）
```

```python
import dis
def f():
    s = "hello"
    s = s + " world"
    return s
dis.dis(f)
```
```
  LOAD_CONST   1 ('hello')
  STORE_FAST   0 (s)            # s 绑到 "hello"

  LOAD_FAST    0 (s)            # 取 s 指向的 "hello"
  LOAD_CONST   2 (' world')
  BINARY_OP    0 (+)            # 拼接，产生一个全新的 str 对象压栈
  STORE_FAST   0 (s)            # s 重新绑到这个新对象（旧的 "hello" 引用 -1）
```

`BINARY_OP +` 后面紧跟 `STORE_FAST s`：先造新串，再让 `s` 改指。原来的 `"hello"` 对象一个字节都没动——这就是为什么"字符串方法返回新串、必须接收返回值"，也是为什么对 `int`/`str` 的操作**看起来**像 Java 的值语义：你根本改不动原对象，只能换一个。错觉的根源就在这里。

对比可变对象的原地修改，字节码完全不同：

```python
def g():
    lst = [1, 2]
    lst.append(3)      # 原地修改，没有 STORE_FAST 重绑
    return lst
```
`lst.append(3)` 编译成方法调用，直接改对象内部，**没有 `STORE_FAST lst`**——名字始终指向同一个对象，只是对象内容变了。

**内存图对比：**
```
不可变重绑：               可变原地改：
  s ──► "hello"  (旧,待回收)   lst ──► [1, 2]
  s ──► "hello world"  (新)    lst ──► [1, 2, 3]   (同一对象，箭头没动)
```

### 3.5 小整数缓存 [-5, 256]：`is` 的"诡异"行为之源

CPython 启动时预先创建好 `[-5, 256]` 范围内的所有整数对象，作为单例缓存（"small int cache"）。任何求得这范围内整数的地方，拿到的都是同一个缓存对象。超出范围则每次新建。

```python
a = 256
b = int("256")      # 用 int() 强制运行期构造，绕开编译期常量折叠
print(a is b)       # True   ← 都是缓存里那唯一的 256 对象
print(id(a) == id(b))  # True

a = 257
b = int("257")
print(a is b)       # False  ← 超出缓存，是两个不同对象
print(a == b)       # True   ← 但值始终相等
```

> 为什么要 `int("256")` 而不直接写 `b = 256`？因为**编译期常量折叠**会更进一步：同一个 code object 里的两个 `257` 字面量会被去重成同一个常量对象，于是 `a = 257; b = 257; a is b` 在一个函数里可能是 `True`，换个上下文又是 `False`。这正说明 `is` 的结果依赖编译器和缓存的实现细节，**不可作为判断依据**。

为什么缓存 `[-5, 256]`？因为这些小整数在循环计数、索引、状态码里出现得最频繁，缓存它们能省下海量重复对象的分配（每个 int 对象 28 字节，见 [05 CPython 内部](05_CPython内部.md) 2.3）。`-5` 是经验下界（常见的小负数）。

### 3.6 字符串驻留（interning）：另一类共享单例

CPython 还会**驻留**一部分字符串，让内容相同的串共享同一对象以省内存、并把字符串比较从逐字符变成比地址。驻留规则大致是：编译期已知的字符串字面量、看起来像标识符的字符串（只含字母数字下划线）通常被驻留；运行期动态拼接/拼装出来的串通常不驻留。

```python
s1 = "hello"
s2 = "hel" + "lo"                    # 编译期常量折叠，和 s1 是同一个
print(s1 is s2)                      # True

s3 = "".join(["h", "e", "l", "l", "o"])  # 运行期构造，未驻留
print(s1 is s3)                      # False
print(s1 == s3)                      # True   ← 值永远相等

import sys
s4 = sys.intern("".join(["h","e","l","l","o"]))  # 手动驻留
print(s1 is s4)                      # True   ← 大量重复字符串时可显式 intern 省内存
```

结论同小整数：**驻留是实现优化，不是语言保证**。是否驻留取决于内容、来源、Python 版本。所以**判断字符串相等永远用 `==`，绝不能用 `is`**。`sys.intern` 偶尔在解析海量重复 token（如列名、标签）时用来省内存并加速比较，属于高级优化手段。

## ④ 使用方法与最佳实践

### 4.1 判空：`is None` / `is not None`

```python
def fetch(uid, conn=None):
    if conn is None:            # ✅ None 是全局单例，is 又快又安全
        conn = get_default_conn()
    ...
```
绝不写 `if conn == None`：`==` 会去调 `__eq__`，自定义对象可能把它重载成对 `None` 返回 `True`（如某些 ORM 字段、numpy 数组甚至会报错或返回数组），骗过你的判空。判 `True`/`False` 单例同理优先用 `is`，但布尔判断通常直接写 `if flag:` 更地道。

### 4.2 比值用 `==`，绝不用 `is` 比数值/字符串

```python
status = int(resp.headers.get("X-Code", "0"))
if status == 200:     # ✅ 永远用 ==
    ...
# if status is 200:   # ❌ 257 之外可能 False，写出随数据而异的幽灵 bug
```
铁律：**`is` 只用于和单例（`None`、有时 `True`/`False`、自定义哨兵对象）比较；一切数值、字符串、容器的相等判断一律 `==`。**

### 4.3 函数内拷贝可变实参，避免污染调用方

```python
def normalize(items):
    items = list(items)     # ✅ 进门先浅拷贝，后续操作不碰调用方的原 list
    items.sort()
    return items

raw = [3, 1, 2]
print(normalize(raw))       # [1, 2, 3]
print(raw)                  # [3, 1, 2]  ← 原列表未被动过
```
若函数语义就是"原地修改"（如 `list.sort()`），则不拷贝、并在文档/命名上讲清楚。

### 4.4 要真副本：`copy` vs `deepcopy`

```python
import copy

original = {"db": {"host": "localhost", "port": 5432}, "retries": 3}

shallow = copy.copy(original)        # 浅拷贝：新外层 dict，内层 dict 仍共享
deep = copy.deepcopy(original)       # 深拷贝：递归复制每一层

original["db"]["host"] = "10.0.0.1"
print(shallow["db"]["host"])         # 10.0.0.1  ← 内层共享，被污染！
print(deep["db"]["host"])            # localhost ← 完全独立
```
常见浅拷贝写法：`list(x)`、`x[:]`、`dict(x)`、`x.copy()`、`set(x)`。**只要结构有嵌套且要彻底隔离，就用 `deepcopy`**；扁平结构浅拷贝足矣（`deepcopy` 有递归开销）。

### 4.5 优先用不可变类型表达"不该被改"的数据

```python
WEEKDAYS = ("Mon", "Tue", "Wed", "Thu", "Fri")   # tuple 而非 list：常量就该不可变
VALID_ROLES = frozenset({"admin", "user", "guest"})  # frozenset：可哈希、不可变
```
不可变类型既防止意外写入，又可作为 dict key / set 元素，还能被安全共享。

### 4.6 自定义类实现 `__eq__` / `__hash__`（引出 [13 章](13_类与对象.md)）

默认情况下，自定义类的实例用**身份**比较（`==` 退化成 `is`），且可哈希（基于 id）。要"值相等"语义就实现 `__eq__`，并**同步实现 `__hash__`** 以保持"相等的对象哈希也相等"，否则对象无法正确放进 `set`/`dict`：

```python
class Point:
    def __init__(self, x, y):
        self.x, self.y = x, y
    def __eq__(self, other):
        return isinstance(other, Point) and (self.x, self.y) == (other.x, other.y)
    def __hash__(self):
        return hash((self.x, self.y))     # 与 __eq__ 用同一组字段

print(Point(1, 2) == Point(1, 2))         # True   值相等
print(len({Point(1, 2), Point(1, 2)}))    # 1      正确去重

# 更省事：@dataclass(frozen=True) 自动生成 __eq__ 和 __hash__（第 13 章）
```
> 注意：只实现 `__eq__` 而不实现 `__hash__`，Python 会把该类的 `__hash__` 置为 `None`，实例变成不可哈希、不能进 set/dict——这是刻意保护"相等性变了，哈希必须同步变"的不变式。

## ⑤ ⚠️ 易错点（Java 思维翻车）

| Java 习惯 | 翻车现场 | 正确做法 / 认知 |
|----------|---------|----------------|
| `b = a` 以为拷了副本 | 改 `b` 把 `a` 也改了，调试半天找不到谁动了数据 | `=` 只绑标签、永不拷贝；要副本用 `list()`/`dict()`/`copy`/`deepcopy` |
| 用 `==` 判 null | `if x == None`，遇到重载 `__eq__` 的对象判错甚至报错 | `if x is None`，单例一律 `is` |
| 用 `is` 比数值 | `a is b` 在 256 是 True、257 是 False，写出"随数据时灵时不灵"的幽灵 bug | 数值/字符串相等一律 `==` |
| 以为字符串可原地改 | 期待 `s.replace("a","b")` 改原串，结果原串没变 | str 不可变，方法返回**新**串，必须 `s = s.replace(...)` 接收 |
| 以为"传基本类型不影响外部、传对象才会" | 困惑于 list 被函数改了、int 却没被改 | 不看"基本/对象"，只看对象**可不可变**；可变就可能被原地改 |
| 浅拷贝当深拷贝用 | `dict(cfg)` 后改副本，内层共享导致原配置被污染 | 嵌套结构用 `copy.deepcopy` |
| 把可变对象当 dict key | `d[[1,2]] = ...` 直接 `TypeError: unhashable type: 'list'` | key 必须不可变可哈希，用 `tuple`/`frozenset` |
| 只重写 `equals` 不管 `hashCode`（Java 也会翻车，Python 更狠） | 自定义类只写 `__eq__`，实例直接不能进 set/dict | `__eq__` 和 `__hash__` 必须成对实现，或用 `@dataclass(frozen=True)` |

### 基础理解

# A
a = [1, 2]; b = a; b.append(3); print(a)      # 输出？

a = (1, 2); b = a; b = b + (3,); print(a)      # 输出？

def zero_first(data):

DEFAULT = {"db": {"host": "localhost", "port": 5432}, "retries": 3}

def make_config(overrides):
    cfg = dict(DEFAULT)        # 本想拷一份默认配置

cfg = make_config({"retries": 5})

### 综合实战
5. 写一个 `freeze(obj)` 的设计思路（给出可运行实现更好）：把一个可能嵌套的 `dict`/`list`/`set` 配置转成**完全不可变**的结构，使任何 `append`/`update`/下标赋值都失败。说明 `dict → 什么`、`list → tuple`、`set → frozenset`、标量保留、如何递归，并说明这样做能预防本章哪些坑。

---

### 参考答案

**1.** A 输出 `[1, 2, 3]`，B 输出 `(1, 2)`。
- A：`list` 可变。`b = a` 是名字绑定（字节码 `STORE_NAME`，无拷贝），`a`、`b` 指向同一个 list 对象。`b.append(3)` **原地修改**这唯一的对象，`a` 这根箭头看到的自然是改后结果。
- B：`tuple` 不可变，没有原地修改的可能。`b = b + (3,)` **造了一个新元组**并让 `b` 重新绑过去（字节码 `BINARY_OP +` 后跟 `STORE`），`a` 仍指向原来那个 `(1, 2)`，毫发无损。
- 一句话：可变对象被原地改、共享者全看见；不可变对象只能造新的、重绑只影响自己这个名字。

**2.** 字符串驻留是 CPython 的实现优化：是否驻留取决于字符串内容（是否像标识符）、来源（编译期字面量 vs 运行期拼接）、Python 版本。它是实现细节而非语言保证——`"abc"` 字面量恰好被驻留所以 `is` 为 `True`，但 `"".join([...])` 拼出的同值串就不驻留、`is` 为 `False`。判断字符串相等必须用比较**值**的 `==`。

**3.** bug 有二：①直接对入参 `data`（可变对象）下标赋值，**原地污染了调用方的原列表**；②号称"返回新列表"，实际返回的还是同一个对象。修复——进门先浅拷贝：
```python
def zero_first(data):
    new = list(data)   # 浅拷贝出独立的外层 list
    new[0] = 0
    return new
```
（若元素本身也是可变对象且需隔离，用 `copy.deepcopy`。）

**4.** `DEFAULT["db"]["host"]` 实际变成 `10.0.0.1`。根因：`dict(DEFAULT)` 是**浅拷贝**——外层是新 dict，但 `cfg["db"]` 和 `DEFAULT["db"]` 仍指向**同一个**内层 dict。调用方改 `cfg["db"]["host"]`，改的就是那个被共享的内层对象。修复用深拷贝：
```python
import copy

def make_config(overrides):
    cfg = copy.deepcopy(DEFAULT)   # 递归复制每一层，内层不再共享
    cfg.update(overrides)
    return cfg
```

**5.** 思路：递归地把每种可变容器换成对应的不可变类型，标量（`int`/`str`/`bool`/`None`）本就不可变直接保留。
```python
from types import MappingProxyType

def freeze(obj):
    if isinstance(obj, dict):
        # dict → 只读视图，且对每个 value 递归 freeze
        return MappingProxyType({k: freeze(v) for k, v in obj.items()})
    if isinstance(obj, (list, tuple)):
        return tuple(freeze(x) for x in obj)      # list/tuple → tuple
    if isinstance(obj, (set, frozenset)):
        return frozenset(freeze(x) for x in obj)  # set → frozenset
    return obj                                     # 标量原样返回

cfg = freeze({"db": {"host": "localhost", "ports": [5432, 5433]}})
# cfg["db"]["host"] = "x"   → TypeError，MappingProxyType 不支持赋值
# cfg["db"]["ports"].append(1)  → AttributeError，tuple 没有 append
```
说明：
- `dict → MappingProxyType`（只读映射视图，不支持写入；若要更强保证可用 `frozenset(items)` 或自定义不可变映射，但会丢失下标读取的便利）。
- `list/tuple → tuple`，`set → frozenset`，递归处理每个元素。
- 能预防本章这些坑：**意外原地修改**（结构只读，写入直接报错）、**浅拷贝共享内层导致污染**（每层都已不可变，无所谓共享）、**把可变对象误当 dict key**（产物全可哈希，能安全作 key/集合元素）。
