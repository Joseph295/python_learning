# 04 · 运行时与内存模型（垃圾回收）⭐

> 你对 JVM 的 GC（分代、Eden/Survivor/Old、Stop-The-World、G1/ZGC）有完整心智。本章告诉你 CPython 用的是**完全不同的另一套**：**引用计数为主 + 循环垃圾回收为辅**。理解这套，你才懂为什么 Python 析构基本是确定性的、`with` 为什么重要、循环引用为什么需要特殊处理。

---

## ① Java 对照表

| 维度 | JVM | CPython |
|------|-----|---------|
| 主机制 | 追踪式 GC（从 GC Root 可达性分析） | **引用计数**（每个对象记录被引用次数） |
| 回收时机 | 不确定，GC 周期触发 | 引用计数归零的**瞬间**立即回收（确定性） |
| 辅助机制 | 分代、并发标记、压缩 | **循环 GC**（gc 模块）专门处理引用环 |
| 析构/清理 | `finalize()`（已废弃，不可靠） | `__del__`（引用计数归零即调，相对可靠但仍有坑） |
| 确定性资源释放 | try-with-resources | `with` 上下文管理器（第 13 章） |
| 弱引用 | `WeakReference`/`SoftReference` | `weakref` 模块 |
| 调优 | `-Xmx`/`-XX:+UseG1GC` 等大量旋钮 | `gc` 模块阈值，旋钮少得多 |
| STW 暂停 | 是核心议题 | 引用计数无 STW；循环 GC 有短暂停 |

---

## ② 设计理念（为什么这么设计）

JVM 用一套**追踪式 GC**（从 GC Root 做可达性分析）解决所有回收问题；CPython 却选了"**引用计数为主 + 循环 GC 为辅**"的双机制。为什么？

- **确定性销毁**：引用计数让对象在最后一个引用消失的**瞬间**就被回收，时机可预测。这对"对象析构即释放资源"的直觉很友好（虽然关键资源仍该用 `with`，见 ⑤）。追踪式 GC 的回收时机则不确定。
- **实现简单、与 C 扩展协作好**：引用计数是局部的、增量的——每次引用增减就地 +1/-1，不需要"停下整个世界扫描堆"。C 扩展只要老老实实 `Py_INCREF`/`Py_DECREF` 就能正确参与内存管理，无需理解全局 GC 算法。这对"Python 当胶水、热点下沉到 C"的定位至关重要（呼应 [05 CPython内部](05_CPython内部.md)）。
- **代价是循环引用**：引用计数有个天生死穴——相互引用成环的对象，计数永远归不了零（见 ③ 的 2.2）。所以 CPython 必须再叠一个**循环 GC** 专门兜底回收这种"不可达的引用环"。这就是为什么是"两套机制"而非一套。

> 一句话：CPython 用引用计数换"确定性 + 简单 + 易与 C 协作"，用循环 GC 补上引用计数回收不了环的漏洞。JVM 用一套追踪式 GC 换吞吐与无需手工计数，代价是回收时机不确定。两种世界观，各有取舍。

## ③ 实现细节（底层怎么实现的）

### 2.1 引用计数：CPython 的主力

CPython 里**每个对象**头部都存了一个**引用计数（refcount）**：有多少个名字/容器槽位指向它。

- 每当多一个引用（赋值、传参、放进容器）→ 计数 +1
- 每当少一个引用（变量离开作用域、`del`、容器移除）→ 计数 -1
- **计数归零的那一刻，对象立即被销毁、内存回收**

```python
import sys

a = []
print(sys.getrefcount(a))   # 2（a 一个 + getrefcount 的参数临时一个）

b = a                       # 多一个引用
print(sys.getrefcount(a))   # 3

del b                       # 少一个引用
print(sys.getrefcount(a))   # 2
```

> `sys.getrefcount(x)` 返回值总比你想的多 1，因为把 `x` 传进函数本身又制造了一个临时引用。

**和 JVM 最大的体感差异：确定性销毁。** 在 CPython 里，一个局部对象在函数返回时（引用归零）几乎立刻被回收，不用等某个不确定的 GC 周期。这也是为什么很多 Python 代码"忘了关文件"也能侥幸工作——文件对象引用归零时 `__del__` 顺手关了它（但**别依赖这个**，见最佳实践）。

### 2.2 引用计数的死穴：循环引用

引用计数有个致命缺陷：**对象互相引用形成环时，计数永远不会归零**，即使外部已经无人可达。

```python
a = {}
b = {}
a["b"] = b      # a 引用 b
b["a"] = a      # b 引用 a，形成环
del a
del b           # 外部引用都删了，但 a、b 互相还引用着对方，refcount 都是 1
                # → 纯引用计数永远回收不了它们 → 内存泄漏
```

JVM 没这问题（可达性分析从 GC Root 出发，环但不可达照样回收）。CPython 为此引入了**第二套机制**。

### 2.3 循环垃圾回收器（cyclic GC / `gc` 模块）

CPython 额外有一个**分代的循环垃圾回收器**，专门检测并回收引用计数处理不了的"不可达引用环"。

- 它只追踪**容器对象**（list/dict/set/自定义实例等可能形成环的东西），不管 int/str 这类不可能成环的。
- 它也是**分代的**（3 代：0/1/2），新对象在第 0 代，熬过几轮回收的晋升到老代，老代扫描频率低——**这个思路和 JVM 分代假设一致**（朝生夕死的对象多）。
- 触发靠分配/释放计数的阈值，不是固定时间。

```python
import gc

print(gc.get_threshold())   # (700, 10, 10) 默认阈值：第0代分配净增 700 触发
gc.collect()                # 手动触发一次完整循环回收，返回回收的不可达对象数
gc.disable()                # 可禁用循环 GC（引用计数仍在工作）
```

> 关键认知：**引用计数 + 循环 GC 是互补的**。99% 的对象靠引用计数即时回收（无 STW、无延迟）；循环 GC 只负责兜底清理引用环。所以 Python 的"GC 调优"远不如 JVM 那么是门学问——大多数时候你根本不用碰它。

### 2.4 `__del__`：终结器（不是 Java 的 finalize 那么糟，但也别滥用）

`__del__` 在对象**引用计数归零、即将销毁时**被调用：

```python
class Resource:
    def __init__(self, name):
        self.name = name
    def __del__(self):
        print(f"释放 {self.name}")

r = Resource("db-conn")
del r            # 立刻打印 "释放 db-conn"（引用计数归零）
```

但有几个坑让它**不适合做资源清理**：
- 涉及**循环引用**时，对象由循环 GC 回收，`__del__` 调用时机变得不确定（旧版本甚至可能不调）。
- 解释器退出时不保证调用所有 `__del__`。
- `__del__` 里抛异常会被忽略，难调试。

> 结论：`__del__` ≈ JVM 的 `finalize`——**能用但不可靠，不要拿它管理文件/连接/锁等关键资源**。确定性清理用 `with`（上下文管理器，第 13 章），它对应 Java 的 try-with-resources，是正道。

### 2.5 弱引用 `weakref`

弱引用**不增加引用计数**，让你引用一个对象又不阻止它被回收。对应 Java 的 `WeakReference`，典型用于缓存、观察者、打破循环引用。

```python
import weakref

class Node:
    pass

n = Node()
w = weakref.ref(n)       # 弱引用，不增加 refcount
print(w())               # <Node object ...>  解引用拿到对象
del n                    # 唯一强引用没了 → 对象被回收
print(w())               # None  弱引用自动失效
```

实战：`weakref.WeakValueDictionary` 做缓存（值被回收后条目自动消失，不会因缓存而泄漏）；父子节点中"子→父"用弱引用打破环。

### 2.6 内存管理的其它要点

- **对象内存池**：CPython 用 `pymalloc` 管理小对象（<512 字节）的分配，分层（arena/pool/block），减少 `malloc` 开销。类比 JVM 的 TLAB 思路但实现不同。
- **`__slots__`**：默认每个实例用一个 `__dict__`（dict）存属性，灵活但费内存。声明 `__slots__` 改用固定槽位，省内存、略快——海量小对象时有用（第 11 章详讲）。
- **释放回操作系统**：Python 释放的内存不一定立刻还给 OS（arena 缓存复用），所以进程 RSS 可能不随对象释放立即下降，这点排查内存时要知道。

## ④ 使用方法与最佳实践

1. **关键资源（文件、socket、连接、锁）用 `with`**，绝不依赖 `__del__` 或引用计数顺手清理。这是和 try-with-resources 等价的正道。
2. **绝大多数时候别碰 `gc` 模块**。需要时的合理场景：长时间运行服务里手动 `gc.collect()`、或确认无环的高性能场景里 `gc.disable()` 减少暂停（如某些批处理）。
3. **警惕循环引用**导致的内存增长：父子/双向结构、回调持有彼此、闭包捕获 self。用 `weakref` 打破环。
4. 排查内存泄漏：`gc.get_objects()`、`tracemalloc`（标准库，对照 JVM 的堆 dump 分析）、`objgraph`（第三方，画引用图）。
5. 海量同构小对象 → 考虑 `__slots__` 或换用 `array`/`numpy`（数据/AI 场景尤其）。

## ⑤ ⚠️ 易错点（Java 思维翻车）

| Java 习惯 | 翻车现场 | 正确做法 |
|----------|---------|---------|
| 依赖 GC 兜底关资源 | 大量打开文件/连接不显式关，靠析构，高负载下句柄耗尽 | 用 `with` 确定性释放 |
| 把 `__del__` 当 try-finally | 在 `__del__` 里关连接，循环引用时不被及时调用 | 用上下文管理器，别在 `__del__` 放关键清理 |
| 以为有环也没事（JVM 习惯） | Python 里大量短命环堆积，循环 GC 压力 / 关 GC 时直接泄漏 | 用 weakref 主动断环 |
| 以为对象释放内存立刻还 OS | 监控看到 RSS 不降，误判泄漏 | 了解 arena 缓存机制，用 tracemalloc 看真实对象 |
| 像调 JVM 一样狂调 GC 参数 | 花时间调 `gc` 阈值收益甚微 | Python GC 旋钮少，多数场景默认即可 |
| 用 `==`/`is` 判对象"是否被回收" | 无法判断 | 用 `weakref`，解引用返回 `None` 即已回收 |

## ⑥ 练习题

### 基础理解
1. 用一句话说清：CPython 为什么需要"引用计数 + 循环 GC"两套机制，而 JVM 只要一套追踪式 GC？
2. 为什么说 Python 的对象销毁"相对确定性"，而 Java 的 `finalize` 不确定？这对资源管理意味着什么？

### 找坑改错
3. 下面的缓存为什么可能导致内存无限增长？如何用标准库改造让"无人使用的对象自动从缓存消失"？
```python
_cache = {}
def get_user(uid):
    if uid not in _cache:
        _cache[uid] = load_user_from_db(uid)
    return _cache[uid]
```
4. 这段树结构在大量创建/丢弃后内存持续上涨，指出根因并给出修复方向：
```python
class Node:
    def __init__(self):
        self.children = []
        self.parent = None
    def add(self, child):
        child.parent = self        # 子持有父（强引用）
        self.children.append(child)# 父持有子（强引用）→ 环
```

### 综合实战
5. 你负责一个 7×24 运行的 Python 服务，监控发现内存缓慢上涨。给出一套排查思路：如何区分"循环引用泄漏"、"全局容器只增不减"、"C 扩展泄漏"，分别用哪些工具（提示：`gc`、`tracemalloc`、`objgraph`）。

---

### 参考答案

**1.** 引用计数无法回收"互相引用但整体不可达"的对象环（计数永不归零），所以需要循环 GC 兜底；JVM 用可达性分析从 GC Root 出发，环只要不可达就能回收，单套机制即可。

**2.** Python 对象在最后一个引用消失（引用计数归零）的瞬间立即销毁并调 `__del__`，时机基本可预测；Java 的 `finalize` 由 GC 在不确定的时间触发，可能很久甚至从不调用。意味着 Python 里非循环对象的清理较及时，但**仍不该依赖它管理关键资源**——确定性释放要用 `with`。

**3.** `_cache` 是强引用的全局 dict，只增不删，所有加载过的 user 永久驻留 → 无限增长。用 `weakref.WeakValueDictionary`：当外部不再强引用某 user 对象时，它被回收，缓存条目自动消失。
```python
import weakref
_cache = weakref.WeakValueDictionary()
```
（注意：值对象必须支持弱引用，且仍被外部使用时才留在缓存。若要"无人用就丢"用此法；若要"限量缓存"用 `functools.lru_cache`。）

**4.** 父子互相强引用形成循环引用。纯引用计数回收不了，得靠循环 GC，回收延迟、`__del__` 时机不定，丢弃大量树时内存上涨。修复：子对父用弱引用——
```python
import weakref
child.parent = weakref.ref(self)   # 访问时 child.parent() 解引用
```
这样"父→子"是强引用维持结构，"子→父"是弱引用不成环。

**5.** 排查思路：
- 先开 `tracemalloc`（启动加 `tracemalloc.start()`），周期性 `take_snapshot()` 对比，看哪些代码行分配的对象数/字节持续增长——定位增长点。
- 用 `gc.collect()` 后看 `gc.garbage`、`len(gc.get_objects())` 趋势；若手动 `collect` 能回收大量对象，倾向**循环引用**问题。
- 若某全局 `list`/`dict`/缓存的 `len()` 单调上涨 → **容器只增不减**，加淘汰策略或弱引用。
- 若 Python 层对象数稳定但进程 RSS 持续涨 → 怀疑 **C 扩展泄漏**（numpy/原生库），用系统级工具（valgrind/`tracemalloc` 看不到 C 分配）和扩展自身的诊断。
- `objgraph.show_growth()` / `show_backrefs()` 可视化哪些类型在增长、被谁引用。
