# 14 · 类型系统与注解 ⭐

> 你在 Java 里被编译器的强类型保护惯了：写错类型过不了 `javac`，泛型 `List<String>` 塞 Integer 直接报错。来到 Python，你会看到 `def f(x: int) -> str` 这种"看着像类型声明"的东西——**但它是一场精心设计的误会**。Python 的类型注解是**可选的、给工具看的、运行时完全不强制的**元数据。本章讲清楚：注解到底存在哪、为什么运行时不检查、mypy/pyright 这些"编译期外挂"怎么工作、`@dataclass` 如何用 `exec` 动态生成 `__init__`（对照 Lombok 的注解处理器），以及怎么把渐进类型（gradual typing）用对、不把它当 Java 泛型理解。

---

## ① Java 对照表

| 概念 | Java | Python | 关键差异 |
|------|------|--------|---------|
| 类型声明 | `String name`（强制，编译期检查） | `name: str`（**注解，运行时不检查**） | Python 注解是元数据，不是约束 |
| 注解/annotation | `@NotNull`、`@Override`（运行时可反射读） | type hint 存进 `__annotations__` | 名字像但角色完全不同 |
| 泛型 | `List<T>`（编译期检查 + **类型擦除**） | `list[T]` / `TypeVar` / `Generic`（**不擦除但运行时也不检查**） | Java 擦除后运行时丢类型；Python 保留 `__args__` 但不据此校验 |
| `Optional<T>` | `Optional<String>`（包装对象） | `str | None`（3.10+）/ `Optional[str]` | Python 不是包装，就是联合类型 |
| interface | `interface Comparable` + `implements` | `Protocol`（结构化，**无需显式实现**） | Python 是鸭子类型的静态版 |
| record / Lombok | `record Point(int x,int y)` / `@Data` | `@dataclass`（运行时 `exec` 生成方法） | Lombok 编译期改字节码；dataclass 运行时生成源码 |
| 枚举值约束 | `enum Color` | `Literal["red","green"]` | Literal 是纯静态约束 |
| 结构化字典 | DTO / `Map<String,Object>` | `TypedDict` | 给 dict 加静态形状，运行时仍是普通 dict |
| 方法重载 | 真重载（按签名分派字节码） | `@overload`（**仅类型提示，无运行时分派**） | Python 只有一个真实现 |
| 类型检查时机 | 编译期，`javac` 内建 | **独立工具** mypy/pyright（编译期外挂） | 解释器自己从不做类型检查 |

> 一句话记住：**Java 的类型是给编译器的约束，Python 的类型是给静态分析工具的注释。** 解释器对它视而不见。

## ② 设计理念：为什么类型是"可选的、给工具看的"

### 2.1 渐进类型（Gradual Typing）

Python 1991 年诞生时是纯动态语言，没有任何类型声明。社区花了二十年才在 **PEP 484（2014，Python 3.5）** 引入类型注解。这个时间差决定了它的设计哲学必须是 **gradual typing（渐进类型）**：

- **既有的几亿行无注解代码必须照常运行**——所以注解只能是**可选**的，不能强制。
- 你可以**只给一部分代码加类型**：核心模块严格标注、脚本随便写，混在一起也能跑。
- 没标注的地方被工具当作 `Any`，"放行"而不报错。

这和 Java"从第一天就强类型、不写类型编译不过"是相反的起点。Java 是"默认严格、无法关闭"；Python 是"默认无类型、按需加码"。

> **核心取舍**：Python 保留了动态语言的灵活（duck typing、运行时改类、`__getattr__` 魔法），又让需要规模化的团队**自愿**引入静态检查。代价是类型系统永远是"第二公民"——解释器不靠它做任何决策。

### 2.2 为什么运行时不强制

让解释器在每次函数调用时校验 `x: int` 会带来三个问题，违背 Python 的设计目标：

1. **性能**：每次调用都要做 `isinstance` 检查（还得递归检查 `list[dict[str, int]]` 这种嵌套），动态语言本就慢，再加这层开销不可接受。
2. **表达力倒退**：duck typing 的精髓是"能嘎嘎叫就是鸭子"。强制 `isinstance` 检查会把"任何有 `.read()` 方法的对象"压缩成"必须是某个具体类"，扼杀多态。
3. **渐进性破坏**：如果运行时强制，那"只标注一半"就会在边界处爆炸，gradual 就无从谈起。

所以 Python 选择：**注解只是元数据，校验交给你愿意时才运行的外部工具**。想要运行时校验？那是**显式**的库（pydantic，见 [26 日期时间序列化pydantic](26_日期时间序列化pydantic.md)）的活，不是语言的活。

### 2.3 为什么用 dataclass 消灭样板

Java 写个数据类要手写或靠 Lombok/record 生成 `构造器/equals/hashCode/toString`。Python 的 `@dataclass`（PEP 557，3.7+）解决同样的痛点，但走的是**运行时生成**而非编译期生成的路。理念是：既然类创建本身就是运行时行为（class 体是可执行代码，见 [13 类与对象](13_类与对象.md)），那就在创建类的那一刻顺手把这些样板方法**用代码生成出来**。你用一行装饰器声明意图，省掉几十行重复代码——这与 Lombok 的目标一致，实现机制却很 Python。

## ③ 实现细节（底层怎么实现的）★本章重点★

### 3.1 注解存在哪：`__annotations__` 字典

类型注解不是语法糖、不会消失，它被求值后存进一个普通 dict——`__annotations__`。函数、类、模块各有一份。

```python
def transfer(from_acct: str, amount: float) -> bool:
    return True

print(transfer.__annotations__)
# 输出: {'from_acct': <class 'str'>, 'amount': <class 'float'>, 'return': <class 'bool'>}
```

注意 `str`、`float` 这些**是真正的类对象**，注解表达式在 `def` 执行时就被**求值**了。变量注解同理：

```python
class Account:
    balance: float          # 只声明，不赋值
    owner: str = "unknown"  # 声明 + 默认值

print(Account.__annotations__)
# 输出: {'balance': <class 'float'>, 'owner': <class 'str'>}
print(Account.owner)        # 输出: unknown
print('balance' in vars(Account))  # 输出: False  纯注解不创建属性
```

> 对照 Java：Java 的 annotation 也能运行时反射读取（`method.getAnnotations()`），这点相似。但 Java annotation 是**元数据标签**（`@Deprecated`），不是类型系统本身；Python 把**类型**也塞进了这个元数据通道。

### 3.2 运行时**完全不强制**——用代码证明

这是 Java 老手最需要在肌肉记忆里改写的一条。注解对解释器就是注释，传错类型照样跑：

```python
def repeat(text: str, times: int) -> str:
    return text * times

# 故意传反：text 传 int=3，times 传 str="abc" —— 解释器毫不在意，照样算
print(repeat(3, "abc"))     # 输出: abcabcabc   (实际执行的是 3 * "abc")
```

注解全标反了却毫无怨言。再看一个错误"延迟爆发"的例子：

```python
def add_tax(price: float) -> float:
    return price * 1.1

result = add_tax("100")     # 传了 str！
# 输出: TypeError: can't multiply sequence by non-int of type 'float'
```

注意：上面报错**不是因为注解校验失败**，而是因为 `"100" * 1.1` 这个运算本身非法。把运算换成合法的，注解错了也毫无声息：

```python
def first(items: list[int]) -> int:
    return items[0]

print(first("hello"))       # 输出: h   传 str 照样跑，返回 str，注解全错也不报
print(first({3: 'a'}))      # 输出: ... 取决于 dict[0]，注解 list[int] 形同虚设
```

**结论**：`def f(x: int)` 不等于 Java 的 `void f(int x)`。Java 编译器拦住你；Python 解释器放你过去，错误要等到某个运算真的不兼容时才以 `TypeError` 形式爆发——或者永远不爆发，悄悄算出错误结果。

### 3.3 `list[int]` 的内部：`__class_getitem__`

`list[int]`、`dict[str, int]` 这种下标语法（3.9+ 内置泛型）是怎么工作的？靠 `__class_getitem__` 这个特殊方法。`list[int]` 实际等价于 `list.__class_getitem__(int)`，返回一个 `types.GenericAlias` 对象——**它保存了参数但不创建新类型**：

```python
ga = list[int]
print(type(ga))             # 输出: <class 'types.GenericAlias'>
print(ga.__origin__)        # 输出: <class 'list'>   原始容器
print(ga.__args__)          # 输出: (<class 'int'>,)  类型参数被保留！

# 关键：它仍然能当普通 list 用，参数被忽略
xs = list[int]()            # 等价于 list()
xs.append("not an int")     # 元素类型？运行时根本不看
print(xs)                   # 输出: ['not an int']
```

> **对照 Java 的类型擦除**：Java 编译后 `List<String>` 和 `List<Integer>` 在字节码里都是裸 `List`，运行时拿不到 `String`（除非用反射技巧）。Python **相反**——`__args__` 把 `int` 保留在对象里，运行时**拿得到**类型参数。但拿得到不等于会校验：Python 保留它**纯粹是给静态工具和反射用的**，解释器自己绝不据此检查元素。所以两者殊途同归：**运行时都不做泛型类型检查**，只是 Java 因为擦除"拿不到"，Python 因为设计哲学"拿得到也不查"。

自定义泛型类用 `Generic` 基类自动获得 `__class_getitem__`：

```python
from typing import TypeVar, Generic

T = TypeVar("T")

class Stack(Generic[T]):
    def __init__(self) -> None:
        self._items: list[T] = []
    def push(self, x: T) -> None:
        self._items.append(x)
    def pop(self) -> T:
        return self._items.pop()

print(Stack[int])           # 输出: __main__.Stack[int]
print(Stack[int].__args__)  # 输出: (<class 'int'>,)
s: Stack[int] = Stack()     # 运行时 Stack[int]() 和 Stack() 行为完全一样
```

### 3.4 `from __future__ import annotations`（PEP 563）：注解字符串化

3.2 里我们看到注解在 `def` 执行时被**求值**。这有两个问题：(1) 前向引用——类还没定义就在注解里用它会 `NameError`；(2) 每次定义函数都求值注解有微小开销。**PEP 563** 提供了惰性方案：加一行 `from __future__ import annotations`，所有注解**不再求值，而是原样存成字符串**。

```python
from __future__ import annotations

def parse(data: dict[str, int]) -> Account:   # Account 在下面才定义
    ...

class Account:
    pass

print(parse.__annotations__)
# 输出: {'data': 'dict[str, int]', 'return': 'Account'}   全是字符串，没求值！
```

没有这一行，`-> Account` 会在 `def parse` 执行时报 `NameError`（Account 尚未定义）。有了它，注解只是字符串，前向引用天然成立。需要拿回真正的类型对象时，用 `typing.get_type_hints()` 按需解析：

```python
import typing
print(typing.get_type_hints(parse))
# 输出: {'data': dict[str, int], 'return': <class '__main__.Account'>}  字符串被解析回真类型
```

> 这相当于把注解从"立即求值"变成"惰性求值的字符串字面量"。框架（如 pydantic、FastAPI）才会真去 `get_type_hints` 解析；普通运行根本不碰它们，所以加这行几乎零成本，还顺手解决前向引用。3.x 后续版本对默认行为有持续演进（PEP 649/749），但 `from __future__ import annotations` 是当前最稳的写法。

### 3.5 mypy / pyright：编译期外挂

既然解释器不检查类型，谁来检查？**独立的静态分析工具**，最主流的是 **mypy**（官方出品）和 **pyright**（微软，VS Code 内置的 Pylance 即基于它）。它们的工作方式：

```
你的 .py 源码
   │  mypy/pyright 解析（自己的 parser，不运行你的代码）
   ▼
构建 AST + 读取所有 __annotations__ + 推断未标注处的类型
   │  类型推断 + 约束求解（类似一个独立的类型检查器）
   ▼
报告类型错误（退出码非 0），但绝不修改/运行你的程序
```

关键认知：

- **它们不导入也不执行你的代码**（纯静态分析），所以没有副作用，能在 CI 里跑。这与 `javac` 内建于编译流程不同——Python 把"编译期类型检查"做成了**可选的、可拔插的外部步骤**。
- 它们做**类型推断**：没标注的局部变量也能推断出类型（`x = 5` 推断为 `int`），不是只看注解。
- 它们的"严格度"可调（`mypy --strict`），团队按需选择渐进类型的程度。

```bash
# mypy 在命令行报错，但你的程序此刻并没有运行
$ mypy app.py
app.py:12: error: Argument 1 to "add_tax" has incompatible type "str"; expected "float"  [arg-type]
Found 1 error in 1 file (checked 1 source file)
```

> 类比：mypy/pyright 之于 Python，像 **ESLint/SpotBugs 之于代码质量**——一个独立的、你选择运行的检查器。Java 的类型检查是 `javac` **强绑定**的；Python 把它解耦成了外挂。

### 3.6 `@dataclass`：用 `exec` 在运行时生成源码 ★深挖★

这是本章最能体现"Python 是动态语言"的机制。Lombok 的 `@Data` 是**编译期**的注解处理器（annotation processor），在 `javac` 编译时介入、往字节码里**写入** `getter/equals/hashCode`。`@dataclass` 完全不同：它在**类创建时（运行时）**，**把样板方法当作 Python 源码字符串拼出来，用 `exec()` 编译执行，再挂到类上**。

```python
from dataclasses import dataclass, field

@dataclass
class Order:
    id: int
    amount: float
    tags: list[str] = field(default_factory=list)

o = Order(1, 9.9)
print(o)                    # 输出: Order(id=1, amount=9.9, tags=[])   __repr__ 是生成的
print(o == Order(1, 9.9))   # 输出: True                                __eq__ 是生成的

import inspect
print(inspect.signature(Order.__init__))
# 输出: (self, id: int, amount: float, tags: list[str] = <factory>) -> None
```

`__init__`、`__repr__`、`__eq__` 你一行没写，它们从哪来？看 CPython `dataclasses.py` 里真实的生成函数（节选其内部 `_create_fn`）：

```python
def _create_fn(name, args, body, *, globals=None, locals=None, ...):
    ...
    args = ','.join(args)
    body = '\n'.join(f'  {b}' for b in body)
    txt = f' def {name}({args}){return_annotation}:\n{body}'   # 拼出函数源码字符串
    ...
    txt = f"def __create_fn__({local_vars}):\n{txt}\n return {name}"
    ns = {}
    exec(txt, globals, ns)            # ★用 exec 把源码字符串编译成真函数★
    return ns['__create_fn__'](**locals)
```

也就是说，`@dataclass` 在装饰 `Order` 时，把 `__init__` 拼成类似这样的源码字符串：

```python
def __init__(self, id: int, amount: float, tags=<factory>):
    self.id = id
    self.amount = amount
    self.tags = tags if tags is not _HAS_DEFAULT_FACTORY else list()
```

然后 `exec` 它、得到真函数、`setattr` 到 `Order` 上。`__repr__` 和 `__eq__` 同理。

> **两种"消灭样板"的对照**：
> | 维度 | Lombok `@Data` | Python `@dataclass` |
> |------|----------------|---------------------|
> | 介入时机 | 编译期（javac annotation processor） | **运行时**（类创建时） |
> | 产物形式 | 字节码里的方法 | `exec` 生成的真 Python 函数对象 |
> | 能否看到生成结果 | 反编译字节码才看得到 | `inspect.getsource` 拿不到（exec 的没源文件），但 `inspect.signature`/字节码可查 |
> | 是否可定制 | 注解参数 | 装饰器参数 `frozen/slots/order/...` + `field()` |
>
> 二者都让你"写一行、得一堆方法"，但 Lombok 是**编译期魔法**，dataclass 是**运行时元编程**——这正是静态语言和动态语言世界观的缩影。

### 3.7 `TypeVar` 与 `Protocol` 的运行时真身

`TypeVar("T")` 创建的是一个**普通的 `TypeVar` 实例对象**，运行时它什么也不约束：

```python
from typing import TypeVar
T = TypeVar("T", bound=int)     # bound 是给静态工具的约束
print(T)                        # 输出: ~T
print(type(T))                  # 输出: <class 'typing.TypeVar'>
print(T.__bound__)              # 输出: <class 'int'>   信息保留着，但没人运行时强制
```

`Protocol`（PEP 544）是"静态版的鸭子类型"：定义一组方法/属性，**任何结构匹配的类自动算实现**，无需 `implements`。默认它只在静态层面起作用；加 `@runtime_checkable` 后 `isinstance` 才能做**仅看方法名存在性**的弱检查（不检查签名）：

```python
from typing import Protocol, runtime_checkable

@runtime_checkable
class Readable(Protocol):
    def read(self) -> str: ...

class FileLike:
    def read(self) -> str: return "data"

print(isinstance(FileLike(), Readable))   # 输出: True   结构匹配，无需继承
print(isinstance([1, 2], Readable))       # 输出: False  list 没有 read
```

> 对照 Java 的 `interface`：Java 必须 `class Foo implements Readable` 显式声明且编译器检查全部方法签名；Python 的 Protocol 是**结构化、隐式、默认静态**的——它把 duck typing 从"运行时碰运气"提升为"静态可检查"，又不丢掉 duck typing 的松耦合。

## ④ 使用方法与最佳实践

### 4.1 基础注解

```python
name: str = "alice"
age: int = 30
ratio: float = 0.95
active: bool = True

def login(user: str, *, remember: bool = False) -> bool:
    return True

# 无返回值用 None（不是 void）
def log(msg: str) -> None:
    print(msg)
```

### 4.2 容器泛型（3.9+ 用内置小写）

```python
# 3.9+：直接用内置类型做泛型，不必从 typing 导入
ids: list[int] = [1, 2, 3]
prices: dict[str, float] = {"apple": 3.5}
coords: tuple[float, float] = (1.0, 2.0)
unique: set[str] = {"a", "b"}

# 不定长同质 tuple 用 ...
row: tuple[int, ...] = (1, 2, 3, 4)
```

> 3.9 以前要写 `from typing import List, Dict` 再用 `List[int]`。**新代码一律用内置小写 `list[int]`**，`typing.List` 已不推荐。

### 4.3 `X | None` 与联合类型（3.10+）

```python
# 3.10+ 推荐写法：用 | 表达联合 / 可空
def find_user(uid: int) -> str | None:        # 可能返回 str 或 None
    return None

def parse(x: int | str) -> float:             # 接受 int 或 str
    return float(x)

# 等价的老写法（3.10 以前 / 兼容场景）
from typing import Optional, Union
def find_user_old(uid: int) -> Optional[str]:    # == str | None
    return None
```

> **`Optional[str]` 就是 `str | None`，不是"包装对象"**。它表达"这里可能是 None"，和 Java `Optional<String>`（一个真包装类，有 `.isPresent()`）**完全不同**——Python 里你拿到的还是 `str` 或 `None` 本身，靠 `if x is not None` 收窄。

### 4.4 `TypeVar` 写泛型函数 / 类

```python
from typing import TypeVar

T = TypeVar("T")

def first(items: list[T]) -> T:          # 返回类型跟着入参元素类型走
    return items[0]

n: int = first([1, 2, 3])                # 工具推断为 int
s: str = first(["a", "b"])               # 工具推断为 str

# 有界 TypeVar：限制为某基类的子类型
from numbers import Number
N = TypeVar("N", bound=Number)
def total(xs: list[N]) -> N:
    return sum(xs)                        # type: ignore  (sum 的标注偏严，演示用)
```

> 3.12+ 还有更简洁的 **PEP 695 语法**：`def first[T](items: list[T]) -> T:` 直接在方括号里声明类型参数，不必预先 `TypeVar`。类似 Java 的 `<T> T first(List<T>)`。

```python
# 3.12+ 新泛型语法
def first[T](items: list[T]) -> T:
    return items[0]

class Stack[T]:                          # 不必继承 Generic[T]
    def __init__(self) -> None:
        self._data: list[T] = []
```

### 4.5 `Protocol` 做结构化接口

```python
from typing import Protocol

class SupportsClose(Protocol):
    def close(self) -> None: ...

def cleanup(resource: SupportsClose) -> None:
    resource.close()                     # 只要有 close() 就能传，无需共同基类

# 数据库连接、文件、socket 都能传——它们结构上都有 close()
```

> 这是给后端代码解耦的利器：定义"需要什么能力"而非"必须是什么类"。比抽象基类（ABC）更松耦合，因为**实现方无需知道 Protocol 的存在**。

### 4.6 `Callable`：函数类型

```python
from typing import Callable

# Callable[[参数类型...], 返回类型]
def apply(fn: Callable[[int, int], int], a: int, b: int) -> int:
    return fn(a, b)

handler: Callable[[str], None] = print    # 接受 str、返回 None 的可调用对象
apply(lambda x, y: x + y, 3, 4)           # 5
```

### 4.7 `Literal`：枚举式取值约束

```python
from typing import Literal

def set_mode(mode: Literal["r", "w", "a"]) -> None:
    ...

set_mode("r")        # OK
set_mode("x")        # mypy 报错：不在允许的字面量集合里（运行时仍放行）

# 常用于代替"魔法字符串"参数
Method = Literal["GET", "POST", "PUT", "DELETE"]
def request(method: Method, url: str) -> None: ...
```

### 4.8 `TypedDict`：给 dict 加静态形状

```python
from typing import TypedDict

class UserDTO(TypedDict):
    id: int
    name: str
    email: str

def make_user(raw: dict) -> UserDTO:
    return {"id": raw["id"], "name": raw["name"], "email": raw["email"]}

u: UserDTO = {"id": 1, "name": "bob", "email": "b@x.com"}
print(type(u))       # 输出: <class 'dict'>   运行时就是普通 dict，零开销
```

> `TypedDict` 适合"形状固定的 JSON / API 响应"，给 dict 静态形状检查但保留 dict 的零成本。需要**运行时校验 + 解析**（外部输入不可信时）则用 pydantic（[26 章](26_日期时间序列化pydantic.md)）。

### 4.9 `@overload`：多签名提示（无运行时分派）

Python 只有**一个真实现**，`@overload` 仅给工具描述"不同入参对应不同返回类型"：

```python
from typing import overload

@overload
def get(key: str) -> str: ...
@overload
def get(key: str, default: int) -> str | int: ...

def get(key, default=None):              # 唯一的真实现
    return {"a": "1"}.get(key, default)

# 工具据 overload 推断：get("a") 返回 str；get("a", 0) 返回 str | int
```

> 对照 Java 真重载：Java 编译期按签名选不同方法体；Python 的 `@overload` **不产生分派**，运行时只跑那个真实现。重载块本身在运行时会被后面的真定义覆盖。

### 4.10 `dataclass` 完整用法

```python
from dataclasses import dataclass, field

@dataclass(frozen=True, slots=True)      # frozen=不可变(类似 record); slots=省内存
class Point:
    x: float
    y: float
    label: str = "origin"                # 默认值
    tags: list[str] = field(default_factory=list)   # 可变默认值必须用 default_factory

p = Point(1.0, 2.0)
print(p)                 # 输出: Point(x=1.0, y=2.0, label='origin', tags=[])
# p.x = 5               # FrozenInstanceError: 不可变
print(p == Point(1.0, 2.0))   # 输出: True   值相等

# 只读派生字段 / 排除某字段不参与 init
@dataclass
class Account:
    owner: str
    _internal_id: int = field(default=0, repr=False, compare=False)
```

| 参数 | 作用 | 类比 |
|------|------|------|
| `frozen=True` | 实例不可变（赋值报错） | Java `record` / `final` 字段 |
| `slots=True`（3.10+） | 用 `__slots__` 省内存、禁动态属性 | 见 [04 运行时与内存模型](04_运行时与内存模型.md) |
| `order=True` | 生成 `<`/`<=`/`>`/`>=` | `Comparable` |
| `field(default_factory=list)` | 每个实例独立的可变默认值 | 防共享可变状态 |
| `field(repr=False)` | 该字段不进 `__repr__` | 隐藏敏感字段 |

### 4.11 何时上类型、何时不上

| 场景 | 建议 |
|------|------|
| 公共 API、库的对外函数 | **必须**标注（别人靠它理解契约） |
| 后端服务核心模块、DTO | **标注** + `mypy --strict` |
| 一次性脚本、notebook 探索 | 可以不标，别为类型纠结 |
| 私有小函数、显而易见的局部变量 | 按需，别过度（`x: int = 5` 多余） |
| 复杂动态逻辑（元编程、`**kwargs` 转发） | 标到能标的程度，剩下用 `Any`/`# type: ignore` 务实放过 |

配 `mypy --strict` 的最小起步：

```ini
# pyproject.toml
[tool.mypy]
strict = true
python_version = "3.12"
warn_unused_ignores = true
```

> 渐进类型的正确姿势：**新代码严格标注、老代码逐步补、工具放 CI 里把关**。不要追求 100% 覆盖率，要追求"边界和契约处类型清晰"。

## ⑤ ⚠️ 易错点（Java 思维翻车）

| Java 习惯 | 翻车现场 | 正确做法 / 认知 |
|----------|---------|----------------|
| 以为注解运行时会校验 | `def f(x: int)` 传 str 不报错，悄悄算出错误结果还以为类型安全 | 注解=元数据，校验靠 mypy；要运行时校验用 pydantic |
| 可变默认值用 `field` 直接给 | `tags: list = []` → `ValueError: mutable default not allowed` | 用 `field(default_factory=list)`，每实例独立 |
| 以为 `list[int]` 运行时检查元素 | 往 `list[int]` 塞 str 照样跑，bug 潜伏到运算时才炸 | 运行时不查元素；靠 mypy 或运行时显式校验 |
| 把 `Optional[T]` 当 Java `Optional` | 调 `.isPresent()`/`.get()` → `AttributeError` | `Optional[T]` 就是 `T | None`，用 `if x is not None` 收窄 |
| 把 `@overload` 当真重载 | 以为运行时按签名分派、写多个函数体 | 只有最后一个是真实现，overload 块仅给工具看 |
| 把 `TypeVar` 当 Java 泛型理解 | 以为运行时会按 `T` 校验/擦除 | Python 不擦除（`__args__` 保留）但运行时也不检查，纯给工具 |
| 以为 mypy 通过=运行不出类型错 | mypy 绿了仍可能 `TypeError`（`Any`、`cast`、外部数据） | mypy 是静态近似，边界输入仍要运行时防御 |
| 用 `==` 比较自己写的类忘了 dataclass | 普通类 `==` 默认比 `id`（同一性），值相等返回 False | 用 `@dataclass` 自动生成基于值的 `__eq__` |
| Protocol 当 interface 要显式 implements | 找不到"implements 关键字"困惑 | Protocol 是结构化的，结构匹配即实现，无需声明 |
| 前向引用直接用未定义的类 | 注解里引用下文的类 → `NameError` | 加 `from __future__ import annotations`，或注解写成字符串 |

一段典型的"Java 味 Python"反面教材：

```python
# ❌ Java 脑回路：以为类型注解 + 重载就万事大吉
from typing import Optional

class UserService:
    def find(self, uid: int) -> Optional[dict]:
        ...
    def find(self, name: str) -> Optional[dict]:   # 以为这是重载——其实覆盖了上一个！
        ...

svc = UserService()
result: dict = svc.find(123)        # 注解说 Optional[dict]，但你当 dict 用
print(result.get("name"))           # 若真返回 None → AttributeError，注解没拦住你
```

问题：(1) 同名方法第二个直接**覆盖**第一个，Python 没有按签名重载；(2) 把 `Optional[dict]` 当 `dict` 用，没做 `None` 检查，注解不会在运行时保护你。

## ⑥ 练习题

### 基础理解

1. 一个函数 `def scale(x: float, factor: int) -> float: ...`，你调用 `scale("hello", "world")`。解释器会在调用时因为类型不符而报错吗？什么时候、以什么形式才可能出错？注解此刻在哪、起了什么作用？

2. 用一句话分别说清：(a) Java 泛型的"类型擦除"和 Python 泛型"运行时不检查"的区别；(b) Lombok `@Data` 和 Python `@dataclass` 在"生成样板代码"上的机制差异。

### 找坑改错

3. 下面的 dataclass 想给每个购物车一个独立的空商品列表，但多个购物车却共享了同一个 list。指出根因并修复：

```python
from dataclasses import dataclass

@dataclass
class Cart:
    user_id: int
    items: list[str] = []          # 想要"默认空列表"

c1 = Cart(1)
c2 = Cart(2)
c1.items.append("book")
print(c2.items)                    # 期望 []，实际会怎样？
```

4. 一个同事写了下面的"运行时类型校验"，声称"加了类型注解就安全了"。指出他对类型注解的两处误解，并说明要真正在运行时校验该怎么做：

```python
def create_order(order_id: int, amount: float, currency: str) -> dict:
    # 直接信任注解，不做任何检查
    return {"id": order_id, "amount": amount * 1.0, "currency": currency.upper()}

# 外部 HTTP 请求传进来的（都是字符串！）
create_order("not-a-number", "abc", 123)
```

### 综合实战

5. 你要给一个下单 API 写 DTO。原始需求：请求体含 `order_id`(int)、`items`(字符串列表)、`amount`(float)、`status`(只能是 `"pending"`/`"paid"`/`"shipped"` 之一)、可选的 `note`(字符串或不传)。请：(a) 用 `dataclass` + 合适的类型注解写出这个 DTO（要求 `items` 默认空列表、`note` 可空、整个对象不可变）；(b) 用 `Literal` 约束 `status`；(c) 说明这个 dataclass 在运行时**不会**做哪些校验，若要在 API 边界做真校验你会用什么（点名第 26 章）。

---

### 参考答案

**1.** 不会在调用时因类型报错——解释器对注解视而不见，`scale("hello", "world")` 会正常进入函数体。出错的时机取决于函数体里的运算：执行到 `x * factor`（即 `"hello" * "world"`）时，因为 `str * str` 非法，才抛 `TypeError: can't multiply sequence by non-int of type 'str'`。这是**运算本身**的错，不是注解校验。注解此刻静静躺在 `scale.__annotations__` 这个 dict 里，运行时不起任何约束作用，只供 mypy/pyright/IDE 等静态工具和反射读取。

**2.** (a) Java 擦除后**运行时拿不到**类型参数（`List<String>` 变裸 `List`），Python **不擦除**（`list[int].__args__` 保留 `int`）但运行时**也不据此检查**——一个是"丢了所以查不了"，一个是"留着但故意不查"，结果都是运行时无泛型检查。(b) Lombok 在**编译期**用 annotation processor 往字节码里写方法；dataclass 在**运行时（类创建时）**把方法源码拼成字符串、用 `exec` 编译成真函数挂到类上。

**3.** 根因：`items: list[str] = []` 这个 `[]` 在**类定义时只创建一次**，被所有未显式传 `items` 的实例**共享**（和 [12 函数](12_函数.md) 的"可变默认参数陷阱"同源）。所以 `c1.items.append("book")` 后 `c2.items` 也变成 `["book"]`。而且实际上 dataclass 会直接在定义时抛 `ValueError: mutable default <class 'list'> for field items is not allowed: use default_factory`——它**特意禁止**你这么写。修复用 `field(default_factory=list)`：

```python
from dataclasses import dataclass, field

@dataclass
class Cart:
    user_id: int
    items: list[str] = field(default_factory=list)   # 每个实例调用一次 list()，独立
```

**4.** 两处误解：(1) 以为 `order_id: int` 等会在运行时**阻止**非 int 传入——不会，`create_order("not-a-number", "abc", 123)` 照常执行，直到 `"abc" * 1.0` 抛 `TypeError`（甚至若某些运算碰巧合法还会算出错误结果）；(2) 以为"加了注解就等于做了校验"——注解只是给静态工具的元数据，解释器从不校验。真要在运行时校验外部输入（HTTP 请求、文件、DB 反序列化），用 **pydantic**（[26 日期时间序列化pydantic](26_日期时间序列化pydantic.md)）：定义 `BaseModel`，它会在实例化时按声明的类型**真正校验并转换**，类型不符抛 `ValidationError`。注解负责"静态契约"，pydantic 负责"运行时把关"，两者分工。

**5.**

```python
from __future__ import annotations
from dataclasses import dataclass, field
from typing import Literal

Status = Literal["pending", "paid", "shipped"]

@dataclass(frozen=True)
class OrderDTO:
    order_id: int
    amount: float
    status: Status                                   # 只能是三个字面量之一（静态约束）
    items: list[str] = field(default_factory=list)   # 默认空列表，每实例独立
    note: str | None = None                          # 可空，默认不传

# 用法
dto = OrderDTO(order_id=1, amount=99.9, status="pending", items=["book"])
print(dto)
# 输出: OrderDTO(order_id=1, amount=99.9, status='pending', items=['book'], note=None)
```

(c) 这个 dataclass 在运行时**不会**做的校验：不会检查 `order_id` 真的是 int、`amount` 真的是 float、`status` 真的在三个字面量里（`OrderDTO(order_id="x", amount="y", status="bogus")` 照样构造成功）、`items` 元素真的是 str。`@dataclass` 只生成 `__init__/__repr__/__eq__`，`frozen` 只保证不可变，`Literal` 只对 mypy 有效。要在 API 边界做**真运行时校验**（拒绝非法请求、自动类型转换、给出结构化错误），用 **pydantic**（[26 日期时间序列化pydantic](26_日期时间序列化pydantic.md)）的 `BaseModel`——把 `status` 写成同样的 `Literal`，pydantic 会在实例化时真正拒绝不合法的值并抛 `ValidationError`。FastAPI 等框架正是用 pydantic 模型自动校验请求体。
