# 19 · 描述符与元类 ⭐

> 这是全书最深的一章。前面 [13 章 类与对象](13_类与对象.md) 留了几个"以后再讲"的坑：`@property` 凭什么能拦截属性访问？普通函数凭什么 `obj.method` 就自动绑了 `self`？`__slots__` 到底是什么？"类也是对象、由 `type` 创建"又意味着什么？答案都在两个机制里——**描述符（descriptor）** 和 **元类（metaclass）**。描述符是"把一次属性访问的行为，抽象成一个可复用对象"；元类是"类的类，控制类本身怎么被造出来"。`@property`/方法绑定/`@classmethod`/`@staticmethod`/`__slots__` 全是描述符；SQLAlchemy 的列、Django 的 Model 字段、pydantic 的校验字段全是描述符 + 元类。看懂这章，你不仅能写出 ORM 级别的"魔法"，更能**读懂这些框架的源码**——因为它们用的就是这套统一机制，而不是什么编译器黑魔法。对你这个 Java 老手，这章对照的是 JVM 里最硬核的部分：动态代理、CGLIB/ASM 字节码增强、注解处理器、反射建类。

## ① Java 对照表

| 概念 | Java | Python | 一句话差异 |
|------|------|--------|-----------|
| 属性访问拦截 | 无语言级；靠 `Proxy`/CGLIB 代理拦 getter/setter | **描述符**（`__get__`/`__set__`/`__delete__`） | Python 把"属性访问行为"做成一等对象，放类上即生效 |
| getter/setter | `getX()`/`setX()` 样板 | `@property`（一个数据描述符） | property 让 `obj.x` 走方法但语法不变 |
| 方法本质 | 类里的 method，绑死在类上 | **函数对象 + `__get__`** 动态绑定 self | 方法 = 非数据描述符返回的绑定方法 |
| 拦截任意属性读 | 无；`InvocationHandler.invoke` | `__getattr__`（兜底）/ `__getattribute__`（全拦） | Python 内建两道拦截钩子 |
| 拦截属性写 | setter / 代理 | `__setattr__` | 写也能整体拦截 |
| 类的类 | `Class<?>`（反射设施，特殊） | `type`（普通元类，可继承） | Python 类是 `type` 的实例，真·一等对象 |
| 运行时定义类 | `Proxy.newProxyInstance` / ASM / ByteBuddy | `type(name, bases, ns)` 一行 | Python 内建动态建类，无需字节码库 |
| 编译期改类 | 注解处理器（APT）、Lombok | 元类 `__new__` / 类装饰器（运行时） | Python 在"类创建时"改，不在编译期 |
| 框架字段映射 | JPA `@Column` + 反射读注解 | ORM 字段 = 描述符，模型类 = 元类造 | Python 用对象本身承载行为，不靠注解+反射 |
| 子类钩子 | 无直接对应（靠抽象方法/SPI） | `__init_subclass__` | 父类能在"每个子类定义时"插一脚 |
| 字段命名感知 | 反射 `field.getName()` | `__set_name__`（类创建时自动喂名字） | 描述符自动知道自己叫什么 |

> 一句话建立映射：**描述符 ≈ JVM 的字段访问拦截 + 动态代理；元类 ≈ JVM 运行时用 ASM/CGLIB 生成类 + 编译期注解处理器改类**。区别是 Java 这些都是"外挂"（代理库、字节码库、APT 插件），Python 把它们做成了语言核心的统一机制。

## ② 设计理念（为什么 Python 这么设计）

### 2.1 为什么需要描述符——把"属性访问的行为"抽象成对象

回顾 Java：当你想让一个字段访问带上副作用（校验、计算、懒加载、记日志），你有几条路：

1. 写 `getX()/setX()`，把逻辑塞进去——但每个字段都要重抄一遍，无法复用。
2. 用 JPA/Hibernate，让 `@Column` 注解 + 反射 + CGLIB 代理在运行时拦截字段访问——这是"框架级"魔法，你看不见也复用不了。
3. 用 Lombok `@Getter`，让注解处理器在编译期生成 getter——又是另一套外挂。

这三条路的共同问题：**"字段访问时该干什么"这件事，没有一个一等的、可复用的载体**。你要么手抄，要么靠框架的黑盒。

Python 的答案是**描述符**：把"当有人读/写这个属性时该执行的逻辑"封装成一个**普通对象**，这个对象实现 `__get__`/`__set__`/`__delete__` 三个钩子之一。把它作为**类属性**放上去，之后任何对该属性的访问都会被这个对象接管。关键收益：

- **可复用**：一个 `Typed(int, min_value=0)` 描述符，可以贴到任意类的任意字段上，校验逻辑写一次用无数次。这正是 ORM/pydantic 的 `IntegerField`/`age: int` 背后的东西。
- **可组合、可见**：描述符是普通对象，你能给它传参、继承它、在 REPL 里检查它。不是藏在框架里的黑魔法。
- **统一**：`@property`、方法绑定、`@classmethod`、`@staticmethod`、`__slots__` 全部用同一套描述符协议实现——Python 没有为它们各加一条语法，而是用一个机制把它们统一了。这是极致的"少而正交"。

### 2.2 为什么"类也是对象、由元类创建"——彻底的对象模型

Java 有 `Class<?>`，但它是**反射设施**：一个特殊的、和普通对象不平权的存在。你不能 `extends Class`，不能自定义"类被加载时发生什么"（除非上字节码 agent）。`Class` 是 JVM 给你的只读窗口。

Python 走到了逻辑的尽头：**类本身就是一个普通对象**。既然是对象，它必有类型——它的类型就是**元类（metaclass）**，默认是 `type`。于是：

```
实例 u   →  它的类型是  →  类 User
类 User  →  它的类型是  →  元类 type
type     →  它的类型是  →  type 自己（递归到底）
```

这不是炫技，而是带来**真正的能力**：既然"类是 `type` 的实例"，那么——

- 造实例时 `__init__` 能定制，**造类时**也应该能定制：重写元类的 `__new__`/`__init__` 就能在"类被创建的那一刻"插手（改方法、加属性、注册到表里）。
- `class Foo: ...` 这个语法，本质上等价于一次函数调用 `Foo = type("Foo", bases, namespace)`。语法只是糖。
- 既然类是对象，就能赋值、传参、塞进容器、在运行时动态造（`type(...)`）——这是装饰器、ORM、依赖注入框架的根基。

**用一套机制实现一切，而非加特殊语法**，是贯穿全章的哲学。Java 为每个需求加一个外挂（代理库、ASM、APT、Lombok）；Python 说"类是对象、属性访问可拦截"，剩下的 property/method/ORM 字段/模型类全是这两条公理的推论。

### 2.3 取舍：威力巨大，但 99% 的人不该碰元类

这套机制的代价是**理解门槛高、滥用危害大**。Tim Peters（Python 之禅作者）有句名言：

> "Metaclasses are deeper magic than 99% of users should ever worry about. If you wonder whether you need them, you don't."
> （元类是 99% 的用户永远不该操心的深度魔法。如果你在纠结要不要用，那就是不需要。）

所以 Python 后来补了**更轻的替代品**：`__init_subclass__`（父类钩进子类创建）、`__set_name__`（描述符自动获知字段名）、类装饰器。它们覆盖了过去 80% 要动用元类的场景。本章会教你元类（你得懂它才能读框架源码），但**最佳实践是优先用轻量替代**——这本身就是 Python 的取舍：把最深的魔法留在工具箱底层，日常用浅一档的工具。

## ③ 实现细节（底层怎么实现的）★本章重点★

这是本章的核心，占最大篇幅。我们从"一次 `obj.x` 到底发生了什么"出发，逐层拆开属性查找算法、描述符协议、方法绑定、`__set_name__`、`__getattr__`/`__getattribute__`/`__setattr__`、元类与类创建全过程，最后串回 [13 章](13_类与对象.md) 的 MRO 与 `__slots__`。

### 3.1 描述符协议：三个钩子

**描述符**是一个实现了以下方法中**至少一个**的对象，且这个对象被放在**类**上（不是实例上，这点至关重要，见 §5）：

```python
__get__(self, obj, objtype=None)   # 读 obj.x 时调用，返回值就是 obj.x 的结果
__set__(self, obj, value)          # 写 obj.x = value 时调用
__delete__(self, obj)              # del obj.x 时调用
```

- `self` 是描述符对象本身（它在类上只有一份，被所有实例共享）。
- `obj` 是正在访问属性的**实例**（通过类直接访问时为 `None`）。
- `objtype` 是实例的类。

按实现的钩子分两类，这个分类决定了优先级（§3.2）：

| 类型 | 实现了 | 优先级 | 例子 |
|------|--------|--------|------|
| **数据描述符**（data descriptor） | `__set__` 或 `__delete__`（通常连带 `__get__`） | **高于**实例 `__dict__` | `property`、`__slots__` 槽、ORM 字段 |
| **非数据描述符**（non-data descriptor） | **只有** `__get__` | **低于**实例 `__dict__` | 普通函数/方法、`@classmethod`、`@staticmethod` |

记住这个区别——它是后面一切的基础：**有 `__set__` 就是数据描述符，优先级压过实例属性；只有 `__get__` 就是非数据描述符，会被实例属性遮蔽。**

### 3.2 属性查找完整算法（本章最重要的一张图）

[13 章 §3.2](13_类与对象.md) 给了简化版。现在上完整版。当你写 `obj.x`，CPython 调用 `type(obj).__getattribute__(obj, 'x')`，它的逻辑（`object.__getattribute__` 的默认实现，对应 CPython 源码 `Objects/object.c` 的 `_PyObject_GenericGetAttrWithDict`）是：

```
obj.x  →  type(obj).__getattribute__(obj, 'x')：

  1. 沿 type(obj) 的 MRO 链查找 'x'，记下找到的类属性 descr（可能没有）
  2. 若 descr 是【数据描述符】(有 __set__/__delete__) → 返回 descr.__get__(obj, type(obj))   ★最高优先级
  3. 否则查 obj.__dict__，命中 → 返回 obj.__dict__['x']                                  ★实例属性
  4. 否则若 descr 是【非数据描述符】(只有 __get__) → 返回 descr.__get__(obj, type(obj))   ★方法走这里
  5. 否则若 descr 存在（普通类属性，非描述符） → 返回 descr
  6. 全都没有 → 调用 type(obj).__getattr__(obj, 'x')（若定义了），否则抛 AttributeError
```

一句话优先级链：

```
数据描述符  >  实例 __dict__  >  非数据描述符 / 类属性  >  __getattr__ 兜底
```

用代码逐步验证这个优先级。先看**数据描述符 vs 非数据描述符在面对同名实例属性时谁赢**：

```python
class DataDesc:                       # 数据描述符：有 __set__
    def __get__(self, obj, objtype=None): return "DATA-DESC"
    def __set__(self, obj, value): obj.__dict__['_d'] = value

class NonDataDesc:                    # 非数据描述符：只有 __get__
    def __get__(self, obj, objtype=None): return "NONDATA-DESC"

class T:
    d = DataDesc()                    # 类属性：数据描述符
    nd = NonDataDesc()                # 类属性：非数据描述符

t = T()
t.__dict__['d'] = "instance-d"        # 强行往实例 dict 塞同名键
t.__dict__['nd'] = "instance-nd"

print(t.d)    # 输出: DATA-DESC      数据描述符赢了实例 __dict__（步骤 2 在步骤 3 前）
print(t.nd)   # 输出: instance-nd    实例 __dict__ 赢了非数据描述符（步骤 3 在步骤 4 前）
```

这就是为什么 `@property`（数据描述符）即使实例 `__dict__` 里有同名键也照样拦截，而**普通方法**（非数据描述符）可以被实例属性"覆盖"掉：

```python
class S:
    def greet(self): return "method"
s = S()
print(s.greet())          # 输出: method        非数据描述符 __get__ 返回绑定方法
s.greet = lambda: "shadowed by instance attr"
print(s.greet())          # 输出: shadowed ...   实例属性遮蔽了方法（非数据描述符优先级低）
```

> 对照 Java：这种"实例字段能遮蔽类里的方法"的事在 Java 里根本不可能——字段和方法在不同命名空间，编译期就分开了。Python 把方法、属性、描述符全塞进同一套 `__dict__` + MRO 查找，于是有了这套精细的优先级规则。

### 3.3 函数为什么能当方法——绑定方法的真相

[13 章 §3.4](13_类与对象.md) 给了结论，这里挖到底。**函数对象本身就是一个非数据描述符**——它实现了 `__get__`（在 C 层是 `func_descr_get`）。当你通过实例访问一个函数类属性时，走的是属性查找的步骤 4，调用函数的 `__get__`，它返回一个**绑定方法（bound method）**对象，把实例预先绑到第一个参数 `self` 上。

```python
class Greeter:
    def hello(self, name):
        return f"Hi {name}, from {id(self)}"

g = Greeter()

# 通过类访问：__get__ 的 obj 为 None，返回函数本身（未绑定）
print(type(Greeter.hello))            # 输出: <class 'function'>

# 通过实例访问：触发 __get__，返回绑定方法
print(type(g.hello))                  # 输出: <class 'method'>

# 手动复现这个过程：函数的 __get__ 就是绑定器
bound = Greeter.hello.__get__(g, Greeter)
print(type(bound))                    # 输出: <class 'method'>
print(bound.__self__ is g)            # 输出: True   绑定方法记着它绑的实例
print(bound.__func__ is Greeter.hello)# 输出: True   也记着原始函数
print(bound("Bob"))                   # 输出: Hi Bob, from ...   self 已自动填上
```

所以 `g.hello("Bob")` 的完整链条是：

```
g.hello            → 属性查找步骤 4：Greeter.hello 是非数据描述符
                   → 调用 Greeter.hello.__get__(g, Greeter)
                   → 返回 method 对象（__self__=g, __func__=原函数）
g.hello("Bob")     → 调用 method → 实际执行 Greeter.hello(g, "Bob")
```

**这就是 `self` 必须显式写出来的底层原因**：它是函数实打实的第一个形参，绑定方法只是替你自动把实例填了进去。也正因为方法是"函数 + `__get__`"，它才能被当一等对象到处传（对照 Java 的 `obj::hello` 方法引用），才能被装饰器包裹。

`@classmethod`/`@staticmethod` 同理，也是描述符，只是 `__get__` 行为不同：

```python
class Demo:
    @staticmethod
    def s(): return "static"
    @classmethod
    def c(cls): return f"cls={cls.__name__}"

# staticmethod 的 __get__ 原样返回函数（不绑任何东西）
print(Demo.__dict__['s'].__get__(None, Demo)())   # 输出: static
# classmethod 的 __get__ 把 cls 绑成类
print(Demo.__dict__['c'].__get__(None, Demo)())    # 输出: cls=Demo
```

> 也就是说，`@staticmethod` 只是一个 `__get__` 不绑 `self`、原样吐出函数的描述符；`@classmethod` 是一个 `__get__` 把第一参数绑成类的描述符。Python 没为它们造新语法，全是描述符协议的不同实现。

### 3.4 property 就是一个数据描述符——手写 mini 版

既然 `@property` 是数据描述符，我们可以**完全手写复刻它**，理解它毫无魔法：

```python
class my_property:
    """复刻内置 property 的核心：一个数据描述符。"""
    def __init__(self, fget=None, fset=None):
        self.fget = fget          # 读时调用的函数
        self.fset = fset          # 写时调用的函数
        self.__doc__ = getattr(fget, "__doc__", None)

    def __set_name__(self, owner, name):
        self.name = name          # 记下自己被赋给哪个属性名（见 §3.5）

    def __get__(self, obj, objtype=None):     # 数据描述符的读
        if obj is None:
            return self           # 通过类访问 Circle.area 时返回描述符本身
        if self.fget is None:
            raise AttributeError(f"property '{self.name}' has no getter")
        return self.fget(obj)     # 调用被 @my_property 装饰的那个方法

    def __set__(self, obj, value):            # 有 __set__ → 数据描述符
        if self.fset is None:
            raise AttributeError(f"can't set attribute '{self.name}'")
        self.fset(obj, value)

    def setter(self, fset):       # 支持 @area.setter 链式语法
        return type(self)(self.fget, fset)


class Circle:
    def __init__(self, radius):
        self._radius = radius

    @my_property                  # 等价于 area = my_property(area)
    def area(self):
        return 3.14159 * self._radius ** 2

    @my_property
    def radius(self):
        return self._radius

    @radius.setter                # 等价于 radius = radius.setter(那个新函数)
    def radius(self, value):
        if value <= 0:
            raise ValueError("半径必须为正")
        self._radius = value


c = Circle(2)
print(round(c.area, 5))           # 输出: 12.56636   走 __get__ → fget
c.radius = 5                      # 走 __set__ → fset（校验）
print(round(c.area, 5))           # 输出: 78.53975
try:
    c.area = 100                  # area 没有 setter
except AttributeError as e:
    print("拒绝:", e)             # 输出: 拒绝: can't set attribute 'area'
try:
    c.radius = -1
except ValueError as e:
    print("校验:", e)             # 输出: 校验: 半径必须为正
```

内置 `property` 比这个多了 `deleter`、文档继承等细节，但**内核就是这么个数据描述符**。理解了这点，[13 章 §4.2](13_类与对象.md) 的 `@property` 对你就不再是黑盒。

### 3.5 `__set_name__`：让描述符知道自己叫什么（写可复用字段）

上面的 `my_property` 每个实例只服务一个固定方法，无所谓名字。但**可复用的字段描述符**有个难题：同一个描述符类要贴到很多不同名字的字段上，它得知道"我这次被赋给了哪个属性名"，好用这个名字去实例 `__dict__` 存值。

3.6 之前的老办法是手动传名字：`name = Field("name")`，又啰嗦又容易写错（名字和变量对不上）。Python 3.6 加了 `__set_name__`：**类创建时，解释器会自动遍历类命名空间里每个描述符，调用 `descr.__set_name__(owner, name)`**，把所属类和属性名喂给它。

```python
class Field:
    def __set_name__(self, owner, name):
        # 类定义时自动被调用，name 就是变量名
        self.name = name
        self.private = f"_{name}"       # 真正存值的私有键
        print(f"  __set_name__: owner={owner.__name__}, name={name!r}")

    def __get__(self, obj, objtype=None):
        if obj is None:
            return self
        return getattr(obj, self.private)

    def __set__(self, obj, value):
        setattr(obj, self.private, value)

print("定义 User 类时：")
class User:
    name = Field()                       # __set_name__ 自动收到 name='name'
    email = Field()                      # 自动收到 name='email'
# 输出:
# 定义 User 类时：
#   __set_name__: owner=User, name='name'
#   __set_name__: owner=User, name='email'

u = User()
u.name = "Alice"
print(u.name, u.__dict__)                # 输出: Alice {'_name': 'Alice'}
```

注意值存在 `obj.__dict__['_name']`，而描述符本身（`Field()` 实例）在类上只有一份、被所有 `User` 实例共享——所以描述符**绝不能把每个实例的状态存在 `self` 上**（那会让所有实例串值，§5 详述），必须存进 `obj.__dict__`。

把校验加上，就是**ORM / pydantic v1 字段的真实套路**——带类型与范围校验的可复用字段：

```python
class Typed:
    """带类型 + 最小值校验的可复用字段描述符。"""
    def __init__(self, expected_type, *, min_value=None):
        self.expected_type = expected_type
        self.min_value = min_value

    def __set_name__(self, owner, name):
        self.name = name
        self.private = f"_{name}"

    def __get__(self, obj, objtype=None):
        if obj is None:
            return self
        return getattr(obj, self.private)

    def __set__(self, obj, value):
        if not isinstance(value, self.expected_type):
            raise TypeError(
                f"{self.name} 需要 {self.expected_type.__name__}, "
                f"得到 {type(value).__name__}")
        if self.min_value is not None and value < self.min_value:
            raise ValueError(f"{self.name} 必须 >= {self.min_value}")
        setattr(obj, self.private, value)


class Product:
    name = Typed(str)
    price = Typed(int, min_value=0)      # 价格必须是非负整数（分）

    def __init__(self, name, price):
        self.name = name                 # 走 Typed.__set__ 校验
        self.price = price

p = Product("Book", 3000)
print(p.name, p.price)                   # 输出: Book 3000
try:
    Product("Book", -5)
except ValueError as e:
    print("校验:", e)                    # 输出: 校验: price 必须 >= 0
try:
    Product(123, 3000)
except TypeError as e:
    print("类型:", e)                    # 输出: 类型: name 需要 str, 得到 int
```

> 对照 Java：这正对应 JPA 的 `@Column(nullable=false)` + Bean Validation 的 `@Min(0)`，但 Java 那套是**注解 + 反射 + 框架在运行时解释注解**；Python 这套是**字段本身就是一个会校验的对象**，行为内聚在描述符里，不需要外部框架来"解释"它。这就是 §2.1 说的"把属性访问行为做成一等对象"的威力。

### 3.6 `__getattr__` / `__getattribute__` / `__setattr__`：拦截层

描述符是"针对某个具体属性"的拦截。还有三个钩子是"针对一个类的所有属性"的整体拦截：

| 钩子 | 触发时机 | 类比 Java |
|------|---------|----------|
| `__getattribute__(self, name)` | **每次**属性读都先走它（无条件） | 动态代理 `invoke` 全拦 |
| `__getattr__(self, name)` | 仅当**常规查找失败**时兜底（属性查找步骤 6） | 缺失字段的回退处理 |
| `__setattr__(self, name, value)` | **每次**属性写都走它 | setter 拦截 |

**`__getattribute__` 是总入口**：§3.2 那套完整算法就是 `object.__getattribute__` 的默认实现。你可以重写它来全面接管属性读——但这是核武器，极易翻车（§5 的递归爆栈）。

**`__getattr__` 是兜底**，只在常规查找（描述符 + `__dict__` + 类属性）全失败时才被调用。它是惰性加载、代理转发最常用的工具：

```python
class LazyConfig:
    """惰性加载：第一次访问某个配置项时才去"加载"，之后缓存进 __dict__。"""
    def __init__(self):
        self._loaded = {}

    def __getattr__(self, name):
        # 只有 self.__dict__ 和类里都找不到 name 时，才会进这里
        if name.startswith("_"):
            raise AttributeError(name)        # 内部属性不走懒加载
        print(f"  [懒加载] 计算 {name} ...")
        value = f"value-of-{name}"            # 真实场景：查 DB / 读远程
        setattr(self, name, value)            # 缓存：写进实例 __dict__
        return value

cfg = LazyConfig()
print(cfg.database_url)   # 输出: [懒加载] 计算 database_url ... \n value-of-database_url
print(cfg.database_url)   # 输出: value-of-database_url   第二次直接命中 __dict__，不再懒加载
```

关键：第一次访问 `database_url`，实例 `__dict__` 没有 → 触发 `__getattr__` → 计算并 `setattr` 缓存。第二次访问，`__dict__` 已有 → 属性查找步骤 3 直接命中，**`__getattr__` 根本不被调用**。这种"找不到才兜底"的特性让它天生适合做缓存/代理。

代理转发（对照 Java 动态代理）：

```python
class LoggingProxy:
    """把所有未知属性访问转发给被包装对象，并记日志。"""
    def __init__(self, target):
        object.__setattr__(self, "_target", target)   # 绕过自己的 __setattr__

    def __getattr__(self, name):
        attr = getattr(self._target, name)            # 转发
        if callable(attr):
            def wrapper(*args, **kwargs):
                print(f"  [日志] 调用 {name}({args})")
                return attr(*args, **kwargs)
            return wrapper
        return attr

proxy = LoggingProxy([1, 2, 3])
proxy.append(4)            # 输出: [日志] 调用 append((4,))
print(proxy._target)      # 输出: [1, 2, 3, 4]
```

**`__setattr__` 拦截所有写**——常用于实现不可变对象、写时校验、追踪脏字段（ORM 的 dirty tracking）：

```python
class Frozen:
    """初始化后禁止再改属性（不可变对象）。"""
    def __init__(self, **kwargs):
        for k, v in kwargs.items():
            object.__setattr__(self, k, v)        # 初始化阶段用 object 的写，绕过拦截
        object.__setattr__(self, "_frozen", True)

    def __setattr__(self, name, value):
        if getattr(self, "_frozen", False):
            raise AttributeError(f"{type(self).__name__} 是不可变的，不能改 {name}")
        object.__setattr__(self, name, value)     # 注意：必须委托给 object，否则无限递归

p = Frozen(x=1, y=2)
print(p.x, p.y)           # 输出: 1 2
try:
    p.x = 99
except AttributeError as e:
    print(e)              # 输出: Frozen 是不可变的，不能改 x
```

> 致命陷阱已埋在注释里：`__setattr__` 内部若直接写 `self.name = value` 会再次触发 `__setattr__`，无限递归爆栈。必须委托给 `object.__setattr__(self, name, value)` 或操作 `self.__dict__[name] = value`。同理 `__getattribute__` 内部要用 `super().__getattribute__(name)`。这是 §5 的头号坑。

### 3.7 元类与类创建全过程

现在进入"类也是对象"的深水。

#### 3.7.1 `class` 语句的真相：等价于一次 `type(...)` 调用

`class Foo:` 不是声明，是**可执行语句**。它做了三件事：

```python
# 你写的：
class Dog:
    legs = 4
    def speak(self):
        return "woof"

# 解释器实际做的（语义等价）：
def speak(self):
    return "woof"
namespace = {"legs": 4, "speak": speak}        # ① 执行类体，收集成命名空间 dict
Dog = type("Dog", (), namespace)                # ② 调用 type(name, bases, namespace) 造类
```

亲手用 `type` 三参数形式动态造类，证明 `class` 只是糖：

```python
def speak(self):
    return f"{self.name} says woof"

# 运行时凭空造一个类，等价于 class Dog(Animal): name=...; speak=...
Animal = type("Animal", (), {})
Dog = type("Dog", (Animal,), {"legs": 4, "speak": speak})

d = Dog()
d.name = "Rex"
print(Dog.__name__)          # 输出: Dog
print(Dog.legs)              # 输出: 4
print(d.speak())             # 输出: Rex says woof
print(Dog.__bases__)         # 输出: (<class '__main__.Animal'>,)
print(isinstance(Dog, type)) # 输出: True   类是 type 的实例
```

> 对照 Java：要在运行时凭空造一个类，你得动用 `ByteBuddy`/`ASM`/`Proxy.newProxyInstance`，手工拼字节码或代理。Python 一行 `type(name, bases, ns)` 搞定——因为"造类"在 Python 里就是"调用元类"这个普通操作。

`type(name, bases, namespace)` 的三个参数：
- `name`：类名字符串（`__name__`）。
- `bases`：基类元组（`__bases__`），决定 MRO。
- `namespace`：类体执行后的命名空间 dict，含方法、类属性。

#### 3.7.2 自定义元类：在类创建时插手

既然 `type` 是元类，**继承 `type` 写一个自定义元类**，就能重写"造类"的过程。元类有三个关键钩子：

| 钩子 | 何时调用 | 能干什么 |
|------|---------|---------|
| `__new__(mcs, name, bases, ns)` | 创建类对象**之前** | 改类的构造（增删方法/属性、改 bases、改名） |
| `__init__(cls, name, bases, ns)` | 类对象创建**之后** | 初始化类（注册、校验已成形的类） |
| `__call__(cls, *args)` | **实例化类时**（`Foo()`） | 控制实例创建（单例、缓存、池化） |

完整跑一遍，看清三个钩子的触发顺序：

```python
class Meta(type):
    def __new__(mcs, name, bases, ns, **kw):
        print(f"  Meta.__new__   造类 {name}（类还没生成）")
        # 在这里可以改 ns，比如给每个方法包一层日志
        return super().__new__(mcs, name, bases, ns)

    def __init__(cls, name, bases, ns, **kw):
        print(f"  Meta.__init__  类 {name} 已生成，做注册/校验")
        super().__init__(name, bases, ns)

    def __call__(cls, *args, **kwargs):
        print(f"  Meta.__call__  要造 {cls.__name__} 的实例了")
        return super().__call__(*args, **kwargs)   # 这一步内部会调 cls.__new__ + __init__

print("=== 定义类时 ===")
class Service(metaclass=Meta):
    def __init__(self):
        print("  Service.__init__ 实例初始化")

print("=== 实例化时 ===")
s = Service()
# 输出:
# === 定义类时 ===
#   Meta.__new__   造类 Service（类还没生成）
#   Meta.__init__  类 Service 已生成，做注册/校验
# === 实例化时 ===
#   Meta.__call__  要造 Service 的实例了
#   Service.__init__ 实例初始化
```

注意时间线：**`Meta.__new__`/`__init__` 在 `class` 语句执行时各跑一次（造类）；`Meta.__call__` 在每次 `Service()` 时跑（造实例）**。这条"元类的 `__call__` 控制实例创建"正是单例模式的实现点：

```python
class Singleton(type):
    _instances = {}
    def __call__(cls, *args, **kwargs):
        if cls not in cls._instances:
            cls._instances[cls] = super().__call__(*args, **kwargs)
        return cls._instances[cls]

class DBConnection(metaclass=Singleton):
    def __init__(self, dsn):
        self.dsn = dsn

a = DBConnection("postgres://...")
b = DBConnection("ignored")
print(a is b)            # 输出: True   全局唯一实例
print(b.dsn)            # 输出: postgres://...   第二次 __init__ 根本没跑
```

实战元类——**自动注册子类**（插件系统、序列化器注册表的经典套路）：

```python
class HandlerMeta(type):
    registry = {}
    def __new__(mcs, name, bases, ns):
        cls = super().__new__(mcs, name, bases, ns)
        if bases:                                   # 跳过基类自身
            key = ns.get("event_type", name)
            mcs.registry[key] = cls
            print(f"  注册 handler: {key} -> {name}")
        return cls

class Handler(metaclass=HandlerMeta):
    pass

class OrderCreated(Handler):
    event_type = "order.created"

class OrderPaid(Handler):
    event_type = "order.paid"

# 输出:
#   注册 handler: order.created -> OrderCreated
#   注册 handler: order.paid -> OrderPaid
print(HandlerMeta.registry["order.paid"])   # 输出: <class '__main__.OrderPaid'>
```

> 对照 Java：这相当于在类加载时自动扫描并注册（Spring 的组件扫描 + `@Component`），但 Spring 靠的是**类路径扫描 + 反射 + 注解处理**这套重型框架机制；Python 用元类，在"类被定义的那一刻"就主动登记，无需扫描器。这正是 §2.2 说的"造类时插手"的能力。

#### 3.7.3 元类的继承规则

一个类的元类，要么显式 `metaclass=` 指定，要么**继承自基类的元类**。这是它和普通继承的本质区别（§5 的坑）：

```python
class Base(metaclass=HandlerMeta): pass
class Sub(Base): pass            # 没写 metaclass，但自动用 HandlerMeta
print(type(Base), type(Sub))    # 输出: <class '...HandlerMeta'> <class '...HandlerMeta'>
```

子类的元类必须是父类元类的（非严格）子类，否则报 `TypeError: metaclass conflict`——这也是多继承时如果两个父类有不同元类会冲突的原因。

### 3.8 `__init_subclass__`：元类的轻量替代

§3.7.2 的"自动注册子类"用元类略重。Python 3.6 提供了 `__init_subclass__`——一个定义在**父类**上的钩子，**每当有子类被创建时自动调用**，覆盖了过去 80% 动用元类的场景，且无需理解元类。它是个隐式的类方法：

```python
class Plugin:
    registry = {}

    def __init_subclass__(cls, /, key=None, **kwargs):
        # 每次有人 class X(Plugin) 时，这里自动跑一次，cls 是新子类
        super().__init_subclass__(**kwargs)
        Plugin.registry[key or cls.__name__] = cls
        print(f"  注册插件: {key or cls.__name__}")

class CsvExporter(Plugin, key="csv"):       # 关键字参数会传给 __init_subclass__
    pass

class JsonExporter(Plugin, key="json"):
    pass

# 输出:
#   注册插件: csv
#   注册插件: json
print(Plugin.registry)
# 输出: {'csv': <class '...CsvExporter'>, 'json': <class '...JsonExporter'>}
```

对比一下决策树：

- 只想在**子类创建时**做点事（注册、校验子类是否实现了某方法）→ **`__init_subclass__`**。
- 只想让**描述符知道自己的名字** → **`__set_name__`**。
- 想改类的整体构造、控制实例创建、给一族类统一织入行为，且上面两个不够用 → 才上**元类**。

> Django 的 `Model` 用元类（`ModelBase`），因为它要把字段描述符收集成 `_meta`、生成 manager、对接 ORM，复杂到 `__init_subclass__` 兜不住。但你自己写库时，先问"`__init_subclass__` 够不够"——通常够。

### 3.9 串回 13 章：`__slots__` 也是描述符，MRO 决定描述符查找链

[13 章 §3.7](13_类与对象.md) 讲 `__slots__` 省内存"按偏移存值"。真相是：**`__slots__` 里每个名字，在类上都生成了一个数据描述符**（C 层的 `member_descriptor`），它的 `__get__`/`__set__` 直接读写实例对象里预留的固定槽位（C 结构体偏移），而不是 `__dict__`。

```python
class Point:
    __slots__ = ("x", "y")

# 每个 slot 名字在类 __dict__ 里是一个 member_descriptor
print(type(Point.__dict__["x"]))            # 输出: <class 'member_descriptor'>
print(hasattr(Point.x, "__get__"),
      hasattr(Point.x, "__set__"))          # 输出: True True   是数据描述符！

p = Point()
p.x = 10
print(p.x)                                  # 输出: 10   走 member_descriptor.__set__/__get__
```

这解释了 `__slots__` 的几个特性：因为槽是**数据描述符**（优先级高于实例 `__dict__`），所以即使你给类额外开了 `__dict__`，slot 字段也优先走槽；因为它直接操作固定偏移而非哈希表，所以省内存、略快。`__slots__`、`@property`、ORM 字段——全是数据描述符这一个机制的不同化身。

而 §3.2 的属性查找第 1 步"沿 MRO 链找描述符"，用的正是 [13 章 §3.5](13_类与对象.md) 的 C3 线性化 MRO。**描述符的查找顺序 = MRO 顺序**：父类和子类若有同名描述符，按 MRO 第一个命中的算。描述符 + MRO + `__getattribute__`，三者合起来就是 Python 对象系统属性访问的完整引擎。

### 3.10 对照 JVM：一张机制映射表

| Python 机制 | 最接近的 JVM 对应 | 关键差异 |
|-------------|------------------|---------|
| 描述符 `__get__`/`__set__` | 字段访问拦截（CGLIB `MethodInterceptor`、JPA 懒加载代理） | Python 是语言核心、贴在类上即生效；JVM 要代理库在运行时织入 |
| `@property` | getter/setter（手写或 Lombok） | Python 调用方语法不变（`obj.x`）；Java getter 是 `obj.getX()` |
| `__getattribute__`/`__getattr__` | 动态代理 `InvocationHandler.invoke` | Python 内建、无需生成代理类 |
| `type(name, bases, ns)` 动态建类 | `Proxy.newProxyInstance` / ASM / ByteBuddy | Python 一行调用；JVM 要拼字节码 |
| 元类 `__new__` 织入行为 | 字节码增强（ASM/CGLIB 在加载期改类） | Python 在"类创建时"用纯 Python 改；JVM 在字节码层改 |
| `__init_subclass__` / 自动注册 | Spring 组件扫描 + `@Component` 注册 | Python 类定义时主动登记；Spring 启动时扫描类路径 |
| ORM 字段=描述符、模型=元类造 | JPA `@Column` + 反射 + 增强代理 | Python 行为内聚在字段对象；JPA 靠注解+反射+框架解释 |

## ④ 使用方法与最佳实践

### 4.1 写一个生产级可复用校验描述符

把 §3.5 的思路工程化：用一个抽象基类描述符封装"存取 + `__set_name__`"样板，子类只需实现 `validate`。这正是 ORM 字段库的设计。

```python
from abc import ABC, abstractmethod

class Validated(ABC):
    """可复用字段描述符基类：子类只写 validate。"""
    def __set_name__(self, owner, name):
        self.private = f"_{name}"
        self.name = name

    def __get__(self, obj, objtype=None):
        if obj is None:
            return self
        return getattr(obj, self.private)

    def __set__(self, obj, value):
        self.validate(value)               # 子类定义的校验
        setattr(obj, self.private, value)

    @abstractmethod
    def validate(self, value):
        ...

class String(Validated):
    def __init__(self, *, minlen=0, maxlen=None):
        self.minlen, self.maxlen = minlen, maxlen
    def validate(self, value):
        if not isinstance(value, str):
            raise TypeError(f"{self.name} 必须是 str")
        if len(value) < self.minlen:
            raise ValueError(f"{self.name} 至少 {self.minlen} 个字符")
        if self.maxlen is not None and len(value) > self.maxlen:
            raise ValueError(f"{self.name} 最多 {self.maxlen} 个字符")

class Number(Validated):
    def __init__(self, *, minval=None, maxval=None):
        self.minval, self.maxval = minval, maxval
    def validate(self, value):
        if not isinstance(value, (int, float)):
            raise TypeError(f"{self.name} 必须是数字")
        if self.minval is not None and value < self.minval:
            raise ValueError(f"{self.name} 不能小于 {self.minval}")
        if self.maxval is not None and value > self.maxval:
            raise ValueError(f"{self.name} 不能大于 {self.maxval}")

class User:
    name = String(minlen=1, maxlen=20)
    age = Number(minval=0, maxval=150)

    def __init__(self, name, age):
        self.name = name
        self.age = age

u = User("Alice", 30)
print(u.name, u.age)                       # 输出: Alice 30
for args in [("", 30), ("Bob", 200), ("Bob", "x")]:
    try:
        User(*args)
    except (TypeError, ValueError) as e:
        print("拒绝:", e)
# 输出:
# 拒绝: name 至少 1 个字符
# 拒绝: age 不能大于 150
# 拒绝: age 必须是数字
```

### 4.2 `@property` 做计算属性与校验（何时用它 vs 描述符）

- **单个属性**的校验/计算，且逻辑不复用 → 用 `@property`（最轻，见 [13 章 §4.2](13_类与对象.md)）。
- **同一种校验逻辑要贴到很多字段/很多类** → 用可复用描述符（§4.1）。property 无法跨字段复用（每个 property 绑死一个 getter），描述符可以。

经验法则：**先 `@property`，发现自己在复制粘贴 property 时，重构成描述符。**

### 4.3 `__getattr__` 做惰性加载与代理

- **惰性加载**：昂贵资源（DB 连接、远程配置、大模型权重）首次访问才初始化，配合 `setattr` 缓存（§3.6）。
- **代理/适配**：包装第三方对象，转发未知属性并加横切逻辑（日志、限流、鉴权）——对照 Java 动态代理，但无需生成代理类。
- **`__getattr__` 只兜底未命中的属性**，已存在的属性不受影响，性能开销只在 miss 时产生，适合做缓存层。

### 4.4 何时用元类（极少）——决策清单

按"代价从低到高"排，能用前面的就别用后面的：

1. 给描述符喂名字 → `__set_name__`。
2. 子类创建时注册/校验/注入 → `__init_subclass__`。
3. 给类做一次性变换（加方法、改属性、包装）→ **类装饰器**（比元类轻，[17 章 装饰器](17_装饰器.md)）。
4. 控制实例创建（单例、缓存）→ 元类 `__call__`，或更简单地用 `__new__`。
5. 以上都兜不住，要给"一整族类"统一织入复杂行为（ORM 模型基类、枚举、ABC）→ 才上**元类**。

记住 Tim Peters 那句"如果你在纠结要不要用元类，你就是不需要"。

### 4.5 读懂框架：ORM 与 pydantic 怎么用这两套机制

这是本章最实用的回报——理解原理后，框架源码不再是天书。

**Django Model**：
```python
class User(models.Model):           # Model 的元类是 ModelBase
    name = models.CharField(max_length=20)   # CharField 是描述符式字段
    age = models.IntegerField()
```
- `models.CharField(...)` 是**字段对象**（描述符思路），承载类型、约束、列映射。
- `Model` 的元类 `ModelBase`（`django/db/models/base.py`）在**类创建时**遍历类体，把所有 `Field` 收集进 `Model._meta.fields`、对接数据库列、生成 manager（`objects`）。
- 你写 `user.name = "x"` 触发字段的赋值逻辑，`User.objects.filter(...)` 用的是元类装好的 manager。

**SQLAlchemy**：
```python
class User(Base):                   # Base 由 declarative_base() / DeclarativeBase 提供（含元类）
    __tablename__ = "users"
    id = Column(Integer, primary_key=True)    # Column 是 InstrumentedAttribute 描述符
    name = Column(String)
```
- `Column(...)` 经元类处理后变成 **`InstrumentedAttribute` 描述符**，`user.name` 的读写被它拦截：读时从实例状态取值，写时记入"脏字段"以便 flush 时生成 UPDATE（dirty tracking，正是 §3.6 `__setattr__` 那类拦截）。
- 类级访问 `User.name == "Alice"` 返回的是 SQL 表达式对象（描述符 `__get__` 在 `obj is None` 时返回特殊对象），这就是 `filter(User.name == "Alice")` 能拼 SQL 的原因。

**pydantic**：
- pydantic v1：字段基于描述符 + 元类 `ModelMetaclass` 收集字段、生成校验器。
- pydantic v2：核心校验逻辑下沉到 Rust 写的 `pydantic-core`，但**模型类仍由元类构造**——元类在类创建时读取类型注解（`__annotations__`）、把每个字段编译成校验 schema。你写 `age: int = Field(ge=0)`，元类负责把它变成运行时校验。

详见 [25 章 数据库与ORM](25_数据库与ORM.md) 和 [26 章 日期时间序列化pydantic](26_日期时间序列化pydantic.md)。**共同套路一句话总结：字段是描述符（承载单个属性的行为），模型类由元类构造（在类创建时收集字段、织入框架能力）。** 你现在能看懂它们了。

## ⑤ ⚠️ 易错点（Java 思维翻车）

| Java 习惯 | 翻车现场 | 正确做法/认知 |
|----------|---------|--------------|
| 在构造器里给字段赋值 | **描述符放到了实例上**：`self.field = Typed(int)` 写在 `__init__` 里，描述符协议**完全不触发**（只有类属性才走描述符），变成普通实例属性 | 描述符**必须定义为类属性**：`class C: field = Typed(int)`。实例上的描述符是死的 |
| 字段值就近存在对象自己身上 | **描述符把实例状态存进 `self`**：`self.value = value`（self 是描述符，类上仅一份），导致所有实例**共享同一个值**、互相串改 | 状态存进**被访问的实例** `obj.__dict__`：`obj.__dict__[self.name] = value` 或 `setattr(obj, self.private, value)` |
| 以为有 `__get__` 就万能 | **数据 vs 非数据描述符搞反**：以为自定义描述符能像 property 一样压过实例属性，结果只写了 `__get__`（非数据描述符），被实例 `__dict__` 遮蔽，校验形同虚设 | 要拦截写、要优先于实例属性，**必须实现 `__set__`**（哪怕只是存值），成为数据描述符 |
| 重写方法拦截一切 | **`__getattribute__`/`__setattr__` 里递归爆栈**：内部写 `self.x` 或 `getattr(self, ...)` 再次触发自己，`RecursionError` | 内部一律走父类：`super().__getattribute__(name)`、`object.__setattr__(self, name, value)`、`self.__dict__[name]=v` |
| 反射/AOP 上来就用最重的工具 | **滥用元类**：能用 `__init_subclass__`/类装饰器/`@property` 的场景硬上元类，代码晦涩、多继承时元类冲突 | "元类是 99% 用户不需要的魔法"。按 §4.4 决策清单从轻到重选 |
| 把元类当普通父类继承 | **元类当基类**：`class Foo(MyMeta)` 想"继承元类的能力"，其实 `MyMeta` 应作为 `metaclass=` 传入，写成基类后 `Foo` 变成了元类本身 | 元类用 `class Foo(metaclass=MyMeta)` 指定，不是 `class Foo(MyMeta)`；区分"是它的实例"和"继承它" |
| `__getattr__` 当成每次都拦 | **以为 `__getattr__` 拦截所有读**：在里面写缓存逻辑却发现已存在的属性不进来；或把它当 `__getattribute__` 用 | `__getattr__` **只在常规查找失败时兜底**；要拦截每次读才用 `__getattribute__`（慎用） |
| `__set_name__` 手动传名字 | 老代码 `Field("name")` 手传名字，和变量名写不一致导致存错 key | 用 `__set_name__` 让解释器在类创建时自动喂名字，杜绝不一致（3.6+） |

补充演示——**最隐蔽的"描述符状态存在 self 上"串值坑**：

```python
class Broken:                      # 错误：描述符把值存在自己身上
    def __set_name__(self, owner, name):
        self.name = name
    def __get__(self, obj, objtype=None):
        if obj is None: return self
        return self.value          # 取的是描述符自己的 value，全实例共享！
    def __set__(self, obj, value):
        self.value = value         # 存在描述符上 → 所有实例共用一份

class Account:
    balance = Broken()

a, b = Account(), Account()
a.balance = 100
b.balance = 999
print(a.balance)                   # 输出: 999   a 的余额被 b 改掉了！串值
```

修复——值存进**被访问的实例**，描述符在类上只当"行为载体"：

```python
class Fixed:
    def __set_name__(self, owner, name):
        self.private = f"_{name}"
    def __get__(self, obj, objtype=None):
        if obj is None: return self
        return getattr(obj, self.private)
    def __set__(self, obj, value):
        setattr(obj, self.private, value)   # 存进 obj，不是 self

class Account:
    balance = Fixed()

a, b = Account(), Account()
a.balance = 100
b.balance = 999
print(a.balance, b.balance)        # 输出: 100 999   各自独立
```

> 这个坑 Java 老手特别容易栽：在 Java 里"字段值存在对象里"是天经地义，但描述符对象在类上只有**一份**、被所有实例共享，把状态存在 `self`（描述符）上就等于存成了"静态字段"。务必存进 `obj`。

## ⑥ 练习题

### 基础理解

**1.** 用自己的话解释属性查找 `obj.x` 的完整优先级链（四个层级），并说明：为什么 `@property` 能在实例 `__dict__` 有同名键时仍然拦截，而普通方法却能被实例属性"覆盖"？这背后是数据描述符和非数据描述符的什么区别？

**2.** `class Foo: pass` 这条语句执行时，等价于调用哪个内置函数、传哪三个参数？据此解释"类也是对象、是 `type` 的实例"是什么意思，并说明元类的 `__new__`/`__init__` 和 `__call__` 分别在什么时刻被调用。

### 找坑改错

**3.**（描述符状态串值）下面这段想用描述符给 `Sensor` 的 `value` 字段加"非负"校验，但两个传感器的读数会互相串改。指出根因，改对，并说明 Java 老手为什么容易栽在这：

```python
class NonNegative:
    def __get__(self, obj, objtype=None):
        return self.val
    def __set__(self, obj, value):
        if value < 0:
            raise ValueError("不能为负")
        self.val = value

class Sensor:
    value = NonNegative()

s1, s2 = Sensor(), Sensor()
s1.value = 10
s2.value = 20
print(s1.value)        # 期望 10，实际是什么？为什么？
```

**4.**（非数据描述符 + 递归）下面有两处坑：一是想做校验却把描述符写成了非数据描述符导致校验被绕过，二是有人试图加 `__setattr__` 做日志却写法错误。指出两个问题并改对：

```python
class Positive:
    def __get__(self, obj, objtype=None):
        return obj.__dict__["_p"]
    # 只有 __get__，没有 __set__

class Order:
    price = Positive()
    def __init__(self, price):
        self.__dict__["_p"] = price
    def __setattr__(self, name, value):
        print(f"set {name}={value}")
        self.name = value          # 想存值
```

### 综合实战

**5.**（描述符 + `__init_subclass__` 做一个 Mini ORM / 校验模型）实现一个轻量的 `Model` 基类，满足：
- 字段用描述符声明，支持类型校验，例如 `name = Field(str)`、`age = Field(int, min_value=0)`；
- 字段用 `__set_name__` 自动获知名字，值存进实例（不串值）；
- `Model` 用 `__init_subclass__`（不要用元类）自动收集每个子类声明的字段，存进 `cls._fields`；
- `Model.__init__(**kwargs)` 按字段赋值并触发校验，缺字段报错；
- 提供 `to_dict()` 返回字段名到值的字典；
- 写一个 `User(Model)` 子类演示：正常创建、类型错误、范围错误、缺字段，并打印 `User._fields`。

说明这个设计对照 Django/SQLAlchemy 是怎么用同样思路（字段=描述符、模型类构造时收集字段）的，以及你用 `__init_subclass__` 而非元类的理由。

---

### 参考答案

**1.** 优先级链：**数据描述符（类 MRO 上、有 `__set__`/`__delete__`）> 实例 `__dict__` > 非数据描述符 / 普通类属性（只有 `__get__` 或无描述符）> `__getattr__` 兜底**。`obj.x` 实际调用 `type(obj).__getattribute__`，它先沿 MRO 找到类属性 descr，若是数据描述符立即用它（压过实例属性），否则查实例 `__dict__`，再否则用非数据描述符/类属性，全失败才调 `__getattr__`。`@property` 实现了 `__set__`（即便只读 property 也有 `__set__` 抛 `AttributeError`），是**数据描述符**，所以排在实例 `__dict__` 前面，必然拦截；普通函数只有 `__get__`，是**非数据描述符**，排在实例 `__dict__` **后面**，所以 `s.greet = something` 在实例 dict 建键后就遮蔽了方法。核心区别就是"有没有 `__set__`/`__delete__`"决定了它在实例属性之前还是之后。

**2.** 等价于 `Foo = type("Foo", (), {})`，三个参数是：类名字符串、基类元组、类体执行后的命名空间 dict。所以"造一个类"就是"调用元类 `type`"，类是 `type` 的实例——`type(Foo)` 返回 `type`，`isinstance(Foo, type)` 为 `True`，类因此也是可赋值、可传参、可动态创建的一等对象。元类钩子时刻：`metaclass.__new__` 和 `__init__` 在 **`class` 语句执行（造类）时**各调一次，`__new__` 在类对象生成前（可改构造）、`__init__` 在类对象生成后（可注册/校验）；`metaclass.__call__` 在**每次实例化 `Foo()` 时**调用，内部再去触发 `Foo.__new__` + `Foo.__init__`，所以它是控制实例创建（单例/缓存）的钩子。

**3.** 根因：值存在了**描述符自己**（`self.val`）身上。描述符 `NonNegative()` 是 `Sensor` 的类属性，**在类上只有一份、被所有实例共享**；`s1.value = 10` 和 `s2.value = 20` 都在改同一个 `self.val`，后写的覆盖前者，故 `s1.value` 实际是 `20`。Java 老手栽在这是因为 Java 里"值存在对象字段里"是本能，而描述符不是普通对象——它在类级共享，存 `self` 上等于存成了静态字段。修复：把值存进**被访问的实例** `obj`：

```python
class NonNegative:
    def __set_name__(self, owner, name):
        self.private = f"_{name}"
    def __get__(self, obj, objtype=None):
        if obj is None: return self
        return getattr(obj, self.private)
    def __set__(self, obj, value):
        if value < 0:
            raise ValueError("不能为负")
        setattr(obj, self.private, value)   # 存进 obj，不是 self

class Sensor:
    value = NonNegative()

s1, s2 = Sensor(), Sensor()
s1.value = 10
s2.value = 20
print(s1.value, s2.value)        # 输出: 10 20   各自独立
```

**4.** 两个问题：
- **问题一**：`Positive` 只有 `__get__`，是**非数据描述符**，优先级**低于**实例 `__dict__`。而 `__init__` 里 `self.__dict__["_p"] = price` 直接写实例 dict，根本没经过任何校验入口；即便有人 `order.price = -5`，因为 `price` 是非数据描述符，赋值会直接在实例 `__dict__` 建 `price` 键遮蔽描述符，校验完全被绕过。要校验**必须加 `__set__`** 成为数据描述符。
- **问题二**：`__setattr__` 里 `self.name = value` 会再次触发 `__setattr__`（写属性就走它），无限递归 `RecursionError`；而且逻辑也错（应存到 `name` 指定的键，不是字面量 `name`）。应委托给 `object.__setattr__` 或写 `self.__dict__`。

修复：

```python
class Positive:
    def __set_name__(self, owner, name):
        self.private = f"_{name}"
    def __get__(self, obj, objtype=None):
        if obj is None: return self
        return getattr(obj, self.private)
    def __set__(self, obj, value):                 # 加上 __set__ → 数据描述符
        if value <= 0:
            raise ValueError(f"{self.private[1:]} 必须为正")
        setattr(obj, self.private, value)

class Order:
    price = Positive()
    def __init__(self, price):
        self.price = price                          # 走 Positive.__set__ 校验

o = Order(100)
print(o.price)                                      # 输出: 100
try:
    Order(-5)
except ValueError as e:
    print(e)                                        # 输出: price 必须为正
```

（若确实需要 `__setattr__` 记日志，正确写法是 `print(...)` 后 `object.__setattr__(self, name, value)`，但本例用数据描述符校验已足够，无需再加 `__setattr__`。）

**5.**

```python
class Field:
    """带类型校验的字段描述符。"""
    def __init__(self, field_type, *, min_value=None, required=True):
        self.field_type = field_type
        self.min_value = min_value
        self.required = required

    def __set_name__(self, owner, name):
        self.name = name
        self.private = f"_{name}"

    def __get__(self, obj, objtype=None):
        if obj is None:
            return self
        return getattr(obj, self.private)

    def __set__(self, obj, value):
        if not isinstance(value, self.field_type):
            raise TypeError(
                f"{self.name} 需要 {self.field_type.__name__}, "
                f"得到 {type(value).__name__}")
        if self.min_value is not None and value < self.min_value:
            raise ValueError(f"{self.name} 必须 >= {self.min_value}")
        setattr(obj, self.private, value)


class Model:
    def __init_subclass__(cls, **kwargs):
        super().__init_subclass__(**kwargs)
        # 子类创建时自动收集本类声明的所有 Field
        cls._fields = {
            name: attr
            for name, attr in vars(cls).items()
            if isinstance(attr, Field)
        }

    def __init__(self, **kwargs):
        for name, field in self._fields.items():
            if name in kwargs:
                setattr(self, name, kwargs[name])     # 触发 Field.__set__ 校验
            elif field.required:
                raise TypeError(f"缺少必填字段: {name}")

    def to_dict(self):
        return {name: getattr(self, name) for name in self._fields}

    def __repr__(self):
        body = ", ".join(f"{k}={v!r}" for k, v in self.to_dict().items())
        return f"{type(self).__name__}({body})"


class User(Model):
    name = Field(str)
    age = Field(int, min_value=0)


print(list(User._fields))                # 输出: ['name', 'age']

u = User(name="Alice", age=30)
print(u)                                 # 输出: User(name='Alice', age=30)
print(u.to_dict())                       # 输出: {'name': 'Alice', 'age': 30}

for kw in [{"name": 123, "age": 30},     # 类型错
           {"name": "Bob", "age": -1},   # 范围错
           {"name": "Bob"}]:             # 缺字段
    try:
        User(**kw)
    except (TypeError, ValueError) as e:
        print("拒绝:", e)
# 输出:
# 拒绝: name 需要 str, 得到 int
# 拒绝: age 必须 >= 0
# 拒绝: 缺少必填字段: age
```

设计说明：**字段 `Field` 是描述符**，承载单个属性的类型/范围校验和存取——对应 Django 的 `CharField`/`IntegerField`、SQLAlchemy 的 `Column`、pydantic 的字段。`__set_name__` 让字段自动获知名字（杜绝手传名字写错）。**`Model` 在子类创建时用 `__init_subclass__` 收集字段**进 `cls._fields`——这正是 Django `ModelBase` 元类、SQLAlchemy declarative 元类做的"遍历类体收集字段"的事，只是它们用元类（因为还要织入 manager、表映射等复杂能力），而我们的需求只是"收集字段"，**用 `__init_subclass__` 更轻、更易懂，无需理解元类**，契合 §4.4 的决策清单。值存进 `obj.__dict__` 而非描述符自身，避免了 §5 的串值坑。理解了这个 50 行的 Mini 版，你再去读 Django/SQLAlchemy/pydantic 的字段与模型源码，会发现它们用的是同一套"描述符 + 类创建时收集"的心智模型，只是工程化程度更高。
