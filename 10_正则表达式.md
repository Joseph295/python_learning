# 10 · 正则表达式 re

> 你在 Java 里用过 `Pattern`/`Matcher`，正则的语法你早就会了——这章不教你正则本身，而是教你 **Python 的 `re` 模块和 Java 的 `java.util.regex` 哪里形似神不同**：`(?P<name>)` 这种 Python 独有的命名组语法、为什么模式字符串前面要加 `r`、`match` 和 `search` 的致命区别、`findall` 在有分组时会偷偷换返回类型，以及两边共有的隐雷——**灾难性回溯（ReDoS）**。看懂底层那套回溯式 NFA 字节码引擎，你就知道什么正则会让 CPU 飙到 100%。

---

## ① Java 对照表

| 维度 | Java `java.util.regex` | Python `re` | 关键差异 |
|------|------|------|------|
| 编译模式 | `Pattern.compile(regex)` | `re.compile(pattern)` | 都返回可复用的编译对象 |
| 从头匹配 | `matcher.lookingAt()` | `re.match(p, s)` / `p.match(s)` | 都**锚定开头**、不要求匹配到尾 |
| 全串匹配 | `matcher.matches()` | `re.fullmatch(p, s)` | 要求整串匹配 |
| 任意位置查找 | `matcher.find()` | `re.search(p, s)` | 扫描全串找第一个 |
| 找全部 | `while (m.find()) {...}` | `re.findall` / `re.finditer` | findall 返回 list，finditer 返回迭代器 |
| 替换全部 | `m.replaceAll(repl)` | `re.sub(p, repl, s)` | Python 的 repl 可以是**函数** |
| 替换并计数 | 无直接对应 | `re.subn` 返回 `(新串, 次数)` | |
| 分割 | `Pattern.split(s)` | `re.split(p, s)` | |
| 命名组定义 | `(?<name>...)` | **`(?P<name>...)`** | **Python 多一个 `P`** |
| 命名组引用 | `\k<name>` / `${name}` | `(?P=name)` 引用、`\g<name>` 替换 | 语法完全不同 |
| 取分组 | `m.group(1)` / `m.group("name")` | `m.group(1)` / `m.group("name")` | 这里一致 |
| 所有命名组 | 无内置（需手动） | `m.groupdict()` → dict | Python 更方便 |
| 忽略大小写 | `Pattern.CASE_INSENSITIVE` | `re.IGNORECASE` / `re.I` | |
| 可读多行模式 | `Pattern.COMMENTS` | `re.VERBOSE` / `re.X` | 允许写空白和注释 |
| 点号匹配换行 | `Pattern.DOTALL` | `re.DOTALL` / `re.S` | 默认都**不**匹配 `\n` |
| 多行锚点 | `Pattern.MULTILINE` | `re.MULTILINE` / `re.M` | `^`/`$` 匹配每行 |
| 字面量转义 | `Pattern.quote(s)` | `re.escape(s)` | |
| 引擎类型 | 回溯式 NFA | **回溯式 NFA** | 同源同病（都会 ReDoS） |

> 一句话记忆点：**Java 是 `(?<name>)`，Python 是 `(?P<name>)`，多一个 `P`**（P 取自 Python，这是 1997 年 Python 给 PCRE 提的扩展，后来反而成了它自己的标志）。还有 **`re.match` 不是"匹配"，是"从开头匹配"**，对应的是 Java 的 `lookingAt()` 而不是 `matches()`——这是 Java 老手最常踩的第一个坑。

## ② 设计理念（为什么 Python 这么设计）

### 2.1 正则是"库"，不是"语言特性"

在 Java 里正则也是标准库（`java.util.regex`），这点和 Python 一致——两者都没把正则做成语言级语法（不像 Perl/JavaScript 有 `/pattern/` 字面量）。但 Python 的克制更彻底：**正则在 Python 里只是 `re` 这一个普通模块**，没有任何语法糖，你必须 `import re`，模式永远是一个**普通字符串**。

这背后是 Python 的核心哲学（见 [01 设计哲学与心智模型](01_设计哲学与心智模型.md)）："There should be one obvious way to do it" 以及对"显式优于隐式"的偏好。把正则做成字符串而非字面量语法，代价是你得手动加 `r"..."`、得 `re.compile`，但好处是：正则没有任何特殊的语言地位，它就是"一个接收字符串参数的函数库"，新手不会被 `/.../g` 这种神秘符号吓到，也不会滥用正则——这与下面 2.4 的"能用 str 方法就别上正则"一脉相承。

### 2.2 回溯式引擎的定位：表达力优先，最坏性能其次

正则引擎有两大流派：

- **回溯式 NFA**（Backtracking）：Java、Python、Perl、PCRE、.NET、JavaScript 都是这派。支持反向引用、前后查找（lookaround）等强力特性，**但最坏情况会指数级爆炸**（ReDoS）。
- **DFA / 自动机式**：Go 的 `regexp`（RE2 引擎）、`grep` 是这派。**保证线性时间**、绝不爆炸，但**不支持反向引用和 lookaround**。

Python 选了回溯式，和 Java 站在同一阵营。这是一个**明确的取舍**：要的是表达力（你能写 `(?P<y>\d+)-\1` 这种反向引用、能写 lookbehind），放弃的是最坏情况的性能保证。Guido 时代的 Python 把正则当成"脚本里方便处理文本的工具"，表达力比抗 DoS 重要。后果就是：**和 Java 一样，Python 的正则能被一行恶意输入打到 CPU 100%**——这不是 bug，是引擎流派的固有代价（详见 ③）。

> 对照记忆：如果你来自需要扛住不可信输入的场景（比如用正则校验用户上传的内容），要知道 **Python/Java 的 `re` 不是为对抗恶意输入设计的**。Go 的 RE2 才是。这是选型时的硬知识。

### 2.3 raw string 与正则：为什么模式要写 `r"..."`

这是 Python 正则**最该先理解**的设计点，Java 里没有完全对应的东西。

正则语言里大量用反斜杠：`\d`(数字)、`\b`(词边界)、`\w`(单词字符)。问题是**反斜杠在 Python 普通字符串里也是转义字符**。于是写一个"匹配数字"的正则会经历**两层转义**：

```
你想要正则引擎看到的：   \d
普通字符串要这样写：     "\\d"   ← Python 先把 \\ 解析成一个 \，引擎才收到 \d
raw string 直接写：      r"\d"   ← r 前缀关闭 Python 的转义，引擎直接收到 \d
```

Java 也有同样的"双层转义"问题——在 Java 里匹配数字得写 `"\\d"`（Java 11 之前没有原始字符串）。Python 用 `r"..."`（raw string，见 [09 字符串、字节与编码](09_字符串字节与编码.md)）**关闭语言层的转义**，让你写的正则和你脑子里想的正则**字字对应**。

所以铁律是：**写正则一律用 `r"..."`**。这不是"建议"，是肌肉记忆——下面 ③ 会从字节码层面解释为什么 `r"\d"` 严格优于 `"\\d"`。

### 2.4 "能用 str 方法解决就别上正则"

Python 的字符串自带一大批方法：`str.startswith`、`str.endswith`、`str.split`、`str.replace`、`str.find`、`in` 运算符（见 [09 字符串、字节与编码](09_字符串字节与编码.md)）。这些是 **C 层实现的、线性时间、绝不回溯爆炸**的操作。

Python 社区有句俗话："Some people, when confronted with a problem, think 'I know, I'll use regular expressions.' Now they have two problems." 设计上的暗示很清楚：**正则是重武器，不是默认选择**。判断固定前缀用 `s.startswith("ERROR")` 比 `re.match(r"ERROR", s)` 更快、更清楚、还不用 `import re`。把正则留给"真的需要模式匹配"的场合。这条理念会贯穿 ④ 和 ⑤。

## ③ 实现细节（底层怎么实现的）★本章重点★

这一节是和普通教程拉开差距的地方。要点：`re` 是**回溯式 NFA 引擎**，模式被**编译成一套自定义字节码**，由 C 写的 `_sre` 引擎逐条解释执行——这套机制和 [05 CPython 内部](05_CPython内部.md) 讲的"源码→字节码→求值循环"是同构的。

### 3.1 从模式到字节码：re 内部有一套自己的编译器

你在 [05 CPython 内部](05_CPython内部.md) 学过：Python 源码会被编译成字节码，由 `ceval` 求值循环执行。**正则也是一模一样的套路**——一个正则模式不是被"逐字符比对"的，而是先**编译成一段正则字节码（SRE bytecode）**，再交给 C 引擎执行。

`re` 模块内部分三层（Python 3.11+ 的模块名）：

- `re._parser`（旧名 `sre_parse`）：把模式字符串解析成**语法树**（类似 [05](05_CPython内部.md) 的 AST 阶段）。
- `re._compiler`（旧名 `sre_compile`）：把语法树编译成 **SRE 字节码**（一个 int 数组）。
- `_sre`：**C 写的虚拟机**，逐条执行那段字节码（类似 `ceval` 求值循环）。

用 `re.DEBUG` 标志可以**亲眼看到这两层产物**——这就是正则版的 `dis`：

```python
import re

re.compile(r'(?P<area>\d{3})-(\d{4})', re.DEBUG)
```

输出（上半是语法树，下半是编译出的字节码）：

```
SUBPATTERN 1 0 0          # 第1个分组（命名组 area），语法树
  MAX_REPEAT 3 3          #   重复 3 到 3 次
    IN
      CATEGORY CATEGORY_DIGIT   # 字符类：数字
LITERAL 45                # 字面量 '-'（ASCII 45）
SUBPATTERN 2 0 0          # 第2个分组
  MAX_REPEAT 4 4
    IN
      CATEGORY CATEGORY_DIGIT

 0. INFO 4 0b0 8 8 (to 5)        # ↓↓↓ 下面是编译出的 SRE 字节码 ↓↓↓
 5: MARK 0                       # 标记分组1的起点
 7. REPEAT_ONE 9 3 3 (to 17)     # 重复匹配（带回溯信息）
11.   IN 4 (to 16)
13.     CATEGORY UNI_DIGIT
16:   SUCCESS
17: MARK 1                       # 标记分组1的终点
19. LITERAL 0x2d ('-')           # 匹配 '-'
21. MARK 2
23. REPEAT_ONE 9 4 4 (to 33)
...
35. SUCCESS
```

`MARK 0`/`MARK 1` 就是记录第 1 组捕获的起止位置（命名组 `area` 在引擎眼里就是"第 1 组"，名字只是个映射表里的别名）。`REPEAT_ONE`、`LITERAL`、`CATEGORY` 就是这套虚拟机的"指令集"。**这和 [05](05_CPython内部.md) 里 `LOAD_FAST`/`BINARY_OP` 是同一种思想**：高层表达式 → 编译成字节码 → 由一个 C 解释循环跑。理解了这点，正则就不再神秘，它就是一台小型字节码虚拟机。

### 3.2 回溯式 NFA：表达力的来源，也是爆炸的来源

`_sre` 引擎是**回溯式**的：遇到量词（`*`/`+`/`{m,n}`）和分支（`|`）时，它会"试一条路，走不通就退回来（backtrack）试另一条"。

以 `a*ab` 匹配 `"aaab"` 为例：`a*` 是贪婪的，先吞掉全部 `aaa`，然后发现后面要的 `a` 没了 → **回溯**，吐出一个 `a`，再试 → 匹配成功。这种"试错-回退"就是回溯。

回溯带来了 DFA 引擎给不了的能力：

- **反向引用** `\1` / `(?P=name)`：匹配"和前面某组捕获到的相同内容"（如 `(\w+)\s+\1` 找重复词），数学上需要回溯，DFA 做不到。
- **lookaround**（前后查找）：`(?=...)`、`(?!...)`、`(?<=...)`、`(?<!...)`。

代价就是下面的灾难性回溯。Java 的引擎同理——**这俩是同病相怜的兄弟**。

### 3.3 灾难性回溯（Catastrophic Backtracking / ReDoS）

这是本章最该刻进脑子的底层知识。**嵌套量词**（一个量词套在另一个量词里，如 `(a+)+`、`(a*)*`、`(\d+)*`）在遇到"几乎匹配但最终失败"的输入时，回溯次数会**随输入长度指数级爆炸**。

看这个经典的"易爆"正则，让读者建立直觉：

```python
import re, time

evil = re.compile(r'(a+)+$')     # 嵌套量词：a+ 再被 + 套一层

for n in range(20, 30):
    s = 'a' * n + 'b'            # n 个 a 后跟一个 b → 永远匹配不上 $
    start = time.perf_counter()
    evil.search(s)
    elapsed = time.perf_counter() - start
    print(f'n={n}  耗时={elapsed:.3f}s')

# 典型输出（机器不同数值不同，但每 +1 大致翻倍）：
# n=20  耗时=0.008s
# n=21  耗时=0.015s
# n=22  耗时=0.031s
# n=23  耗时=0.062s
# n=24  耗时=0.125s
# n=25  耗时=0.255s   ← 每多一个字符，时间翻倍
# n=26  耗时=0.510s
# n=27  耗时=1.030s
# ...   n=35 就要几分钟，n=40 跑到天荒地老
```

**为什么爆炸？** 对于 `(a+)+`，"n 个 a"可以被切分成内层 `a+` 的各种组合（`aaa` = `a|a|a` 或 `aa|a` 或 `a|aa` 或 `aaa`……），切分方式有 **2^(n-1)** 种。末尾的 `b` 让 `$` 永远失败，引擎被迫**穷举所有切分方式**才能确认"匹配不上"，于是 O(2^n) 爆炸。

这就是 **ReDoS（Regular expression Denial of Service）**：攻击者只要构造一个几十字节的字符串，就能让你的服务器某个线程 CPU 跑满、卡死。**Java 完全一样**——`Pattern.compile("(a+)+$").matcher(evilInput)` 同样会爆。

**怎么识别危险模式**（这是要带走的能力）：

1. **嵌套量词**：`(x+)+`、`(x*)*`、`(x+)*`、`(.*)*`。
2. **量词作用于有重叠的分支**：`(a|a)*`、`(a|ab)*`。
3. 上面这些 + **后面跟一个常常失败的锚点或字面量**（`$`、`!` 等），触发"几乎匹配但失败"。

**怎么修**：

- 去掉嵌套：`(a+)+` 想表达的往往就是 `a+`，直接写 `a+`。
- 把"重叠的可选"写成不重叠：用更精确的字符类。
- 用**占有量词/原子组**避免回溯——可惜 **Python 标准 `re` 直到 3.11 才支持**占有量词 `a++` 和原子组 `(?>...)`（PEP 见 3.11 release note）。3.11+ 可以写 `(?>a+)+` 或 `a++` 让内层一旦匹配就不交还、彻底掐断回溯。3.11 之前只能靠重写正则或换 `regex` 第三方库。
- 真要扛恶意输入：上 `regex` 库的超时参数、或在另一进程跑、或换 RE2（`google-re2` 的 Python 绑定）。

```python
# Python 3.11+ 用占有量词根治上面的 evil 正则：
safe = re.compile(r'(a++)+$')   # a++ 占有匹配，不回溯 → 不再爆炸
# 或原子组：re.compile(r'(?>a+)+$')
```

### 3.4 模块级函数的编译缓存（`re._cache`，默认 512）

`re.match(p, s)` 这类**模块级函数**每次都要先把模式 `p` 编译成字节码。如果每次调用都重新编译，循环里就慢死了。CPython 的对策：**`re` 模块内部维护一个编译缓存**。

```python
import re
re.purge()                       # 清空缓存，干净起步
for p in ['error', 'warn', 'info']:
    re.compile(p)
print(len(re._cache))            # → 3：编译过的模式被缓存了
print(re._MAXCACHE)              # → 512：默认最多缓存 512 个不同模式
```

机制要点：

- 缓存键是 `(类型, 模式串, flags)`，所以**同一个模式字符串第二次用就命中缓存、不再重新编译**。
- 上限 `_MAXCACHE = 512`。**一旦缓存满了，CPython 会直接清空整个缓存**（`_cache.clear()`，不是 LRU 淘汰），重新开始攒。
- 缓存只对**模式字符串**有效。如果你**每次循环动态拼出不同的模式**（比如把变量塞进正则），缓存命中率为 0，还会反复把缓存撑满清空——**比手动 `re.compile` 一次再复用慢得多**。

> 实战结论：在热点/循环里，**显式 `re.compile` 一次、复用编译对象**，别依赖那个 512 的内部缓存（它是个全局共享、会被别处挤掉、满了就整体清空的兜底设施，不是给你做性能保证的）。这和 Java 里"`Pattern.compile` 一次、复用 `Pattern`"是同一条最佳实践。

### 3.5 为什么 `r"\d"` 严格优于 `"\\d"`（字节码视角）

表面上 `r"\d"` 和 `"\\d"` **给到正则引擎的字符串完全一样**（都是反斜杠 + d 两个字符），编译出的 SRE 字节码也一模一样。那为什么还说前者"严格优于"？

```python
import re
print(r"\d" == "\\d")            # → True，运行期两者等价
```

原因在**正确性的鲁棒性**，而不是性能：

1. **少一层心算**。`"\\d"` 要你脑内先做"Python 转义：`\\`→`\`"，再想"引擎转义：`\d`→数字"。两层转义叠加，人极易算错（尤其是 `\\\\` 这种四连反斜杠匹配一个字面反斜杠时）。
2. **避开"未知转义"的静默陷阱**。Python 对字符串里**不认识的转义序列**（如 `"\d"`）的处理：`\d` 不是 Python 合法转义，于是 Python 把它**原样保留**为 `\d`——但这是**已被废弃的行为**，从 3.6 起会发 `DeprecationWarning`，未来可能变成报错。

```python
import warnings
warnings.simplefilter('error')   # 把警告升级为异常来演示
try:
    pat = "\d+"                  # 非 raw：Python 觉得 \d 是可疑转义
except SyntaxWarning as e:
    print("触发警告:", e)        # 在新版 Python 上 \d 这类会告警
# r"\d+" 永远不会有这个问题
```

3. **`\b` 的致命歧义**。在正则里 `\b` 是"词边界"；但在 Python 普通字符串里 `\b` 是 ASCII 8（退格符）！所以 `"\bword"` 给引擎的是 `退格符 + word`，而 `r"\bword"` 给引擎的才是正则的"词边界 + word"。这个坑**只有 raw string 能根治**：

```python
print("\bword")                  # → 退格符在前（普通字符串解释了 \b）
print(repr("\bword"))            # → '\x08word'  ← \b 变成了 0x08！
print(repr(r"\bword"))           # → '\\bword'   ← raw 保留为反斜杠 b，正则才认
```

> 一句话：`r"\d"` 和 `"\\d"` 编译结果相同，但 `r"..."` **让你写的字符 = 引擎看到的字符**，消除了一整类转义算错和 `\b`/`\n` 这类歧义。这就是"严格优于"——不是更快，是**更不容易错**。

## ④ 使用方法与最佳实践

### 4.1 核心语法速查（Java 老手快速对表）

正则语法 Python 和 Java 99% 相同，这里只做速查，重点标 Python 特有项。

| 类别 | 写法 | 含义 |
|------|------|------|
| 字符类 | `\d \w \s` | 数字 / 单词字符(含下划线) / 空白；大写 `\D \W \S` 取反 |
| | `[a-z0-9_]` | 自定义字符类；`[^...]` 取反 |
| 量词（贪婪） | `* + ? {m,n}` | 0+ / 1+ / 0或1 / m到n 次 |
| 量词（懒惰） | `*? +? ?? {m,n}?` | 加 `?` 变最小匹配（见 4.2） |
| 量词（占有，3.11+） | `*+ ++ ?+` | 不回溯（防 ReDoS，见 3.3） |
| 锚点 | `^ $` | 行首 / 行尾（`re.M` 下匹配每行） |
| | `\b \B` | 词边界 / 非词边界 |
| | `\A \Z` | 串首 / 串尾（不受 `re.M` 影响） |
| 分组 | `(...)` | 捕获组 |
| | `(?:...)` | **非**捕获组（只分组不捕获，省内存） |
| | `(?P<name>...)` | **命名捕获组（Python 特有 `P`）** |
| 反向引用 | `(?P=name)` 或 `\1` | 引用前面组捕获到的内容 |
| 前查找 | `(?=...)` / `(?!...)` | 后面跟着 / 不跟着（不消耗字符） |
| 后查找 | `(?<=...)` / `(?<!...)` | 前面是 / 不是（定长） |
| 内联标志 | `(?i)` / `(?im)` | 在模式内开标志 |

### 4.2 贪婪 vs 懒惰：`.*` 和 `.*?`

默认量词是**贪婪**的——尽可能多吞。加 `?` 变**懒惰**——尽可能少吞。这是提取 HTML/标签时的头号实用知识：

```python
import re
html = '<a>first</a><a>second</a>'

print(re.findall(r'<a>.*</a>', html))    # 贪婪：['<a>first</a><a>second</a>']  吞过头！
print(re.findall(r'<a>.*?</a>', html))   # 懒惰：['<a>first</a>', '<a>second</a>']  正确
```

> 贪婪 `.*` 会一路吞到最后一个 `</a>` 再回溯，常把多条记录粘成一条。要"每条最短匹配"用 `.*?`。但注意懒惰量词也可能引入回溯开销，最优解常常是**用排除字符类** `[^<]*` 直接表达"非 `<` 的字符"，无需回溯。

### 4.3 编译复用（对照 `Pattern.compile`）

```python
import re

# 推荐：模式编译一次，复用编译对象（尤其在循环/热点里）
LOG_LINE = re.compile(r'^(?P<level>\w+)\s+(?P<msg>.+)$')

for line in ['ERROR db down', 'INFO started']:
    m = LOG_LINE.match(line)
    if m:
        print(m.group('level'), '->', m.group('msg'))
# ERROR -> db down
# INFO -> started
```

编译对象的方法（`p.match`/`p.search`/`p.findall`/`p.sub`...）和模块级函数同名，少传一个模式参数，且必然命中"自己"无需查 512 缓存。

### 4.4 命名组与 `groupdict`（比 Java 顺手）

```python
import re
URL = re.compile(r'(?P<scheme>https?)://(?P<host>[^/:]+)(?::(?P<port>\d+))?(?P<path>/\S*)?')

m = URL.match('https://api.example.com:8443/v1/users')
print(m.group('host'))     # → api.example.com
print(m.group('port'))     # → 8443
print(m.groupdict())
# → {'scheme': 'https', 'host': 'api.example.com', 'port': '8443', 'path': '/v1/users'}
```

`groupdict()` 一把把所有命名组转成 dict——Java 里没有内置等价物，得自己一个个 `m.group("name")`。这是 Python 处理结构化文本的利器。

### 4.5 `re.sub` 用函数替换（Java 要绕一大圈）

`re.sub` 的替换项**可以是一个函数**，每个匹配都调用它，传入 `Match` 对象、返回替换串。这在 Java 里得用 `Matcher.appendReplacement` 循环手写，Python 一行搞定：

```python
import re

text = 'order 1001 and order 1002'

# 把每个订单号 +1（替换项是函数）
def bump(m):
    return str(int(m.group()) + 1)

print(re.sub(r'\d+', bump, text))      # → order 1002 and order 1003

# 替换串里引用命名组用 \g<name>
print(re.sub(r'(?P<y>\d{4})-(?P<m>\d{2})',
             r'\g<m>/\g<y>', '2026-06'))   # → 06/2026
```

`re.subn` 同时返回替换次数：`re.subn(r'\d+', 'N', text)` → `('order N and order N', 2)`。

### 4.6 `re.VERBOSE`：写人能读懂的正则

复杂正则一行写出来没人维护得了。`re.VERBOSE`（`re.X`）允许**忽略模式里的空白、写 `#` 注释**——正则版的"格式化 + 注释"：

```python
import re

IPV4 = re.compile(r'''
    ^
    (?P<a>\d{1,3}) \.      # 第一段
    (?P<b>\d{1,3}) \.      # 第二段
    (?P<c>\d{1,3}) \.
    (?P<d>\d{1,3})
    $
''', re.VERBOSE)

print(IPV4.match('192.168.0.1').groupdict())
# → {'a': '192', 'b': '168', 'c': '0', 'd': '1'}
```

注意：VERBOSE 下**真正要匹配的空格**要写成 `\ ` 或 `[ ]` 或 `\s`，否则会被忽略。对应 Java 的 `Pattern.COMMENTS`。

### 4.7 组合标志、`finditer`、`split`

```python
import re

# 多标志用 | 组合；finditer 给出每个 Match（含位置）
log = "WARN disk full\nERROR db down\nINFO ok"
pat = re.compile(r'^(\w+)\s', re.MULTILINE | re.IGNORECASE)
for m in pat.finditer(log):
    print(m.start(), m.group(1))    # 0 WARN / 15 ERROR / 28 INFO（带偏移）

# split：用正则切分（多个分隔符一把切）
print(re.split(r'[,\s]+', 'a, b,c   d'))   # → ['a', 'b', 'c', 'd']
```

`finditer` 对应 Java 的 `while(m.find())` 循环，但更省内存（惰性迭代）；`findall` 一次性返回 list。

### 4.8 最佳实践清单

1. **模式一律 `r"..."`**（3.5 的理由）。
2. **能用 `str` 方法就别上正则**：`startswith`/`endswith`/`in`/`split`/`replace` 更快更清楚（2.4）。
3. **循环/热点里 `re.compile` 一次复用**，别靠内部 512 缓存（3.4）。
4. **要找全部第一反应 `finditer`**（省内存、带位置），需要 list 才 `findall`。
5. **警惕嵌套量词**，写完危险正则用 3.3 的耗时实验自测，或上占有量词。
6. **复杂正则用 `re.VERBOSE`** 加注释，否则三个月后你自己都看不懂。
7. **校验类正则尽量锚定** `^...$`（用 `re.fullmatch` 更直接），避免"只匹配了一部分还以为通过"。

## ⑤ ⚠️ 易错点（Java 思维翻车）

| Java 习惯 | 翻车现场 | 正确做法/认知 |
|----------|---------|---------|
| 直接写 `"\\d+\\b"`（Java 没原始串） | 漏一个反斜杠、或 `\b` 被当退格符，正则莫名匹配不上 | **永远 `r"\d+\b"`**；`r` 关闭语言层转义（3.5） |
| 以为 `re.match` = "匹配字符串" | `re.match(r'world', 'hello world')` 返回 `None`，以为正则写错了 | `match` 只从**开头**匹配（= Java `lookingAt`）；任意位置查找用 `re.search` |
| 用 `.matches()` 心智写校验 | 想整串匹配却用了 `match`，结果 `'123abc'` 也"通过"了数字校验 | 整串匹配用 `re.fullmatch` 或显式 `^...$` |
| 默认 `.*` 贪婪没意识 | `<a>.*</a>` 把整段 HTML 的多条记录吞成一条 | 用懒惰 `.*?` 或排除类 `[^<]*`（4.2） |
| 以为 `findall` 总返回字符串列表 | 模式里**一加分组**，`findall` 返回值突变成**元组列表**，下游 `for x in result: x.upper()` 直接崩 | 有分组想要整段匹配，用 `finditer`+`m.group(0)`；或把不需要捕获的组写成 `(?:...)` |
| 以为 `.` 匹配任意字符（含换行） | 跨行日志用 `re.search(r'A.*B', text)` 匹配不到（中间有 `\n`） | `.` **默认不匹配 `\n`**；要跨行加 `re.DOTALL`（Java 的 `DOTALL` 同理） |
| 命名组照抄 Java `(?<name>)` | `(?<name>\d+)` 在 Python 报 `error: unknown extension ?<n`（除非是 lookbehind） | Python 是 **`(?P<name>...)`**，引用 `(?P=name)`，替换 `\g<name>` |
| 以为 `^`/`$` 天然按行 | 多行文本里 `^ERROR` 只匹配到全文开头那一处 | `^`/`$` 默认按**整串**；按行要 `re.MULTILINE` |
| 把不可信输入喂给带嵌套量词的正则 | 一条 `'a'*40+'b'` 让某线程 CPU 100%、服务卡死（ReDoS） | 避免 `(x+)+` 类嵌套量词；3.11+ 用占有量词 `x++`；扛 DoS 换 RE2（3.3） |
| 循环里反复 `re.compile(动态拼的模式)` | 缓存命中率 0、512 满了整体清空，比预编译慢几个量级 | 模式固定就 `compile` 一次复用；模式动态就想办法收敛成有限几个（3.4） |

### 基础理解

**1.** 下面四行代码哪些能匹配成功（返回非 `None`）？逐行说明 `match`/`search`/`fullmatch` 的区别。
```python
import re
print(re.match(r'\d+', '2026 年'))
print(re.match(r'\d+', '年 2026'))
print(re.search(r'\d+', '年 2026'))
print(re.fullmatch(r'\d+', '2026 年'))
```

**2.** 为什么写 Python 正则强烈建议用 `r"..."`？举出 `"\bword"` 和 `r"\bword"` 给正则引擎的字符串分别是什么，并说明哪个是"词边界 + word"。

### 找坑改错

**3.（findall 陷阱）** 这段想提取所有邮箱，结果 `print` 出来是一堆元组、`.split('@')` 还报错。指出根因并给出两种修法。
```python
import re
text = "联系 corey295m@gmail.com 或 admin@example.org"
emails = re.findall(r'(\w+)@(\w+\.\w+)', text)
for e in emails:
    user, domain = e.split('@')   # AttributeError: 'tuple' object has no attribute 'split'
    print(user, domain)
```

**4.（ReDoS 识别）** 下面这个"校验用户名"的正则在生产上偶发把某个 worker 的 CPU 打满，输入是用户可控的。指出它**为什么危险**、什么样的输入会触发、以及在 Python 3.11+ 怎么改。
```python
import re
USERNAME = re.compile(r'^(\w+\s?)+$')   # 期望：单词加可选空格，整串
USERNAME.match('a a a a a a a a a a a a a a a a a a a a a a a a a a !')
```

### 综合实战

**5.（解析日志行提取字段）** 你有一批 Nginx 风格的访问日志，每行如下。用一个带**命名组**和 `re.VERBOSE` 的正则解析出 `ip`、`time`、`method`、`path`、`status`、`size`，用 `finditer` 遍历整个多行字符串，打印每条的 `method path -> status`，并统计有多少条 `status` 是 5xx。
```
log = '''127.0.0.1 - - [28/Jun/2026:10:00:01 +0000] "GET /api/users HTTP/1.1" 200 1234
10.0.0.5 - - [28/Jun/2026:10:00:02 +0000] "POST /api/login HTTP/1.1" 503 57
192.168.1.9 - - [28/Jun/2026:10:00:03 +0000] "GET /favicon.ico HTTP/1.1" 404 0'''
```

### 参考答案

**1.**
- `re.match(r'\d+', '2026 年')` → **匹配成功**（`match` 从开头匹配，开头就是 `2026`，不要求到串尾）。
- `re.match(r'\d+', '年 2026')` → **`None`**（`match` 锚定开头，开头是"年"不是数字）。
- `re.search(r'\d+', '年 2026')` → **匹配成功**（`search` 扫描全串，在中间找到 `2026`）。
- `re.fullmatch(r'\d+', '2026 年')` → **`None`**（`fullmatch` 要求**整串**都匹配，但串里有" 年"，不全是数字）。
> 记忆：`match`=从头(Java `lookingAt`)，`search`=任意位置(Java `find`)，`fullmatch`=整串(Java `matches`)。

**2.** 因为反斜杠在 Python 普通字符串里也是转义字符，不用 `r` 会发生"双层转义"，极易算错，且有歧义。具体到 `\b`：
- `"\bword"`：Python 把 `\b` 解释成 **ASCII 8 退格符**，引擎收到的是 `\x08word`——根本不是词边界。
- `r"\bword"`：`r` 关闭语言层转义，引擎收到的是字面的 `\b` + `word`，这才是正则的**词边界 + word**。
所以 `r"\bword"` 才是对的。

**3.** 根因：`findall` 在**模式含有捕获组**时，返回的是**每个匹配的"组元组"列表**（这里两个组 → 每个元素是 `(user, domain)` 元组），不是整段匹配字符串。所以 `e` 是元组，没有 `.split`。两种修法：

修法 A——既然组已经拆好了，直接解包，别再 split：
```python
for user, domain in emails:      # emails 本就是 [(user, domain), ...]
    print(user, domain)
```

修法 B——把组改成**非捕获组** `(?:...)`，让 `findall` 回到返回整段字符串：
```python
emails = re.findall(r'(?:\w+)@(?:\w+\.\w+)', text)   # → ['corey295m@gmail.com', 'admin@example.org']
for e in emails:
    user, domain = e.split('@')
    print(user, domain)
```
（更稳妥的是用 `finditer` 配合 `m.group(0)` 拿整段匹配。）

**4.** 危险原因：`(\w+\s?)+` 是**嵌套量词**——`\w+` 外面又套了一层 `+`。对于一长串能被内层 `\w+` 以多种方式切分的字符（如很多个被空格分隔的字母），再在**末尾放一个必然失败的字符**（这里 `!` 不是 `\w` 也不是 `\s`，导致 `$` 永远匹配不上），引擎会被迫穷举所有切分组合，回溯次数随长度指数级增长 → CPU 打满（ReDoS）。
触发输入：`'a ' * n + '!'` 这种"一堆合法单词 + 一个非法收尾字符"，n 越大越慢。
修法（Python 3.11+）：用**占有量词**掐断回溯，或重写成不嵌套：
```python
USERNAME = re.compile(r'^(?:\w+\s?)++$')   # 3.11+ 占有量词 ++，不回溯
# 或更干净地不嵌套量词、直接表达意图：
USERNAME = re.compile(r'^\w+(?:\s\w+)*$')  # 单词 (空格 单词)*，无嵌套量词，线性
```
3.11 之前只能靠后一种重写，或用第三方 `regex` 库的 `timeout`、或换 RE2。

**5.** 参考实现：
```python
import re

log = '''127.0.0.1 - - [28/Jun/2026:10:00:01 +0000] "GET /api/users HTTP/1.1" 200 1234
10.0.0.5 - - [28/Jun/2026:10:00:02 +0000] "POST /api/login HTTP/1.1" 503 57
192.168.1.9 - - [28/Jun/2026:10:00:03 +0000] "GET /favicon.ico HTTP/1.1" 404 0'''

LINE = re.compile(r'''
    ^
    (?P<ip>\d+\.\d+\.\d+\.\d+)      \s -\ -\ 
    \[(?P<time>[^\]]+)\]            \s
    "(?P<method>[A-Z]+)\ (?P<path>\S+)\ HTTP/[\d.]+"  \s
    (?P<status>\d{3})              \s
    (?P<size>\d+)
    $
''', re.VERBOSE | re.MULTILINE)

errors = 0
for m in LINE.finditer(log):
    d = m.groupdict()
    print(f"{d['method']} {d['path']} -> {d['status']}")
    if d['status'].startswith('5'):
        errors += 1

print('5xx 数量:', errors)

# 输出:
# GET /api/users -> 200
# POST /api/login -> 503
# GET /favicon.ico -> 404
# 5xx 数量: 1
```
要点：①命名组 + `groupdict()` 让字段提取自描述；②`re.VERBOSE` 下真正要匹配的空格写成 `\ ` 或 `\s`（注意 `-\ -\ ` 那段）；③`re.MULTILINE` 让 `^`/`$` 按行锚定，配合 `finditer` 逐行匹配；④判断 5xx 直接 `status.startswith('5')` 比再写正则更清楚（呼应 2.4"能用 str 方法就别上正则"）。
