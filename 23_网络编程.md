# 23 · 网络编程：从 socket 到 WSGI/ASGI ⭐

> 你在 Java 里用过 `Socket`/`ServerSocket` 写过 TCP，用过 NIO 的 `Selector` 做多路复用，用 Netty 撸过高并发服务器，用 `HttpURLConnection` 发过请求，更在 Servlet 容器（Tomcat）里写过无数 `doGet/doPost`。Python 这一整套你都能找到对应物，而且——好消息——**心智模型几乎一模一样**，因为大家都封装的是同一套 BSD socket 系统调用。本章从最底层的 `socket` 一路往上爬：字节收发、粘包半包、`selectors` 多路复用，最后讲清 Python 后端世界最重要的两个"接口规范"——**WSGI（同步）和 ASGI（异步）**，它们就是 Python 版的 Servlet 规范。看懂它们，你才明白 Flask/Django/FastAPI 和 gunicorn/uvicorn 是怎么拼在一起的。

---

## ① Java 对照表

| 概念 / 任务 | Java | Python | 关键差异 |
|------------|------|--------|---------|
| TCP 客户端 | `new Socket(host, port)` | `socket.socket()` + `.connect()` | 都是 BSD socket 薄封装，API 几乎同名 |
| TCP 服务端 | `ServerSocket` + `.accept()` | `socket()` + `.bind()/.listen()/.accept()` | Python 把 listen socket 和普通 socket 用同一个类 |
| UDP | `DatagramSocket` + `DatagramPacket` | `socket(AF_INET, SOCK_DGRAM)` | Python 用 `sendto/recvfrom`，无独立 packet 类 |
| 读写流 | `getInputStream()/getOutputStream()` | `.recv(n)` / `.sendall(b)` | **Python 直接收发 `bytes`，没有 Stream 包装层** |
| NIO 多路复用 | `Selector` + `SelectionKey` | `selectors.DefaultSelector` | Python 封装更薄，底层都是 epoll/kqueue |
| 高层非阻塞框架 | Netty（EventLoop + 编解码器） | `asyncio` + Protocol/Streams | 见 [21 异步编程asyncio](21_异步编程asyncio.md) |
| 现成 TCP 服务端骨架 | 自己写线程池 | `socketserver.ThreadingTCPServer` | 标准库自带，类比一个迷你 Tomcat connector |
| HTTP 客户端 | `HttpURLConnection` / `HttpClient` | `http.client` / 第三方 `httpx`、`requests` | 标准库 `http.client` 偏底层，生产用第三方 |
| HTTP 服务端（玩具） | 内嵌 Jetty | `http.server` | 仅供本地调试/静态文件，**不可上生产** |
| **应用↔容器接口规范** | **Servlet API**（`HttpServlet`） | **WSGI**（同步）/ **ASGI**（异步） | 本章核心：解耦"框架"和"服务器" |
| Web 服务器 / 容器 | Tomcat / Jetty / Undertow | gunicorn / uvicorn / hypercorn | 实现 WSGI/ASGI 规范、负责 socket 与并发 |
| 应用框架 | Spring MVC | Flask/Django（WSGI）、FastAPI/Starlette（ASGI） | 框架只写 `app`，不碰 socket |

> 一句话建立映射：**`socket` ≈ `java.net.Socket`，`selectors` ≈ NIO `Selector`，WSGI/ASGI ≈ Servlet 规范，gunicorn/uvicorn ≈ Tomcat。**

## ② 设计理念（为什么 Python 这么设计）

### 2.1 `socket` 是 OS 网络原语的薄封装，故意"不抽象"

Java 的 `java.net.Socket` 在 BSD socket 之上还盖了一层：给你 `InputStream`/`OutputStream`，把"流"的抽象塞进来，让你以为在读一个有头有尾的字节流对象。Python 的 `socket` 几乎**不做这层抽象**——它就是 C 语言 `socket()`/`bind()`/`listen()`/`accept()`/`send()`/`recv()` 这套 POSIX 系统调用的逐一映射，连函数名都懒得改。

这是 Python 标准库一贯的取舍：**底层模块只做"够用的薄封装"，把抽象留给上层和生态**。好处是你只要懂 BSD socket（你在大学网络课学过的那套），就直接会用 Python socket，几乎零学习成本；坏处是它把"TCP 是字节流、没有消息边界"这件事**赤裸裸地暴露给你**，不像 Java 的 Stream 让你产生"我在读一个对象"的错觉。后面 §3 的粘包/半包问题，本质就是这层"不抽象"的直接后果。

### 2.2 为什么 Python 后端需要 WSGI/ASGI 这层标准接口

这是本章最值得 Java 老手深思的设计。回想 Java 世界：你写 Web 应用从不直接碰 socket，你写的是 **Servlet**（或 Spring MVC，它底层也是 Servlet）。Servlet 是一份**规范**（`javax.servlet` / `jakarta.servlet`），它规定了"应用"和"容器"之间的契约：

- 容器（Tomcat）负责监听端口、管理线程池、解析 HTTP、把请求封装成 `HttpServletRequest`；
- 应用（你的 `HttpServlet`）只管实现 `service(req, resp)`；
- 因为有这份规范，**同一个 war 包能在 Tomcat、Jetty、Undertow 上跑**，框架和服务器互相解耦。

Python 早期没有这个规范，结果是灾难：每个 Web 框架都绑死一个自己的服务器，你换框架就得换部署方式，服务器作者要为每个框架适配。于是社区在 2003 年定了 **PEP 333 / WSGI**（Web Server Gateway Interface），干的事和 Servlet 规范一模一样：

> **WSGI = Python 版的 Servlet 规范。** 它规定"Web 应用"是一个可调用对象 `app(environ, start_response)`，"服务器"负责调用它。有了这层契约，**任何 WSGI 框架（Flask/Django）能跑在任何 WSGI 服务器（gunicorn/uWSGI）上**——和 war 包跑在任意容器里是同一种解耦。

```
   Java 世界                         Python 世界
┌──────────────┐               ┌──────────────┐
│ Spring MVC   │  ← 框架 →      │ Flask/Django │
├──────────────┤               ├──────────────┤
│ Servlet 规范 │  ← 契约 →      │ WSGI / ASGI  │
├──────────────┤               ├──────────────┤
│ Tomcat       │  ← 容器 →      │ gunicorn     │
├──────────────┤               ├──────────────┤
│   socket     │  ← OS 原语 →   │   socket     │
└──────────────┘               └──────────────┘
```

### 2.3 同步 WSGI vs 异步 ASGI 的分野

WSGI 有一个致命的时代局限：它是**同步**的。`app(environ, start_response)` 是个普通函数，调一次返回一次响应——这套模型天然是"一请求一线程/进程"，**没法表达长连接、WebSocket、服务器推送（SSE）、HTTP/2**，因为这些都需要在一次连接里持续地、双向地、异步地收发事件。

这正是 Servlet 3.0 引入异步、Netty 大行其道的同一个动机。Python 社区在 `async`/`await`（见 [21 异步编程asyncio](21_异步编程asyncio.md)）成熟后，定义了 **ASGI**（Asynchronous Server Gateway Interface）作为 WSGI 的异步继任者：

> **ASGI = 异步版的 WSGI。** 应用是一个**协程** `async app(scope, receive, send)`，用 `receive`/`send` 两个异步通道收发"事件"，因此天然支持长连接、WebSocket、流式响应。FastAPI/Starlette 就架在 ASGI 上，uvicorn 是最流行的 ASGI 服务器。

分野记牢：**纯请求-响应、CPU 偏多、生态老 → WSGI 足够；需要高并发 I/O、WebSocket、流式 → ASGI。** 这和"Servlet 同步线程模型 vs Netty 事件循环"是同一道分水岭。

## ③ 实现细节（底层怎么实现的）★本章重点★

### 3.1 socket 对象 = 文件描述符（fd）的封装

在 Linux/macOS 上，**socket 本质是一个文件描述符（fd）**——和打开的文件、管道一样，都是内核 fd 表里的一个整数。Python 的 `socket` 对象就是对这个 fd 的封装，你能直接看到它：

```python
import socket

s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
print(s.fileno())   # → 3   (一个整数 fd；0/1/2 被 stdin/out/err 占了)
print(type(s))      # → <class 'socket.socket'>
s.close()
```

- `AF_INET` = IPv4 地址族（`AF_INET6` 是 IPv6）；
- `SOCK_STREAM` = TCP（面向连接、字节流）；`SOCK_DGRAM` = UDP（数据报）。

因为 socket 是 fd，它和 [22 文件IO与系统交互](22_文件IO与系统交互.md) 里的文件 fd 在内核层是同一套机制——这也是为什么 `selectors`/`epoll` 能同时监听文件和 socket。`send`/`recv` 在 C 层就是 `write(fd, ...)`/`read(fd, ...)` 的网络版兄弟，**操作的单位是字节，不是字符**（呼应 [09 字符串字节与编码](09_字符串字节与编码.md)：网络收发永远是 `bytes`）。

### 3.2 阻塞 vs 非阻塞：一个 fd 标志位的差别

socket 默认是**阻塞**的：`recv()` 在没数据时会让当前线程睡死（内核挂起线程直到数据到达）；`accept()` 在没连接时同样阻塞。这对应 Java 的 `Socket` 默认行为。

```python
s.setblocking(False)   # 切成非阻塞；底层是给 fd 设 O_NONBLOCK 标志
```

非阻塞模式下，`recv()` 没数据时**不睡，立刻抛 `BlockingIOError`**（errno = EAGAIN/EWOULDBLOCK），让你"轮询不到就去干别的"。这正是事件循环和 NIO 的根基：**非阻塞 + 多路复用**，一个线程伺候成千上万个连接。`asyncio` 底层的 socket 全是非阻塞的。

| 模式 | `recv()` 无数据时 | 对应 Java | 用途 |
|------|------------------|-----------|------|
| 阻塞（默认） | 线程挂起等待 | `Socket` 默认 / BIO | 一连接一线程，简单 |
| 非阻塞 | 立刻抛 `BlockingIOError` | `SocketChannel.configureBlocking(false)` | 配合 selector 做多路复用 |

### 3.3 TCP 粘包 / 半包：被故意暴露的真相 ★

这是 Java BIO 老手最容易翻车、也是面试必问的点。**TCP 是字节流，不是消息流——它不保证你一次 `recv` 拿到的就是发送方一次 `send` 的内容。** 内核的发送/接收缓冲区会把数据攒着、拆着、合着发：

- **半包**：发送方 `send` 了 100 字节，你一次 `recv(1024)` 可能只拿到 60 字节（剩下的还在路上）。
- **粘包**：发送方连发两条消息 `"AB"` `"CD"`，你一次 `recv` 可能拿到 `"ABCD"` 糊在一起。

```python
# 错误认知：一次 recv 收全 —— 这是 Java BIO 老手的头号坑
data = conn.recv(1024)          # ❌ 可能只收到半条消息，或粘了两条
```

**根因**：`recv(n)` 的语义是"最多返回 n 字节，到了多少给多少（至少 1 字节，0 表示对端关闭）"。它**不等你凑满 n**。所以 TCP 应用必须**自己定义消息边界**，常见两招：

1. **分隔符**：用 `\n` 之类的定界符（HTTP 头就用 `\r\n`、Redis 协议用 `\r\n`）。
2. **长度前缀**：消息前面先发 4 字节长度，收方先读长度再读够那么多字节（Netty 的 `LengthFieldBasedFrameDecoder` 就干这个）。

这件事 Netty 用**解码器（ByteToMessageDecoder）**帮你做了，让你以为收到的是完整消息；Python 的裸 socket **不替你做**，你必须手写循环。§4.2 给出标准的"按长度前缀分帧"实现。

```python
# 正确：循环 recv 直到凑够 n 字节，解决半包
def recv_exactly(conn, n):
    buf = bytearray()
    while len(buf) < n:
        chunk = conn.recv(n - len(buf))
        if not chunk:                 # 收到空 bytes = 对端已关闭连接
            raise ConnectionError("peer closed before sending enough data")
        buf.extend(chunk)
    return bytes(buf)
```

> **`send` 也有同样问题**：`send(b)` 返回**实际发出的字节数**，可能小于 `len(b)`（发送缓冲区满了）。所以发数据要么循环 `send`，要么直接用 **`sendall()`**——它内部帮你循环到全部发完。生产代码一律用 `sendall`。

### 3.4 selectors：基于 epoll/kqueue 的单线程多路复用

要在一个线程里同时盯着成千上万个 socket"谁有数据了"，靠轮询每个 socket 是 O(n) 的浪费。OS 提供了高效的**就绪通知**机制：Linux 的 `epoll`、macOS/BSD 的 `kqueue`、老式的 `select/poll`。Python 标准库 `selectors` 模块在它们之上提供统一封装——**这就是 Java NIO `Selector` 的对应物**，而且底层就是同一批系统调用。

```python
import selectors
sel = selectors.DefaultSelector()   # Linux 自动选 epoll，macOS 选 kqueue
print(type(sel).__name__)           # → EpollSelector 或 KqueueSelector
```

核心模型（和 NIO 一一对应）：

| 步骤 | Python `selectors` | Java NIO |
|------|-------------------|----------|
| 注册关注的 fd + 事件 | `sel.register(sock, EVENT_READ, data)` | `channel.register(selector, OP_READ, attach)` |
| 阻塞等就绪 | `events = sel.select(timeout)` | `selector.select()` |
| 拿到就绪集合 | 返回 `(key, mask)` 列表 | `selector.selectedKeys()` |
| 取回附带数据 | `key.data` | `key.attachment()` |

`epoll` 之所以高效：注册一次后内核维护就绪链表，`select()` 返回的是**只含就绪 fd 的列表**（O(就绪数) 而非 O(总数)）。这正是 [21 异步编程asyncio](21_异步编程asyncio.md) 里 asyncio 事件循环的引擎——`asyncio` 的默认事件循环底层就是用 `selectors` 驱动的。完整 echo 服务器见 §4.4。

### 3.5 WSGI 接口：拆开看它到底长什么样 ★

WSGI 规范（PEP 3333）极简，核心就一句话：**一个 WSGI 应用是一个可调用对象，签名为 `app(environ, start_response)`**。

- **`environ`**：一个 `dict`，装着这次请求的所有信息——**等价于 Servlet 的 `HttpServletRequest`**，只不过摊平成了字典。键名沿用 CGI 传统：`REQUEST_METHOD`（`"GET"`）、`PATH_INFO`（`"/users"`）、`QUERY_STRING`、`HTTP_*`（每个请求头，如 `HTTP_USER_AGENT`）、`wsgi.input`（请求体的字节流文件对象）等。
- **`start_response`**：一个由服务器传进来的回调，你用它**发送状态行和响应头**——等价于设置 `HttpServletResponse` 的 status 和 headers。签名 `start_response(status, response_headers)`，如 `start_response("200 OK", [("Content-Type", "application/json")])`。
- **返回值**：一个**可迭代的 `bytes`**（通常是一个 `[b"..."]` 列表）——就是响应体。注意是 `bytes` 不是 `str`（呼应 09 章：出口处 encode）。

把这三件事拼起来，你就能手写一个**不依赖任何框架**的最小 WSGI 应用，并用标准库自带的参考服务器 `wsgiref` 跑起来：

```python
# minimal_wsgi.py —— 一个手写的最小 WSGI 应用，返回 JSON
import json
from wsgiref.simple_server import make_server

def app(environ, start_response):
    # environ 就是 Python 版的 HttpServletRequest（摊平成 dict）
    method = environ["REQUEST_METHOD"]
    path = environ.get("PATH_INFO", "/")

    body = json.dumps({"method": method, "path": path, "msg": "hello WSGI"})
    body_bytes = body.encode("utf-8")        # 出口必须 encode 成 bytes

    # start_response 等价于设置 response 的 status + headers
    start_response("200 OK", [
        ("Content-Type", "application/json; charset=utf-8"),
        ("Content-Length", str(len(body_bytes))),
    ])
    return [body_bytes]                       # 返回可迭代的 bytes

if __name__ == "__main__":
    with make_server("127.0.0.1", 8000, app) as httpd:
        print("WSGI serving on http://127.0.0.1:8000")
        httpd.serve_forever()
        # 另开终端: curl http://127.0.0.1:8000/users
        # → {"method": "GET", "path": "/users", "msg": "hello WSGI"}
```

> **顿悟时刻**：Flask 的 `@app.route` 装饰器、Django 的 `urls.py`，最终都收敛成这么一个 `app(environ, start_response)` 可调用对象交给 gunicorn 调用。框架做的是路由、参数解析、模板这些"上层便利"，**底下的契约就是这 30 行**。这正是 Servlet 与 Spring MVC 的关系。

### 3.6 ASGI 接口：协程 + 双向事件通道 ★

ASGI 把 WSGI 异步化，签名变成 **`async def app(scope, receive, send)`**：

- **`scope`**：一个 `dict`，描述这条**连接**的元信息（`scope["type"]` 是 `"http"`/`"websocket"`/`"lifespan"`，加上 method、path、headers 等）。它比 environ 更"连接级"，因为一条连接（尤其 WebSocket）可以有多个来回。
- **`receive`**：一个**异步可调用**，`await receive()` 拿到一个**事件 dict**（如 `{"type": "http.request", "body": b"...", "more_body": False}`）。这是收。
- **`send`**：一个**异步可调用**，`await send({...})` 发出一个事件（先发 `http.response.start` 带状态和头，再发一个或多个 `http.response.body`）。这是发。

为什么要拆成 `receive`/`send` 两个异步通道？因为这样才能表达**长连接里的多次双向交互**：WebSocket 可以收一条发一条、来回无数次；流式响应可以分多次 `send` body。这正是 Netty 的 `ChannelHandlerContext.read()/write()` 事件模型的 Python 版。

```python
# minimal_asgi.py —— 手写最小 ASGI 应用，返回 JSON（用 uvicorn 跑）
import json

async def app(scope, receive, send):
    assert scope["type"] == "http"           # 只处理 HTTP（忽略 websocket/lifespan）

    # 把请求体读完（可能分多帧，要循环——又是"半包"思想）
    body = b""
    while True:
        event = await receive()
        body += event.get("body", b"")
        if not event.get("more_body", False):
            break

    payload = json.dumps({
        "method": scope["method"],
        "path": scope["path"],
        "received_bytes": len(body),
        "msg": "hello ASGI",
    }).encode("utf-8")

    # 先发响应头事件
    await send({
        "type": "http.response.start",
        "status": 200,
        "headers": [(b"content-type", b"application/json; charset=utf-8")],
    })
    # 再发响应体事件
    await send({
        "type": "http.response.body",
        "body": payload,
    })

# 运行: uvicorn minimal_asgi:app --port 8001
# 另开终端: curl http://127.0.0.1:8001/items
# → {"method": "GET", "path": "/items", "received_bytes": 0, "msg": "hello ASGI"}
```

注意 ASGI 的 headers 是 **`bytes` 的元组列表**（`(b"content-type", b"...")`），而 WSGI 用 `str`——这是 ASGI 更贴近字节、更适合异步流式的设计。

### 3.7 uvicorn / gunicorn 在这层的角色

| 工具 | 实现的规范 | 角色（对照 Java） | 典型用法 |
|------|-----------|------------------|---------|
| **gunicorn** | WSGI（主）| 进程管理器 + WSGI 服务器，类比 Tomcat 的多线程/多进程模型 | `gunicorn myapp:app -w 4` 起 4 个 worker 进程 |
| **uvicorn** | ASGI | 单进程异步 ASGI 服务器（基于 `uvloop`+`httptools`），类比 Netty | `uvicorn myapp:app` |
| **gunicorn + uvicorn worker** | ASGI | gunicorn 管进程、每个 worker 是 uvicorn 异步循环 | `gunicorn myapp:app -k uvicorn.workers.UvicornWorker -w 4` |

它们干的是 Tomcat 干的活：**监听端口、accept 连接、解析 HTTP 字节、按 WSGI/ASGI 契约调用你的 `app`、把返回值编码成 HTTP 响应字节发回**。因为多核下 GIL（见 [20 并发与GIL](20_并发与GIL.md)）限制单进程并行，gunicorn 用**多进程 worker** 压满多核——这和 Tomcat 用线程池的思路不同，是 Python 的特色。

## ④ 使用方法与最佳实践

### 4.1 TCP echo 服务端 + 客户端（可跑）

```python
# tcp_echo_server.py —— 阻塞式单连接 echo（最简版）
import socket

with socket.socket(socket.AF_INET, socket.SOCK_STREAM) as srv:
    srv.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)  # 见 §5，必加
    srv.bind(("127.0.0.1", 9000))
    srv.listen(5)                       # backlog=5，等待队列长度
    print("listening on 9000 ...")
    conn, addr = srv.accept()           # 阻塞直到有连接；返回新 socket
    with conn:
        print("connected by", addr)
        while True:
            data = conn.recv(1024)      # 收字节
            if not data:                # 空 bytes = 对端关闭
                break
            conn.sendall(data)          # 原样回发；用 sendall 不用 send
    print("client gone")
```

```python
# tcp_echo_client.py
import socket

with socket.socket(socket.AF_INET, socket.SOCK_STREAM) as c:
    c.connect(("127.0.0.1", 9000))
    c.sendall("你好 socket".encode("utf-8"))   # 出口 encode
    reply = c.recv(1024)
    print(reply.decode("utf-8"))                # 入口 decode → 你好 socket
```

`with` 块退出时自动 `close()`（socket 实现了上下文管理器，见 [15 异常与上下文管理器](15_异常与上下文管理器.md)），避免 fd 泄漏。

### 4.2 自定义消息分帧（长度前缀，解决粘包/半包）★

工业界最常用的分帧法：**4 字节大端长度前缀 + 负载**。用 `struct` 打包长度（对照 Java 的 `DataOutputStream.writeInt` + `ByteBuffer`）。

```python
import struct

def send_msg(conn, payload: bytes):
    """发一帧：4 字节长度前缀（大端）+ 负载"""
    header = struct.pack(">I", len(payload))   # >I = 大端无符号 4 字节，同 Java writeInt
    conn.sendall(header + payload)

def recv_msg(conn) -> bytes | None:
    """收一帧：先读 4 字节长度，再读够那么多字节"""
    header = recv_exactly(conn, 4)
    if header is None:
        return None
    (length,) = struct.unpack(">I", header)
    return recv_exactly(conn, length)

def recv_exactly(conn, n: int) -> bytes | None:
    buf = bytearray()
    while len(buf) < n:
        chunk = conn.recv(n - len(buf))
        if not chunk:
            return None                        # 对端中途关闭
        buf.extend(chunk)
    return bytes(buf)
```

这套 `send_msg`/`recv_msg` 就是 Netty `LengthFieldPrepender` + `LengthFieldBasedFrameDecoder` 的手写等价物。**任何裸 TCP 应用协议都得自带这样的分帧逻辑。**

### 4.3 UDP 用法（无连接）

```python
# udp_server.py
import socket
srv = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)   # SOCK_DGRAM = UDP
srv.bind(("127.0.0.1", 9100))
data, addr = srv.recvfrom(2048)      # 一次 recvfrom = 一个完整数据报（无粘包！）
print(f"from {addr}: {data.decode()}")
srv.sendto(b"ack", addr)             # 回给来源地址
```

```python
# udp_client.py
import socket
c = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
c.sendto(b"ping", ("127.0.0.1", 9100))   # 无需 connect，直接 sendto
print(c.recvfrom(2048)[0].decode())       # → ack
```

UDP 的 `recvfrom` **一次对应一个完整数据报**——没有粘包问题（但可能丢包、乱序、超过 MTU 被分片）。对照 Java `DatagramSocket`/`DatagramPacket`。

### 4.4 selectors 单线程多路复用 echo（可同时服务多个客户端）

```python
# selector_echo_server.py —— 单线程伺候 N 个连接，NIO 风格
import selectors, socket

sel = selectors.DefaultSelector()

def accept(srv):
    conn, addr = srv.accept()
    conn.setblocking(False)
    sel.register(conn, selectors.EVENT_READ, read)   # 注册新连接，回调设为 read
    print("accepted", addr)

def read(conn):
    try:
        data = conn.recv(1024)
    except BlockingIOError:
        return
    if data:
        conn.sendall(data)
    else:                              # 对端关闭
        sel.unregister(conn)
        conn.close()

srv = socket.socket()
srv.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
srv.bind(("127.0.0.1", 9000))
srv.listen()
srv.setblocking(False)
sel.register(srv, selectors.EVENT_READ, accept)      # 监听 socket 也注册进去

print("selector echo on 9000")
while True:
    for key, _mask in sel.select(timeout=None):      # 阻塞等任意 fd 就绪
        callback = key.data                          # 取回注册时存的回调
        callback(key.fileobj)                        # 分派
```

一个线程、无阻塞、靠 epoll 伺候任意多连接——**这就是 asyncio 事件循环去掉协程糖之后的样子**。生产中你不会手写这个，但理解它才理解 asyncio。

### 4.5 HTTP 客户端：`http.client`（标准库）

```python
import http.client, json

conn = http.client.HTTPSConnection("httpbin.org")
conn.request("GET", "/get?lang=python")
```

`http.client` 偏底层（对照 `HttpURLConnection` 那种繁琐感）。**生产中绝大多数人用第三方 `httpx`（支持同步+异步）或 `requests`**——三行搞定，自动处理连接池、重定向、编码（详见 [24 HTTP与Web框架](24_HTTP与Web框架.md)）。

### 4.6 HTTP 服务端：`http.server`（仅供调试）

### 4.7 异步网络：asyncio streams（高层 API）

写异步 TCP，别碰裸 socket，用 `asyncio` 的 **streams** 高层 API（`StreamReader`/`StreamWriter`），它把"非阻塞 socket + 事件循环"全包了（见 [21 异步编程asyncio](21_异步编程asyncio.md)）：

```python
# async_echo_server.py
import asyncio

async def handle(reader: asyncio.StreamReader, writer: asyncio.StreamWriter):
    while data := await reader.read(1024):   # await，不阻塞线程
        writer.write(data)
        await writer.drain()                 # 等发送缓冲排空（背压）
    writer.close()

async def main():
    server = await asyncio.start_server(handle, "127.0.0.1", 9000)
    async with server:
        await server.serve_forever()

asyncio.run(main())
```

一个事件循环、单线程、能扛上万并发连接——这是 Python 写高并发网络服务的地道方式，对标 Netty。

### 最佳实践清单

1. **裸 socket 只用于学习/特殊协议**；HTTP 用 httpx/框架，高并发用 asyncio streams。
2. **永远用 `sendall` 不用 `send`**；收数据永远**循环 recv + 自定义分帧**，绝不假设一次收全。
3. **永远用 `with` 管理 socket**（自动 close），避免 fd 泄漏。
4. 服务端 `bind` 前**必加 `SO_REUSEADDR`**（见 §5）。
5. **边界处显式 encode/decode**（呼应 09 章），中间逻辑只碰 `str`。
6. **生产部署用成熟服务器**（gunicorn/uvicorn），不要拿 `http.server` 或裸 socket 上线。
7. 阻塞 socket 一定设**超时** `sock.settimeout(seconds)`，否则可能永久卡死。

## ⑤ ⚠️ 易错点（Java 思维翻车）

| Java 习惯 | 翻车现场 | 正确做法 / 认知 |
|----------|---------|----------------|
| 以为一次 `recv` 收全（像 `InputStream.read` 读一个完整对象） | 高负载/大消息时收到半条或两条粘一起，解析报错 | TCP 是**字节流无边界**：循环 `recv` + 自定义分帧（长度前缀/分隔符） |
| 把 `recv` 的返回当字符串处理 | `TypeError`，或对 `bytes` 调字符串方法失败 | socket 收发的是 **`bytes`**，入口 `decode`、出口 `encode`（[09 字符串字节与编码](09_字符串字节与编码.md)） |
| 用 `send` 以为一次发完（像 `write`） | 大数据时只发出一部分，对端收不全 | `send` 可能短写，**一律用 `sendall`** |
| 忘了 close（依赖 GC/finalizer） | fd 泄漏，最终 `Too many open files` | 用 `with socket.socket() as s:` 或 `try/finally` 显式 close |
| 重启服务端立刻 bind 同端口 | `OSError: [Errno 48] Address already in use`（TIME_WAIT 占着） | bind 前 `setsockopt(SOL_SOCKET, SO_REUSEADDR, 1)` |
| 阻塞 socket 不设超时（以为会自己断） | `recv`/`connect` 永久挂起，线程卡死 | `sock.settimeout(5)`，超时抛 `socket.timeout` 自己处理 |
| 拿 `http.server` 或裸 socket 上生产 | 单线程、无并发、无安全，被打挂 | 生产用 gunicorn/uvicorn（实现 WSGI/ASGI 的成熟服务器） |
| 把 WSGI `app` 返回 `str` | 服务器报类型错误 | WSGI 返回**可迭代的 `bytes`**（如 `[b"..."]`） |
| 在 ASGI/asyncio 里调阻塞的 socket | 阻塞整个事件循环，所有连接卡住 | 异步上下文里用 `asyncio` streams 或 `loop.sock_*`，绝不混入阻塞调用 |

## ⑥ 练习题

### 基础理解

1. 用一句话分别解释：为什么 TCP 服务端必须"循环 `recv`"，而 UDP 服务端一次 `recvfrom` 就能拿到完整数据？（从"流 vs 数据报"角度作答）

2. WSGI 应用的签名是 `app(environ, start_response)`，ASGI 是 `async app(scope, receive, send)`。分别说明：(a) `environ` 对应 Servlet 的什么对象？(b) 为什么 ASGI 要把收发拆成 `receive`/`send` 两个异步通道，而 WSGI 一个返回值就够？

### 找坑改错

3. 下面这段"Java 味"的 Python TCP 服务端代码有**至少三处**会翻车的地方，请逐一指出并改正：

```python
import socket

server = socket.socket()
server.bind(("127.0.0.1", 9000))
server.listen()
conn, addr = server.accept()
msg = conn.recv(1024)                  # 假设一次收全

conn.send("ok")                        # 回个 ok
# 程序结束，没 close
```

4. 一个同事写了按长度前缀分帧的收包函数，但偶发"收到的消息少了几个字节/解析错位"。看出 bug 了吗？

```python
def recv_msg(conn):

    (length,) = struct.unpack(">I", header)
```

### 综合实战

5. 写一个**最小 ASGI 应用**（不用任何 Web 框架），实现一个简单的 JSON API：对 `GET /time` 返回当前服务器时间 `{"now": "<ISO8601>"}`，对其它路径返回 404 和 `{"error": "not found"}`。要求用 uvicorn 能跑起来，并用 `curl` 验证。

---

### 参考答案

**1.** TCP 是**字节流**，`recv(n)` 只保证"最多返回 n 字节、到多少给多少"，不保证一次给全一条应用消息（可能半包/粘包），所以要循环收到边界为止；UDP 是**数据报**，`recvfrom` 一次正好返回一个完整数据报（发方一次 `sendto` 的内容），有天然边界，故无需循环拼接（但可能丢包/乱序）。

**2.** (a) `environ` 对应 `HttpServletRequest`（把请求方法、路径、查询串、请求头、请求体都摊平进一个 dict；同时 `start_response` 对应设置 `HttpServletResponse` 的 status 和 headers）。(b) WSGI 是**同步请求-响应**模型，一次调用返回一个响应体就结束，无法表达长连接里的多次双向交互；ASGI 面向**长连接/WebSocket/流式**，一条连接可能有多个收发事件（收一条发一条、分多帧发 body），所以必须用 `receive`/`send` 两个异步通道持续地、双向地传递事件。

**3.** 至少三处问题：
- **没设 `SO_REUSEADDR`**：重启会 `Address already in use`。加 `server.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)`（bind 前）。
- **`recv(1024)` 假设一次收全**：TCP 半包/粘包，应循环 recv 或按协议分帧。
- **`"收到：" + msg`**：`msg` 是 `bytes`，不能和 `str` 直接拼接，会 `TypeError`。应 `msg.decode("utf-8")`。
- **`conn.send("ok")`**：发的必须是 `bytes`，且 `send` 可能短写。应 `conn.sendall("ok".encode())` 或 `conn.sendall(b"ok")`。
- **没有 close**：fd 泄漏。应对 `conn`、`server` 用 `with`，或 `try/finally` close。

改正版（要点）：
```python
import socket
with socket.socket() as server:
    server.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
    server.bind(("127.0.0.1", 9000))
    server.listen()
    conn, addr = server.accept()
    with conn:
        msg = conn.recv(1024)                 # 真实场景仍需循环/分帧
        print("收到：" + msg.decode("utf-8"))
        conn.sendall(b"ok")
```

**4.** 两处都犯了"假设 `recv` 一次收全"的错：`conn.recv(4)` 可能返回不足 4 字节（半包），`conn.recv(length)` 同样可能返回不足 `length` 字节。结果是长度解析错位、负载被截断。修复：两处都改用循环到收满的 `recv_exactly`（见 §4.2），且要处理 `recv` 返回空 `bytes`（对端关闭）的情况。

**5.** 最小 ASGI JSON API：

```python
# time_api.py —— uvicorn time_api:app --port 8002
import json
from datetime import datetime, timezone

async def app(scope, receive, send):
    assert scope["type"] == "http"

    # 把请求体读干净（即便 GET 也要消费，遵守 ASGI 协议）
    while True:
        event = await receive()
        if not event.get("more_body", False):
            break

    if scope["method"] == "GET" and scope["path"] == "/time":
        status = 200
        payload = {"now": datetime.now(timezone.utc).isoformat()}
    else:
        status = 404
        payload = {"error": "not found"}

    body = json.dumps(payload).encode("utf-8")
    await send({
        "type": "http.response.start",
        "status": status,
        "headers": [(b"content-type", b"application/json; charset=utf-8")],
    })
    await send({"type": "http.response.body", "body": body})

# 运行: uvicorn time_api:app --port 8002
# 验证: curl http://127.0.0.1:8002/time
#   → {"now": "2026-06-28T12:00:00+00:00"}
# 验证: curl -i http://127.0.0.1:8002/nope
#   → HTTP/1.1 404 Not Found ... {"error": "not found"}
```

要点：判断 `scope["method"]` 和 `scope["path"]` 路由；用 `datetime.now(timezone.utc).isoformat()` 生成 ISO8601；响应体 `encode` 成 `bytes`；先 `http.response.start`（带 status 和 bytes 头）后 `http.response.body`。这就是 FastAPI 路由 `GET /time` 时，剥掉框架糖之后底层真正发生的事。
