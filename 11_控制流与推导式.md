# 11 · 控制流与推导式

> 控制流是你最熟的领域——`if`/`for`/`while` 你闭着眼都会写。但 Python 在这里埋了几处"形似神不同"：缩进就是语法、任何对象都能当布尔判断、`for`/`while` 能跟 `else`、循环和过滤被压缩进一行"推导式"、还有 3.10 才补上的 `match` 结构化模式匹配。本章的目标不是教你写循环，而是让你把 Java 的肌肉记忆换成 Pythonic 的写法——并看清推导式在 CPython 3.12 里到底编成了什么。

---

## ① Java 对照表

| 概念 | Java | Python | 一句话差异 |
|------|------|--------|-----------|
| 条件 | `if/else if/else` | `if/elif/else` | `else if` 合成一个词 `elif`；条件不要括号，靠**缩进**分块 |
| 三元 | `c ? a : b` | `a if c else b` | 顺序是"值-条件-值"，可读但反直觉 |
| 增强 for | `for (T x : coll)` | `for x in coll` | Python 没有 C 风格 `for(i=0;i<n;i++)` |
| 计数循环 | `for (int i=0;i<n;i++)` | `for i in range(n)` | 用 `range` 生成序列，而非手动维护下标 |
| 带索引遍历 | `for (int i...) coll.get(i)` | `for i, x in enumerate(coll)` | `enumerate` 同时给索引和元素 |
| 并行遍历 | 手动多个索引 | `for a, b in zip(xs, ys)` | `zip` 把多个序列拉成元组流 |
| `while` | `while (cond)` | `while cond:` | 一样 |
| `do-while` | `do{}while(c)` | **无** | 用 `while True: ... if not c: break` 模拟 |
| `break`/`continue` | 同名 | 同名 | 一样；但多了 `for-else`/`while-else` |
| 集合转换 | Stream `.map().filter().collect()` | 列表/字典/集合推导式 | 推导式是声明式的"内建 Stream" |
| 惰性流 | `Stream`（惰性） | 生成器表达式 `(... for ...)` | 圆括号版推导式，惰性、省内存 |
| 赋值即表达式 | `while ((n=read())!=-1)` | `while (n := read()) != -1` | 海象 `:=`（3.8+）让赋值能当表达式用 |
| 模式匹配 | Java 21 `switch` + 模式 | `match/case`（3.10+） | 都能解构，但 Python 的 `case` 默认是**绑定**不是比较 |
| 真值判断 | 只有 `boolean`/`Boolean` 能进 `if` | **任何对象**都能进 `if` | 空容器/0/None/空串 为"假"，靠 `__bool__`/`__len__` |
| switch 穿透 | `case` 不 `break` 会穿透 | `case` **不穿透**，无需 `break` | 匹配一个就跳出 |

## ② 设计理念（为什么 Python 这么设计）

### 缩进即语法：把"排版好习惯"升级成"编译器强制"

Java 用 `{}` 划分代码块，缩进只是给人看的——所以才有无数团队为大括号换行风格吵架，也才有 `goto fail;` 那种因为缺一对括号导致的著名安全漏洞。Python 直接取消了块定界符，**用缩进本身表达层级**。这不是为了少打字，而是一个哲学选择（见 [01 设计哲学与心智模型](01_设计哲学与心智模型.md)：可读性高于一切）：代码"看起来的结构"和"实际的结构"被强制对齐，再也不会出现"缩进骗了你眼睛、括号才是真相"的情况。

代价是缩进从"风格"变成了"语义错误源"——空格和 Tab 混用、对齐错一格，都是 `IndentationError`。对 Java 老手来说这需要一点适应，但好处是：你读任何 Python 代码,层级一目了然，全社区缩进风格统一（PEP 8 规定 4 空格）。

### 真值的"鸭子化"：任何对象都能回答"你真吗？"

Java 的 `if` 只接受 `boolean`，`if ("")`、`if (list)` 都编译不过。Python 反过来：**任何对象放进 `if` 都有真假**。这延续了"一切皆对象 + 鸭子类型"的思路——与其问"这是不是布尔值"，不如问"这个对象自认为是真还是假"。容器空了就是假、数字 0 是假、`None` 是假、非空就是真。于是 `if not users:`（列表空吗）、`if config:`（字典有内容吗）成了地道写法，比 Java 的 `if (list.isEmpty())`、`if (map != null && !map.isEmpty())` 短得多。这套"语义真值"是 Python 简洁的来源之一，但也是 Java 老手的头号翻车点（`x = 0` 时 `if x:` 为假，见 ⑤）。

### 推导式：函数式与命令式之间的折中

"从一个集合造另一个集合"是编程里最高频的操作。Java 8 给的答案是 Stream：`list.stream().map(...).filter(...).collect(toList())`——函数式、链式、但啰嗦，且要理解 `Collector`。Python 给的答案是**推导式**：`[f(x) for x in xs if cond(x)]`——它把 `for` 循环和 `if` 过滤压成一行，读起来几乎就是数学里的集合构造式 `{f(x) | x ∈ xs, cond(x)}`。

推导式是个精妙的折中：它有命令式 `for` 的直白（你一眼看出在循环），又有函数式的声明性（你描述的是"想要什么"而非"怎么一步步建")。它比 `map`/`filter` 更易读（不用写 lambda），又比手写 `for`+`append` 更紧凑、更快（少了反复的方法查找，见 ③）。这就是为什么 Pythonic 代码里推导式无处不在，而 `map`/`filter` 反而少见。

### for-else：给"搜索循环"一个语法位置

`for-else` 是 Python 最让人困惑的设计之一，理解它的关键是：**这个 `else` 应该读成 "no break"**。循环正常跑完（没被 `break` 打断）才执行 `else`。它专为"在序列里找东西"的模式设计：找到就 `break`，没找到就走 `else` 做兜底。Java 里你得引入一个 `boolean found = false;` 标志变量，Python 把这个标志"内建"进了语法。设计意图是好的（消除标志变量），但名字起得糟（`else` 极易误解），所以工业界态度两极——下文会给出何时用、何时别用。

### match：结构化解构，而非"高级 switch"

Java 的 `switch`（哪怕加了 Java 21 的模式匹配）骨子里还是"拿一个值跟一堆分支比相等"。Python 3.10 的 `match` 从一开始就不是为"比相等"设计的，而是为**结构化解构**：它看的是数据的"形状"——这是不是一个三元素序列？是不是一个带 `name` 键的字典？是不是一个 `Point` 实例、它的 `.x` 是多少？匹配成功的同时,把里面的部件**绑定**到变量上。这套思路来自函数式语言（ML/Haskell/Rust）的代数数据类型匹配，处理嵌套 JSON、AST、协议消息这类"形状已知的数据"时极其顺手。理解这个定位，才不会把 `match` 当成 `if/elif` 的语法糖去用（那确实不值得）。

## ③ 实现细节（底层怎么实现的）★本章重点★

### 3.1 真值判断协议：先 `__bool__`，再 `__len__`，都没有就为真

Python 对"`x` 是真是假"有一套明确的查找协议。`bool(x)`（以及 `if x:`、`while x:`、`not x`）按顺序问对象：

1. 有 `__bool__` 方法吗？有就调它，返回的必须是 `True`/`False`。
2. 没有 `__bool__`，那有 `__len__` 吗？有就调它，`len == 0` 为假，非 0 为真。
3. 两个都没有？那么对象**恒为真**（绝大多数自定义对象走这条）。

这条规则直接解释了内置类型的真值：

```python
# 这些都是"假值"（falsy）
print(bool(None))      # → False   None 有 __bool__ 返回 False
print(bool(False))     # → False
print(bool(0))         # → False   int.__bool__：0 为假
print(bool(0.0))       # → False
print(bool(""))        # → False   空 str：__len__ == 0
print(bool([]))        # → False   空 list：__len__ == 0
print(bool({}))        # → False   空 dict
print(bool(set()))     # → False
print(bool(range(0)))  # → False

# 非空 / 非零都是"真值"（truthy）
print(bool("0"))       # → True    字符串 "0" 非空！是真（Java 老手易错）
print(bool([0]))       # → True    含一个元素的列表，非空即真
print(bool(-1))        # → True    非零即真，负数也真
```

自定义类怎么控制自己的真假：

```python
class Cart:
    def __init__(self):
        self.items = []
    def __len__(self):                 # 没写 __bool__，于是真值看 __len__
        return len(self.items)

cart = Cart()
print(bool(cart))                      # → False  空购物车（len==0）
cart.items.append("book")
print(bool(cart))                      # → True   有商品了
# 于是可以写 `if cart:` 而不是 `if len(cart.items) > 0:`
```

**字节码层面**，`if`/`while`/`and`/`or` 并不会傻乎乎地先算出一个 `bool` 对象再判断，而是用专门的条件跳转指令直接根据真值跳转。看一段：

```python
import dis
def first_nonempty(a, b):
    if a:            # a 真值为真则用 a，否则用 b
        return a
    return b
dis.dis(first_nonempty)
```

3.12 下的关键指令（简化）：

```
  LOAD_FAST                0 (a)
  POP_JUMP_IF_FALSE   ...  (to L1)   # 弹出 a，按其"真值"决定是否跳转
  LOAD_FAST                0 (a)
  RETURN_VALUE
L1:
  LOAD_FAST                1 (b)
  RETURN_VALUE
```

`POP_JUMP_IF_FALSE` 这条指令内部会对栈顶对象走上面的真值协议（必要时调 `__bool__`/`__len__`），为假就跳走。这就是为什么 `if x:` 对任何对象都成立——真值判断是 VM 指令的内建语义，不是某个类型的特权。对照 JVM：`ifeq`/`ifne` 只接受栈顶的 `int`（boolean 在字节码层是 int），编译器在 `javac` 阶段就保证了进来的一定是布尔值，VM 不做"问对象"这种动态分派。

### 3.2 推导式有独立作用域：一个隐藏函数（以及 3.12 的反转）

这是 Python 3 相对 Python 2 的一个重要修正，也是面试常考点。**Python 3 的推导式有自己的独立作用域，循环变量不会泄漏到外面**：

```python
x = "外层的 x"
squares = [x for x in range(3)]    # 这个 x 是推导式内部的
print(squares)                      # → [0, 1, 2]
print(x)                            # → 外层的 x   （没被覆盖！）
```

在 **Python 2** 里，列表推导式（不含 genexpr）是会泄漏的——上面 `print(x)` 会打出 `2`，循环变量污染了外层。Python 3 修了这个坑。

**它是怎么做到隔离的？** Python 3 的做法是：编译器把推导式的循环体**编译成一个匿名的嵌套函数**，每次求推导式就调用这个隐藏函数，循环变量自然是那个函数的局部变量，调用结束就销毁，不可能泄漏。用 `dis` 能直接看到这个隐藏函数被 `MAKE_FUNCTION` 造出来（这是 Python 3.0–3.11 的实现，下面是该模型的字节码）：

```python
import dis
def build(n):
    return [i * i for i in range(n)]
dis.dis(build)
```

3.11 及之前会看到外层函数里出现：

```
  LOAD_CONST     (<code object <listcomp> ...>)   # 一个独立的 code object！
  MAKE_FUNCTION                                    # 把它造成函数
  LOAD_GLOBAL    (range)
  ...
  CALL_FUNCTION                                    # 调用这个隐藏函数来跑循环
```

也就是说，推导式和你手写的 `def` 一样有自己的栈帧——这正是隔离的根源。

**Python 3.12 的反转（PEP 709：推导式内联）**：从 3.12 起，CPython 为了性能把**列表/字典/集合推导式"内联"**进了外层函数——不再每次都 `MAKE_FUNCTION` + 建栈帧 + 函数调用，而是把循环体直接展开在当前帧里。这让推导式快了约一倍（省掉了一次函数调用的开销）。**但隔离语义被刻意保留了**：编译器在内联时仍然把循环变量当作"隐藏的临时变量"处理，进入推导式前保存可能的同名变量、退出后恢复，所以 `x` 照样不泄漏。换句话说：**3.12 改了实现（去掉了隐藏函数的运行时开销），没改语义（作用域隔离还在）**。

一个重要例外：**生成器表达式 `(... for ...)` 没有被内联**，它依旧编译成一个带独立栈帧的函数。原因是生成器必须能"挂起/恢复"（惰性逐个产出），天然需要自己的帧。你能在字节码里看到它仍然 `MAKE_FUNCTION`，循环体里用的是 `YIELD_VALUE`：

```python
import dis
def gen(n):
    return sum(i * i for i in range(n))
dis.dis(gen)
# 外层仍可见 MAKE_FUNCTION + 一个 <genexpr> code object；
# 其内部循环体以 YIELD_VALUE 逐个产出，而非 LIST_APPEND 一次性建表
```

### 3.3 四种推导式编出的不同"收集"指令

四种推导式语法几乎一样，区别在于"往哪收集"，字节码里对应不同的收集指令：

| 写法 | 结果类型 | 收集指令（概念） |
|------|---------|-----------------|
| `[x for x in it]` | `list` | `BUILD_LIST` + `LIST_APPEND` |
| `{x for x in it}` | `set` | `BUILD_SET` + `SET_ADD` |
| `{k: v for ...}` | `dict` | `BUILD_MAP` + `MAP_ADD` |
| `(x for x in it)` | 生成器 | `YIELD_VALUE`（惰性，不收集） |

前三种是**急切求值**：循环立刻跑完，一次性建出整个容器。第四种生成器表达式是**惰性求值**：调用时几乎不干活，只返回一个生成器对象，每次 `next()` 才推进一步、产出一个值。这意味着内存占用从"整个结果集"降到"一次一个元素"。

```python
import sys
lst = [i for i in range(1_000_000)]        # 立刻建出百万元素列表
gen = (i for i in range(1_000_000))        # 几乎不占内存，还没开始算
print(sys.getsizeof(lst))                  # → 8000056 左右（约 8MB）
print(sys.getsizeof(gen))                  # → 200 左右（一个生成器对象的壳）
```

生成器表达式的惰性机制（迭代器协议、`yield`、`send`）属于 [16 迭代器与生成器](16_迭代器与生成器.md) 的深水区，本章只用它"省内存"的特性，不深挖。心智模型：**list 推导式 = 把整碗饭端上来；生成器表达式 = 一口一口现做**。

### 3.4 海象运算符 `:=`：赋值表达式

普通赋值 `x = 1` 是**语句**，没有值，不能嵌进表达式（Java 里 `x = 1` 是有值的表达式，`while ((n = read()) != -1)` 合法）。Python 早年故意把赋值设计成语句，就是为了堵死 `if (x = 0)` 这种"本想写 `==` 却写成 `=`"的经典 C/Java bug。

3.8 引入的海象运算符 `:=` 在保留这层保护的同时，开了一个**显式**的口子：`(n := read())` 是表达式，值就是赋给 `n` 的值。它必须用 `:=`（不可能和 `==` 手滑混淆），且常需括号，所以不会重蹈 C 的覆辙。底层上它就是"求值 + `STORE_FAST` 到变量 + 把值留在栈上"，没有魔法。

### 3.5 match 编译成"判形状 + 解构 + 跳转"

`match` 不是哈希跳转表（Java `switch` 对 int/String 会编成 `tableswitch`/`lookupswitch` 跳转表），而是**自上而下逐个 case 尝试**，每个 case 编成一串"检查 + 解构"的字节码，失败就跳到下一个 case。3.10+ 为模式匹配新增了一批专用指令：

| 指令 | 作用 |
|------|------|
| `MATCH_SEQUENCE` | 栈顶是否是序列（且非 str/bytes） |
| `MATCH_MAPPING` | 栈顶是否是映射（dict 类） |
| `MATCH_KEYS` | 映射里是否含指定的一组键 |
| `MATCH_CLASS` | 栈顶是否是某类的实例，并按 `__match_args__` 取属性 |
| `GET_LEN` | 取长度，用于序列模式的元素个数检查 |

一个结构模式 `case [a, b]:` 大致编译成：`MATCH_SEQUENCE`（是序列吗）→ `GET_LEN` + 比较（长度是 2 吗）→ 逐个 `STORE_FAST` 解构出 `a`、`b`。类模式 `case Point(x=px, y=py):` 编译成 `MATCH_CLASS`（是 Point 吗）→ 取 `.x`/`.y` 绑定到 `px`/`py`。**这就是"结构化解构"在 VM 层的样子：先验形状，再拆零件**。因为是顺序尝试而非跳表，`match` 在分支极多时不一定比 `dict` 分派快——它的价值在解构表达力，不在速度。

## ④ 使用方法与最佳实践

### 4.1 真值惯用法：别和 `None`/`len` 较劲

```python
users = get_users()       # 返回 list

# ✅ Pythonic
if not users:             # 空列表为假
    return "no users"
if users:                 # 非空为真
    process(users)

# ❌ Java 味（啰嗦且有时不安全）
if len(users) == 0: ...
if users != None: ...     # 还得另判空
```

**但注意区分"空"和"None"两种语义**。如果 `None`（没传/没查到）和空列表（查到但为空）要区别对待，就必须显式判 `None`：

```python
def render(items=None):
    if items is None:          # 明确：参数没传
        items = load_default()
    if not items:              # 明确：传了但是空的
        return "empty"
    return ", ".join(items)
```

判 `None` 永远用 `is None`/`is not None`，不用 `== None`（`is` 是身份比较，更快也更准，见 [03 变量与对象模型](03_变量与对象模型.md)）。

### 4.2 用 `enumerate` 取代 `range(len())`

需要下标时，不要回到 Java 的 `for(int i=0; i<n; i++)`：

```python
rows = ["alice", "bob", "carol"]

# ✅ enumerate，可指定起始值
for i, name in enumerate(rows, start=1):
    print(f"{i}. {name}")
# → 1. alice / 2. bob / 3. carol

# ❌ Java 味
for i in range(len(rows)):
    print(f"{i+1}. {rows[i]}")     # 反复 rows[i] 索引，啰嗦易错
```

### 4.3 用 `zip` 并行遍历多个序列

```python
ids   = [101, 102, 103]
names = ["alice", "bob", "carol"]
scores = [90, 85, 88]

for uid, name, score in zip(ids, names, scores):
    print(f"#{uid} {name}: {score}")
# → #101 alice: 90 ...

# zip 在最短序列耗尽时停止；要求等长可用 strict（3.10+）
for a, b in zip([1, 2, 3], [10, 20], strict=False):  # 默认行为：到 [10,20] 用完就停
    ...
# zip(..., strict=True) 长度不等会抛 ValueError —— 适合"本应等长"的数据校验
# 构造 dict 的经典写法：
mapping = dict(zip(names, scores))    # → {'alice': 90, 'bob': 85, 'carol': 88}
```

### 4.4 四种推导式速查

```python
nums = [1, 2, 3, 4, 5, 6]

# list 推导式：转换 + 过滤
evens_sq = [n * n for n in nums if n % 2 == 0]     # → [4, 16, 36]

# set 推导式：去重
parities = {n % 2 for n in nums}                   # → {0, 1}

# dict 推导式：建映射
square_map = {n: n * n for n in nums}              # → {1: 1, 2: 4, ...}

# 生成器表达式：惰性，喂给聚合函数（注意没有外层方括号）
total = sum(n * n for n in nums)                   # → 91，不建中间列表

# 嵌套推导式（= 嵌套 for，从左到右就是循环嵌套顺序）
matrix = [[1, 2], [3, 4], [5, 6]]
flat = [x for row in matrix for x in row]          # → [1, 2, 3, 4, 5, 6]
#       ↑收集  ↑外层for      ↑内层for
```

### 4.5 推导式 vs `map`/`filter`

```python
# 三种等价写法，求偶数的平方
data = range(10)
a = [x * x for x in data if x % 2 == 0]                       # ✅ 首选：推导式
b = list(map(lambda x: x * x, filter(lambda x: x % 2 == 0, data)))  # ❌ 啰嗦
import operator
# map/filter 在"函数已经现成"时才划算：
names = [" Alice ", " Bob "]
clean = list(map(str.strip, names))                          # ✅ map 直接传内置方法，干净
```

经验法则：**有转换逻辑/过滤条件 → 推导式**；**只是把一个现成函数套到每个元素上 → `map(func, it)` 也很干净**。要写 `lambda` 的 `map`/`filter` 基本都该改成推导式。

### 4.6 生成器表达式省内存：处理大数据流

```python
# 统计一个大日志文件里 ERROR 行数 —— 不要一次读进内存
with open("app.log", encoding="utf-8") as f:
    error_count = sum(1 for line in f if "ERROR" in line)   # 逐行流式，内存恒定

# 处理百万级记录求平均，避免建中间列表
def avg_latency(records):                # records 可能是个大迭代器
    total = n = 0
    for lat in (r["latency_ms"] for r in records if r["ok"]):
        total += lat
        n += 1
    return total / n if n else 0.0
```

凡是结果要"喂给 `sum`/`max`/`any`/`all`/`join`/`for`"而不需要保留整个列表的场景，用生成器表达式而非 list 推导式，内存从 O(n) 降到 O(1)。

### 4.7 海象运算符的合理场景

```python
import re

# 场景一：避免重复计算 / 重复调用
data = [1, 2, 3, 4, 5, 6, 7, 8]
result = [y for x in data if (y := x * x) > 10]    # y 只算一次，又能在条件和结果里用
# → [16, 25, 36, 49, 64]

# 场景二：while 读取循环（终于能写 Java 那种 while((n=read())!=...) 了）
def read_chunks(stream, size=4096):
    while chunk := stream.read(size):     # 读到空（falsy）就停
        yield chunk

# 场景三：正则匹配后立刻用
line = "status=200 latency=35ms"
if (m := re.search(r"status=(\d+)", line)):
    print("status:", m.group(1))          # → status: 200
```

别滥用：海象是为"消除重复计算/精简控制流"准备的，把整段逻辑硬塞进一行只会降低可读性。

### 4.8 match：解构序列 / 字典 / 类（3.10+）

`match` 的主场是处理"形状已知"的数据。下面用一个后端常见场景——分发不同类型的事件消息：

```python
def handle_event(event: dict) -> str:
    match event:
        # 映射模式：匹配含特定键的 dict，并解构出值
        case {"type": "click", "x": x, "y": y}:
            return f"click at ({x}, {y})"
        case {"type": "key", "code": code}:
            return f"key {code}"
        # 带守卫（guard）：模式匹配后再加 if 条件
        case {"type": "scroll", "delta": d} if d > 0:
            return "scroll up"
        case {"type": "scroll"}:
            return "scroll down"
        # 通配：_ 匹配任何东西且不绑定
        case _:
            return "unknown event"

print(handle_event({"type": "click", "x": 10, "y": 20}))   # → click at (10, 20)
print(handle_event({"type": "scroll", "delta": 3}))        # → scroll up
print(handle_event({"type": "wat"}))                       # → unknown event
```

序列解构与类解构：

```python
from dataclasses import dataclass

# 序列模式：* 收集剩余
def describe(cmd: list) -> str:
    match cmd:
        case []:
            return "empty"
        case [single]:
            return f"one arg: {single}"
        case ["move", x, y]:
            return f"move to {x},{y}"
        case [action, *rest]:               # *rest 吃掉剩下所有
            return f"{action} with {len(rest)} args"

print(describe(["move", 3, 4]))             # → move to 3,4
print(describe(["draw", "a", "b", "c"]))    # → draw with 3 args

@dataclass
class Point:
    x: int
    y: int

def quadrant(p):
    match p:
        # 类模式：位置参数按 __match_args__（dataclass 自动生成）匹配
        case Point(0, 0):
            return "origin"
        case Point(x=px, y=py) if px > 0 and py > 0:
            return "Q1"
        case Point():                       # 是 Point 但不满足上面
            return "elsewhere"
        case _:
            return "not a point"

print(quadrant(Point(0, 0)))                # → origin
print(quadrant(Point(3, 5)))                # → Q1
```

何时用 `match`：解构嵌套数据（JSON 消息、AST 节点、协议帧）、按"形状"分发。何时**别**用：简单的值相等分发（`if/elif` 或 dict 派发更清楚）、只有两三个分支。`match` 不是性能优化（它是顺序尝试，不是跳表），它的价值是解构表达力。

### 4.9 for-else：找元素的兜底

```python
def find_admin(users):
    for u in users:
        if u["role"] == "admin":
            print("found admin:", u["name"])
            break
    else:                          # 读作 "no break"：循环没被 break 打断才执行
        print("no admin found")    # 即一个都没找到的兜底

find_admin([{"name": "a", "role": "user"}])    # → no admin found
```

`while-else` 同理：`while` 条件自然变假退出（非 `break`）才走 `else`。**坦白说**：`for-else` 可读性争议大，很多团队禁用它。更清晰的替代常常是把搜索抽成函数用 `return`，或用生成器 + `next(..., default)`：

```python
admin = next((u for u in users if u["role"] == "admin"), None)
if admin is None:
    print("no admin found")
```

## ⑤ ⚠️ 易错点（Java 思维翻车）

| Java 习惯 | 翻车现场 | 正确做法 / 认知 |
|----------|---------|----------------|
| `if (x != null)` / `if (x == null)` | 写成 `if x != None:` | 用 `if x is not None:` / `is None`（身份比较，更快更准） |
| `if (!list.isEmpty())` | 写成 `if len(lst) > 0:` | 直接 `if lst:`（非空为真）；空判 `if not lst:` |
| `if (x)` 只对布尔成立 | `x = 0` 时 `if x:` 当成"无值"漏处理，但 0 是有效数据 | 数字/可能为 0 的值，显式 `if x is not None:` 或 `if x != 0:` |
| `for(int i=0;i<n;i++) a.get(i)` | `for i in range(len(a)): a[i]` 啰嗦易错 | 直接 `for x in a:`；要下标用 `enumerate(a)` |
| Stream 里做副作用习以为常 | 在推导式里 `[print(x) for x in xs]` / 改外部状态 | 推导式只用来"造集合"；纯副作用用普通 `for` 循环 |
| `switch(x){case FOO:...}` 字面量比较 | `match` 里 `case FOO:`（FOO 是变量），以为在比相等 | **裸名字是捕获不是比较**！见下方专题 |
| 遍历时增删（`ConcurrentModificationException` 会提醒你） | Python 边遍历边改容器，静默出错或漏元素 | 遍历副本 `for x in list(d):` 或建新容器 |
| switch 要写 `break` 防穿透 | 在 `case` 里写 `break` | `match` 不穿透，`break` 在 `case` 里是语法错误 |
| `do{}while` | 找 `do-while` 语法 | 用 `while True: ... if cond: break` |
| 三元 `c ? a : b` | 写成 `c ? a : b` 报错 | Python 是 `a if c else b`（值-条件-值） |

### 专题：match 的"名字捕获 vs 常量比较"陷阱

这是 `match` 最危险的坑，Java 老手几乎必踩。在 `match` 里，**一个裸名字（小写标识符）是"捕获模式"——它会匹配任何值并把值绑定到这个名字，而不是拿这个名字的当前值去比较**：

```python
PENDING = "pending"
ACTIVE = "active"

def status_label(status):
    match status:
        case PENDING:          # ❌ 陷阱！这不是"等于 PENDING"
            return "等待中"
        case ACTIVE:           # ❌ 永远到不了这里
            return "活跃"
        case _:
            return "未知"

print(status_label("active"))   # → 等待中  ？！
```

为什么？`case PENDING:` 里的 `PENDING` 被当成**捕获变量**：它匹配*任意* `status`，并把 `status` 的值重新绑定给局部 `PENDING`。所以第一个 case 永远命中，后面是死代码（Python 其实会对这种"不可达 case"发警告/错误）。

**正确写法**：要和常量比较，必须用"带点的名字"（属性访问形式），因为带点的名字是**值模式**（取该名字的值来比较），不是捕获：

```python
from enum import Enum

class Status(Enum):
    PENDING = "pending"
    ACTIVE = "active"

def status_label(status: Status):
    match status:
        case Status.PENDING:      # ✅ 带点 → 值模式，真的在比较
            return "等待中"
        case Status.ACTIVE:       # ✅
            return "活跃"
        case _:
            return "未知"

print(status_label(Status.ACTIVE))   # → 活跃
```

记忆口诀：**`match` 里裸小写名 = 捕获（绑定），带点名/字面量 = 比较**。要按常量分发，把常量放进 `enum` 或用带点的限定名；否则就老实用 `if/elif x == PENDING`。

### 专题：迭代中修改容器

```python
# ❌ 边遍历边删 dict，行为未定义 / RuntimeError
counts = {"a": 0, "b": 3, "c": 0}
for k, v in counts.items():
    if v == 0:
        del counts[k]            # RuntimeError: dictionary changed size during iteration

# ✅ 遍历副本，或推导式建新容器
counts = {k: v for k, v in counts.items() if v != 0}     # 推导式过滤，最 Pythonic
# 或：for k in list(counts):  if counts[k]==0: del counts[k]
```

Java 的 `ConcurrentModificationException` 会在你犯错时立刻抛出；Python 对 list 边遍历边删则可能**静默跳过元素**（不报错但结果错），更阴险：

```python
nums = [1, 2, 2, 3, 2]
for n in nums:
    if n == 2:
        nums.remove(n)           # ❌ 静默漏删！下标随删除移动
print(nums)                      # → [1, 3, 2]   还剩一个 2，没删干净
# ✅ nums = [n for n in nums if n != 2]
```

events = [

    {"kind": "weird"},
]

### 参考答案

**1.**
```
bool(0)       → False   (__bool__: int 0 为假)
bool(0.0)     → False   (__bool__: float 0.0 为假)
bool("")      → False   (__len__ == 0)
bool("False") → True    (__len__ == 5，非空字符串！内容是什么无关)
bool([])      → False   (__len__ == 0)
bool([[]])    → True    (__len__ == 1，含一个元素[空列表]，外层非空即真)
bool({})      → False   (__len__ == 0)
bool(None)    → False   (__bool__: None 恒为假)
bool(-1)      → True    (__bool__: 非零即真)
bool(range(0))→ False   (__len__ == 0，空 range)
```
数字和 `None`/`bool` 走 `__bool__`；容器（str/list/dict/set/range）走 `__len__`。陷阱：`"False"`、`"0"`、`[[]]`、`[0]` 都是**真**，因为它们非空。

**2.** Python 3 把推导式的循环体编译成一个**独立作用域**（3.11 及之前用一个隐藏的嵌套函数 + 独立栈帧实现，循环变量是该函数的局部变量，跑完即销毁，所以不泄漏到外层）。PEP 709（3.12）把 list/dict/set 推导式**内联**进外层函数，去掉了每次创建隐藏函数和栈帧的运行时开销（快约一倍），但通过"进入前保存、退出后恢复同名变量"的方式**保留了作用域隔离语义**。所以它改的是**实现**（性能），不是**语义**（`x` 照样不泄漏）。注意生成器表达式未被内联，仍是独立帧。

**3.** 问题与修复：
- `for i in range(len(records)): r = records[i]` —— Java 味下标遍历，改 `for r in records:`。
- `r["score"] != None` —— 改 `r["score"] is not None`。
- `len(r["name"]) > 0` —— 改 `if r["name"]:`（非空字符串为真）。
- `len(result) == 0` —— 改 `if not result:`。
- 三层嵌套 `if` 可合并；整体可用推导式。Pythonic 版：
```python
def summarize(records):
    result = [
        r["name"]
        for r in records
        if r["score"] is not None and r["name"] and r["score"] >= 60
    ]
    return result if result else "no one passed"
```
（注：若 `score` 可能为 `0` 且要参与判断，`is not None` 正确区分了"没分"和"0 分"，这正是不能用 `if r["score"]:` 的原因。）

**4.** 根因：`case GET:` / `case POST:` 里的 `GET`、`POST` 是**裸名字**，在 `match` 中被当作**捕获模式**——`case GET:` 匹配*任意* `method` 并把它绑定给局部变量 `GET`，所以第一个 case 永远命中，返回 `"read"`，后面是死代码。修复：用带点的限定名（值模式）才是比较。用 enum 最清晰：
```python
from enum import Enum
class Method(Enum):
    GET = "GET"
    POST = "POST"

def dispatch(method):
    match method:
        case Method.GET:    # 带点 → 值模式（真比较）
            return "read"
        case Method.POST:
            return "write"
        case _:
            return "other"
print(dispatch(Method.POST))   # → write
```
若坚持用字符串，直接 `if/elif method == "GET"` 反而更合适——这种纯值分发本就不该用 `match`。

**5.** 参考实现：
```python
def analyze(events):
    # 生成器表达式：逐个解构、过滤，不建中间大列表
    def parsed():
        for e in events:
            match e:
                case {"level": ("error" | "warn") as level, "kind": kind} as ev:
                    # 按 kind 进一步解构出 message
                    match ev:
                        case {"kind": "http", "status": s, "path": p}:
                            msg = f"HTTP {s} {p}"
                        case {"kind": "db", "ms": ms} if ms > 1000:
                            msg = f"slow query {ms}ms"
                        case {"kind": "db"}:
                            msg = "db event"
                        case {"kind": "task", "name": n, "reason": reason}:
                            msg = f"task {n} failed: {reason}"
                        case _:
                            msg = "(no detail)"
                    yield (kind, msg, level)
                case _:
                    # 缺 level / 形状不符（如 {"kind":"weird"}）直接跳过
                    continue

    pairs = []
    error_count = 0
    for kind, msg, level in parsed():     # 单次流式遍历
        pairs.append((kind, msg))
        if level == "error":              # 也可写 error_count += level == "error"
            error_count += 1
    return pairs, error_count


events = [
    {"kind": "http", "level": "error", "status": 500, "path": "/api/x"},
    {"kind": "http", "level": "info",  "status": 200, "path": "/health"},
    {"kind": "db",   "level": "warn",  "query": "SELECT ...", "ms": 1200},
    {"kind": "task", "level": "error", "name": "sync", "reason": "timeout"},
    {"kind": "weird"},
]
print(analyze(events))
# → ([('http', 'HTTP 500 /api/x'), ('db', 'slow query 1200ms'),
#     ('task', 'task sync failed: timeout')], 2)
```
要点：①用 `("error" | "warn")` **或模式** + `as level` 同时过滤并绑定 level；②映射模式 `{"kind": "http", "status": s, ...}` 解构关心字段；③守卫 `if ms > 1000` 区分慢查询；④通配 `case _` 兜掉 `{"kind":"weird"}` 这种缺字段的；⑤`parsed()` 是生成器，`for` 单次流式消费，不堆叠多份中间列表，内存友好。
