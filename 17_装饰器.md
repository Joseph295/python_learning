# 17 · 装饰器 ⭐

> 你在 Spring 里写 `@Transactional`、`@Cacheable`、`@RestController` 写得飞起，第一眼看到 Python 的 `@app.get("/")`、`@property`、`@lru_cache` 会本能地想："哦，这是注解。" **这是本章要纠正的最大误会。** Java 注解是贴在代码上的一张静止标签（元数据），靠反射在运行时被人读取；Python 装饰器是一个**会立刻执行、当场把你的函数/类换成另一个对象**的高阶函数。`@deco` 只是 `func = deco(func)` 的语法糖——一行赋值，定义时就跑完了。看懂这个等式，property、lru_cache、FastAPI 路由、retry、计时器，全都通了。

---

## ① Java 对照表

| 维度 | Java 注解 / AOP | Python 装饰器 |
|------|----------------|---------------|
| 本质 | **元数据**，贴在元素上的静态标签 | **高阶函数**，对函数/类做立即变换 |
| 何时"生效" | 编译期保留 + **运行时靠反射读取**才有意义 | **定义时立即执行**：`def` 一跑完就 `f = deco(f)` |
| 谁来解释它 | 框架/容器（Spring、JUnit）用反射扫描并织入 | 解释器，无需框架——语言级机制 |
| 改变原方法吗 | **不改**字节码（标准注解只是标记），由代理/织入间接增强 | **直接替换**：原函数对象被包装函数取代 |
| 织入方式 | 动态代理 / CGLIB / AspectJ 编译期织入 | 闭包包裹（wrapper 闭包持有原函数引用） |
| 带参数 | `@RequestMapping(value="/x", method=GET)` | `@retry(times=3)`——**多一层函数调用**返回真正的装饰器 |
| 堆叠 | 多个注解无明确顺序语义 | `@a` `@b` 有严格顺序：`a(b(f))`，自底向上 |
| 典型代表 | `@Transactional` `@Cacheable` `@Override` | `@property` `@lru_cache` `@staticmethod` `@dataclass` |
| 能拿到元信息吗 | `method.getAnnotation(...)` 反射 | 包装后默认**丢失**元信息，要 `functools.wraps` 找回 |
| 心智锚点 | "给方法贴标签，容器看标签干活" | "**`f = deco(f)`**，当场把 f 换掉" |

> 一句话定调：Java 的 `@Transactional` 是**声明意图**，真正开事务的是 Spring 的代理；Python 的 `@timer` 是**当场动手**，装饰器函数自己就把计时逻辑包进去了。Java 注解不写框架配合就什么都不发生；Python 装饰器一旦写上，定义那一刻就已经改变了世界。

## ② 设计理念（为什么 Python 这么设计）

### 2.1 装饰器是"一等函数 + 闭包"的语法糖

[12 函数](12_函数.md) 讲过 Python 函数是**一等公民**：能赋值给变量、能当参数传、能当返回值、能在函数里定义函数并捕获外层变量（闭包）。装饰器没有引入任何新机制，它**只是这两件事的组合的一个语法糖**：

```python
def deco(func):              # 接收一个函数
    def wrapper(*args, **kwargs):
        print("before")
        result = func(*args, **kwargs)   # 闭包捕获了 func
        print("after")
        return result
    return wrapper           # 返回一个新函数

def greet():
    print("hello")

greet = deco(greet)          # ← 这就是装饰器的全部真相
greet()
# 输出:
# before
# hello
# after
```

`@deco` 写在 `def greet` 上面，效果和最后那行 `greet = deco(greet)` **一模一样**：

```python
@deco
def greet():
    print("hello")
```

Python 在 2.4 加入 `@` 语法（PEP 318），动机直白：`greet = deco(greet)` 这行写在函数**定义之后、且离 def 很远**，可读性差——读者得先看完整个函数体，到末尾才发现"哦它被包了一层"。把 `@deco` 提到函数头顶，**装饰意图一眼可见**。所以 `@` 纯粹是为了可读性的糖，没有任何新语义。

### 2.2 横切关注点：比 AOP 更轻、更显式

装饰器解决的问题和 Spring AOP 一模一样：**横切关注点（cross-cutting concerns）**——日志、计时、缓存、权限校验、重试、事务，这些逻辑散落在无数业务方法里，又跟业务本身无关。你不想把 `log.info("enter")` 抄进每个方法。

Java 的答案是 AOP：定义切面（Aspect）、切点（Pointcut）、通知（Advice），由容器在运行时用动态代理把横切逻辑"织入"目标方法周围。功能强大，但**重**：你需要理解代理对象、自调用失效、织入时机、`@EnableAspectJAutoProxy` 一整套机制，而且**这一切是"隐式"发生的**——看方法源码根本看不出它被增强了。

Python 装饰器是同一思想的**轻量、显式**版本：

- **轻量**：不需要容器、不需要代理框架，语言自带。一个普通函数就是一个"切面"。
- **显式**：`@retry` 明明白白写在函数头上，谁读代码都知道这个函数被增强了，增强逻辑就在 `retry` 函数里，跳过去就能看。不像 AOP 那样"切点表达式匹配到了才生效，匹配错了静默失效"。

代价是：Python 装饰器没有 AOP 那种"用切点表达式批量匹配一组方法"的能力——你得**逐个**把 `@retry` 贴到要增强的函数上。但对绝大多数场景，这种"显式、就地"反而更符合 Python "明确优于隐晦"（[01 设计哲学与心智模型](01_设计哲学与心智模型.md)）的哲学。

### 2.3 不改源码地增强行为

装饰器的核心价值主张：**在不修改原函数源码、也不改变所有调用点的前提下，增强它的行为**（开闭原则）。

```python
# 第三方库给你的函数，你不能改它的源码
def fetch_price(symbol): ...

# 你想给它加缓存，但不想动它一行，也不想改所有调用处
fetch_price = lru_cache(fetch_price)   # 包一层，调用点照旧 fetch_price("AAPL")
```

这正是 `@property` `@classmethod` `@dataclass` 这些**语言/标准库核心特性都用装饰器实现**的原因：它们要在"定义类/函数时"对其做变换，而装饰器恰好是"定义时执行的变换器"。装饰器不是一个边角语法，它是 Python 对象模型的**主干语法**之一。

## ③ 实现细节（底层怎么实现的）★本章重点★

### 3.1 装饰发生在"定义时"，不是"调用时"——用字节码与执行顺序证明

Java 老手最容易错的直觉：以为 `@deco` 像注解一样"附着着，等调用时才起作用"。**错。装饰器在 `def` 语句执行完的那一瞬间就调用了。** 用一段会打印副作用的装饰器证明：

```python
def deco(func):
    print(f"装饰器正在执行，包装 {func.__name__}")   # 定义时就会打印
    def wrapper(*args, **kwargs):
        return func(*args, **kwargs)
    return wrapper

print("=== 开始定义函数 ===")

@deco
def task():
    print("task 运行")

print("=== 函数已定义，但还没调用 ===")
task()
print("=== task 调用完毕 ===")
# 输出:
# === 开始定义函数 ===
# 装饰器正在执行，包装 task     ← 注意！这行在"定义已完成"之前就打印了
# === 函数已定义，但还没调用 ===
# task 运行
# === task 调用完毕 ===
```

"装饰器正在执行"出现在"函数已定义"之前，**铁证：装饰发生在定义时**。这跟 Java 注解的行为根本不同——Java 注解贴上去，没人反射读它就永远不发生任何事。

从字节码看更清楚。`@deco` 编译出来的指令和手写 `task = deco(task)` 几乎一致：

```python
import dis

source = """
@deco
def task():
    pass
"""
dis.dis(compile(source, "<demo>", "exec"))
# 关键指令（简化）:
#   MAKE_FUNCTION              # 先把 def task 的函数体做成函数对象
#   LOAD_NAME    deco          # 加载装饰器
#   ...
#   CALL                       # 调用 deco(原函数)   ← 立即调用
#   STORE_NAME   task          # 把返回值存回名字 task
```

可以看到流程是：**先 `MAKE_FUNCTION` 造出原始函数 → 立刻 `CALL` 装饰器 → `STORE_NAME` 把结果绑回同一个名字**。所谓"装饰"，就是在 `def` 这条语句里多插了一次函数调用和一次重新绑定。模块导入时这段就跑了，与你日后是否调用 `task` 无关。

### 3.2 闭包如何保存被包装函数与配置（`__closure__`，呼应 12 章）

`wrapper` 凭什么能在以后调用时还记得 `func`？靠**闭包**（[12 函数](12_函数.md)）。`func` 是外层函数 `deco` 的局部变量，`wrapper` 引用了它，于是 `func` 被"关进"了 `wrapper` 的闭包里，存在 `wrapper.__closure__` 的 cell 中：

```python
def deco(func):
    def wrapper(*args, **kwargs):
        return func(*args, **kwargs)
    return wrapper

def hello(): ...
wrapped = deco(hello)

print(wrapped.__closure__)                       # (<cell at 0x...: function object>,)
print(wrapped.__closure__[0].cell_contents)      # <function hello at 0x...>
print(wrapped.__closure__[0].cell_contents is hello)   # True
```

`wrapper.__closure__[0].cell_contents` 就是被包装的原函数 `hello`。装饰器**没有任何魔法**，就是一个闭包变量。带配置的装饰器同理——配置参数也被关进闭包，随包装函数一起活着（见 3.4）。这也是和 Java 的体感分水岭：Spring 把增强逻辑放在**独立的代理对象**里；Python 把它放在**闭包**里，原函数和增强逻辑被同一个 `wrapper` 对象绑在一起。

### 3.3 `functools.wraps`：不加它，元信息全丢

包装之后，外界看到的是 `wrapper`，不再是原函数。于是原函数的 `__name__`、`__doc__`、`__module__`、类型注解、签名**全部被 `wrapper` 的盖掉**：

```python
def deco(func):
    def wrapper(*args, **kwargs):
        """wrapper 的文档"""
        return func(*args, **kwargs)
    return wrapper

@deco
def compute_tax(amount: float) -> float:
    """计算税额"""
    return amount * 0.06

print(compute_tax.__name__)   # wrapper          ← 名字没了！
print(compute_tax.__doc__)    # wrapper 的文档    ← 文档没了！
```

后果很现实：日志里函数名全打成 `wrapper`；`help(compute_tax)` 看到的是错的；调试器、`inspect.signature`、Sphinx 文档、pytest 报错全失真。在 Java 里你绝不会遇到这种事——代理对象再怎么包，反射 `method.getName()` 拿到的还是原方法名。

解药是 `functools.wraps`，它本身是个装饰器，**把原函数的元信息复制到 wrapper 上**：

```python
import functools

def deco(func):
    @functools.wraps(func)          # ← 关键
    def wrapper(*args, **kwargs):
        return func(*args, **kwargs)
    return wrapper

@deco
def compute_tax(amount: float) -> float:
    """计算税额"""
    return amount * 0.06

print(compute_tax.__name__)      # compute_tax   ✓
print(compute_tax.__doc__)       # 计算税额        ✓
print(compute_tax.__wrapped__)   # <function compute_tax ...>  指回原函数
```

`wraps` 干了三件事：

1. 复制 `__name__` `__qualname__` `__doc__` `__module__` `__dict__` 等（`WRAPPER_ASSIGNMENTS` / `WRAPPER_UPDATES`）。
2. 设置 `wrapper.__wrapped__ = func`，**留一条回到原函数的路**——`inspect.signature` 会顺着它拿到真实签名，`inspect.unwrap()` 能层层剥回原函数。
3. 因此 `inspect.signature(compute_tax)` 仍能显示 `(amount: float) -> float`，而不是 `(*args, **kwargs)`。

> 规矩：**写函数装饰器，几乎永远要加 `@functools.wraps(func)`。** 不加不会报错，但会埋下名字/文档/签名失真的隐患，这是新手装饰器最常见的低级 bug。

### 3.4 带参数的装饰器：返回装饰器的"工厂"，三层嵌套

`@retry(times=3)` 和 `@retry` 的区别藏着一个易混的层级。关键认知：**`@` 后面跟的整个表达式，其求值结果才是真正的装饰器。**

- `@retry`：`retry` 本身就是装饰器，`task = retry(task)`。
- `@retry(times=3)`：先求值 `retry(times=3)`，**它的返回值**才是装饰器，即 `task = retry(times=3)(task)`。

所以带参数的装饰器必须**比普通装饰器多一层**，是三层嵌套——一个"返回装饰器的工厂函数"：

```python
import functools

def retry(times):                       # 第①层：工厂，接收配置参数
    def decorator(func):                # 第②层：真正的装饰器，接收函数
        @functools.wraps(func)
        def wrapper(*args, **kwargs):   # 第③层：包装，接收调用参数
            for attempt in range(1, times + 1):
                try:
                    return func(*args, **kwargs)
                except Exception as e:
                    print(f"第 {attempt} 次失败: {e}")
                    if attempt == times:
                        raise
        return wrapper
    return decorator                    # 工厂返回装饰器

@retry(times=3)
def flaky():
    raise ValueError("网络抖动")

# 等价于：flaky = retry(times=3)(flaky)
```

三层各管一件事，对号入座就不会乱：

| 层 | 名字 | 接收什么 | 何时执行 | 闭包里存了什么 |
|----|------|---------|---------|----------------|
| ① | `retry` | 配置参数 `times` | 写 `@retry(times=3)` 时 | —— |
| ② | `decorator` | 被装饰函数 `func` | 紧接着对 `func` 应用时 | `times` |
| ③ | `wrapper` | 实际调用参数 `*args` | 每次调用 `flaky()` 时 | `func`、`times` |

配置参数 `times` 被关进 ② 和 ③ 的闭包里，活得和包装函数一样久。这就是 3.2 说的"闭包保存配置"。

> 兼容"带括号"和"不带括号"两种写法（`@cache` 和 `@cache()` 都能用）需要额外判断，标准做法是用 `functools.partial` 或检测第一个参数是不是 callable，进阶时再说，初学先固定一种写法。

### 3.5 堆叠：自底向上 = `a(b(f))`

多个装饰器叠在一起，**离函数最近的先生效**（自底向上应用），等价于函数式的嵌套调用：

```python
def bold(func):
    @functools.wraps(func)
    def wrapper(*a, **k):
        return "<b>" + func(*a, **k) + "</b>"
    return wrapper

def italic(func):
    @functools.wraps(func)
    def wrapper(*a, **k):
        return "<i>" + func(*a, **k) + "</i>"
    return wrapper

@bold
@italic
def text():
    return "hi"

print(text())     # <b><i>hi</i></b>
```

读法：

```python
@bold
@italic
def text(): ...

# 完全等价于：
text = bold(italic(text))
```

**应用顺序自底向上**：先 `italic(text)` 得到内层包装，再 `bold(...)` 包外面 → `bold(italic(text))`。
**运行顺序自顶向下**：调用时先进 `bold` 的 wrapper（加 `<b>`），它再调 `italic` 的 wrapper（加 `<i>`），最后到原函数。像剥洋葱，外层先碰到。

实战意义：顺序往往**有语义**。比如认证和日志：

```python
@require_auth      # 外层：先验权限
@log_call          # 内层：再记日志
def transfer(): ...
```

这样未授权请求被 `require_auth` 直接挡掉，**不会**进入 `log_call`。若把两者对调，则会先记录日志再校验权限——日志里就会出现一堆未授权的噪声。堆叠顺序写反是真实会出 bug 的地方。

### 3.6 类装饰器与基于类的装饰器（`__call__`）

"类装饰器"有两个意思，别混：

**(A) 装饰器是个函数，但作用对象是一个类**——对类做变换。`@dataclass` 就是典型：接收类、给它生成 `__init__`/`__repr__` 等，返回（同一个）类。

```python
def add_repr(cls):
    def __repr__(self):
        fields = ", ".join(f"{k}={v!r}" for k, v in self.__dict__.items())
        return f"{cls.__name__}({fields})"
    cls.__repr__ = __repr__       # 直接往类上塞方法
    return cls

@add_repr
class Point:
    def __init__(self, x, y):
        self.x, self.y = x, y

print(Point(1, 2))    # Point(x=1, y=2)
```

**(B) 装饰器本身是一个类**——用类的实例来当包装器，靠 `__call__` 让实例可调用（[13 类与对象](13_类与对象.md)）。`__init__` 接住被装饰函数，`__call__` 实现包装逻辑。适合需要**保存状态**的场景：

```python
import functools

class CountCalls:
    def __init__(self, func):
        functools.update_wrapper(self, func)   # 相当于实例版的 wraps
        self.func = func
        self.count = 0

    def __call__(self, *args, **kwargs):
        self.count += 1
        print(f"第 {self.count} 次调用 {self.func.__name__}")
        return self.func(*args, **kwargs)

@CountCalls
def ping():
    return "pong"

ping(); ping()
# 输出:
# 第 1 次调用 ping
# 第 2 次调用 ping
print(ping.count)   # 2   ← 状态存在实例属性里，比闭包里的 nonlocal 更直观
```

`@CountCalls` 等价于 `ping = CountCalls(ping)`，于是 `ping` 现在是一个 `CountCalls` **实例**；`ping()` 触发 `__call__`。基于类的写法在需要多个状态/方法、或想暴露 `ping.count` 这种附加接口时比闭包清晰（闭包要用 `nonlocal` 攒状态，读写都别扭）。

### 3.7 装饰方法：self 的传递与描述符（引用 19 章）

装饰**实例方法**时，初学最大的困惑是"self 去哪了"。其实没什么特别——`wrapper(*args, **kwargs)` 的 `args[0]` 就是 `self`，原样透传即可，**只要你用 `*args` 接收，self 自然在里面**：

```python
import functools

def log_method(func):
    @functools.wraps(func)
    def wrapper(self, *args, **kwargs):     # 显式写 self 更清楚（也可并入 *args）
        print(f"调用 {type(self).__name__}.{func.__name__}")
        return func(self, *args, **kwargs)  # self 原样传回去
    return wrapper

class Account:
    def __init__(self, balance):
        self.balance = balance

    @log_method
    def deposit(self, amount):
        self.balance += amount
        return self.balance

acc = Account(100)
print(acc.deposit(50))
# 输出:
# 调用 Account.deposit
# 150
```

为什么不用特殊处理 `self`？因为方法的本质是**描述符**（[19 描述符与元类](19_描述符与元类.md)）。装饰器在类体里执行时，`deposit` 还只是个**普通函数**（`wrapper`），装饰器照常把它换成新函数。等到 `acc.deposit` 访问时，函数对象作为描述符的 `__get__` 才把 `acc` 绑定成 `self` 自动传入——这一步发生在装饰**之后**，装饰器根本不参与，所以它对方法和普通函数一视同仁。

唯一要记住的两条规则：

- **`wrapper` 一定要用 `*args` 接住所有位置参数**，否则 `self` 会被吞掉，`func(self, ...)` 少了 self 直接报错。
- 写**带参数**装饰器装饰方法时，三层嵌套照旧，`self` 仍然落在最内层 `wrapper` 的 `*args` 里，不需要在工厂层或装饰器层操心。

> 注意：`@property`、`@staticmethod`、`@classmethod` 这三个**本身就是描述符**，不是普通函数装饰器，它们的"魔法"全在 `__get__` 里（19 章详解）。这里只需知道：它们也是用 `@` 语法、走 `f = deco(f)` 的同一套规则贴上去的。

## ④ 使用方法与最佳实践

### 4.1 四个最常用的手写装饰器

**计时器**（性能排查必备）：

```python
import functools, time

def timer(func):
    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        start = time.perf_counter()
        try:
            return func(*args, **kwargs)
        finally:
            elapsed = (time.perf_counter() - start) * 1000
            print(f"{func.__name__} 耗时 {elapsed:.2f} ms")
    return wrapper

@timer
def crunch():
    return sum(i * i for i in range(1_000_000))

crunch()   # crunch 耗时 ~30.00 ms
```

**日志**：

```python
def log_call(func):
    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        sig = ", ".join([*map(repr, args), *(f"{k}={v!r}" for k, v in kwargs.items())])
        print(f"→ {func.__name__}({sig})")
        result = func(*args, **kwargs)
        print(f"← {func.__name__} 返回 {result!r}")
        return result
    return wrapper

@log_call
def add(a, b):
    return a + b

add(2, 3)
# → add(2, 3)
# ← add 返回 5
```

**带参数的重试**（生产级，含退避，见 ⑥ 练习答案的完整版）：见 3.4 的 `retry`，实战通常加 `delay`、`backoff`、`exceptions=(...)` 指定捕获哪些异常。

**缓存**：自己别手写，直接用标准库 `functools.lru_cache`（见 4.3）。

### 4.2 务必用 `wraps`，并保留类型注解（ParamSpec 简述）

`@functools.wraps(func)` 是肌肉记忆，前面强调过不再赘述。这里补一个**类型注解**的坑：如果你的项目用 mypy/pyright 做静态检查（[14 类型系统与注解](14_类型系统与注解.md)），一个写法粗糙的装饰器会把被装饰函数的签名"抹平"成 `(*args, **kwargs)`，类型检查器就再也看不出参数类型了。

Python 3.10+ 用 `ParamSpec`（PEP 612）解决：它能把"原函数的参数列表"捕获并原样转发给 wrapper，让类型检查器知道签名没变：

```python
from typing import Callable, TypeVar, ParamSpec
import functools

P = ParamSpec("P")          # 捕获参数规格
R = TypeVar("R")            # 捕获返回类型

def timer(func: Callable[P, R]) -> Callable[P, R]:
    @functools.wraps(func)
    def wrapper(*args: P.args, **kwargs: P.kwargs) -> R:
        return func(*args, **kwargs)
    return wrapper

@timer
def greet(name: str, times: int = 1) -> str:
    return (name + "! ") * times

greet(123)        # mypy 报错：Argument 1 has incompatible type "int"; expected "str"
                  # ← 注解被完整保留，静态检查照常生效
```

`Callable[P, R] -> Callable[P, R]` 明确告诉类型系统："装饰前后签名一致"。不写 ParamSpec 也能运行，但会丢失 IDE 补全和类型检查，团队项目建议补上。

### 4.3 标准库里你天天用的装饰器

| 装饰器 | 来源 | 作用 | 对应 Java |
|--------|------|------|----------|
| `@property` | 内置 | 把方法伪装成属性，支持 getter/setter（19 章） | getter/setter + Lombok `@Getter` |
| `@staticmethod` | 内置 | 静态方法，不接收 self | `static` 方法 |
| `@classmethod` | 内置 | 类方法，第一个参数是 `cls` | 静态工厂方法 |
| `@functools.lru_cache` | functools | 带容量上限的记忆化缓存（[16 迭代器与生成器](16_迭代器与生成器.md) 已讲内部） | `@Cacheable` |
| `@functools.cache` | functools | `lru_cache(maxsize=None)` 的简写（3.9+） | 无界缓存 |
| `@functools.cached_property` | functools | 惰性计算一次并缓存到实例（见下） | `@Lazy` 字段 |
| `@dataclasses.dataclass` | dataclasses | 自动生成 `__init__`/`__repr__`/`__eq__` 等 | Lombok `@Data` / record |
| `@functools.wraps` | functools | 写装饰器时复制元信息 | —— |
| `@abc.abstractmethod` | abc | 标记抽象方法 | `abstract` 方法 |
| `@functools.singledispatch` | functools | 按第一个参数类型分派（函数重载） | 方法重载 |

`@property` 三件套——把"看起来像字段访问、其实跑了方法"的逻辑藏起来：

```python
class Circle:
    def __init__(self, radius):
        self._radius = radius

    @property
    def radius(self):                 # 读：c.radius
        return self._radius

    @radius.setter
    def radius(self, value):          # 写：c.radius = 5
        if value <= 0:
            raise ValueError("半径必须为正")
        self._radius = value

    @property
    def area(self):                   # 只读派生属性
        return 3.14159 * self._radius ** 2

c = Circle(2)
print(c.area)        # 12.56636，像访问字段，其实算出来的
c.radius = 5         # 走 setter，带校验
# c.radius = -1      # ValueError: 半径必须为正
```

注意 `@radius.setter` 是 `property` 对象自带的方法返回的装饰器——又一次印证"装饰器只是普通的对象方法调用"。

`@cached_property`——第一次访问时计算，之后直接读缓存（**没有锁开销、没有 maxsize**，专为"实例上算一次就不变"的派生属性设计）：

```python
import functools

class Dataset:
    def __init__(self, rows):
        self.rows = rows

    @functools.cached_property
    def stats(self):
        print("正在计算统计量……")      # 只会打印一次
        return {"count": len(self.rows), "sum": sum(self.rows)}

d = Dataset([1, 2, 3, 4])
print(d.stats)    # 正在计算统计量…… → {'count': 4, 'sum': 10}
print(d.stats)    # {'count': 4, 'sum': 10}   ← 不再打印，直接用缓存
```

它和 `@property` 的区别：`property` 每次访问都重算；`cached_property` 算一次就把结果**写进实例的 `__dict__`**（所以要求实例没有 `__slots__` 或显式留槽）。与 `lru_cache` 的区别：`cached_property` 缓存在**实例**上（实例没了缓存就没了，不会内存泄漏），`lru_cache` 缓存在**函数**上（小心长生命周期下缓存 self 导致实例无法回收）。

### 4.4 框架里的装饰器：FastAPI 路由

Web 框架把装饰器用到了极致。FastAPI 的路由注册（[24 HTTP与Web框架](24_HTTP与Web框架.md)）：

```python
from fastapi import FastAPI

app = FastAPI()

@app.get("/users/{uid}")
async def get_user(uid: int):
    return {"uid": uid}
```

对照 Spring 的 `@GetMapping("/users/{uid}")`：**两者长得像，机制却相反**。Spring 的 `@GetMapping` 是注解（元数据），真正注册路由的是 Spring 容器启动时扫描注解、反射读取后建立映射。FastAPI 的 `@app.get(...)` 是**带参数的装饰器**：`app.get("/users/{uid}")` 立即返回一个装饰器，**当场**把 `get_user` 函数登记进 `app` 的路由表，然后原样返回该函数。没有反射、没有扫描，导入这个模块、执行到 `def get_user` 时，路由就已经注册好了。理解这点，你就明白为什么 FastAPI 必须 `import` 到含路由的模块、为什么路由"装饰即注册"。

### 4.5 何时用装饰器，何时用上下文管理器

两者都能做"前后包裹"，别用错（[15 异常与上下文管理器](15_异常与上下文管理器.md)）：

| 想包裹的是…… | 用 | 原因 |
|--------------|-----|------|
| **一个完整函数**的每次调用 | 装饰器 | 一次贴上，所有调用都被增强 |
| **某段代码块 / 某个资源的生命周期** | `with`（上下文管理器） | 作用于具体的代码区间，不是整个函数 |
| 计时整个函数 | 装饰器 `@timer` | 自动套住函数体 |
| 只计时函数里**某几行** | `with timer_block():` | 装饰器粒度太粗 |
| 给一类函数统一加重试/缓存 | 装饰器 | 横切关注点的典型形态 |
| 确保文件/连接/锁释放 | `with` | 资源的确定性释放是上下文管理器的主场 |

口诀：**装饰器套"函数"，with 套"代码块/资源"**。两者还能组合——`contextlib.contextmanager` 本身就是个装饰器，把生成器变成上下文管理器。

## ⑤ ⚠️ 易错点（Java 思维翻车）

| Java 习惯 | 翻车现场 | 正确做法 / 认知 |
|----------|---------|----------------|
| 把装饰器当注解，以为只是"贴个标签" | 以为 `@deco` 不影响函数，调用时才发现函数行为/返回值变了，甚至函数被换成了别的对象 | 牢记 `f = deco(f)`：**定义时立即替换**，装饰器返回什么，函数名就指向什么 |
| 忘了 `functools.wraps` | 日志里函数名全是 `wrapper`，`help()`/`inspect.signature`/Sphinx 全失真，pytest 报错对不上号 | 写函数装饰器**永远**加 `@functools.wraps(func)` |
| 带参 / 不带参装饰器写混 | `@retry`（少一层）当 `@retry(times=3)` 用，报 `func is not callable`；或反过来 `@retry()` 漏写参数 | 带参装饰器是**三层**嵌套（工厂→装饰器→包装）；`@retry` 直接装饰，`@retry(...)` 先调用再装饰 |
| 堆叠顺序凭感觉 | `@log` 写在 `@require_auth` 外面，未授权请求也被记了一堆日志 | 自底向上 `a(b(f))`：离函数最近的先生效，顺序有语义，认证类放外层 |
| 以为装饰器里的变量像实例字段一样可随便改 | 在 wrapper 里 `count += 1` 报 `UnboundLocalError`（闭包变量默认只读） | 用 `nonlocal count`，或干脆改用基于类的装饰器把状态放实例属性 |
| 装饰方法时 wrapper 写成 `wrapper()` 不收参数 | `self` 没地方接，调用方法直接 `TypeError: takes 0 positional arguments` | wrapper **必须** `(*args, **kwargs)`，`self` 会作为 `args[0]` 自动透传 |
| 误以为 `lru_cache` 装饰实例方法没问题 | `@lru_cache` 把 `self` 当 key 缓存，**强引用 self**，实例永远无法回收 → 内存泄漏；且要求参数可哈希 | 实例级缓存用 `@cached_property`；要带参缓存可用 `lru_cache` 但注意 self 生命周期，或用 weakref 方案 |
| 在装饰器里直接 `return func`（忘了包一层） | 增强逻辑根本没生效，函数和没装饰一样 | 装饰器必须**返回 wrapper**，不是返回原 func（除非你故意只想注册副作用） |

### 基础理解

**1.** 不运行代码，说出下面的输出，并解释为什么"装饰器执行"这行出现在那个位置：

```python
def trace(func):
    print(f"装饰 {func.__name__}")
    def wrapper(*a, **k):
        print("调用前")
        return func(*a, **k)
    return wrapper

print("A")
@trace
def job():
    print("job 体")
print("B")
job()
```

**2.** `@functools.wraps(func)` 到底复制了哪些东西？如果不加，下面三件事分别会怎样：`logging` 打印的函数名、`help(f)` 的输出、`inspect.signature(f)` 的结果？另外 `f.__wrapped__` 是用来干什么的？

### 找坑改错

**3.** 一位 Java 同事写了下面这个"带参数的权限装饰器"，他坚持说"和 Spring 的 `@PreAuthorize` 一样用"。运行 `@require_role("admin")` 装饰的函数时直接报错。指出**两处**问题并改对：

import functools

def require_role(role):

**4.** 下面的方法计数器，作者想统计每个实例各自调了多少次 `process`，但运行后 `count` 永远是 1（或报错）。找出根因并修复：

```python
def counter(func):
    count = 0
    def wrapper(self, *args, **kwargs):
        count += 1                          # 想累加
        print(f"第 {count} 次")
        return func(self, *args, **kwargs)
    return wrapper

class Worker:
    @counter
    def process(self, item):
        return item.upper()
```

### 综合实战

**5.** 写一个生产级的 `retry` 装饰器，满足：
- 参数：`times`（最大尝试次数）、`delay`（初始等待秒数）、`backoff`（每次失败后 delay 乘以的倍数）、`exceptions`（只捕获这些异常类型，默认 `Exception`）。
- 行为：失败后等待 `delay` 秒再试，每次失败 `delay *= backoff`（指数退避）；用尽次数后抛出最后一次的异常。
- 必须保留被装饰函数的元信息和类型注解（用 `ParamSpec`）。
- 写一个会前两次失败、第三次成功的示例函数验证它。

### 参考答案

**1.** 输出：

```
A
装饰 job          ← 装饰器在 def job 执行完那一刻立即调用，发生在 print("B") 之前
B
调用前
job 体
```

`@trace` 等价于 `job = trace(job)`，`trace` 在**定义 `job` 时就被调用**（不是等到 `job()` 才调），所以 "装饰 job" 紧跟在 "A" 之后、"B" 之前打印。这正是装饰器与 Java 注解的本质区别——注解贴上不读就什么都不发生，装饰器定义时当场执行。

**2.** `@functools.wraps(func)` 把原函数的 `__name__`、`__qualname__`、`__doc__`、`__module__`、`__annotations__`、`__dict__` 复制到 wrapper，并设置 `wrapper.__wrapped__ = func`。不加的话：
- `logging` 用 `func.__name__` 会打印成 `"wrapper"`，所有被装饰函数日志名字撞一起。
- `help(f)` 显示 wrapper 的（通常为空的）文档和 `(*args, **kwargs)` 签名，看不到真实信息。
- `inspect.signature(f)` 返回 `(*args, **kwargs)` 而非原始参数列表（除非靠 `__wrapped__` 找回）。
`f.__wrapped__` 指回原始未包装的函数，`inspect.signature` 会顺着它取真实签名，`inspect.unwrap(f)` 能层层剥离回最里层的原函数——这是调试和内省装饰链的关键。

**3.** 两处问题：
- **第一处**：`@functools.wraps(func)` 里的 `func` 此刻还不存在。带参数装饰器是三层结构，`func` 要由**中间那层 `decorator(func)`** 接收，当前少了这一层。
- **第二处**：少了 `decorator` 这一层导致整个结构错位——`require_role` 直接返回了 `wrapper`，于是 `@require_role("admin")` 实际变成 `delete_user = wrapper`，而 `wrapper` 引用的 `func` 根本没绑定。

正确写法（补上中间层）：

```python
import functools

def require_role(role):                     # ① 工厂，收 role
    def decorator(func):                    # ② 装饰器，收 func ← 之前缺这层
        @functools.wraps(func)
        def wrapper(*args, **kwargs):       # ③ 包装，收调用参数
            if current_user_role() != role:
                raise PermissionError("无权限")
            return func(*args, **kwargs)
        return wrapper
    return decorator

@require_role("admin")
def delete_user(uid):
    print(f"删除 {uid}")
```

记忆点：带参装饰器 = `工厂(role) → 装饰器(func) → 包装(*args)`，`@functools.wraps(func)` 永远在拿到 `func` 的那一层（②）里。

**4.** 根因有两个层面：
- **直接错误**：`count += 1` 在 `wrapper` 内对**外层闭包变量** `count` 赋值，Python 会把 `count` 当作 wrapper 的局部变量，读它时还没赋值 → `UnboundLocalError`。要用 `nonlocal count` 才能修改闭包变量。
- **设计错误**：即便加了 `nonlocal`，`count` 是定义在 `counter(func)` 里的、**所有实例共享同一个** wrapper 闭包，统计的是"这个方法被所有实例调用的总次数"，而非"每个实例各自的次数"。要按实例统计，应把计数存到 `self` 上。

修复（按实例统计，存到实例属性）：

```python
import functools

def counter(func):
    @functools.wraps(func)
    def wrapper(self, *args, **kwargs):
        attr = f"_{func.__name__}_count"
        n = getattr(self, attr, 0) + 1
        setattr(self, attr, n)               # 计数存在实例上，各实例独立
        print(f"第 {n} 次")
        return func(self, *args, **kwargs)
    return wrapper

class Worker:
    @counter
    def process(self, item):
        return item.upper()

w1, w2 = Worker(), Worker()
w1.process("a"); w1.process("b"); w2.process("c")
# 第 1 次 / 第 2 次 / 第 1 次   ← w2 独立计数
```

（若确实想统计"全局总调用次数"，用 `nonlocal` 或基于类的装饰器把 `count` 放实例属性 `self.count` 即可，见 3.6 的 `CountCalls`。）

**5.** 生产级 `retry`：

```python
import functools
import time
from typing import Callable, TypeVar, ParamSpec

P = ParamSpec("P")
R = TypeVar("R")

def retry(
    times: int = 3,
    delay: float = 0.5,
    backoff: float = 2.0,
    exceptions: tuple[type[Exception], ...] = (Exception,),
) -> Callable[[Callable[P, R]], Callable[P, R]]:
    """失败重试 + 指数退避。"""
    def decorator(func: Callable[P, R]) -> Callable[P, R]:
        @functools.wraps(func)
        def wrapper(*args: P.args, **kwargs: P.kwargs) -> R:
            current_delay = delay
            for attempt in range(1, times + 1):
                try:
                    return func(*args, **kwargs)
                except exceptions as e:
                    if attempt == times:                 # 最后一次：不再吞，抛出
                        print(f"{func.__name__} 第 {attempt} 次仍失败，放弃")
                        raise
                    print(f"{func.__name__} 第 {attempt} 次失败({e!r})，"
                          f"{current_delay:.1f}s 后重试")
                    time.sleep(current_delay)
                    current_delay *= backoff             # 指数退避
            raise AssertionError("unreachable")          # 让类型检查器满意
        return wrapper
    return decorator


# —— 验证：前两次失败，第三次成功 ——
_calls = 0

@retry(times=3, delay=0.1, backoff=2.0, exceptions=(ConnectionError,))
def fetch(url: str) -> str:
    global _calls
    _calls += 1
    if _calls < 3:
        raise ConnectionError("503")
    return f"{url} 的数据"

print(fetch("http://api/x"))
# fetch 第 1 次失败(ConnectionError('503'))，0.1s 后重试
# fetch 第 2 次失败(ConnectionError('503'))，0.2s 后重试
# http://api/x 的数据
```

要点回顾：①三层结构（`retry` 工厂 → `decorator` → `wrapper`）；②`exceptions` 参数让你只重试该重试的异常（如网络错误），别把 `ValueError` 这种逻辑错也盲目重试；③`current_delay *= backoff` 实现指数退避，避免雪崩；④最后一次失败 `raise` 把原异常透传，不掩盖错误；⑤`functools.wraps` + `ParamSpec` 保住元信息和类型。这就是 `tenacity` 库的最小内核——生产中直接用 `tenacity` 即可，但理解它的骨架来自这一章。
