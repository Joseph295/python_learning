# 29 · 工程化最佳实践

> 你在 Java 世界里有一整套"成年人的工程纪律"：`checkstyle` + `spotbugs` 卡门禁、`google-java-format` 统一风格、`slf4j`/`logback` 分级打日志、`@Value`/`@ConfigurationProperties` 把配置从代码里抽出来、`picocli` 写 CLI、`mvn release` + Nexus 发版、JCA 做加密。换到 Python，这些事**一件都不会少**——少的只是你对工具的熟悉度。本章把这套纪律一一映射过来，并深挖几个 Java 老手最容易"想当然"的点：**`logging` 的 Logger/Handler/Formatter 层级与传播为什么会导致重复输出或干脆不输出**、**ruff 为什么用 Rust 写就能快两个数量级**、**`pydantic-settings` 如何从环境变量校验出配置对象**、**打包发布从 `pyproject.toml` 到 wheel 上 PyPI 的完整链路**（承接 [02 环境与工具链](02_环境与工具链.md) 的 wheel 知识），以及 **为什么口令绝不能用 `random` 生成、绝不能存明文/MD5**。看完这章，你写的 Python 项目就能像你的 Java 项目一样"可以交给团队、可以上生产"。

## ① Java 对照表

| 你在 Java 里的 | Python 对应物 | 关键差异 |
|----------------|--------------|---------|
| `checkstyle`（风格）+ `spotbugs`/`PMD`（缺陷） | **`ruff`**（lint）+ `mypy`（类型，见 [14 类型系统与注解](14_类型系统与注解.md)） | ruff 一个工具收编几十个老 linter，Rust 写，快 10～100× |
| `google-java-format` / `spotless` | **`ruff format`** / `black` | 都是"零配置、不吵架"的固执格式化器；ruff format 与 black 风格兼容 |
| Maven/Gradle 多 module 标准目录 `src/main/java` | **src 布局** `src/yourpkg/` | 把包挪进 `src/`，强制"装了才能 import"，杜绝"误导入源码树" |
| `application.yml` + `@Value`/`@ConfigurationProperties` | **环境变量 + `pydantic-settings`** | 12-factor：配置走环境变量，不进代码不进仓库 |
| `slf4j` 门面 + `logback` 实现 | 标准库 **`logging`**（门面+实现一体） | `logging` 自带分级/Handler/Formatter，无需选实现 |
| `logback.xml` / `log4j2.xml` | `logging.config.dictConfig({...})` | 用 dict（或 ini）配置，不用 XML |
| `picocli` / `args4j` | **`typer`** / `click` / 标准库 `argparse` | typer 用类型注解声明参数，像 picocli 用注解 |
| `mvn deploy` → Nexus/Artifactory | **`python -m build`** → **`twine upload`** → PyPI | 先产 wheel/sdist，再上传 |
| JCA：`MessageDigest`/`SecureRandom`/`Cipher` | `hashlib` / **`secrets`** / `cryptography` | `secrets` 才是密码学安全随机（≈`SecureRandom`），`random` 不是 |
| `BCryptPasswordEncoder`（Spring Security） | `bcrypt` / `argon2-cffi` 库 | 口令哈希用专门库，别手搓 |
| `jjwt` / `nimbus-jose-jwt` | `PyJWT` (`import jwt`) | JWT 用成熟库，别手拼 base64 |
| Git 提交前的 pre-commit hook / `mvn verify` | **`pre-commit`** 框架 | 提交即跑 ruff/mypy/格式化，挡住坏代码 |

> 一句话：**Java 的工程纪律靠"一堆各管一段的工具 + XML 配置"，Python 现代实践靠"少数几个固执而极快的工具 + `pyproject.toml` 一处配置"。** 心智不变，工具换皮。

## ② 设计理念（为什么 Python 这么设计）

### 2.1 ruff：把 lint + format 收敛成一个极快的工具

Java 这边你大概率同时挂着 `checkstyle`（命名、空格、import 顺序）、`spotbugs`（空指针、资源泄漏）、`PMD`（坏味道），三个工具三套配置三遍解析源码。Python 早年也一样碎：`flake8`（风格）+ `pyflakes`（错误）+ `isort`（import 排序）+ `pydocstyle`（文档）+ `pyupgrade`（语法升级）……每个都是独立的 Python 进程，慢且配置打架。

**ruff 的设计目标就是"一个工具替掉这一堆"**：它用 Rust 重新实现了上述几十个 linter 的几百条规则，外加一个与 black 兼容的格式化器（`ruff format`）。理念有两条：

1. **收敛**：一份 `[tool.ruff]` 配置、一次源码解析、一套缓存，搞定 lint + 排序 + 格式化 + 语法现代化。
2. **快到没有借口**：慢的工具开发者就懒得在保存时跑、懒得进 CI 门禁。ruff 把整个仓库的检查压到毫秒级，于是"每次保存都检查"成为可能——这才能真正改变行为。

> 对照 Java：相当于有人把 checkstyle + spotbugs + PMD + 自动 import 整理 + 自动升级语法**合并成一个原生二进制**，还快到可以挂进编辑器实时跑。

### 2.2 12-factor：配置必须走环境变量，绝不进代码

Spring 体系里你早就习惯了"配置和代码分离"——`application.yml`、`@Value("${db.url}")`、不同 profile 对应不同环境。但 YAML 文件本身常常**还是进了 git 仓库**，于是数据库密码、API key 跟着代码一起被提交，这是事故高发区。

[12-factor app](https://12factor.net/config) 的第 3 条把这件事推到极致：**配置 = 随环境变化的一切，必须存进环境变量，而不是配置文件，更不是代码**。理由：

- **同一份构建物（artifact）跑所有环境**：dev/staging/prod 用同一个 wheel/镜像，差异只在注入的环境变量。对照 Java 你不想为每个环境打一个 jar。
- **密钥天然不进仓库**：环境变量不在代码树里，就不会被 `git add` 误提交。
- **十二要素同源**：容器编排（K8s Secret、ECS task definition）注入环境变量是标准做法，与平台无缝。

Python 这边把这套理念工具化的就是 **`pydantic-settings`**：它像 `@ConfigurationProperties` 一样把一组环境变量**绑定并校验**成一个强类型配置对象，缺了必填项启动即报错（fail-fast），而不是运行到一半才 NPE。

### 2.3 logging 分级 + 结构化优于 print

`System.out.println` 调试 Java？你不会。你用 slf4j，因为它能**分级**（DEBUG/INFO/WARN/ERROR）、能**按类配不同输出**、能**结构化**、能在生产关掉 DEBUG。

Python 新手最大的坏习惯就是用 `print` 当日志。`print` 的问题和 `System.out.println` 一模一样：

- **无级别**：线上没法只看 ERROR、关掉 DEBUG。
- **无来源**：不知道是哪个模块、哪一行、什么时间打的。
- **无路由**：不能让 ERROR 进文件+告警、INFO 进 stdout。
- **写死 stdout**：库作者用 `print` 等于绑架了调用方的输出。

标准库 `logging` 把 slf4j + logback 的能力**合二为一**内置：分级、Logger 层级、Handler 路由、Formatter 渲染。**结构化日志（每条日志是一个 JSON 对象）** 则是云原生时代的必需——ELK/Loki/CloudWatch 要的是可检索的字段，不是一行人类散文。

### 2.4 显式依赖与可复现（呼应 02 章）

工程化的底座是 [02 环境与工具链](02_环境与工具链.md) 讲过的**可复现性**：锁文件锁住整棵依赖树，CI 从锁文件还原。本章的打包发布、CLI、配置全部建立在这个底座上——一个不锁依赖的项目，谈不上任何"最佳实践"。

### 2.5 "为人类写 CLI"

picocli 的哲学是"声明式描述命令，框架负责解析、校验、生成帮助"。Python 的 `typer`（基于 `click`）把这个理念和**类型注解**结合：你写 `def main(name: str, count: int = 1)`，typer 自动把它变成一个带 `--help`、带类型校验、带错误提示的命令行程序。好的 CLI 应该像好的 API：参数自解释、错误信息友好、`--help` 即文档。

## ③ 实现细节（底层怎么实现的）★本章重点★

### 3.1 logging 体系：Logger / Handler / Formatter / Filter 的层级与传播

这是本章最容易"翻车"的地方，必须拆到底。Java 老手把 `logging` 当 slf4j 用没问题，但**传播（propagate）机制**和 logback 不完全一样，配错就会重复输出或干脆不输出。

**四个核心对象（各司其职）：**

| 对象 | 职责 | logback 对应 |
|------|------|-------------|
| `Logger` | 你调用的入口（`logger.info(...)`），按名字组织成树 | `Logger` |
| `Handler` | 决定日志**去哪**（控制台、文件、网络、邮件） | `Appender` |
| `Formatter` | 决定日志**长什么样**（时间、级别、消息模板） | `Layout`/`Encoder` |
| `Filter` | 在 Logger 或 Handler 上做**细粒度放行/拦截** | `Filter` |

**Logger 是一棵树，用名字的点号分层：**

```python
import logging

# 名字用点号分层，形成树：root → "orders" → "orders.service"
root = logging.getLogger()                 # root logger（名字是 ""）
svc = logging.getLogger("orders.service")  # 它的祖先是 "orders"，再上是 root
print(svc.parent)        # <Logger orders (WARNING)>  ← 父节点
print(svc.name)          # orders.service
```

这棵树和 Java logger 的层级一致（`com.acme.orders.Service` 继承 `com.acme.orders` 的配置）。

**★传播（propagate）：一条日志如何向上冒泡★**

一条日志记录（`LogRecord`）的处理流程是：

1. 在**某个 Logger** 上调用 `.info(msg)`，先和该 Logger 的**有效级别**比较，不够级别直接丢弃。
2. 过了级别，记录交给**该 Logger 自己挂的所有 Handler** 处理。
3. **然后沿父链向上冒泡（propagate=True，默认）**：把同一条记录交给父 Logger 的 Handler、祖父的 Handler……一直到 root。**注意：冒泡时只跑祖先的 Handler，不再检查祖先 Logger 的级别。**

```python
import logging, sys

# root 配一个 Handler 输出到 stdout
logging.basicConfig(level=logging.INFO,
                    format="%(name)s | %(levelname)s | %(message)s",
                    stream=sys.stdout)

log = logging.getLogger("orders.service")
log.info("order created")
# 输出: orders.service | INFO | order created
# ↑ "orders.service" 自己没挂 Handler，记录冒泡到 root，由 root 的 Handler 打印
```

**经典翻车①：重复输出。** 如果你给某个子 Logger **也**加了 Handler，又没关掉传播，那这条记录会被子 Handler 打一遍、再冒泡到 root 打一遍：

```python
import logging, sys

logging.basicConfig(level=logging.INFO, stream=sys.stdout,
                    format="ROOT: %(message)s")
log = logging.getLogger("orders")
log.addHandler(logging.StreamHandler(sys.stdout))   # ★给子 logger 又加了一个 Handler
log.handlers[-1].setFormatter(logging.Formatter("CHILD: %(message)s"))

log.warning("disk almost full")
# 输出:
# CHILD: disk almost full     ← orders 自己的 Handler
# ROOT: disk almost full      ← 冒泡到 root 又打一遍  ★重复！
```

修复：要么子 Logger **不要**自己加 Handler（让它冒泡到 root 统一处理），要么加了就把 `log.propagate = False` 关掉冒泡。这正是 logback 里 additivity 的对应物。

**经典翻车②：什么都不输出。** root logger 默认级别是 `WARNING`，且默认没有 Handler（直到你调用 `basicConfig` 或加一个）。如果你从不配置、直接 `logging.getLogger("x").info(...)`，INFO 低于 root 的 WARNING，且没有 Handler，于是**静默丢弃**：

```python
import logging
logging.getLogger("orders").info("nothing happens")  # 无任何输出
# 原因：从未 basicConfig，root 无 Handler 且级别 WARNING > INFO
```

**为什么库要用 `getLogger(__name__)`？**

`__name__` 在模块 `orders/service.py` 里就是字符串 `"orders.service"`。库作者用 `logging.getLogger(__name__)` 拿到的 Logger 名字**自动**与包结构对齐，于是：

- 库**只产生**日志，**绝不**自己配 Handler、不设级别、不决定去哪——把这些决策权留给**应用**（root 由谁配由 app 决定）。这等价于 slf4j 库只依赖门面、由最终应用挑 logback 配置。
- 应用可以精确控制某个库的噪音：`logging.getLogger("noisy_lib").setLevel(logging.WARNING)`，因为名字层级可寻址。

```python
# 库代码的标准姿势（mylib/client.py）
import logging
logger = logging.getLogger(__name__)   # → "mylib.client"

def fetch(url: str) -> None:
    logger.debug("fetching %s", url)   # 用 %s 占位，不要 f-string（见 ④ / ⑤）
```

**结构化日志（JSON）怎么配？**

默认 Formatter 输出人类散文。要输出 JSON（每条一个对象），自定义一个 Formatter，把 `LogRecord` 的字段序列化：

```python
import logging, json, sys, datetime

class JsonFormatter(logging.Formatter):
    def format(self, record: logging.LogRecord) -> str:
        payload = {
            "ts": datetime.datetime.fromtimestamp(
                record.created, datetime.timezone.utc).isoformat(),
            "level": record.levelname,
            "logger": record.name,
            "msg": record.getMessage(),        # 已展开 %s 参数
        }
        # 透传 extra={...} 里挂的业务字段
        if hasattr(record, "order_id"):
            payload["order_id"] = record.order_id
        if record.exc_info:                    # 异常堆栈也进 JSON
            payload["exc"] = self.formatException(record.exc_info)
        return json.dumps(payload, ensure_ascii=False)

handler = logging.StreamHandler(sys.stdout)
handler.setFormatter(JsonFormatter())
logging.basicConfig(level=logging.INFO, handlers=[handler])

log = logging.getLogger("orders.service")
log.info("order paid", extra={"order_id": "A-1001"})
# 输出: {"ts": "2026-06-28T...+00:00", "level": "INFO", "logger": "orders.service", "msg": "order paid", "order_id": "A-1001"}
```

生产里通常直接用 `python-json-logger`（`pip install python-json-logger`）或 `structlog`（`pip install structlog`）省掉手写，但原理就是上面这套：**Formatter 把 LogRecord 渲染成 JSON 字符串**。

### 3.2 ruff 为什么快：Rust + 单遍 + 缓存

`flake8` 跑一个中型仓库要几秒到几十秒，ruff 通常几十毫秒，差两个数量级。原因不是玄学：

1. **Rust 原生二进制，零解释器启动**：`flake8`/`pylint` 是 Python 写的，每次跑要起解释器、import 一堆插件模块；ruff 编译好直接执行。
2. **一次解析、多规则复用**：传统工具链是 `isort` 解析一遍、`flake8` 再解析一遍、`pyupgrade` 又解析一遍。ruff 把源码解析成 AST **一次**，几百条规则全在这一棵 AST 上跑。
3. **并行 + 文件级缓存**：多核同时检查不同文件；没改动的文件命中缓存直接跳过。
4. **基于 RustPython 的解析器**：高度优化的词法/语法分析，比 Python 的 `ast` 模块快得多。

> 心智模型：ruff 之于 flake8，正如 [02 章](02_环境与工具链.md) 里 uv 之于 pip——**用 Rust 重写 + 缓存 + 并行**，把"慢到不想跑"变成"快到可以每次保存都跑"。

### 3.3 pydantic-settings：从环境变量加载并校验

`pip install pydantic-settings`。它构建在 pydantic v2（Rust 核心，见 [26 日期时间序列化pydantic](26_日期时间序列化pydantic.md)）之上，做的事相当于 Spring 的 `@ConfigurationProperties` + JSR-303 校验：

```python
# pip install pydantic-settings
from pydantic import Field, PostgresDsn
from pydantic_settings import BaseSettings, SettingsConfigDict


class Settings(BaseSettings):
    model_config = SettingsConfigDict(
        env_prefix="APP_",        # 只认 APP_ 前缀的环境变量
        env_file=".env",          # 本地开发从 .env 读（生产用真·环境变量）
        env_file_encoding="utf-8",
    )

    # 字段名 -> 环境变量 APP_DATABASE_URL（大小写不敏感）
    database_url: PostgresDsn                      # 必填；缺了启动即报错
    port: int = 8000                               # 有默认值 -> 可选
    debug: bool = False                            # "true"/"1"/"yes" 自动转 bool
    api_key: str = Field(..., min_length=20)       # 带校验：长度不足直接报错


settings = Settings()      # ★实例化即"加载+校验"
print(settings.port, settings.debug)
```

**底层加载顺序（优先级，高 → 低）：**

1. 初始化时显式传的参数 `Settings(port=9000)`
2. **环境变量**（`APP_PORT=...`）
3. `.env` 文件
4. 字段默认值

实现机制：`BaseSettings.__init__` 时，pydantic-settings 收集所有字段名，逐个去环境（`os.environ`）和 `.env` 里按 `env_prefix + 字段名` 查找，拿到的是**字符串**，再交给 pydantic 的**类型强制（coercion）+ 校验器**把 `"8000"` 转成 `int(8000)`、把 `"true"` 转成 `True`、把不合法的 URL 直接抛 `ValidationError`。**这就是 fail-fast**：配置错了，进程在启动的第一秒就崩，而不是在某个深夜的请求里崩。

```python
# 演示 fail-fast（环境里没设 APP_DATABASE_URL）
# settings = Settings()
# → pydantic_core.ValidationError: 1 validation error for Settings
#   database_url  Field required [type=missing, ...]
```

对照 Java：这正是 `@Validated @ConfigurationProperties` 在应用启动时校验 `application.yml` 的行为——只不过 Python 这边数据源默认是**环境变量**而非 YAML。

### 3.4 打包流程：pyproject build backend → wheel/sdist → twine 上 PyPI

承接 [02 章 3.3](02_环境与工具链.md) 讲过的 wheel/sdist 格式，这里讲**怎么生产并发布**它们。整条链路对标 Maven 的 `package` + `deploy`：

```
pyproject.toml          # 声明元数据 + 构建后端（build backend）
   │  python -m build   # 调用后端，产出两件分发物
   ▼
dist/
 ├── yourpkg-0.1.0-py3-none-any.whl   # wheel：装好即用的 zip（对照已编译 jar）
 └── yourpkg-0.1.0.tar.gz             # sdist：源码归档（对照 source jar）
   │  twine upload dist/*             # 上传到 PyPI（对照 mvn deploy 到 Nexus）
   ▼
PyPI（pypi.org）          # 公共仓库；别人 pip install yourpkg 就能装
```

**关键概念：构建后端（build backend）。** `pyproject.toml` 的 `[build-system]` 段声明用谁来构建——`hatchling`、`setuptools`、`flit`、`pdm` 等都行。`python -m build`（`pip install build`）是个**前端**，它读这段配置、在隔离环境里装好后端、调用后端的 `build_wheel`/`build_sdist` 钩子产出文件。这套"前端/后端分离"由 PEP 517/518 规定，相当于 Maven 的"`mvn` 命令 + 可插拔的打包插件"。

```toml
# pyproject.toml —— 一个可发布的最小配置
[project]
name = "orders-cli"                 # PyPI 上的包名（全局唯一）
version = "0.1.0"
description = "Order management CLI"
readme = "README.md"
requires-python = ">=3.12"
authors = [{ name = "You", email = "you@example.com" }]
dependencies = ["typer>=0.12", "pydantic-settings>=2.0"]

[project.scripts]                   # ★装完后生成的命令行入口（对照 jar 的 Main-Class）
orders = "orders_cli.main:app"      # 终端敲 `orders` 就调用 orders_cli/main.py 里的 app

[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"
```

```bash
pip install build twine             # 安装打包前端 + 上传工具
python -m build                     # 产出 dist/*.whl 和 dist/*.tar.gz
twine check dist/*                  # 校验元数据（README 渲染、字段完整性）
twine upload dist/*                 # 上传到 PyPI（先用 TestPyPI 演练：--repository testpypi）
```

> wheel 文件名 `yourpkg-0.1.0-py3-none-any.whl` 里的 `py3-none-any` 表示纯 Python、不限平台（对照 [02 章](02_环境与工具链.md) 里 numpy 的 `cp312-...-arm64`）。纯 Python 包一个 wheel 通吃所有平台；含 C 扩展才需要为每个平台分别构建。

### 3.5 secrets vs random：密码学安全随机

这是安全章最该刻进 DNA 的一条。Java 里你知道：生成 token/盐/会话 ID 要用 `SecureRandom`，**不能**用 `new Random()`，因为后者是可预测的伪随机（线性同余，给几个输出就能反推种子）。

Python 完全平行：

| | `random` 模块 | `secrets` 模块 |
|---|---|---|
| 算法 | Mersenne Twister（梅森旋转） | 操作系统 CSPRNG（`os.urandom` → `/dev/urandom`/`BCryptGenRandom`） |
| 可预测性 | **可预测**：观察到 624 个输出即可重建内部状态、预测后续 | 密码学安全，不可预测 |
| 用途 | 模拟、洗牌、采样、游戏 | token、盐、口令重置码、API key、会话 ID |
| Java 对应 | `java.util.Random` | `java.security.SecureRandom` |

```python
import secrets, random

# ❌ 错误：random 生成 token，可被预测 → 会话劫持
bad_token = "".join(random.choice("0123456789abcdef") for _ in range(32))

# ✅ 正确：secrets，操作系统级 CSPRNG
good_token = secrets.token_urlsafe(32)     # URL 安全的随机串
print(good_token)        # 例: 'kJ3v...约 43 字符'
print(secrets.token_hex(16))               # 32 位十六进制
print(secrets.token_bytes(16))             # 16 字节原始随机

# 安全比较（防时序攻击）：比对 token/HMAC 用 compare_digest，不要用 ==
import hmac
assert hmac.compare_digest(good_token, good_token)
```

`secrets` 本质是 `random.SystemRandom`（用 `os.urandom`）的便捷封装。`os.urandom(n)` 直接读操作系统的 CSPRNG，这就是 `SecureRandom` 的等价物。

### 3.6 hashlib 加盐哈希口令（对照明文/MD5 反例）

存储用户口令是事故重灾区。三条铁律：

1. **永不存明文**（被拖库直接全员裸奔）。
2. **永不用快速哈希**（MD5/SHA-1/SHA-256 单次）——它们为**速度**设计，GPU 每秒能算几十亿次，彩虹表+暴力破解分分钟。
3. **必须加盐 + 慢哈希**：每个用户一个随机盐（防彩虹表），用**故意很慢**的算法（PBKDF2/scrypt/bcrypt/argon2，提高单次成本）。

标准库 `hashlib` 提供 PBKDF2（对标 Java JCA 的 `PBKDF2WithHmacSHA256`）：

```python
import hashlib, secrets, hmac

def hash_password(password: str) -> tuple[bytes, bytes]:
    salt = secrets.token_bytes(16)                       # ★每个用户独立随机盐
    dk = hashlib.pbkdf2_hmac(
        "sha256", password.encode(), salt,
        iterations=600_000,                              # ★故意拉高迭代数，变慢
    )
    return salt, dk

def verify(password: str, salt: bytes, expected: bytes) -> bool:
    dk = hashlib.pbkdf2_hmac("sha256", password.encode(), salt, 600_000)
    return hmac.compare_digest(dk, expected)             # ★恒定时间比较

salt, stored = hash_password("hunter2")
print(verify("hunter2", salt, stored))   # True
print(verify("wrong",   salt, stored))   # False
```

**反例对照（务必认得）：**

```python
import hashlib
# ❌ 灾难级：明文
db_password = "hunter2"
# ❌ 灾难级：单次 MD5/SHA256，无盐，GPU 秒破
bad = hashlib.md5("hunter2".encode()).hexdigest()
bad2 = hashlib.sha256("hunter2".encode()).hexdigest()
```

生产**首选** `bcrypt`（`pip install bcrypt`）或 `argon2-cffi`（`pip install argon2-cffi`，argon2 是当前密码哈希竞赛冠军），它们把盐和参数**编码进结果字符串**，更难用错：

```python
# pip install bcrypt
import bcrypt
hashed = bcrypt.hashpw(b"hunter2", bcrypt.gensalt(rounds=12))  # 盐内嵌在结果里
print(hashed)                                  # b'$2b$12$....'
print(bcrypt.checkpw(b"hunter2", hashed))      # True
```

对标 Spring Security 的 `BCryptPasswordEncoder`——同一个 bcrypt 算法、同样的 `$2b$` 格式，可互相验证。

### 3.7 JWT 用成熟库

JWT 就是 `header.payload.signature` 三段 base64url，手拼**极易**出安全漏洞（最著名的 `alg: none` 绕过、用 HMAC 密钥当 RSA 公钥的混淆攻击）。用 `PyJWT`（`pip install PyJWT`，对标 `jjwt`）：

```python
# pip install PyJWT
import jwt, datetime

SECRET = "load-this-from-env-never-hardcode"   # 实际从 pydantic-settings 读

token = jwt.encode(
    {"sub": "user-42",
     "exp": datetime.datetime.now(datetime.timezone.utc)
            + datetime.timedelta(hours=1)},
    SECRET, algorithm="HS256",
)
print(token)                                   # eyJ...

decoded = jwt.decode(token, SECRET, algorithms=["HS256"])  # ★显式锁定算法白名单
print(decoded["sub"])                          # user-42
# 过期/签名错误会抛 jwt.ExpiredSignatureError / jwt.InvalidSignatureError
```

关键：`decode` 时**显式传 `algorithms=[...]` 白名单**，杜绝 `alg: none` 攻击。这和 `jjwt` 强制你指定算法是同一道防线。

### 4.4 logging 正确配置（别用 print 做日志）

应用入口处用 `dictConfig` 一次性配好（对标 `logback.xml`），之后各模块只管 `getLogger(__name__)`：

```python
# src/orders_api/logging_setup.py
import logging.config

def setup_logging(level: str = "INFO") -> None:
    logging.config.dictConfig({
        "version": 1,
        "disable_existing_loggers": False,
        "formatters": {
            "plain": {"format": "%(asctime)s | %(levelname)-7s | %(name)s | %(message)s"},
        },
        "handlers": {
            "console": {"class": "logging.StreamHandler",
                        "formatter": "plain", "stream": "ext://sys.stdout"},
        },
        "root": {"level": level, "handlers": ["console"]},   # 只在 root 配 Handler
        "loggers": {
            "noisy_lib": {"level": "WARNING"},   # 单独压某个吵闹的库
        },
    })

# 模块里：
logger = logging.getLogger(__name__)
logger.info("service started")           # ✅ 不是 print
logger.warning("retrying %s (attempt %d)", "payment", 2)   # %s 占位，惰性求值
```

要点：**只在 root 挂 Handler**，子 Logger 靠传播冒泡上来——避免 3.1 的重复输出。记异常用 `logger.exception("failed")`（在 except 块里自动带堆栈）。

### 4.8 口令与 token 速查

- **口令存储**：`bcrypt` 或 `argon2-cffi`，**绝不**明文/MD5/单次 SHA。
- **token/盐/会话 ID**：`secrets`，**绝不** `random`。
- **JWT**：`PyJWT`，`decode` 时锁 `algorithms=[...]`。
- **对称/非对称加密、证书**：`cryptography` 库（`pip install cryptography`，对标整个 JCA）。
- **比对密文/token**：`hmac.compare_digest`，**绝不** `==`（防时序攻击）。

## ⑤ ⚠️ 易错点（Java 思维翻车）

| Java 习惯 | 翻车现场 | 正确做法 / 认知 |
|----------|---------|----------------|
| 调试随手 `System.out.println`，以为 Python `print` 同理 | 用 `print` 当生产日志：无级别、无来源、无法路由、写死 stdout，线上没法关 DEBUG | 用 `logging`，模块里 `getLogger(__name__)`，入口 `dictConfig` 统一配（见 4.4） |

| logback 里 logger 自带 appender 很正常 | 给子 logger 加了 Handler 又没关 `propagate`，每条日志打两遍（重复输出） | 只在 root 配 Handler，或加了就 `logger.propagate = False`（见 3.1） |
| 从不显式配置就指望有输出（log4j 有默认配置） | 没 `basicConfig`/`dictConfig`，root 级别 WARNING 且无 Handler，`info()` 静默丢失 | 应用入口必须配一次 logging，否则 INFO/DEBUG 全丢 |
| `new Random()` 生成会话 ID 觉得够随机 | 用 `random` 生成 token/重置码，可被预测 → 会话劫持、账户接管 | token/盐/密钥一律 `secrets`（≈`SecureRandom`），见 3.5 |
| `MessageDigest.getInstance("MD5")` 存口令 | 明文或单次 MD5/SHA 存口令，无盐，拖库后 GPU 秒破 | `bcrypt`/`argon2`，加盐慢哈希；标准库退一步用 `pbkdf2_hmac` 高迭代（3.6） |
| `if (token.equals(expected))` 比对密文 | 用 `==` 比 token/HMAC，比较耗时随匹配长度变化，泄露信息（时序攻击） | `hmac.compare_digest(a, b)` 恒定时间比较 |
| 日志消息用字符串拼接 `log.info("x=" + x)` | `logger.info(f"user {expensive()}")`：f-string **总会**先求值，即便该级别被关 | 用 `%s` 占位：`logger.info("user %s", expensive())`，惰性求值，级别没开就不算 |
| 项目根直接放包，靠"能 import"就行 | 平铺布局测试碰巧能跑，打包却漏了子包/数据文件，用户装上 import 失败 | 用 src 布局 + `pip install -e .`，测的就是用户装到的（4.2） |
| 手拼 JWT / 自己 base64 | 漏判 `alg: none` 或不校验过期，认证被绕过 | 用 `PyJWT`，`decode` 锁 `algorithms` 白名单（3.7） |
| `requirements.txt` 不锁版本就发布 | 依赖漂移，"我机器能跑"，用户/CI 装到不同版本崩溃 | 锁文件锁全树（见 [02 章](02_环境与工具链.md)），CI 从锁还原 |

### 参考答案

**1.** 流程：在 `logging.getLogger("a.b.c")` 上调 `.warning(msg)` → 先与该 logger 的**有效级别**比较，过关则生成一条 `LogRecord` → 交给 `a.b.c` 自己挂的 Handler 处理 → 若 `propagate=True`（默认），同一条记录**沿父链向上冒泡**给 `a.b`、`a`、root 的 Handler 依次处理（**冒泡时不再检查祖先 logger 的级别**，只跑它们的 Handler）。每个 Handler 用自己的 Formatter 渲染、Filter 过滤。

库用 `getLogger(__name__)`：① 名字自动与包结构对齐（`a/b/c.py` → `"a.b.c"`），应用可按层级精确调每个库的级别；② 库**只产日志**，把"输出到哪、什么级别、什么格式"的决策权留给最终应用（在 root 配），等价于 slf4j 库只依赖门面、由 app 选 logback 配置。库自己加 Handler 会绑架调用方输出、还容易造成重复打印。

**2.** `random`（Mersenne Twister）是可预测的伪随机：观察到足够多输出就能重建内部状态、预测后续，**绝不能**用于安全场景；适合模拟、洗牌、随机采样。`secrets` 基于操作系统 CSPRNG（`os.urandom`），不可预测，用于 token、盐、口令重置码、会话 ID、API key。对应 Java：`random`≈`java.util.Random`，`secrets`≈`java.security.SecureRandom`。例：洗牌用 `random.shuffle(deck)`；生成会话 token 用 `secrets.token_urlsafe(32)`。

**3.** 问题与修复：

- **硬编码密钥** `API_KEY`：必须从环境变量读（`pydantic-settings` 或 `os.environ`），并把含密钥的文件 `.gitignore`。
- **`random` 生成 user_id**：若作安全标识应改 `secrets.token_hex/​token_urlsafe`；纯展示用 id 也建议用 UUID/secrets，避免可预测。
- **MD5 存口令、无盐、单次**：改 `bcrypt`/`argon2`，或退一步 `hashlib.pbkdf2_hmac("sha256", pw, salt, 600_000)` + 每用户随机盐。
- **`print` 当日志**：改 `logging.getLogger(__name__).info("registering %s id=%s", username, user_id)`，用占位符。
- **`==` 比对哈希**：改 `hmac.compare_digest(...)`，防时序攻击。

修正版要点：

```python
import secrets, hashlib, hmac, logging
from orders.config import get_settings   # API_KEY 从这里(环境变量)读

logger = logging.getLogger(__name__)

def register(username: str, password: str):
    user_id = secrets.token_hex(8)
    salt = secrets.token_bytes(16)
    pw_hash = hashlib.pbkdf2_hmac("sha256", password.encode(), salt, 600_000)
    logger.info("registering %s id=%s", username, user_id)
    save_to_db(username, salt, pw_hash, user_id)
    return user_id

def login(password: str, salt: bytes, stored_hash: bytes) -> bool:
    candidate = hashlib.pbkdf2_hmac("sha256", password.encode(), salt, 600_000)
    return hmac.compare_digest(candidate, stored_hash)
```

**4.** 原因：`basicConfig` 给 **root** 加了一个 StreamHandler；随后又给子 logger `"orders"` **额外**加了一个 Handler，且 `propagate` 默认为 `True`。于是 `logger.info(...)` 先被 `orders` 自己的 Handler 打一次，再冒泡到 root 被 root 的 Handler 打一次 → 重复。修复任选其一：① 删掉给 `orders` 单独加 Handler 的代码，只靠 `basicConfig` 的 root Handler；② 若确需 `orders` 用专属格式，则加完 Handler 后设 `logger.propagate = False`，不再冒泡到 root。
