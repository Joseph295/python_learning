# 项目一：图书管理 REST API

## 1.1 需求

做一个图书馆的图书管理服务，提供 REST API：

- `POST /books`：新增图书（校验 `title` 非空 ≤200、`author` 非空、`year` 在 1450~2100、`isbn` 唯一）。
- `GET /books/{id}`：按 id 查，不存在返回 404。
- `GET /books?author=&limit=&offset=`：列表，支持按作者过滤 + 分页。
- `PATCH /books/{id}`：部分更新。
- `DELETE /books/{id}`：删除，成功 204，不存在 404。
- `GET /healthz`：健康检查（给 K8s 探针用）。

非功能需求（这才是"工程"的部分）：分层（路由 / 服务 / 仓储）、配置走环境变量、结构化日志、有单元测试和接口测试、能用 Docker 跑起来。

## 1.2 目录结构（src 布局）

采用 [29 工程化最佳实践](29_工程化最佳实践.md) §4.2 推荐的 **src 布局**——把包放进 `src/`，强制"先 `pip install -e .` 装进环境才能 import"，杜绝"测试碰巧能 import 到源码树"的打包假象（对照 Maven 的 `src/main/java`）：

**分层的意义**（Java 老手熟得不能再熟，但要点明 Python 里的落地）：

```
HTTP 请求
   │
   ▼  routers/      ← Web 层：只管 HTTP（解析参数、校验、状态码）。对应 @RestController
   ▼  service.py    ← 服务层：业务规则、事务边界、跨仓储编排。对应 @Service
   ▼  repository.py ← 仓储层：纯 CRUD，只和 Session 打交道。对应 @Repository
   ▼  models.py     ← ORM 实体。对应 @Entity
   ▼  数据库
```

每层只依赖下一层、只通过明确的接口交互。好处和 Java 一样：**可测**（每层能单独测、下层可 mock）、**可换**（换数据库只动仓储层）、**可读**（职责单一）。下面逐文件给完整代码。

## 1.4 配置层：config.py（呼应 26/29）

```python
# src/bookapi/config.py
from functools import lru_cache

from pydantic_settings import BaseSettings, SettingsConfigDict


class Settings(BaseSettings):
    """从环境变量 / .env 读取并校验配置，对照 Spring 的 @ConfigurationProperties。"""

    # env_prefix="BOOKAPI_" → 读 BOOKAPI_DATABASE_URL 等环境变量
    model_config = SettingsConfigDict(
        env_prefix="BOOKAPI_", env_file=".env", extra="ignore"
    )

    database_url: str = "sqlite:///./books.db"   # 默认本地 sqlite 文件库
    log_level: str = "INFO"
    debug: bool = False
    sql_echo: bool = False                        # True 时打印 SQLAlchemy 生成的 SQL


@lru_cache                                        # 进程内只构造一次（配置是单例）
def get_settings() -> Settings:
    return Settings()
```

逐点讲解：

- **`BaseSettings` 从环境变量读配置并按类型校验**——这是 [26 日期时间序列化pydantic](26_日期时间序列化pydantic.md) §4.7 讲的 pydantic-settings。声明 `debug: bool = False`，环境变量 `BOOKAPI_DEBUG=true` 会被 pydantic 自动转成 `True`；`sql_echo` 同理。配置写错（比如给一个非法值）**进程启动第一秒就崩**（fail-fast），而不是深夜某个请求里崩——这正是 [29 工程化最佳实践](29_工程化最佳实践.md) §3.3 强调的好处。
- **`@lru_cache` 让 `get_settings()` 成为单例**——这是 [17 装饰器](17_装饰器.md) 的记忆化装饰器在工程里的实用落地，等价于 Spring 容器里的单例 bean。FastAPI 的 `Depends(get_settings)` 每次拿到的都是同一个对象。
- **配置从代码里抽出来、用环境变量注入**——呼应 [30 部署与可观测性](30_部署与可观测性.md) §2.4 的"镜像不可变、配置运行时注入"：同一个 Docker 镜像，开发/预发/生产只是注入不同的 `BOOKAPI_DATABASE_URL`。

## 1.5 日志层：logging_setup.py（呼应 29）

```python
# src/bookapi/logging_setup.py
import logging
from logging.config import dictConfig


def setup_logging(level: str = "INFO") -> None:
    """用 dictConfig 一次性配置 logging，对照 logback.xml。"""
    dictConfig({
        "version": 1,
        "disable_existing_loggers": False,
        "formatters": {
            # 生产建议 JSON；这里给一个带时间/级别/logger 名的人类可读格式
            "default": {
                "format": "%(asctime)s %(levelname)-7s [%(name)s] %(message)s",
            },
        },
        "handlers": {
            "console": {
                "class": "logging.StreamHandler",
                "stream": "ext://sys.stdout",   # ★日志写 stdout，容器友好（30 章）
                "formatter": "default",
            },
        },
        "root": {"level": level, "handlers": ["console"]},
        # 让 uvicorn 的访问日志也走我们的 handler，避免格式分裂
        "loggers": {
            "uvicorn.error": {"level": level},
            "uvicorn.access": {"level": "WARNING"},
        },
    })


logger = logging.getLogger("bookapi")   # 名字与包对齐，便于精确控噪（29 章 §3.1）
```

> 这里用到了 [29 工程化最佳实践](29_工程化最佳实践.md) §3.1：用 `dictConfig` 配置（对照 `logback.xml`），日志写 **stdout**（容器里由编排平台收集，[30 部署与可观测性](30_部署与可观测性.md)），用 `getLogger("bookapi")` 让 logger 名字与包结构对齐。生产里把 `formatters.default` 换成 §29 的 `JsonFormatter` 就得到结构化日志。**注意 `%(message)s` 配 `logger.info("x=%s", x)` 用 `%` 占位而非 f-string**——这是 29 章反复强调的：日志参数惰性求值，级别没到就不拼字符串。

## 1.6 持久层：db.py + models.py（呼应 25）

```python
# src/bookapi/db.py
from collections.abc import Iterator

from sqlalchemy import create_engine
from sqlalchemy.orm import Session, sessionmaker

from .config import get_settings

_settings = get_settings()

# ★Engine 是进程级单例：建一次，全程序共享它的连接池（25 章 §3.5、⑤）
# 绝不要每个请求 create_engine —— 那等于每次重建整个连接池
engine = create_engine(
    _settings.database_url,
    echo=_settings.sql_echo,
    pool_pre_ping=True,                 # 取连接前 ping，剔除被 DB 掐断的死连接（生产必开）
    # sqlite 多线程访问需要这个；Postgres/MySQL 不需要
    connect_args=(
        {"check_same_thread": False}
        if _settings.database_url.startswith("sqlite")
        else {}
    ),
)

SessionFactory = sessionmaker(bind=engine, expire_on_commit=False)


def get_session() -> Iterator[Session]:
    """FastAPI 依赖：每个请求一个 Session，用完即弃（25 章 ⑤：Session 非线程安全）。"""
    session = SessionFactory()
    try:
        yield session                   # 把 session 交给路由/服务（呼应 15 章 yield 式资源管理）
    finally:
        session.close()                 # 请求结束确定性关闭，连接还回池
```

```python
# src/bookapi/models.py
from datetime import datetime

from sqlalchemy import DateTime, Integer, String, func
from sqlalchemy.orm import DeclarativeBase, Mapped, mapped_column


class Base(DeclarativeBase):
    pass


class Book(Base):
    __tablename__ = "books"

    id: Mapped[int] = mapped_column(Integer, primary_key=True)
    title: Mapped[str] = mapped_column(String(200))
    author: Mapped[str] = mapped_column(String(120), index=True)   # 按作者查 → 建索引
    year: Mapped[int] = mapped_column(Integer)
    isbn: Mapped[str] = mapped_column(String(20), unique=True)     # 唯一约束（数据库层兜底）
    created_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True), server_default=func.now()
    )

    def __repr__(self) -> str:
        return f"Book(id={self.id!r}, title={self.title!r}, isbn={self.isbn!r})"
```

讲解：

- **`Engine` 进程级单例 + 内置连接池**——[25 数据库与ORM](25_数据库与ORM.md) §3.5 的核心：`Engine` 内置 `QueuePool`（对照 HikariCP），整个应用建一个、所有 Session 共享。`pool_pre_ping=True` 是生产标配。
- **`get_session()` 是个生成器依赖**——`yield session` 把会话交出去，`finally` 里 `close()` 保证连接确定性归还连接池。这把 [15 异常与上下文管理器](15_异常与上下文管理器.md) 的 yield 式资源管理、[16 迭代器与生成器](16_迭代器与生成器.md) 的生成器、[24 HTTP与Web框架](24_HTTP与Web框架.md) 的 `Depends` 三者串起来了——FastAPI 进入时 `next()` 拿到 session，请求结束后再驱动一次走到 `finally`。**每请求一个 Session、用完即弃**，正是 25 章 ⑤ 强调的"Session 非线程安全，绝不能跨请求共享"。
- **2.0 声明式模型**——`Mapped[int]` + `mapped_column` 是 [25 数据库与ORM](25_数据库与ORM.md) §4.2 的 SQLAlchemy 2.0 风格，类型注解既给 mypy 看也被 ORM 元类读取。`author` 上的 `index=True`、`isbn` 上的 `unique=True` 是数据库层约束，和 pydantic 的应用层校验**双保险**（应用层先挡，数据库兜底防并发竞态）。

## 1.7 校验层：schemas.py（呼应 14/26）

注意区分两套"模型"：`models.py` 是 **ORM 实体**（和数据库表对应），`schemas.py` 是 **pydantic DTO**（和 HTTP 请求/响应对应）。Java 里你也会把 `@Entity` 和 `@RequestBody DTO` 分开——**别让数据库实体直接当 API 契约**。

```python
# src/bookapi/schemas.py
from datetime import datetime

from pydantic import BaseModel, ConfigDict, Field


class BookCreate(BaseModel):
    """POST 请求体：所有约束声明在类型里，pydantic 自动校验（违反 → 422）。"""
    title: str = Field(min_length=1, max_length=200)
    author: str = Field(min_length=1, max_length=120)
    year: int = Field(ge=1450, le=2100)
    isbn: str = Field(min_length=10, max_length=20)


class BookUpdate(BaseModel):
    """PATCH 请求体：所有字段可选，只更新传了的（部分更新）。"""
    title: str | None = Field(default=None, min_length=1, max_length=200)
    author: str | None = Field(default=None, min_length=1, max_length=120)
    year: int | None = Field(default=None, ge=1450, le=2100)


class BookOut(BaseModel):
    """响应体：从 ORM 对象读取字段并序列化。"""
    # from_attributes=True 让 pydantic 能从 ORM 对象（按属性）构造，而不只是 dict（26 章）
    model_config = ConfigDict(from_attributes=True)

    id: int
    title: str
    author: str
    year: int
    isbn: str
    created_at: datetime
```

> 这里用到了 [14 类型系统与注解](14_类型系统与注解.md)（类型即 schema）和 [26 日期时间序列化pydantic](26_日期时间序列化pydantic.md)：①`Field(ge=..., le=...)` 把约束声明进类型，FastAPI 自动校验、非法返回 422、`/docs` 自动展示约束；②`BookUpdate` 全字段 `| None` 默认 `None`，实现"部分更新"语义；③`BookOut` 的 `ConfigDict(from_attributes=True)` 是关键——它让 pydantic 能直接从 SQLAlchemy 的 `Book` ORM 对象（按属性 `.id`/`.title`）构造响应模型，省掉手动转 dict。`created_at` 是带时区的 `datetime`，pydantic 序列化成 ISO 8601 字符串（26 章 §4 的日期序列化）。

## 1.8 领域异常：exceptions.py（呼应 15）

```python
# src/bookapi/exceptions.py
class DomainError(Exception):
    """业务异常基类——服务层抛它，Web 层统一翻译成 HTTP 状态码。"""


class BookNotFound(DomainError):
    def __init__(self, book_id: int) -> None:
        self.book_id = book_id
        super().__init__(f"book {book_id} not found")


class DuplicateISBN(DomainError):
    def __init__(self, isbn: str) -> None:
        self.isbn = isbn
        super().__init__(f"isbn {isbn} already exists")
```

> 这里用到了 [15 异常与上下文管理器](15_异常与上下文管理器.md) 的异常设计：**服务层只抛领域异常**（`BookNotFound`/`DuplicateISBN`），不关心 HTTP——这和 Java 里 service 抛 `BookNotFoundException`、由 `@ControllerAdvice` 翻译成 404 是同一个分层原则。Web 层（§1.11）会用 FastAPI 的 `exception_handler` 把它们映射成状态码。这样服务层可独立于 HTTP 测试。

## 1.9 仓储层：repository.py（呼应 25）

仓储层是**纯数据访问**：只和 `Session` 打交道，不含业务规则、不提交事务（事务边界归服务层）。

```python
# src/bookapi/repository.py
from collections.abc import Sequence

from sqlalchemy import select
from sqlalchemy.orm import Session

from .models import Book


class BookRepository:
    """纯 CRUD，对照 Spring Data 的 Repository（但这里手写，更透明）。"""

    def __init__(self, session: Session) -> None:
        self.session = session

    def add(self, book: Book) -> Book:
        self.session.add(book)          # 登记进工作单元，flush 时才发 INSERT（25 章 §3.3）
        self.session.flush()            # flush 拿到数据库生成的自增 id，但不提交
        return book

    def get(self, book_id: int) -> Book | None:
        return self.session.get(Book, book_id)        # 命中身份映射时不查库（25 章 §3.4）

    def get_by_isbn(self, isbn: str) -> Book | None:
        return self.session.scalars(
            select(Book).where(Book.isbn == isbn)     # 参数化 SQL，天然免疫注入（25 章 §3.7）
        ).first()

    def list(
        self, author: str | None = None, limit: int = 20, offset: int = 0
    ) -> Sequence[Book]:
        stmt = select(Book).order_by(Book.id)
        if author is not None:
            stmt = stmt.where(Book.author == author)  # 可组合的查询构造
        stmt = stmt.limit(limit).offset(offset)       # 分页
        return self.session.scalars(stmt).all()

    def delete(self, book: Book) -> None:
        self.session.delete(book)
```

> 这里用到了 [25 数据库与ORM](25_数据库与ORM.md) 的工作单元：`add()` 只登记不发 SQL，`flush()` 发出 INSERT 并回填自增 `id`（但**不 commit**，事务还没结束）；`session.get(Book, id)` 利用身份映射；`select(Book).where(Book.isbn == isbn)` 生成**参数化 SQL**（`:isbn_1` 占位符），所以 ORM 查询天然防 SQL 注入（25 章 §3.7 的描述符魔法）。仓储层不提交——**谁开启事务谁提交**，这是下面服务层的职责。

## 1.10 服务层：service.py（呼应 15/25）

服务层编排业务规则、**划定事务边界**：

```python
# src/bookapi/service.py
import logging
from collections.abc import Sequence

from .exceptions import BookNotFound, DuplicateISBN
from .models import Book
from .repository import BookRepository
from .schemas import BookCreate, BookUpdate

logger = logging.getLogger("bookapi.service")


class BookService:
    def __init__(self, repo: BookRepository) -> None:
        self.repo = repo

    def create(self, data: BookCreate) -> Book:
        # 业务规则：ISBN 不能重复（应用层先查，数据库 unique 约束兜底防并发）
        if self.repo.get_by_isbn(data.isbn) is not None:
            raise DuplicateISBN(data.isbn)
        book = Book(**data.model_dump())          # pydantic → dict → ORM 对象（26 章）
        self.repo.add(book)
        self.repo.session.commit()                # ★服务层提交事务（显式事务边界，25 章 §2.3）
        logger.info("book created id=%s isbn=%s", book.id, book.isbn)
        return book

    def get(self, book_id: int) -> Book:
        book = self.repo.get(book_id)
        if book is None:
            raise BookNotFound(book_id)           # 抛领域异常，Web 层翻译成 404
        return book

    def list(
        self, author: str | None, limit: int, offset: int
    ) -> Sequence[Book]:
        return self.repo.list(author=author, limit=limit, offset=offset)

    def update(self, book_id: int, data: BookUpdate) -> Book:
        book = self.get(book_id)                  # 复用 get（不存在自动抛 404）
        # exclude_unset=True：只取请求里实际传了的字段（部分更新的关键，26 章）
        patch = data.model_dump(exclude_unset=True)
        for field, value in patch.items():
            setattr(book, field, value)           # 触发 ORM 脏检查（25 章 §3.3）
        self.repo.session.commit()
        logger.info("book updated id=%s fields=%s", book_id, list(patch))
        return book

    def delete(self, book_id: int) -> None:
        book = self.get(book_id)
        self.repo.delete(book)
        self.repo.session.commit()
        logger.info("book deleted id=%s", book_id)
```

讲解：

- **事务边界在服务层显式提交**——[25 数据库与ORM](25_数据库与ORM.md) §2.3 强调的"事务边界更显式"：一个业务操作 = 一个 `commit()`。这比 Spring 的 `@Transactional` 注解更显式（你能一眼看到事务在哪结束）。若 `commit()` 前抛异常，`get_session()` 依赖的 `session.close()` 会触发回滚（未提交的改动丢弃）。
- **`data.model_dump()` 把 pydantic 模型转成 dict** 再 `Book(**dict)` 建 ORM 对象——[26 日期时间序列化pydantic](26_日期时间序列化pydantic.md) 的 `model_dump()`。`update` 里 `model_dump(exclude_unset=True)` 只取**请求实际传了的字段**，这是实现 PATCH 部分更新的命门。
- **`setattr(book, field, value)` 触发脏检查**——[25 数据库与ORM](25_数据库与ORM.md) §3.3：改 ORM 对象属性会被 `InstrumentedAttribute` 描述符标记为脏，`commit()` 时只对脏列发 UPDATE。这就是 [19 描述符与元类](19_描述符与元类.md) 的描述符机制在框架里的真实落地。
- **抛领域异常而非 `HTTPException`**——服务层不依赖 FastAPI，纯业务，便于 §1.13 单测时直接断言 `pytest.raises(BookNotFound)`。

## 1.11 装配层：dependencies.py + main.py（呼应 24/15）

`dependencies.py` 把"会话 → 仓储 → 服务"用依赖注入串起来：

```python
# src/bookapi/dependencies.py
from typing import Annotated

from fastapi import Depends
from sqlalchemy.orm import Session

from .db import get_session
from .repository import BookRepository
from .service import BookService


def get_repository(
    session: Annotated[Session, Depends(get_session)],
) -> BookRepository:
    return BookRepository(session)


def get_service(
    repo: Annotated[BookRepository, Depends(get_repository)],
) -> BookService:
    return BookService(repo)


# 类型别名：路由里写 ServiceDep 即可注入 BookService（FastAPI 0.95+ 的 Annotated 写法）
ServiceDep = Annotated[BookService, Depends(get_service)]
```

```python
# src/bookapi/main.py
from contextlib import asynccontextmanager

from fastapi import FastAPI, Request
from fastapi.responses import JSONResponse

from .config import get_settings
from .db import Base, engine
from .exceptions import BookNotFound, DuplicateISBN
from .logging_setup import logger, setup_logging
from .routers import books, health


@asynccontextmanager
async def lifespan(app: FastAPI):
    # 启动：配置日志、建表（生产用 Alembic 迁移，这里 demo 直接 create_all）
    settings = get_settings()
    setup_logging(settings.log_level)
    Base.metadata.create_all(engine)
    logger.info("bookapi started, db=%s", settings.database_url)
    yield
    # 关闭：清理资源（连接池等）
    engine.dispose()
    logger.info("bookapi stopped")


app = FastAPI(title="Book API", version="0.1.0", lifespan=lifespan)

app.include_router(health.router)
app.include_router(books.router)


# ── 领域异常 → HTTP 状态码的统一翻译（对照 Spring @ControllerAdvice）──
@app.exception_handler(BookNotFound)
async def handle_not_found(request: Request, exc: BookNotFound) -> JSONResponse:
    return JSONResponse(status_code=404, content={"detail": str(exc)})


@app.exception_handler(DuplicateISBN)
async def handle_duplicate(request: Request, exc: DuplicateISBN) -> JSONResponse:
    return JSONResponse(status_code=409, content={"detail": str(exc)})
```

讲解：

- **`lifespan` 异步上下文管理器管理应用生命周期**——[24 HTTP与Web框架](24_HTTP与Web框架.md) §答案3、[30 部署与可观测性](30_部署与可观测性.md) §4.6 的 lifespan 模式：启动时建表/配日志、关闭时 `engine.dispose()` 释放连接池。这是 [15 异常与上下文管理器](15_异常与上下文管理器.md) 的 `@asynccontextmanager`（async 版上下文管理器）的工程应用。
- **依赖链 service → repository → session**——FastAPI 的 `Depends` 是**函数式、请求级**的依赖注入（24 章 §3.3）。每个请求 FastAPI 递归求值：先建 session、再包成 repository、再包成 service 注进路由。和 Spring 启动时建容器不同，这是**每请求按需调用**——更轻、更好测（直接传 mock）。`Annotated[..., Depends(...)]` 是现代 FastAPI 的标准写法，类型别名 `ServiceDep` 让路由签名干净。
- **异常处理器集中翻译领域异常 → HTTP**——`@app.exception_handler(BookNotFound)` 把服务层抛的 `BookNotFound` 统一转成 404、`DuplicateISBN` 转 409，对照 Spring 的 `@ControllerAdvice` + `@ExceptionHandler`。**业务层不知道 HTTP，Web 层不知道业务细节**，干净分层。

## 1.12 路由层：routers/books.py + health.py（呼应 24/30）

```python
# src/bookapi/routers/books.py
from typing import Annotated

from fastapi import APIRouter, Query, status

from ..dependencies import ServiceDep
from ..schemas import BookCreate, BookOut, BookUpdate

router = APIRouter(prefix="/books", tags=["books"])


@router.post("", status_code=status.HTTP_201_CREATED)
def create_book(data: BookCreate, service: ServiceDep) -> BookOut:
    return service.create(data)            # 返回 ORM 对象，FastAPI 按 BookOut 序列化


@router.get("/{book_id}")
def get_book(book_id: int, service: ServiceDep) -> BookOut:
    return service.get(book_id)            # 不存在 → 服务层抛 BookNotFound → 404


@router.get("")
def list_books(
    service: ServiceDep,
    author: str | None = Query(default=None),
    limit: Annotated[int, Query(ge=1, le=100)] = 20,
    offset: Annotated[int, Query(ge=0)] = 0,
) -> list[BookOut]:
    return list(service.list(author=author, limit=limit, offset=offset))


@router.patch("/{book_id}")
def update_book(book_id: int, data: BookUpdate, service: ServiceDep) -> BookOut:
    return service.update(book_id, data)


@router.delete("/{book_id}", status_code=status.HTTP_204_NO_CONTENT)
def delete_book(book_id: int, service: ServiceDep) -> None:
    service.delete(book_id)               # 204 无 body
```

```python
# src/bookapi/routers/health.py
from fastapi import APIRouter

router = APIRouter(tags=["health"])


@router.get("/healthz")
def liveness() -> dict[str, str]:
    # liveness：只回答"进程还活着吗"，绝不依赖 DB（30 章 §4.3）
    return {"status": "ok"}
```

讲解：

- **路由层只管 HTTP**——参数来源由类型注解决定（[24 HTTP与Web框架](24_HTTP与Web框架.md) §4.4）：`book_id: int` 在路径里 → 路径参数；`author: str | None = Query(...)` → 查询参数；`data: BookCreate`（pydantic 模型）→ 请求体；`service: ServiceDep` → 依赖注入。`Query(ge=1, le=100)` 给分页参数加边界校验。
- **返回 ORM 对象，FastAPI 自动按 `BookOut` 序列化**——靠 `BookOut` 的 `from_attributes=True`（§1.7）。响应模型由返回注解 `-> BookOut` 声明，FastAPI 据此过滤字段、生成 OpenAPI 文档。
- **`@router.post` 是装饰器，导入即注册**（24 章 §3.3）——不是 Spring 那种"注解 + 反射扫描"，是导入 `books.py` 时**当场**登记进路由表，所以 `main.py` 必须 `include_router(books.router)` 把它挂上。
- **健康检查 `/healthz` 只回答存活、不碰 DB**——[30 部署与可观测性](30_部署与可观测性.md) §4.3 的铁律：liveness 探针依赖 DB 会导致 DB 抖一下 K8s 把所有 Pod 杀光重启，雪上加霜。

# 项目二：日志分析数据管道

## 2.1 需求

运维给你一个几个 GB 的 Nginx 访问日志，要你出一份分析报告：

- 各 **HTTP 状态码**的请求数分布。
- **流量最大的前 10 个 URL 路径**（按累计响应字节）。
- **请求最多的前 10 个客户端 IP**。
- 按**分钟**聚合的请求量时间序列。

约束（这才是重点）：**文件大到不能整个读进内存**，必须流式处理；日志里可能有非法编码的字节，不能因为一行坏数据就崩。这正是"数据/脚本"方向最常见的形态——一个健壮的、内存 O(1) 的离线分析脚本。

技术选型对照：在 Java 里你可能上 Spark 或写个 `BufferedReader` + 正则；Python 单机版用 **生成器流式读取 + 正则解析 + `Counter` 聚合 + pandas 汇总**，几十行搞定，内存占用与文件大小无关。

## 2.2 完整代码

```python
# log_pipeline.py —— 运行：python log_pipeline.py access.log
# 依赖：pip install pandas
import re
import sys
from collections import Counter, defaultdict
from collections.abc import Iterator
from datetime import datetime
from pathlib import Path

import pandas as pd

# ── 1. 正则：解析 Nginx combined 日志格式（10 章）────────────────────
# 形如：1.2.3.4 - - [28/Jun/2026:10:00:01 +0000] "GET /api/books HTTP/1.1" 200 1234 ...
LOG_PATTERN = re.compile(
    r'(?P<ip>\S+) \S+ \S+ '
    r'\[(?P<time>[^\]]+)\] '
    r'"(?P<method>\S+) (?P<path>\S+) [^"]*" '
    r'(?P<status>\d{3}) (?P<bytes>\d+|-)'
)


class ParsedLine:
    """一条解析后的日志（用 __slots__ 省内存，13/04 章）。"""
    __slots__ = ("ip", "minute", "path", "status", "bytes")

    def __init__(self, ip: str, minute: str, path: str, status: int, n_bytes: int):
        self.ip = ip
        self.minute = minute
        self.path = path
        self.status = status
        self.bytes = n_bytes


# ── 2. 惰性读取 + 解析：生成器逐行产出，内存 O(1)（16/22 章）──────────
def read_lines(path: Path) -> Iterator[str]:
    """逐行惰性读取，绝不一次性 read() 整个文件。"""
    # encoding + errors="replace"：遇到非法字节用 � 替换而不抛异常（09 章）
    with path.open("r", encoding="utf-8", errors="replace") as f:
        yield from f                      # 文件对象本身可迭代，逐行流式产出


def parse_lines(lines: Iterator[str]) -> Iterator[ParsedLine]:
    """把原始行解析成结构化对象；解析失败的行跳过。"""
    for line in lines:
        m = LOG_PATTERN.match(line)
        if m is None:
            continue                      # 脏数据直接跳过，不让一行坏数据搞垮整个管道
        raw_bytes = m["bytes"]
        n_bytes = 0 if raw_bytes == "-" else int(raw_bytes)
        # 把时间截断到分钟，作为时间序列的桶；解析失败也跳过
        try:
            dt = datetime.strptime(m["time"], "%d/%b/%Y:%H:%M:%S %z")
        except ValueError:
            continue
        minute = dt.strftime("%Y-%m-%d %H:%M")
        yield ParsedLine(m["ip"], minute, m["path"], int(m["status"]), n_bytes)


# ── 3. 聚合统计：Counter / defaultdict 单遍累计（06/07 章）─────────────
def aggregate(records: Iterator[ParsedLine]) -> dict:
    status_counter: Counter[int] = Counter()
    ip_counter: Counter[str] = Counter()
    bytes_by_path: defaultdict[str, int] = defaultdict(int)
    requests_by_minute: Counter[str] = Counter()
    total = 0

    for r in records:                     # 单遍遍历生成器，边读边聚合（流式，不囤数据）
        total += 1
        status_counter[r.status] += 1
        ip_counter[r.ip] += 1
        bytes_by_path[r.path] += r.bytes
        requests_by_minute[r.minute] += 1

    return {
        "total": total,
        "status": status_counter,
        "top_ips": ip_counter,
        "bytes_by_path": bytes_by_path,
        "by_minute": requests_by_minute,
    }


# ── 4. 用 pandas 汇总输出（27 章）──────────────────────────────────────
def report(stats: dict) -> None:
    print(f"总请求数: {stats['total']:,}\n")

    # 状态码分布：Counter → DataFrame
    status_df = (
        pd.Series(stats["status"], name="count")
        .sort_index()
        .rename_axis("status")
        .reset_index()
    )
    status_df["pct"] = (status_df["count"] / stats["total"] * 100).round(2)
    print("=== 状态码分布 ===")
    print(status_df.to_string(index=False))

    # Top 10 流量路径
    top_paths = pd.Series(stats["bytes_by_path"], name="bytes").nlargest(10)
    print("\n=== Top 10 流量路径（字节）===")
    print(top_paths.to_string())

    # Top 10 IP
    top_ips = pd.Series(dict(stats["top_ips"].most_common(10)), name="requests")
    print("\n=== Top 10 客户端 IP ===")
    print(top_ips.to_string())

    # 按分钟的请求量时间序列：算 QPM 峰值与均值
    minute_s = pd.Series(stats["by_minute"]).sort_index()
    print("\n=== 请求量（按分钟）===")
    print(f"峰值: {minute_s.max()} / 分钟（在 {minute_s.idxmax()}）")
    print(f"均值: {minute_s.mean():.1f} / 分钟")


def main(log_path: str) -> None:
    path = Path(log_path)                 # pathlib：OO 路径（22 章）
    if not path.exists():
        sys.exit(f"文件不存在: {path}")
    # 管道串联：读 → 解析 → 聚合 → 报告。每一段都是惰性的，数据像水流过管道
    lines = read_lines(path)
    records = parse_lines(lines)
    stats = aggregate(records)
    report(stats)


if __name__ == "__main__":
    if len(sys.argv) != 2:
        sys.exit("用法: python log_pipeline.py <access.log>")
    main(sys.argv[1])
```

## 2.3 逐段讲解：每步用到哪一章

- **正则解析（[10 正则表达式](10_正则表达式.md)）**：`LOG_PATTERN` 用**命名组** `(?P<ip>...)` 把日志行拆成字段，`m["status"]` 按名取。预编译 `re.compile(...)` 一次、复用千万次（10 章强调的编译缓存）。脏行 `match` 返回 `None`，直接 `continue` 跳过——**健壮性**就在这里。

- **惰性流式读取（[16 迭代器与生成器](16_迭代器与生成器.md)、[22 文件IO与系统交互](22_文件IO与系统交互.md)）**：`read_lines` 用 `yield from f` 逐行产出，文件对象本身可迭代且**流式**——处理 100GB 文件内存占用仍是 O(1)，而不是 O(文件大小)。这是 16 章开篇就强调的生成器杀手锏。整个管道 `read → parse → aggregate` 是一串生成器，数据**一行一行流过**，任何时刻内存里只有一行 + 几个累加器，不囤积。对照 Java：相当于 `Stream<String>` 的惰性管道，但写起来更轻。

- **编码处理（[09 字符串字节与编码](09_字符串字节与编码.md)）**：`path.open("r", encoding="utf-8", errors="replace")` 显式指定编码（22 章铁律：文本模式永远写 `encoding`），`errors="replace"` 让非法字节变成 `�` 而非抛 `UnicodeDecodeError`——**真实日志里混入坏字节是常态**，脚本不能因一行坏数据全盘崩溃。这正是 09 章讲的编解码错误处理策略。

- **聚合统计（[06 内置数据结构](06_内置数据结构.md)、[07 collections模块](07_collections模块.md)）**：`Counter` 计数状态码/IP/分钟桶（`counter[k] += 1` 自动从 0 起），`defaultdict(int)` 累计每路径字节数（不用先判 key 是否存在）。`Counter.most_common(10)` 直接取 Top N（底层是堆，比排序整个字典高效）。这是 07 章 `Counter`/`defaultdict` 的典型实战。

- **`__slots__` 省内存（[13 类与对象](13_类与对象.md)、[04 运行时与内存模型](04_运行时与内存模型.md)）**：`ParsedLine` 用 `__slots__` 取消每个实例的 `__dict__`，海量小对象时内存显著降低。虽然这里数据流过即弃，但展示了"高频小对象用 `__slots__`"的工程意识。

- **pandas 汇总（[27 标准库与数据AI生态](27_标准库与数据AI生态.md)）**：聚合产生的 `Counter`/`dict` 直接喂给 `pd.Series`，用 `nlargest(10)`、`sort_index()`、向量化算百分比 `count / total * 100`、`idxmax()` 找峰值时刻。这是 27 章"能向量化就别按行循环"的体现——**前面用生成器流式聚合到小字典，最后才上 pandas 做汇总**，而不是把几个 GB 全塞进 DataFrame（那会 OOM）。这个"流式聚合 + 小结果集再用 pandas"的分工是处理大数据的关键模式。

- **pathlib + CLI（[22 文件IO与系统交互](22_文件IO与系统交互.md)）**：`Path(log_path)` 面向对象路径、`.exists()` 判存在，`sys.argv` 取命令行参数。

### 参考答案要点

**1.** 仓储层加 `count`：
```python
from sqlalchemy import func, select
def count(self, author: str | None = None) -> int:
    stmt = select(func.count()).select_from(Book)
    if author is not None:
        stmt = stmt.where(Book.author == author)
    return self.session.scalar(stmt) or 0
```
泛型分页模型：
```python
from typing import Generic, TypeVar
from pydantic import BaseModel
T = TypeVar("T")
class Page(BaseModel, Generic[T]):
    items: list[T]
    total: int
    limit: int
    offset: int
```
路由返回 `Page[BookOut]`，service 同时调 `repo.list(...)` 和 `repo.count(...)`。**关键**：`count` 用 `select(func.count())` 在数据库侧计数，**绝不能 `len(repo.list(...))`** 把全表拉进内存再数（典型新手错误）。

**2.** `lru_cache` 不适合是因为它**无法按 key 失效**——更新/删除某本书后，缓存里的旧值还在，返回脏数据。手写带失效的缓存：
```python
_cache: dict[int, BookOut] = {}
# get 时：命中返回；未命中查库后写入
# update/delete 时：_cache.pop(book_id, None) 清除该条
```
这暴露了缓存的核心难题——**缓存失效**（Phil Karlton 名言："计算机科学两大难题之一"）。生产用 Redis + TTL + 写时失效。

**3.** 改异步要点：`create_async_engine`、`AsyncSession`、依赖用 `async def get_session()` + `async with`、所有 `session.xxx()` 前加 `await`、查询用 `await session.scalars(...)`。**最大的坑**：异步下属性访问不能触发惰性 SQL，关系字段必须 `options(selectinload(...))` 显式预加载，否则 `MissingGreenlet` 报错。这把 25 章 §3.9 的"性能优化"升级成了"async 下的硬性要求"。

**6.**
```python
import gzip
def read_lines(path: Path) -> Iterator[str]:
    opener = gzip.open if path.suffix == ".gz" else open
    with opener(path, "rt", encoding="utf-8", errors="replace") as f:
        yield from f
```
管道其余部分（`parse_lines`/`aggregate`/`report`）**完全不动**——因为它们只依赖"一个产出字符串的迭代器"这个抽象，不关心数据来自普通文件还是 gzip。这就是 16 章生成器"用迭代器协议解耦数据源与处理逻辑"的实战价值。

**8.**
```python
from collections import Counter
from concurrent.futures import ProcessPoolExecutor
from functools import reduce
import operator

def analyze_one(path_str: str) -> Counter:
    # 单个文件的解析+聚合，返回 Counter（可跨进程 pickle）
    ...

def parallel(paths: list[str]) -> Counter:
    with ProcessPoolExecutor() as ex:
        partials = ex.map(analyze_one, paths)
    return reduce(operator.add, partials, Counter())   # Counter 支持 + 合并
```
**为什么多进程**：日志解析是 **CPU 密集**（正则匹配、字符串处理），多线程会被 GIL 卡住无法真并行（[20 并发与GIL](20_并发与GIL.md)）；多进程各有独立解释器和 GIL，能吃满多核。代价是进程间数据要 pickle、`Counter` 结果通过返回值传回主进程合并。这正是 20 章"CPU 密集用多进程、I/O 密集用多线程/async"决策的实战落地。
