# 22 · 文件 IO 与系统交互

> 你在 Java 里玩转 `java.nio.file.Files/Path`、`InputStream/Reader` 的装饰器栈、`try-with-resources`、`ProcessBuilder`。Python 这套**形似神不同**：文件对象是一座 `raw → buffered → text` 的三层 IO 塔，`pathlib.Path` 用面向对象路径干掉了字符串拼接，`subprocess` 用"参数列表"从根上堵死了 shell 注入。本章把这些机制拆到底层，并把每一个"Java 老手会踩的坑"标红——其中**不写 `encoding` 跨平台乱码**和 **`shell=True` 注入 / PIPE 死锁**是重灾区。

## ② 设计理念（为什么 Python 这么设计）

### 2.1 文本与二进制彻底分家

Java 早期把"字节"和"字符"混在一起埋了无数雷（`new FileReader` 用平台默认编码、`String.getBytes()` 不带 charset），后来靠 `Charset` 慢慢收口。Python 3 干脆**从类型层面把两者切开**：

- **二进制模式**（`"rb"/"wb"`）：读写的是 `bytes`，纯粹的字节搬运，没有任何编解码，跨平台逐字节一致。
- **文本模式**（`"r"/"w"`）：读写的是 `str`，文件对象在边界上**自动做编解码**（`bytes ↔ str`）和**换行符转换**（`\r\n ↔ \n`）。

这正是 [09 字符串字节与编码](09_字符串字节与编码.md) 里 `str`（Unicode 码点序列）与 `bytes`（字节序列）二分法在 IO 层的延伸：**编解码只发生在文本模式文件对象这一层**。理解了这条边界，你就懂为什么"读图片/protobuf 必须用 `rb`"、"读 CSV/JSON/日志用文本模式但必须带 `encoding`"。

### 2.2 pathlib：用对象取代字符串拼接

Java 早就用 `Path` 取代了 `new File("a" + File.separator + "b")` 这种易错拼接。Python 的老 API `os.path` 还停留在"路径就是字符串、用函数操作"的时代（`os.path.join`、`os.path.basename`），3.4 才引入 `pathlib`，把路径升级成**对象**：

```python
from pathlib import Path
cfg = Path.home() / "app" / "config.yaml"   # / 运算符拼接，自动用对的分隔符
cfg.parent.mkdir(parents=True, exist_ok=True)
cfg.write_text("debug: true\n", encoding="utf-8")
```

理念是：路径的所有操作（取父目录、换后缀、判断存在、读写、遍历）都该是路径对象的**方法**，而不是散落在 `os`、`os.path`、`glob`、`shutil` 里的一堆函数。这跟 Java 的 `Path` 哲学一致——只是 Python 多给了 `/` 运算符这个糖。

### 2.3 subprocess：用参数列表堵死 shell 注入

最关键的安全理念。在 shell 里 `rm -rf $dir`，如果 `dir` 是用户输入的 `; rm -rf /`，就是灾难。`subprocess` 的默认姿势是**传一个参数列表**而不是一整行命令串：

```python
subprocess.run(["git", "clone", repo_url])   # repo_url 永远是"一个参数"，不会被 shell 解析
```

操作系统的 `execvp` 直接拿这个列表当 `argv`，**根本不经过 shell**，所以 `repo_url` 里有空格、分号、`$()` 都只是普通字符。只有当你显式 `shell=True` 时才会退化成"拼一行交给 `/bin/sh`"——那才有注入风险。这跟 Java 的 `ProcessBuilder(List<String> command)` 是一个思路：**命令和参数结构化，不做字符串拼接**。

### 2.4 with：确定性关闭，呼应上下文管理器

Python 的文件对象是上下文管理器，`with open(...)` 在块退出时**确定性地** `close()`（flush 缓冲 + 释放 fd），无论正常结束还是抛异常。这是 [15 异常与上下文管理器](15_异常与上下文管理器.md) 的核心机制在 IO 上的应用，等价于 Java 的 try-with-resources。注意：虽然 [04 运行时与内存模型](04_运行时与内存模型.md) 讲过引用计数归零时 `__del__` 会顺手关文件，但那**不确定、不可依赖**——`with` 才是正道。

## ③ 实现细节（底层怎么实现的）★本章重点★

### 3.1 文件对象是一座三层 IO 塔

`open()` 返回的不是一个简单对象，而是 `io` 模块搭起来的**三层装饰器栈**——和 Java 的 `new BufferedReader(new InputStreamReader(new FileInputStream(f), UTF_8))` 是同构的，只是 Python 默认帮你套好了：

```
┌─────────────────────────────────────────┐
│ TextIOWrapper   ← 文本层：编解码 + 换行转换 │  收发 str
├─────────────────────────────────────────┤
│ BufferedReader/Writer  ← 缓冲层：批量读写  │  收发 bytes
├─────────────────────────────────────────┤
│ FileIO (raw)    ← 裸层：直接系统调用 read/write │  收发 bytes
└─────────────────────────────────────────┘
        ↓ 持有一个 OS 文件描述符 fd
```

你可以亲手把这座塔拆开看：

```python
f = open("/tmp/demo.txt", "w", encoding="utf-8")
print(type(f))              # <class '_io.TextIOWrapper'>   文本层
print(type(f.buffer))       # <class '_io.BufferedWriter'>  缓冲层
print(type(f.buffer.raw))   # <class '_io.FileIO'>          裸层
print(f.buffer.raw.fileno())# 一个整数，就是 OS 文件描述符 fd（如 3）
print(f.encoding)           # utf-8   编解码就发生在 TextIOWrapper
f.close()
```

对照 Java 的装饰器流栈，映射关系一目了然：

| Python 层 | 收发类型 | 职责 | Java 对应 |
|-----------|---------|------|----------|
| `TextIOWrapper` | `str` | 编解码、换行转换 | `InputStreamReader`/`OutputStreamWriter` |
| `BufferedReader/Writer` | `bytes` | 缓冲、批量 syscall | `BufferedInputStream`/`BufferedOutputStream` |
| `FileIO`（raw） | `bytes` | 直接 `read(2)`/`write(2)` | `FileInputStream`/`FileOutputStream` |

- **二进制模式** `open(p, "rb")` 省掉最顶层，直接给你 `BufferedReader`（收发 `bytes`）。
- `open(p, "rb", buffering=0)` 连缓冲层也省掉，直接给你 `FileIO`——裸 syscall，每次 `read` 都进内核。

### 3.2 缓冲：全缓冲 / 行缓冲 / 无缓冲与 flush

缓冲层的存在是为了减少昂贵的系统调用——攒够一批再 `write(2)`，和 JVM 的 `BufferedOutputStream` 动机完全一样。`open` 的 `buffering` 参数控制策略：

| `buffering` 值 | 行为 | 何时用 |
|----------------|------|--------|
| `-1`（默认） | 二进制：全缓冲（块大小 `io.DEFAULT_BUFFER_SIZE`，通常 8 KiB）；文本写到 tty 时行缓冲 | 绝大多数场景 |
| `0` | 无缓冲（仅二进制模式允许） | 要求每次写立即落 syscall |
| `1` | 行缓冲（仅文本模式）：遇 `\n` 就 flush | 日志、实时输出 |
| `>1` | 指定字节数的全缓冲 | 调优 |

```python
import time
# 全缓冲：数据先攒在用户态缓冲里，close/flush 时才真正写盘
with open("/tmp/log.txt", "w", encoding="utf-8") as f:
    f.write("started\n")
    # 此刻文件里很可能还是空的——数据在缓冲区
    f.flush()              # 手动强制刷盘，等价于 Java 的 flush()
    # os.fsync(f.fileno())  # 更狠：连 OS page cache 也刷到物理磁盘
```

> **坑预告**：子进程或崩溃前没 flush，缓冲区里的数据会丢。`with` 块正常退出会自动 flush+close，但 `os._exit()`、段错误、被 `kill -9` 不会。日志场景常用 `buffering=1`（行缓冲）或每条 `flush()`。

### 3.3 为什么文本模式**必须**显式写 encoding

这是 Java 老手最容易栽的坑，根子在于：**`TextIOWrapper` 不指定 `encoding` 时，用的是"本地化默认编码"`locale.getpreferredencoding()`，而它依赖运行环境**。

```python
import locale
print(locale.getpreferredencoding(False))
# 你的 Mac/Linux:  UTF-8
# 同事的中文 Windows: cp936 (GBK!)   ← 同一份代码，不同机器解出不同编码
```

后果：你在 Mac 上 `open("data.csv").read()` 一切正常，部署到中文 Windows 服务器或 CI 上，默认编码变成 GBK，读 UTF-8 文件直接 `UnicodeDecodeError` 或读出乱码。这跟 Java 早期 `new FileReader`（用平台默认 charset）的历史包袱一模一样。

Python 社区为此推了 **PEP 597**：从 3.10 起，不传 `encoding` 时可开启 `EncodingWarning` 警告你（`python -X warn_default_encoding`）；PEP 686 计划未来把默认改成 UTF-8。但**现在的纪律就一条**：文本模式永远显式写 `encoding="utf-8"`。

```python
# ❌ 埋雷：跨平台行为不一致
text = open("config.yaml").read()
# ✅ 显式编码：到哪都一样
text = open("config.yaml", encoding="utf-8").read()
```

详见 [09 字符串字节与编码](09_字符串字节与编码.md)。换行符也类似：文本模式默认开 **universal newlines**——读时把 `\r\n`/`\r` 统一成 `\n`，写时把 `\n` 翻成平台换行（Windows 上变 `\r\n`）。想要逐字节原样，用 `newline=""`（CSV 场景必须）或干脆二进制模式。

### 3.4 pathlib 就是 os.path 的 OO 封装

`pathlib` 不是另起炉灶，它内部大量复用 `os`/`os.path`。`PurePath` 负责纯路径字符串运算（拆分、拼接、判断），`Path` 在其上加了**会碰文件系统**的方法（`exists`、`stat`、`open`、`iterdir`）。`Path` 在 POSIX 上实例化成 `PosixPath`，Windows 上是 `WindowsPath`，由它们决定分隔符与大小写规则。对照表：

| 字符串时代（os.path） | OO 时代（pathlib） |
|----------------------|-------------------|
| `os.path.join(a, b)` | `Path(a) / b` |
| `os.path.basename(p)` | `Path(p).name` |
| `os.path.dirname(p)` | `Path(p).parent` |
| `os.path.splitext(p)[1]` | `Path(p).suffix` |
| `os.path.exists(p)` | `Path(p).exists()` |
| `os.path.abspath(p)` | `Path(p).resolve()` |
| `glob.glob(pat)` | `Path(...).glob(pat)` |

```python
from pathlib import Path
p = Path("/var/log/app/access.log")
print(p.name)     # access.log
print(p.stem)     # access
print(p.suffix)   # .log
print(p.parent)   # /var/log/app
print(p.parts)    # ('/', 'var', 'log', 'app', 'access.log')
print(p.with_suffix(".log.gz"))  # /var/log/app/access.log.gz
```

`Path` 对象可以直接传给几乎所有标准库 IO 函数（`open`、`shutil`、`json.load` 配合 `open` 等），因为它实现了 `os.PathLike` 协议（`__fspath__`）——等价于 Java 里 `Path` 能被 `Files` 各方法接受。

### 3.5 subprocess 的 fork+exec、管道与两大杀手

`subprocess` 底层在 POSIX 上走经典的 **fork + exec**：父进程 `fork(2)` 出子进程，子进程里 `execvp(2)` 用新程序镜像替换自己，父进程拿到子进程 pid 后 `waitpid(2)` 收尸。这正是 Java `ProcessBuilder` 在 JVM 里干的事。

**管道（PIPE）** 的本质是内核里一个**有限容量的缓冲区**（Linux 默认 64 KiB）。`stdout=subprocess.PIPE` 让子进程的标准输出连到这个管道，父进程从另一头读。容量有限这件事，就是著名的 **PIPE 死锁** 根源：

```
子进程拼命往 stdout 管道写  ──►  管道写满（64KiB）  ──►  子进程 write() 阻塞，不再继续
父进程却还卡在 proc.stdout.read() 等着读 stderr / 或先 wait() ──►  双方互等 ──► 死锁
```

```python
import subprocess
# ❌ 死锁写法：子进程输出超过管道容量，又先 wait() 再读
proc = subprocess.Popen(["big_output_cmd"], stdout=subprocess.PIPE)
proc.wait()                 # 子进程被管道堵住写不动，永远不退出 → 这里永久阻塞
data = proc.stdout.read()   # 到不了这一行

# ✅ 正解：用 communicate()，它在内部用线程/select 同时排空所有管道
proc = subprocess.Popen(["big_output_cmd"], stdout=subprocess.PIPE, stderr=subprocess.PIPE)
out, err = proc.communicate(timeout=30)   # 安全：边读边排空，不会堵
```

`subprocess.run(..., capture_output=True)` 内部就是用 `communicate()`，所以高层 API 天然没这个坑——**这就是优先用 `run` 而非裸 `Popen` 的实现层理由**。

**`shell=True` 的注入风险**：传 `shell=True` 时，Python 把你的命令串交给 `/bin/sh -c`，由 shell 做分词、通配展开、变量替换、命令替换。用户可控的内容混进去就能注入：

```python
filename = "report.txt; rm -rf ~"      # 来自用户输入
# ❌ 灾难：shell 会把分号后的命令也执行掉
subprocess.run(f"cat {filename}", shell=True)
# ✅ 列表参数 + 无 shell：filename 永远是"一个参数"，分号只是普通字符
subprocess.run(["cat", filename])      # 报错"找不到名为 'report.txt; rm -rf ~' 的文件"，但不会删你家
```

## ④ 使用方法与最佳实践

### 4.1 读写文件：永远 with + 显式 encoding

```python
from pathlib import Path

# 写文本
with open("/tmp/users.csv", "w", encoding="utf-8", newline="") as f:
    f.write("id,name\n1,张三\n2,李四\n")

# 读文本
with open("/tmp/users.csv", encoding="utf-8") as f:
    content = f.read()          # 一次读全部成 str

# pathlib 的一行流（内部自动 open+close），小文件首选
text = Path("/tmp/users.csv").read_text(encoding="utf-8")
Path("/tmp/out.txt").write_text("done\n", encoding="utf-8")
```

### 4.2 大文件：按行惰性读，别一次 read()

文件对象本身就是**迭代器**（呼应 [16 迭代器与生成器](16_迭代器与生成器.md)）——`for line in f` 一次只把一行读进内存，处理几十 GB 日志也不会爆内存。这是和 Java `BufferedReader.lines()`（返回 `Stream<String>`）对应的惰性姿势：

```python
# ✅ 惰性逐行：内存恒定，能处理任意大文件
total = 0
with open("/var/log/huge.log", encoding="utf-8") as f:
    for line in f:                       # 每次迭代读一行，O(1) 内存
        if "ERROR" in line:
            total += 1
print(f"ERROR 行数: {total}")

# ❌ 反例：一次性读爆内存
# data = open("/var/log/huge.log").read()   # 100GB 文件 → MemoryError
```

### 4.3 二进制读写

```python
# 读图片/任意二进制：rb 模式收发 bytes
data: bytes = Path("/tmp/logo.png").read_bytes()
print(data[:8])              # b'\x89PNG\r\n\x1a\n'  PNG 魔数

# 分块拷贝大二进制文件，内存恒定
with open("/tmp/src.bin", "rb") as src, open("/tmp/dst.bin", "wb") as dst:
    while chunk := src.read(1024 * 1024):   # := 海象运算符(3.8+)，每次读 1MB
        dst.write(chunk)
```

### 4.4 pathlib 全家桶

```python
from pathlib import Path

base = Path("/tmp/myapp")
base.mkdir(parents=True, exist_ok=True)      # mkdir -p，已存在不报错
(base / "data").mkdir(exist_ok=True)

# 写几个文件
for i in range(3):
    (base / "data" / f"part-{i}.txt").write_text(f"chunk {i}\n", encoding="utf-8")

# 判断
cfg = base / "config.yaml"
print(cfg.exists(), cfg.is_file(), base.is_dir())   # False False True

# 遍历（iterdir 当前层；glob 单层通配；rglob 递归）
for p in (base / "data").iterdir():
    print(p.name)                            # part-0.txt ...
print(list(base.rglob("*.txt")))             # 递归找所有 .txt，等价 Files.walk + 过滤

# 取信息
p = base / "data" / "part-0.txt"
print(p.stat().st_size, "bytes")             # 文件大小
print(p.resolve())                           # 绝对路径（解析 .. 和符号链接）
```

### 4.5 shutil：高层文件操作

```python
import shutil
from pathlib import Path

shutil.copy2("/tmp/a.txt", "/tmp/b.txt")     # 复制内容+元数据(时间戳等)，类似 Files.copy(COPY_ATTRIBUTES)
shutil.copytree("/tmp/myapp", "/tmp/backup", dirs_exist_ok=True)  # 递归复制整个目录树
shutil.move("/tmp/b.txt", "/tmp/archive/")   # 移动/重命名（跨盘自动 copy+delete）
shutil.rmtree("/tmp/backup")                 # 递归删目录（Java 要手写递归或 walkFileTree）
print(shutil.which("git"))                   # 在 PATH 里找可执行文件，类似 `which`
print(shutil.disk_usage("/"))                # 磁盘总量/已用/剩余
```

> `shutil.rmtree` 是 `rm -rf` 的威力，没有回收站。删之前务必校验路径，别用拼接出来的、可能为空的变量。

### 4.6 subprocess.run：check / capture_output / timeout / 列表参数

`subprocess.run` 是 3.5+ 的高层 API，覆盖 90% 需求，记住这套参数组合：

```python
import subprocess

result = subprocess.run(
    ["git", "-C", "/tmp/myapp", "status", "--porcelain"],  # ✅ 列表参数，防注入
    capture_output=True,    # 捕获 stdout/stderr（内部用 communicate，无 PIPE 死锁）
    text=True,              # 用文本模式解码成 str（否则是 bytes）；可加 encoding="utf-8"
    timeout=10,             # 超时(秒)，超了抛 TimeoutExpired 并杀子进程
    check=True,             # 返回码非 0 时抛 CalledProcessError（类似 Java 自己判 exitValue）
)
print("返回码:", result.returncode)   # 0
print("输出:", result.stdout)
print("错误:", result.stderr)

# 处理失败：check=True 时用 try 捕获
try:
    subprocess.run(["false"], check=True)
except subprocess.CalledProcessError as e:
    print(f"命令失败，返回码 {e.returncode}")   # 命令失败，返回码 1
```

需要**管道连接两个命令**（`ps aux | grep python`）时，不要用 `shell=True`，而是把两个进程的 PIPE 接起来——或者更地道地：用第一个命令的输出在 Python 里过滤：

```python
# 地道做法：命令1输出交给 Python 处理，不需要 shell 管道
ps = subprocess.run(["ps", "aux"], capture_output=True, text=True)
lines = [ln for ln in ps.stdout.splitlines() if "python" in ln]
print(f"找到 {len(lines)} 个 python 进程")
```

### 4.7 tempfile：临时文件/目录

```python
import tempfile, os
from pathlib import Path

# 临时文件：默认关闭即删除（delete=True）
with tempfile.NamedTemporaryFile(mode="w", suffix=".json",
                                 encoding="utf-8", delete=True) as tf:
    tf.write('{"ok": true}')
    tf.flush()
    print("临时文件:", tf.name)          # /var/folders/.../tmpXXXX.json
    # 在 with 块内文件存在，可被别的进程读
# 出块即删

# 临时目录：with 退出时整个目录树自动 rmtree
with tempfile.TemporaryDirectory() as td:
    work = Path(td)
    (work / "scratch.txt").write_text("temp data", encoding="utf-8")
    print("临时目录:", td)
# 出块即删，无需手动清理——比 Java 的 deleteOnExit 更确定
```

> 安全提示：别自己用 `"/tmp/" + name` 拼临时文件名（可预测，有符号链接攻击/竞态风险）。`tempfile` 用 `O_CREAT|O_EXCL` 原子创建唯一文件名，是安全做法。

## ⑤ ⚠️ 易错点（Java 思维翻车）

| Java 习惯 | 翻车现场 | 正确做法/认知 |
|----------|---------|--------------|
| `new FileReader(f)` 不带 charset 也能跑 | `open("x.csv")` 不写 `encoding`，Mac 上正常，中文 Windows 上 `UnicodeDecodeError`/乱码 | 文本模式**永远** `encoding="utf-8"`；二进制内容用 `"rb"` |
| 信任 GC/finally 兜底关流 | 不用 `with` 直接 `f = open(...)`，高并发下 fd 耗尽（`Too many open files`） | 一律 `with open(...) as f:` 确定性关闭 |
| `Runtime.exec("cmd " + userInput)` 拼命令 | `subprocess.run(f"cat {name}", shell=True)`，`name` 含 `; rm -rf` 被注入 | 用**列表参数** `run(["cat", name])`，默认不过 shell |
| 裸 `Popen` 后先 `wait()` 再读 stdout | 子进程输出 > 64KiB 管道写满，双方互等，**永久死锁** | 用 `run(capture_output=True)` 或 `proc.communicate()` |
| `"dir" + "/" + name` 字符串拼路径 | Windows 上分隔符错、`..`/重复斜杠没归一、易拼错 | 用 `Path(dir) / name`，跨平台正确 |
| `Files.readAllLines` 习惯，整文件读进来 | `open(huge).read()` 把几十 GB 日志一次塞内存 → `MemoryError` | `for line in f` 惰性逐行；二进制分块 `read(size)` |
| 假设相对路径相对于脚本所在目录 | `open("config.yaml")` 相对的是**进程 cwd**，换目录启动就找不到 | 用 `Path(__file__).parent / "config.yaml"` 锚定脚本位置 |
| 硬编码 `\` 或 `/` 分隔符 | 路径在另一平台跑不通 | 让 `pathlib` 处理分隔符；`os.sep` 仅供查看 |
| 以为 `write` 立刻落盘 | 全缓冲下程序崩溃/被 kill，缓冲区数据丢失 | 关键写入后 `f.flush()`（必要时 `os.fsync`）；或行缓冲 |
| `sys.argv` 当成 Java 的 `args`，从 `[0]` 取参数 | 把脚本名当成了第一个参数 | 真正参数是 `sys.argv[1:]`；CLI 用 `argparse` |

### 基础理解

### 找坑改错

**3.** 下面这段"调用 ffmpeg 转码并捕获日志"的代码有**两个**严重问题（一个安全、一个会挂死），请指出并改正：
```python
import subprocess

        f"ffmpeg -i {user_filename} -y out.mp4",

        stdout=subprocess.PIPE,

    return proc.stdout.read()
```

### 综合实战

**5.** 写一个脚本 `log_report.py`，接收一个目录作为命令行参数，**递归**统计该目录下所有 `.log` 文件里包含 `"ERROR"` 的行数，按"文件 → 错误行数"汇总打印，并把汇总结果写入该目录下的 `error_summary.txt`（UTF-8）。要求：用 `pathlib`、惰性逐行读（不得一次性 `read()`）、显式编码、`with` 管理文件、能处理几个 GB 的大日志。

---

### 参考答案

**1.** 三层：最底层 `FileIO`（raw，收发 `bytes`，直接做 `read(2)`/`write(2)` 系统调用）；中间 `BufferedReader/Writer`（收发 `bytes`，攒批减少 syscall）；最顶层 `TextIOWrapper`（收发 `str`，负责按 `encoding` 编解码 + 换行符转换）。可用 `f.buffer`、`f.buffer.raw` 逐层取出验证。读 PNG 必须用 `"rb"`：图片是任意字节，文本模式会试图把这些字节按某编码解码成 `str`（多半 `UnicodeDecodeError`），且换行符转换会把 `0x0D 0x0A` 篡改，破坏二进制内容。二进制数据必须原样逐字节读写。

**2.** **B 会出问题。** B 没写 `encoding`，用的是 `locale.getpreferredencoding()`——开发机是 UTF-8，中文 Windows 是 GBK(cp936)。若 `report.txt` 是 UTF-8 编码，在 GBK 环境下会按 GBK 解码 UTF-8 字节，导致乱码或 `UnicodeDecodeError`。A 显式 `encoding="utf-8"`，到哪都一致。改法：给 B 加上 `encoding="utf-8"`（并建议用 `with`）。

**3.** 问题一（安全）：`shell=True` + f-string 拼接 `user_filename`，若文件名是 `x.mp4; rm -rf ~` 会被 shell 当成两条命令执行，命令注入。问题二（死锁）：`Popen` 后先 `proc.wait()` 再 `proc.stdout.read()`——ffmpeg 的日志走 stderr/stdout，输出量大时管道（64KiB）写满，子进程阻塞在 write 永不退出，`wait()` 永久挂死。改正：
```python
import subprocess
def transcode(user_filename):
    result = subprocess.run(
        ["ffmpeg", "-i", user_filename, "-y", "out.mp4"],  # 列表参数，不过 shell
        capture_output=True, text=True, timeout=600,        # 内部 communicate，无死锁
    )
    if result.returncode != 0:
        raise RuntimeError(f"转码失败: {result.stderr[-500:]}")
    return result.stderr   # ffmpeg 进度/日志默认在 stderr
```

**4.** 风险：① `os.system` 把参数拼进 shell，`user_dir` 含 `; ` 或 `../../` 可注入/越权删除（如 `user_dir = "../../etc"`）；② `rm -rf` 无任何校验。安全改写：
```python
import shutil
from pathlib import Path

BASE = Path("/data/tmp").resolve()
def clean(user_dir: str):
    target = (BASE / user_dir).resolve()
    # 关键：校验 target 确实在 BASE 之内，挡掉 ../ 越权
    if not target.is_relative_to(BASE):     # Python 3.9+
        raise ValueError(f"非法路径: {user_dir}")
    if target == BASE:
        raise ValueError("拒绝清空根目录")
    if target.is_dir():
        shutil.rmtree(target)
        target.mkdir()                       # 清空后重建空目录
```

**5.** 参考实现：
```python
#!/usr/bin/env python3
"""log_report.py: 递归统计目录下 .log 文件的 ERROR 行数。
用法: python3 log_report.py /var/log/myapp
"""
import sys
from pathlib import Path

def count_errors(log_path: Path) -> int:
    n = 0
    # 惰性逐行：内存恒定，可处理 GB 级文件；errors="replace" 容忍坏字节
    with open(log_path, encoding="utf-8", errors="replace") as f:
        for line in f:
            if "ERROR" in line:
                n += 1
    return n

def main() -> int:
    if len(sys.argv) != 2:
        print("用法: python3 log_report.py <目录>", file=sys.stderr)
        return 2
    root = Path(sys.argv[1])
    if not root.is_dir():
        print(f"不是目录: {root}", file=sys.stderr)
        return 1

    results: dict[Path, int] = {}
    for log_file in sorted(root.rglob("*.log")):   # 递归找所有 .log
        results[log_file] = count_errors(log_file)

    total = sum(results.values())
    summary = root / "error_summary.txt"
    with open(summary, "w", encoding="utf-8") as out:
        for path, cnt in results.items():
            line = f"{path}: {cnt}"
            print(line)                            # 打印到屏幕
            out.write(line + "\n")                 # 同时写汇总文件
        out.write(f"---\nTOTAL ERROR LINES: {total}\n")
    print(f"---\n共 {total} 条 ERROR，汇总已写入 {summary}")
    return 0

if __name__ == "__main__":
    sys.exit(main())
```
要点：`rglob("*.log")` 递归匹配；`for line in f` 惰性读，GB 级日志内存恒定；全程显式 `encoding="utf-8"`；`with` 管理读和写两个文件；`argv[1]` 取参数（`argv[0]` 是脚本名）；返回码用 `sys.exit` 传给 shell，便于自动化串联。生产里参数解析更建议用 `argparse`。
