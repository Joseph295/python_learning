# 13 · 类与对象 ⭐

> 你写了十几年 Java 类，`class`/`extends`/`@Override`/`equals`/`hashCode` 刻进肌肉记忆。Python 的 `class` 长得很像，但**神不同**：没有真正的 private、构造器不能重载、`self` 要手写、运算符可以重载、可以多继承且靠 MRO+C3 算法决定调用顺序、类本身也是对象。本章告诉你这些"形似神不同"的差异底层是怎么实现的——属性查找走 `__dict__` 和 MRO 链、`super()` 不是"调父类"而是"沿 MRO 找下一个"、`__slots__` 怎么省内存、`__x` 的"private"只是改名魔术。看懂这章，你才能写出地道的 Python 类，而不是"带 self 的 Java 类"。

## ② 设计理念（为什么 Python 这么设计）

### 2.1 为什么没有真正的 private——"我们都是成年人"

Java 用 `private`/`protected`/`public` 由**编译器强制**封装边界：你访问不到别人的私有字段，编译都过不去。Python 的设计哲学相反——**"We're all consenting adults here"（我们都是自愿的成年人）**：约定优于强制。

- 一个前导下划线 `_x` 是**约定**："这是内部实现，外人别碰"。但语言**不拦你**，你硬要访问 `obj._x` 完全能访问。
- 两个前导下划线 `__x` 触发**名称改写（name mangling）**，把 `__x` 改成 `_ClassName__x`。这**不是为了保密**，而是为了避免子类不小心覆盖父类的内部属性（见 §3.8）。它仍然能被访问（用改写后的名字），所以也不是真 private。

背后的取舍：Python 认为**强封装的收益配不上它的成本**。库作者用文档和下划线表达意图，使用者自负其责。这换来的是极高的灵活性——你能 monkey-patch、能在测试里访问内部状态、能在调试时窥探一切。对 Java 老手这是巨大的心态转变：**Python 不保护你免于自己，它信任你**。

### 2.2 为什么用 @property 取代 getter/setter

Java 充斥着这种样板：

```java
private String name;
public String getName() { return name; }
public void setName(String n) { this.name = n; }
```

理由是"将来可能要加校验/计算，先封装好以免改 API"。Python 不需要这种**预防性封装**，因为它有 `@property`：你**一开始就用公开属性 `obj.name`**，等真的需要校验或计算时，再把 `name` 变成 `@property`——**调用方代码一行不用改**。

```python
user.name = "Alice"   # 一开始是普通属性
# 后来加了 @property 做校验，调用方写法完全不变
```

这就是 Python 的"**统一访问原则**"（Uniform Access Principle）：属性访问和方法调用在语法上无法区分，所以你可以自由地在两者间切换而不破坏接口。Java 因为 `obj.name` 和 `obj.getName()` 语法不同，只能一开始就上 getter/setter 防御。Python 把这笔"封装税"推迟到真正需要时再交。

### 2.3 运算符重载：让自定义类型像内置类型

Java 几乎不允许运算符重载（只有 `+` 对字符串是特例），理由是"防止滥用导致可读性灾难"。Python 选择**全面开放**：你可以让自定义的 `Money` 用 `+`、`Vector` 用 `*`、矩阵用 `@`、自定义容器用 `len()`/`in`/`[]`/`for`。

哲学是：**内置类型不该有特权**。`list` 能用 `[]` 和 `len()`，凭什么你的 `Matrix` 不能？Python 通过**魔术方法（dunder methods，`__xxx__`）协议**让一切类型平权——`len(x)` 其实是调 `x.__len__()`，`a + b` 其实是调 `a.__add__(b)`。这就是 numpy/pandas 能让 `df['col'] * 2`、`arr[mask]` 这么自然的根基。代价是确实可能被滥用，但 Python 把判断权交给程序员（又是"成年人"哲学）。

### 2.4 多继承 + MRO，取代接口

Java 是**单实现继承 + 多接口**：一个类只能 `extends` 一个父类（避免"钻石问题"的实现冲突），但能 `implements` 多个接口（接口原本没实现，无冲突）。Java 8 之后接口有了 `default` 方法，钻石问题部分回归，用显式 `Interface.super.foo()` 解决。

Python 选择**完全的多继承**：一个类能继承多个有实现的父类。钻石问题（多个父类有同名方法）用 **MRO（Method Resolution Order，方法解析顺序）+ C3 线性化算法**自动解决——把继承图拍平成一个确定的线性顺序，沿这个顺序查方法。配合 `super()`（沿 MRO 找下一个，而非"调父类"），实现**协作式多继承（cooperative multiple inheritance）**，典型应用是 **mixin**（混入类，给类"拼装"能力）。取舍：更灵活，但 MRO 和 `super()` 的语义比 Java 复杂，用不好会踩坑（见 §5）。

### 2.5 为什么 self 要显式

Java 的 `this` 是隐式的，方法里直接写 `name` 就访问字段。Python 强制方法第一个参数是 `self`，访问属性必须 `self.name`。这不是设计缺陷，是刻意为之：

- **"Explicit is better than implicit"（显式优于隐式）**——Python 之禅。`self.name` 一眼看出是实例属性，`name` 是局部变量，无歧义。
- **方法只是个普通函数**，恰好第一个参数是实例。`obj.method()` 等价于 `Class.method(obj)`，self 的显式让这个等价关系一目了然（见 §3.4 绑定方法）。这种"方法即函数"的统一让装饰器、把方法当回调传递等操作非常自然。

## ③ 实现细节（底层怎么实现的）★本章重点★

这一段是本章的核心。理解了属性查找机制和 MRO，你就理解了 Python 对象系统的 90%。

### 3.1 实例属性存在 `__dict__` 里（一个 dict）

Java 对象的字段在编译期就定死了：类有几个字段、每个字段的偏移量，对象布局是固定的连续内存。访问 `obj.x` 是**按固定偏移取值**，极快。

Python 默认完全不同：**每个实例的属性存在一个名为 `__dict__` 的字典里**。

```python
class User:
    def __init__(self, name, age):
        self.name = name      # 等价于 self.__dict__['name'] = name
        self.age = age

u = User("Alice", 30)
print(u.__dict__)             # 输出: {'name': 'Alice', 'age': 30}

u.email = "a@x.com"           # 运行时随便加属性！Java 做不到
print(u.__dict__)             # 输出: {'name': 'Alice', 'age': 30, 'email': 'a@x.com'}
```

后果（Java 老手必须消化）：
- **属性不是编译期定死的**，是运行时往 dict 里塞键值。所以你能动态加属性、删属性。
- **每个实例都背着一个 dict**，比 Java 的紧凑对象**费内存**（dict 有哈希表开销）。这正是 `__slots__` 要解决的问题（见 §3.7）。
- **访问 `obj.x` 是一次 dict 查找**（哈希），不是按偏移取值，比 Java 慢。

类本身也有 `__dict__`，存的是方法和类属性：

```python
print(User.__dict__.keys())
# 输出: dict_keys(['__module__', '__init__', '__dict__', '__weakref__', '__doc__'])
#       方法 __init__ 在这里；name/age 不在，因为它们是实例属性
```

### 3.2 属性查找顺序（核心机制）

当你写 `obj.x` 时，Python 按一个**确定的顺序**去找 `x`。简化版（完整版涉及描述符，见 §3.3）：

```
obj.x 的查找顺序：
  1. 在 type(obj) 的 MRO 链上找 x，且 x 是"数据描述符"   → 用它（最高优先级）
  2. 在 obj.__dict__ 里找 x                              → 用它（实例属性）
  3. 在 type(obj) 的 MRO 链上找 x（类属性、方法、非数据描述符）→ 用它
  4. 都没有 → 调 __getattr__（如果定义了）→ 否则抛 AttributeError
```

记住这个**实例属性会"遮蔽"类属性**的规则（第 2 步在第 3 步之前）：

```python
class Config:
    timeout = 30              # 类属性

c = Config()
print(c.timeout)             # 输出: 30   实例 __dict__ 没有，去类里找到
c.timeout = 99               # 在实例 __dict__ 里建了一个 timeout，遮蔽类属性
print(c.timeout)             # 输出: 99   先在实例 __dict__ 找到
print(Config.timeout)        # 输出: 30   类属性本身没变
print(c.__dict__)            # 输出: {'timeout': 99}
```

这跟 Java 的字段隐藏（field hiding）表面像，但机制完全不同：Java 是编译期按静态类型决定访问哪个字段；Python 是运行时按 dict 查找顺序决定。

### 3.3 数据描述符为什么优先（引出第 19 章）

第 1 步提到"数据描述符优先于实例属性"，这是 `@property` 能工作的原因。**描述符（descriptor）**是定义了 `__get__`/`__set__`/`__delete__` 的对象，放在**类**上。同时定义了 `__set__`（或 `__delete__`）的叫**数据描述符**，优先级高于实例 `__dict__`；只有 `__get__` 的叫**非数据描述符**（如普通函数/方法），优先级低于实例 `__dict__`。

`@property` 创建的就是一个数据描述符，所以即使实例 `__dict__` 里有同名键，访问时也走 property 的 `__get__`。这套机制是属性查找的底层引擎——**方法绑定、`@property`、`@classmethod`/`@staticmethod` 全是描述符协议实现的**。本章只需知道"数据描述符优先"这个结论；完整的描述符协议见 [19 章 描述符与元类](19_描述符与元类.md)。

### 3.4 `obj.method()` 如何绑定 self——绑定方法（bound method）

这是"方法即函数"的关键。函数对象是**非数据描述符**（实现了 `__get__`）。当你通过实例访问一个方法时，描述符协议把它**包装成"绑定方法"**，自动把实例塞进第一个参数：

```python
class Greeter:
    def hello(self, name):
        return f"Hi {name}, I'm {id(self)}"

g = Greeter()

# 通过类访问：拿到的是普通函数（unbound），要手动传实例
print(type(Greeter.hello))        # 输出: <class 'function'>
print(Greeter.hello(g, "Bob"))    # 手动传 self

# 通过实例访问：拿到的是绑定方法（bound method），self 已绑好
m = g.hello
print(type(m))                    # 输出: <class 'method'>
print(m.__self__ is g)            # 输出: True   绑定方法记着它的实例
print(m("Bob"))                   # 不用传 self，已经绑了 g
```

所以 `g.hello("Bob")` 的本质是：`Greeter.hello` 是函数 → 通过 `g` 访问触发 `__get__` → 返回一个绑定了 `g` 的 method 对象 → 调用它时自动 `Greeter.hello(g, "Bob")`。**这就是为什么 self 要显式写出来**——它就是函数的第一个真实参数，绑定方法只是帮你自动填上了。

绑定方法是一等对象，可以当回调到处传（对照 Java 的方法引用 `obj::hello`）：

```python
callbacks = [g.hello]    # 把绑定方法存起来，self 跟着走
print(callbacks[0]("Eve"))
```

### 3.5 MRO 与 C3 线性化算法（钻石继承）

多继承时，"`obj.x` 在 MRO 链上找"的那个**链**怎么排？用 **C3 线性化算法**。它保证：① 子类排在父类前面；② 父类的相对顺序保持（按声明的 `class C(A, B)` 顺序）；③ 单调性（子类的 MRO 与父类的 MRO 一致）。如果无法满足，定义类时直接报 `TypeError`。

经典的**钻石继承**例子：

```python
class Base:
    def who(self): return "Base"

class A(Base):
    def who(self): return "A"

class B(Base):
    def who(self): return "B"

class D(A, B):               # 钻石：D → A,B → Base
    pass

print([c.__name__ for c in D.__mro__])
# 输出: ['D', 'A', 'B', 'Base', 'object']
print(D().who())             # 输出: A   沿 MRO 第一个有 who 的是 A
```

`D.__mro__` 就是查找链：`D → A → B → Base → object`。找 `who` 时按这个顺序，第一个命中的是 `A`。注意 `Base` 在 `A` 和 `B` **都之后**——C3 保证公共父类排在所有子类之后，所以 `Base.who` 不会被提前选中。

C3 的直觉算法：`L[D] = D + merge(L[A], L[B], [A, B])`，merge 时反复取"出现在某个列表头部、且不出现在任何列表尾部"的类。手算一遍：

```
L[Base] = [Base, object]
L[A]    = [A, Base, object]
L[B]    = [B, Base, object]
L[D]    = D + merge([A,Base,object], [B,Base,object], [A,B])
        = D, A + merge([Base,object], [B,Base,object], [B])
          （Base 不能取：它在第二个列表的尾部 → 跳过，取 B）
        = D, A, B + merge([Base,object], [Base,object], [])
        = D, A, B, Base, object
```

**对照 Java 如何避开钻石问题**：Java 单实现继承，根本不会有"两个父类都带实现的同名方法"。接口的 `default` 方法若冲突，编译器**强制你手动 override 并用 `Interface.super.foo()` 显式选**——把责任推给程序员。Python 反过来，用 C3 **自动**给出一个确定顺序，不报错就按 MRO 走。各有取舍：Java 更安全更啰嗦，Python 更灵活但要求你懂 MRO。

### 3.6 `super()` 不是"调父类"，是"沿 MRO 找下一个"（协作式多继承）

这是 Java 老手最大的认知陷阱。Java 的 `super.foo()` 明确指向**直接父类**。Python 的 `super().foo()` 指向 **MRO 链上当前类的下一个**——这取决于**实例的实际类型**，不是当前类的父类！

```python
class Base:
    def __init__(self):
        print("Base.__init__")

class A(Base):
    def __init__(self):
        print("A.__init__ start")
        super().__init__()        # 不一定调 Base！取决于实例的 MRO
        print("A.__init__ end")

class B(Base):
    def __init__(self):
        print("B.__init__ start")
        super().__init__()
        print("B.__init__ end")

class D(A, B):
    def __init__(self):
        print("D.__init__ start")
        super().__init__()        # MRO 下一个是 A
        print("D.__init__ end")

D()
```

输出：

```
D.__init__ start
A.__init__ start
B.__init__ start         ← 注意！A 里的 super() 调到了 B，不是 Base
B.__init__ end
A.__init__ end
D.__init__ end
```

关键观察：`A` 的 `super().__init__()` 调到了 **`B`** 而不是 `Base`！因为 `D` 的 MRO 是 `[D, A, B, Base, object]`，在 `A` 之后的下一个是 `B`。**`A` 单独使用时 `super()` 才是 `Base`，但作为 `D` 的一部分时 `super()` 是 `B`**。这就是协作式多继承——每个类把 `super()` 当成"把活儿传给链上下一个"，整条链每个 `__init__` 都被调一次（钻石的公共父类 `Base` 只调一次，避免了 Java 朴素调用会调两次的问题）。

**实践铁律**：多继承体系里**每个类的每个协作方法都必须调 `super()`**，否则链会断、某些父类初始化被跳过（见 §5）。

`super()` 在 3.x 里是"魔法"——无参形式靠编译器在 code object 里塞一个隐藏的 `__class__` 闭包单元，让它知道"当前类是谁"，再结合 `self.__class__.__mro__` 找下一个。等价的显式写法是 `super(A, self)`。

### 3.7 `__slots__`：去掉 `__dict__`，改用固定槽位（呼应第 04 章）

如 §3.1 所说，每个实例背一个 `__dict__` 很费内存。声明 `__slots__` 让 Python **不创建 `__dict__`**，改用类似 Java 字段的**固定槽位**（在类型对象里预留偏移，实例按偏移存值）：

```python
import sys

class PointDict:                 # 默认：有 __dict__
    def __init__(self, x, y):
        self.x = x
        self.y = y

class PointSlots:                # 固定槽位，无 __dict__
    __slots__ = ("x", "y")
    def __init__(self, x, y):
        self.x = x
        self.y = y

p1 = PointDict(1, 2)
p2 = PointSlots(1, 2)
print(hasattr(p1, "__dict__"))   # 输出: True
print(hasattr(p2, "__dict__"))   # 输出: False
print(p2.x, p2.y)                # 输出: 1 2   照常用

p1.z = 99                        # OK，dict 随便加
try:
    p2.z = 99                    # 报错！槽位里没 z
except AttributeError as e:
    print("slots 拒绝:", e)      # 输出: slots 拒绝: 'PointSlots' object has no attribute 'z'
```

`__slots__` 的效果（数据/AI 海量小对象场景尤其重要）：
- **省内存**：没有 per-instance dict，几百万个对象能省下可观内存（常省 40%~50%）。
- **略快**：属性访问走固定槽位（类似按偏移），比 dict 查找快一点。
- **代价**：不能动态加属性（`p2.z = 99` 报错）；多继承时多个非空 `__slots__` 有冲突限制；默认也不能弱引用（需显式加 `"__weakref__"`）。

这与 [04 章 运行时与内存模型](04_运行时与内存模型.md) §2.6 提到的 `__slots__` 省内存呼应，也是 [14 章 dataclass](14_类型系统与注解.md) 里 `@dataclass(slots=True)` 的底层。

### 3.8 名称改写 `__x` → `_ClassName__x`

两个前导下划线（且**最多一个**尾随下划线）的属性名会被编译器**改写**：在类体内写 `self.__x`，实际存的键是 `_ClassName__x`。

```python
class Account:
    def __init__(self):
        self.__balance = 100      # 实际存为 _Account__balance

a = Account()
print(a.__dict__)                 # 输出: {'_Account__balance': 100}
# print(a.__balance)              # AttributeError：没有叫 __balance 的属性
print(a._Account__balance)        # 输出: 100   用改写后的名字照样访问到——可见不是真 private
```

**它的真正目的不是保密，是防子类意外覆盖**。设想父类内部用 `self.__cache`，子类作者不知情也用了 `self.__cache`——若不改写，子类会覆盖父类的内部状态导致 bug。改写后父类的是 `_Parent__cache`、子类的是 `_Child__cache`，互不干扰：

```python
class Parent:
    def __init__(self):
        self.__id = "parent-internal"   # _Parent__id
    def parent_id(self):
        return self.__id                # 编译期就绑定成 _Parent__id

class Child(Parent):
    def __init__(self):
        super().__init__()
        self.__id = "child-internal"    # _Child__id，不会覆盖父类的 _Parent__id
    def child_id(self):
        return self.__id

c = Child()
print(c.parent_id(), c.child_id())      # 输出: parent-internal child-internal
```

**对照 Java**：Java 的 `private` 是编译器强制的访问控制，外部真的访问不到。Python 的 `__x` 只是个**改名规则**，知道规则就能绕过——所以它解决的是"命名冲突"而非"访问控制"。约定上，对外内部成员用**单**下划线 `_x` 表"私有"，**双**下划线 `__x` 留给"我真的不想被子类碰"的少数情况。

### 3.9 `type(obj)` 与"类也是对象"（引出第 19 章）

Java 里 `obj.getClass()` 返回一个 `Class<?>` 对象，但 `Class` 是特殊的反射设施。Python 走得更彻底：**类本身就是一个普通对象，是 `type` 的实例**。

```python
class User: pass

u = User()
print(type(u))            # 输出: <class '__main__.User'>   实例的类型是 User
print(type(User))        # 输出: <class 'type'>            User 的类型是 type！
print(isinstance(User, type))   # 输出: True
```

`User` 这个类是 `type` 创建的对象，可以被赋值、传参、动态创建（`type("Name", bases, dict)`）。这就是为什么 Python 能做装饰器、能在运行时改类、能用元类拦截类的创建。"类是对象、`type` 是元类"是 [19 章 描述符与元类](19_描述符与元类.md) 的主题，本章点到为止。

## ④ 使用方法与最佳实践

### 4.1 `__init__` 初始化、实例属性 vs 类属性

```python
class BankAccount:
    bank_name = "PyBank"          # 类属性：所有实例共享（类比 Java static 字段）
    interest_rate = 0.03          # 共享常量

    def __init__(self, owner, balance=0):
        self.owner = owner        # 实例属性：每个实例独有
        self.balance = balance

a1 = BankAccount("Alice", 100)
a2 = BankAccount("Bob", 200)
print(a1.bank_name, a2.bank_name)   # 输出: PyBank PyBank   共享同一个
BankAccount.interest_rate = 0.05    # 改类属性，影响所有实例
print(a1.interest_rate)             # 输出: 0.05
```

- **类属性**用于：共享配置、常量、计数器（对照 Java `static` 字段）。注意**可变类属性是大坑**，见 §5。
- **实例属性**全部在 `__init__` 里初始化好，别在其他方法里"首次出现"——否则可读性差，且对象状态不明确。

### 4.2 `@property`：计算属性与校验

```python
class Circle:
    def __init__(self, radius):
        self.radius = radius      # 触发下面的 setter 做校验

    @property
    def radius(self):             # getter
        return self._radius

    @radius.setter
    def radius(self, value):      # setter：集中校验
        if value <= 0:
            raise ValueError("半径必须为正")
        self._radius = value

    @property
    def area(self):               # 只读计算属性，没有 setter
        return 3.14159 * self._radius ** 2

c = Circle(5)
print(c.area)                     # 输出: 78.53975   像访问属性一样，其实是计算
c.radius = 10                     # 走 setter 校验
print(c.area)                     # 输出: 314.159
try:
    c.radius = -1                 # 校验拦截
except ValueError as e:
    print(e)                      # 输出: 半径必须为正
# c.area = 100                    # AttributeError：area 没有 setter，只读
```

最佳实践：**一开始用普通属性，需要校验/计算时再升级为 `@property`**，调用方零改动（呼应 §2.2）。

### 4.3 `@classmethod` 做替代构造器，`@staticmethod` 做工具函数

Java 没有"多个构造器重载"的优雅替代，常用静态工厂 `User.of(...)`。Python 因为 `__init__` 不能重载，**替代构造器是 `@classmethod` 的核心用途**：

```python
from datetime import date

class Employee:
    def __init__(self, name, birth_year):
        self.name = name
        self.birth_year = birth_year

    @classmethod
    def from_csv_row(cls, row):           # cls 是"被调用的类"，支持继承
        name, year = row.split(",")
        return cls(name.strip(), int(year))   # 用 cls() 而非 Employee()

    @classmethod
    def from_age(cls, name, age):         # 另一个"构造器"
        return cls(name, date.today().year - age)

    @staticmethod
    def is_valid_year(year):              # 不依赖 self/cls 的工具函数
        return 1900 <= year <= date.today().year

e1 = Employee.from_csv_row("Alice, 1990")
e2 = Employee.from_age("Bob", 30)
print(e1.name, e1.birth_year)             # 输出: Alice 1990
print(e2.birth_year)                      # 输出: 1996  （2026-30）
print(Employee.is_valid_year(1850))       # 输出: False
```

**为什么 classmethod 用 `cls()` 而不是写死 `Employee()`**：如果有子类 `Manager(Employee)`，`Manager.from_csv_row(...)` 会让 `cls` 是 `Manager`，返回 `Manager` 实例。写死 `Employee()` 就破坏了继承——这正是 classmethod 比 staticmethod 优越之处。

| | 拿到 | 用途 | Java 类比 |
|---|---|---|---|
| 实例方法 | `self` | 操作实例 | 普通方法 |
| `@classmethod` | `cls` | 替代构造器、操作类级状态 | `static` 工厂方法（但知道实际类） |
| `@staticmethod` | 都不拿 | 逻辑相关但不依赖状态的工具 | `static` 工具方法 |

### 4.4 魔术方法：让自定义类型像内置类型

#### `__repr__` vs `__str__`（对照 `toString`）

- `__repr__`：给**开发者**看，目标是**无歧义**，最好能粘回代码重建对象。`repr(x)`、交互式回显、容器内元素、调试器都用它。
- `__str__`：给**最终用户**看，目标是**可读**。`print(x)`、`str(x)`、f-string 默认用它（没定义则回退到 `__repr__`）。

```python
class Money:
    def __init__(self, cents, currency="USD"):
        self.cents = cents
        self.currency = currency

    def __repr__(self):                   # 开发者友好、可重建
        return f"Money(cents={self.cents}, currency={self.currency!r})"

    def __str__(self):                    # 用户友好
        return f"{self.cents / 100:.2f} {self.currency}"

m = Money(1999)
print(repr(m))                            # 输出: Money(cents=1999, currency='USD')
print(str(m))                             # 输出: 19.99 USD
print([m])                                # 输出: [Money(cents=1999, currency='USD')]  容器用 repr
```

**最佳实践：至少实现 `__repr__`**。没有它，对象打印成 `<__main__.Money object at 0x...>`，调试时一无所知。`__str__` 可选。

#### `__eq__` 与 `__hash__` 必须配对（对照 `equals`/`hashCode`）

和 Java"重写 `equals` 必须重写 `hashCode`"的铁律一样，但 Python 的坑更隐蔽：**只要你定义了 `__eq__`，Python 会自动把 `__hash__` 设为 `None`，对象立刻变成不可哈希**（不能进 set、不能做 dict 键）。

```python
class Money:
    def __init__(self, cents, currency="USD"):
        self.cents = cents
        self.currency = currency

    def __eq__(self, other):
        if not isinstance(other, Money):
            return NotImplemented          # 让 Python 尝试反向比较，而非直接 False
        return (self.cents, self.currency) == (other.cents, other.currency)

    def __hash__(self):                    # 必须配对！否则不可哈希
        return hash((self.cents, self.currency))   # 与 __eq__ 用相同字段

a = Money(1999)
b = Money(1999)
print(a == b)                              # 输出: True   值相等
print(a is b)                              # 输出: False  不是同一对象
print({a, b})                              # 输出: {Money(cents=1999, currency='USD')}  去重成一个
print(len({a: 1, b: 2}))                   # 输出: 1   同键
```

铁律（同 Java）：
- `a == b` 为真 ⟹ `hash(a) == hash(b)` 必须相等。所以 `__hash__` 和 `__eq__` **用同一组字段**。
- `__eq__` 里类型不匹配返回 `NotImplemented`（不是 `False`），给对方一个反向比较的机会。
- **可变对象不要做 dict 键**：如果用于哈希的字段会变，对象在容器里会"丢失"。要么字段设计成不可变，要么干脆不实现 `__hash__`。

> 预告：[14 章 dataclass](14_类型系统与注解.md) 的 `@dataclass(frozen=True)` 自动生成配套的 `__eq__` 和 `__hash__`，省掉这些样板——但你得先懂这章的原理才知道它在帮你做什么。

#### `__lt__` 等富比较 + 排序（对照 `Comparable`）

实现 `__lt__`（小于）就能用 `sorted()`/`min()`/`max()`。配合 `functools.total_ordering` 可由 `__eq__`+`__lt__` 自动补全其余比较运算符：

```python
from functools import total_ordering

@total_ordering
class Version:
    def __init__(self, major, minor):
        self.major, self.minor = major, minor

    def __eq__(self, other):
        return (self.major, self.minor) == (other.major, other.minor)

    def __lt__(self, other):
        return (self.major, self.minor) < (other.major, other.minor)

    def __repr__(self):
        return f"v{self.major}.{self.minor}"

    __hash__ = None    # 这里不需要哈希就显式声明（演示用）

vs = [Version(1, 2), Version(1, 0), Version(2, 1)]
print(sorted(vs))               # 输出: [v1.0, v1.2, v2.1]
print(max(vs))                  # 输出: v2.1   total_ordering 补全了 > 的逻辑
```

#### `__add__` 运算符重载

```python
class Vector:
    def __init__(self, x, y):
        self.x, self.y = x, y
    def __add__(self, other):              # self + other
        return Vector(self.x + other.x, self.y + other.y)
    def __mul__(self, scalar):             # self * scalar
        return Vector(self.x * scalar, self.y * scalar)
    def __repr__(self):
        return f"Vector({self.x}, {self.y})"

v = Vector(1, 2) + Vector(3, 4)
print(v)                                   # 输出: Vector(4, 6)
print(Vector(1, 2) * 3)                    # 输出: Vector(3, 6)
```

#### 容器协议：`__len__`/`__getitem__`/`__contains__`/`__iter__`/`__call__`

让自定义类支持 `len()`、`[]`、`in`、`for`、`()`：

```python
class Playlist:
    def __init__(self, songs):
        self._songs = list(songs)

    def __len__(self):                     # len(pl)
        return len(self._songs)

    def __getitem__(self, index):          # pl[0]、pl[1:3]（切片）、支持 for 兜底
        return self._songs[index]

    def __contains__(self, song):          # song in pl
        return song in self._songs

    def __iter__(self):                    # for s in pl（优先于 __getitem__）
        return iter(self._songs)

    def __call__(self, n):                 # pl(n) —— 实例可调用，像函数
        return self._songs[:n]

pl = Playlist(["A", "B", "C", "D"])
print(len(pl))                             # 输出: 4
print(pl[1])                               # 输出: B
print(pl[1:3])                             # 输出: ['B', 'C']
print("C" in pl)                           # 输出: True
print([s.lower() for s in pl])             # 输出: ['a', 'b', 'c', 'd']  可迭代
print(pl(2))                               # 输出: ['A', 'B']  实例当函数调用
```

> `__call__` 的妙用：让实例像函数（有状态的"函数对象"，对照 Java 的函数式接口实现类）。`__getitem__`/`__iter__` 是迭代器协议的入口，完整的迭代器/生成器机制见 [16 章 迭代器与生成器](16_迭代器与生成器.md)。

### 4.5 用 ABC 定义契约（抽象基类）

`abc.ABC` + `@abstractmethod` 对应 Java 的 `abstract class`/`interface`：定义契约、强制子类实现、不能直接实例化抽象类。

```python
from abc import ABC, abstractmethod

class PaymentGateway(ABC):                 # 抽象基类 = 契约
    @abstractmethod
    def charge(self, amount: int) -> str:  # 子类必须实现
        ...

    @abstractmethod
    def refund(self, txn_id: str) -> bool:
        ...

    def describe(self):                    # 可以有具体方法（像 Java abstract class）
        return f"Gateway: {type(self).__name__}"

class StripeGateway(PaymentGateway):
    def charge(self, amount):
        return f"stripe-txn-{amount}"
    def refund(self, txn_id):
        return True

s = StripeGateway()
print(s.charge(100))                       # 输出: stripe-txn-100
print(s.describe())                        # 输出: Gateway: StripeGateway

try:
    PaymentGateway()                       # 不能实例化抽象类
except TypeError as e:
    print(e)                               # 输出: Can't instantiate abstract class PaymentGateway ...
```

如果子类漏实现某个 `@abstractmethod`，实例化时直接 `TypeError`——契约被强制。注意：Python 的 ABC 是**运行时**检查（实例化那一刻），不是 Java 的编译期检查。

### 4.6 enum 枚举

```python
from enum import Enum, auto

class Status(Enum):
    PENDING = auto()                       # auto() 自动赋值 1, 2, 3
    ACTIVE = auto()
    CLOSED = auto()

print(Status.ACTIVE)                       # 输出: Status.ACTIVE
print(Status.ACTIVE.name)                  # 输出: ACTIVE
print(Status.ACTIVE.value)                 # 输出: 2
print(Status(2))                           # 输出: Status.ACTIVE   按值查找
print(Status["ACTIVE"])                    # 输出: Status.ACTIVE   按名查找
print(list(Status))                        # 输出: [Status.PENDING, Status.ACTIVE, Status.CLOSED]

# 枚举是单例，可以用 is 比较
print(Status.ACTIVE is Status.ACTIVE)      # 输出: True

# 带方法的枚举（比 Java 弱，但够用）
class Priority(Enum):
    LOW = 1
    HIGH = 3
    def is_urgent(self):
        return self.value >= 3
print(Priority.HIGH.is_urgent())           # 输出: True
```

进阶：`IntEnum`（值是 int，可与整数比较）、`Flag`/`IntFlag`（位标志，可 `|` 组合）、`StrEnum`（3.11+，值是 str）。**最佳实践：用枚举取代"魔法字符串/魔法数字"常量**，获得类型安全、自动补全和可读的 repr。

### 4.7 组合优于继承、super() 协作

- **组合优于继承**：和 Java 一样的原则，且 Python 鸭子类型让组合更自然——不需要共同基类，只要对象有需要的方法即可。继承主要用于"真 is-a"和 mixin。
- **mixin（混入类）**：用多继承给类拼装正交能力。命名常以 `Mixin` 结尾，自身不独立实例化：

```python
class JsonMixin:                           # 正交能力：序列化
    def to_json(self):
        import json
        return json.dumps(self.__dict__)

class TimestampMixin:                      # 正交能力：时间戳
    def touch(self):
        import time
        self.updated_at = time.time()

class Document(JsonMixin, TimestampMixin):
    def __init__(self, title):
        self.title = title

d = Document("Report")
d.touch()
print(d.to_json())                         # 输出: {"title": "Report", "updated_at": 17...}
```

- **协作式 super()**：多继承体系里每个类的协作方法都调 `super()`，让 MRO 链上每个类都参与（见 §3.6）。

## ⑤ ⚠️ 易错点（Java 思维翻车）

| Java 习惯 | 翻车现场 | 正确做法/认知 |
|----------|---------|--------------|
| 把可变对象当 `static` 字段初始化 | **可变类属性被所有实例共享**：`class C: items = []`，一个实例 `self.items.append(x)` 后所有实例的 `items` 都变了 | 可变默认值放 `__init__` 里：`self.items = []`（每实例独立）；类属性只放不可变的常量/配置 |
| 方法里直接写字段名（`this` 隐式） | **忘了 self**：`def area(self): return r * r`（`r` 未定义或撞到全局），或定义方法漏写 `self` 参数 | 实例属性一律 `self.x`，每个实例方法第一参数写 `self` |
| 重写 `equals` 配 `hashCode` | **实现 `__eq__` 没实现 `__hash__`**：对象变不可哈希，进 set/dict 报 `TypeError: unhashable type` | `__eq__` 和 `__hash__` 配对实现，用相同字段；或用 `@dataclass(frozen=True)` 自动生成 |
| 以为 `private` 真的私有 | **以为 `__x` 是真 private**：试图靠它保密，结果 `obj._Cls__x` 直接读到；或子类里 `self.__x` 莫名访问不到父类的 | `__x` 只是改名防冲突，非访问控制；对外"私有"用单下划线 `_x` 约定 |
| `super.foo()` 调直接父类 | **多继承不调 super() 破坏 MRO**：某个类 `__init__` 里没调 `super().__init__()`，导致 MRO 链后面的父类没被初始化 | 协作体系里每个类的协作方法都调 `super()`；理解 super 是"MRO 下一个"不是"父类" |
| 把 Python enum 当 Java enum 那样用 | **enum 当普通类用**：`Status.ACTIVE.value` 拿来直接算术、或在枚举体里写 `__init__` 想加构造参数却搞错 | 用 `.name`/`.value`/`is` 比较；需要带数据的枚举用规范写法（`value` 元组或 `__init__`）；魔法常量优先用 enum |
| 以为 `__init__` 能重载 | 定义两个 `__init__` 想重载，**后一个直接覆盖前一个** | 用默认参数 / `@classmethod` 替代构造器；或 `functools.singledispatchmethod` |
| 类属性当"实例计数器"却忘了作用域 | `self.count += 1` 想累加类属性，实际在实例 `__dict__` 新建了 count 遮蔽类属性 | 改类级状态用 `type(self).count += 1` 或 `ClassName.count += 1` |

补充演示——最隐蔽的"可变类属性共享"：

```python
class ShoppingCart:
    items = []                  # 危险！类属性，所有实例共享同一个 list
    def add(self, item):
        self.items.append(item)

c1, c2 = ShoppingCart(), ShoppingCart()
c1.add("apple")
print(c2.items)                 # 输出: ['apple']   c2 莫名其妙有了 c1 的东西！
print(c1.items is c2.items)     # 输出: True        是同一个 list
```

修复：把 `items` 放进 `__init__`，每个实例独立持有：

```python
class ShoppingCart:
    def __init__(self):
        self.items = []         # 每个实例自己的 list
    def add(self, item):
        self.items.append(item)
```

### 综合实战

**5.**（带值相等语义的领域对象）后端开发中你需要一个 `Money` 值对象（value object），要求：
- 用整数 `cents`（分）存储金额，附带 `currency`（如 "USD"），避免浮点误差；
- **值相等**语义：`cents` 和 `currency` 都相同即相等，且可放进 `set`/做 `dict` 键；
- 支持 `+`（同币种相加，不同币种抛异常）；
- 支持按金额排序（`sorted`）；
- 有开发友好的 `__repr__` 和用户友好的 `__str__`；
- 金额一旦创建不可变（提示：可用 `@property` 只读、或预告 14 章的 `frozen`）。

写出这个类，并演示相等、入集合、相加、排序、异常路径。说明这个设计对照 Java 里你会怎么实现（构造、`equals`/`hashCode`、`Comparable`、不可变）。

**1.**

```python
class Temperature:
    def __init__(self, celsius=0.0):
        self._celsius = celsius

    @property
    def celsius(self):
        return self._celsius

    @celsius.setter
    def celsius(self, value):
        self._celsius = value

    @property
    def fahrenheit(self):                 # 计算属性：读时换算
        return self._celsius * 9 / 5 + 32

    @fahrenheit.setter
    def fahrenheit(self, value):          # 写时换算回摄氏度
        self._celsius = (value - 32) * 5 / 9

t = Temperature(100)
print(t.fahrenheit)                       # 输出: 212.0
t.fahrenheit = 32
print(t.celsius)                          # 输出: 0.0
```

用的是 `@property` 数据描述符机制（§3.3）。对照 Java：你会写 `getCelsius/setCelsius/getFahrenheit/setFahrenheit` 四个方法，调用方写 `t.getFahrenheit()`；Python 用 `t.fahrenheit` 属性语法，且后续把字段改成计算逻辑时调用方零改动（统一访问原则）。

**2.** `@classmethod` 第一个参数是 `cls`（被调用的**类**本身），`@staticmethod` 不接收 `self` 也不接收 `cls`（就是个挂在类命名空间下的普通函数）。典型用途：classmethod 做替代构造器（`from_xxx`）、操作类级状态；staticmethod 做与类逻辑相关但不依赖任何实例/类状态的工具函数。用 `cls(...)` 而非写死 `ClassName(...)` 是为了**继承友好**：子类 `Sub.from_xxx()` 调用时 `cls` 是 `Sub`，能正确返回 `Sub` 实例；写死父类名会让子类调用也返回父类实例，破坏多态。

**3.** 根因：`members = []` 是**类属性**，所有实例共享同一个 list（§3.1、§5）。`a.add_member` 里 `self.members.append` 没有给实例新建属性，而是**就地修改了那个共享 list**（读 `self.members` 走类属性，append 改的是它本身），所以 `b.members` 也看到 `["Alice"]`。打印结果是 `['Alice']` 而非 `[]`。修复——把列表放进 `__init__` 成为实例属性：

```python
class Team:
    def __init__(self, name):
        self.name = name
        self.members = []                 # 每个实例独立的 list
    def add_member(self, person):
        self.members.append(person)
```

注意区分：`self.members.append(x)`（就地改，命中共享类属性）和 `self.members = [...]`（赋值，才会在实例 dict 新建属性遮蔽类属性）——前者正是陷阱所在。

**4.** 问题：一旦定义了 `__eq__`，Python 自动把 `__hash__` 置为 `None`，`Point` 变不可哈希，放进 `set` 抛 `TypeError: unhashable type: 'Point'`（§4.4）。Java 老手栽在这里是因为 Java 里不重写 `hashCode` 只是"哈希行为不理想"（仍能进 HashSet，只是按对象身份散列），**不会直接报错**；Python 却是直接禁止哈希，行为更激进。修复——配对实现 `__hash__`，用与 `__eq__` 相同的字段：

```python
class Point:
    def __init__(self, x, y):
        self.x, self.y = x, y
    def __eq__(self, other):
        if not isinstance(other, Point):
            return NotImplemented
        return (self.x, self.y) == (other.x, other.y)
    def __hash__(self):
        return hash((self.x, self.y))     # 与 __eq__ 同字段
    def __repr__(self):
        return f"Point({self.x}, {self.y})"

pts = {Point(1, 2), Point(1, 2), Point(3, 4)}
print(len(pts))                           # 输出: 2   正确去重
```

```python
from functools import total_ordering

@total_ordering
class Money:
    __slots__ = ("_cents", "_currency")          # 省内存 + 阻止随意加属性（弱不可变）

    def __init__(self, cents: int, currency: str = "USD"):
        object.__setattr__(self, "_cents", int(cents))
        object.__setattr__(self, "_currency", currency)

    # --- 只读属性，实现不可变 ---
    @property
    def cents(self): return self._cents
    @property
    def currency(self): return self._currency

    # --- 值相等 + 哈希配对 ---
    def __eq__(self, other):
        if not isinstance(other, Money):
            return NotImplemented
        return (self._cents, self._currency) == (other._cents, other._currency)

    def __hash__(self):
        return hash((self._cents, self._currency))

    # --- 排序（同币种才有意义，这里简化按 cents）---
    def __lt__(self, other):
        if not isinstance(other, Money) or self._currency != other._currency:
            return NotImplemented
        return self._cents < other._cents

    # --- 加法：同币种相加 ---
    def __add__(self, other):
        if not isinstance(other, Money):
            return NotImplemented
        if self._currency != other._currency:
            raise ValueError(f"币种不匹配: {self._currency} + {other._currency}")
        return Money(self._cents + other._cents, self._currency)

    def __repr__(self):
        return f"Money(cents={self._cents}, currency={self._currency!r})"

    def __str__(self):
        return f"{self._cents / 100:.2f} {self._currency}"


a = Money(1999)
b = Money(1999)
print(a == b, hash(a) == hash(b))         # 输出: True True
print({a, b})                             # 输出: {Money(cents=1999, currency='USD')}  去重
print(a + Money(1))                       # 输出: Money(cents=2000, currency='USD')
print(str(a + Money(1)))                  # 输出: 20.00 USD
print(sorted([Money(300), Money(100), Money(200)]))
# 输出: [Money(cents=100, ...), Money(cents=200, ...), Money(cents=300, ...)]
try:
    Money(100, "USD") + Money(100, "EUR")
except ValueError as e:
    print(e)                              # 输出: 币种不匹配: USD + EUR
```

对照 Java：你会写一个 `final class Money`，字段 `private final long cents; private final String currency;`，构造器赋值，重写 `equals`+`hashCode`（用 `Objects.equals`/`Objects.hash`），`implements Comparable<Money>` 写 `compareTo`，加法返回 new 对象保持不可变，`toString` 给展示。Python 这边：值相等用 `__eq__`+`__hash__` 配对、排序用 `__lt__`+`@total_ordering`（自动补全其余比较）、不可变用 `__slots__`+只读 `@property`、加法用 `__add__` 运算符重载让 `a + b` 自然可读。**更进一步**，[14 章](14_类型系统与注解.md) 的 `@dataclass(frozen=True, slots=True)` 能自动生成 `__init__`/`__repr__`/`__eq__`/`__hash__`，把这段样板压到几行——但理解了本章的底层，你才知道它替你做了什么。
