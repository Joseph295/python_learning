# 28 · 测试 ⭐

> 你写过几千个 JUnit 用例，`@Test`/`@BeforeEach`/`assertEquals`/Mockito 早已是肌肉记忆。本章告诉你 Python 测试是**另一套美学**：用**裸 `assert`** 取代几十个断言方法、用**fixture 依赖注入**取代生命周期注解、用**约定优于配置**自动发现用例。重点会挖一个让 Java 老手震惊的底层魔法——**pytest 在导入时改写你的 AST**，这才是它敢只用一个 `assert` 关键字就把失败信息打印得比 AssertJ 还详细的原因。再加上 mock 的"属性替换"本质、hypothesis 的反例缩小算法，你就理解了 Python 测试栈的全貌。

## ① Java 对照表

| 维度 | Java 生态 | Python 生态 | 一句话差异 |
|------|-----------|-------------|-----------|
| 主流测试框架 | JUnit 5 / TestNG | **pytest**（第三方，事实标准）/ `unittest`（标准库） | pytest 不是标准库但人人用；unittest 像 JUnit 3 的移植 |
| 测试方法标记 | `@Test` 注解 | **`test_` 前缀**的函数（约定，无注解） | 约定优于配置：靠命名自动发现 |
| 断言 | `assertEquals(a,b)` / AssertJ `assertThat` | **裸 `assert a == b`** | pytest 重写 AST，裸 assert 也能打印详细中间值 |
| 前置/后置 | `@BeforeEach`/`@AfterEach`/`@BeforeAll` | **fixture**（依赖注入） + `yield` | fixture 按参数名注入，比注解更灵活、可组合 |
| 共享配置 | 基类 / `@ExtendWith` / Spring `TestConfiguration` | **`conftest.py`**（自动发现，无需 import） | 放对目录就生效，靠文件位置决定作用范围 |
| 参数化 | `@ParameterizedTest` + `@MethodSource` | **`@pytest.mark.parametrize`** | 一个装饰器搞定，比 JUnit 的 source 样板少 |
| 分组/条件执行 | `@Tag` / `@Disabled` / `Assumptions` | **`@pytest.mark.<name>`** / `@pytest.mark.skip` | marker 自定义灵活，配 `-m` 表达式筛选 |
| Mock | Mockito `mock()`/`when()`/`verify()` | **`unittest.mock`** / `monkeypatch` fixture | mock 本质是运行时替换属性，结束自动还原 |
| 异常断言 | `assertThrows(E.class, ...)` | **`with pytest.raises(E):`** | 上下文管理器包住，还能断言消息 |
| 覆盖率 | JaCoCo | **`coverage.py`** / `pytest-cov` | 行/分支覆盖，输出 HTML/term |
| 属性测试 | jqwik | **hypothesis** | 自动生成输入 + 缩小反例 |
| 多环境矩阵跑测 | Maven Surefire + profiles | **tox** / **nox** | 在多个 Python 版本/依赖组合下跑测试 |
| 构建集成 | `mvn test` / `gradle test` | `pytest` / `tox` | pytest 直接当命令跑，无需构建工具 |

> 心智锚点：**pytest ≈ "JUnit5 + AssertJ + Mockito + 参数化"全家桶，但 API 极简**。Java 靠注解和方法堆出能力，pytest 靠"函数 + fixture 注入 + AST 魔法"把样板压到最低。

## ② 设计理念（为什么 Python 这么设计）

### 2.1 裸 assert 取代断言方法：把样板还给语言

JUnit 为什么需要 `assertEquals`、`assertTrue`、`assertThat(x).isEqualTo(y)` 这一大家子方法？因为 Java 的 `assert` 关键字**默认关闭**（要 `-ea`），而且失败时只抛 `AssertionError`，**不告诉你 `a` 和 `b` 实际是什么**。所以 JUnit/AssertJ 必须自己实现断言库，把"期望值 vs 实际值"格式化出来。

pytest 反其道而行：**就用 Python 内置的 `assert` 语句**。

```python
def test_total():
    cart = {"apple": 3, "pear": 2}
    assert sum(cart.values()) == 6   # 直接写，失败时 pytest 会告诉你左边算出来是 5
```

代价是：内置 `assert` 失败默认只抛裸 `AssertionError`，看不到中间值。pytest 用一个底层魔法解决了这点（见 ③ 的 AST 重写），让裸 assert 的失败信息**比 AssertJ 还详细**——你不必学几十个断言方法，一个 `assert` 配任意 Python 表达式就够了。这是 pytest 设计哲学的核心：**不发明 DSL，复用语言本身**。

### 2.2 fixture 依赖注入：比生命周期注解更可组合

JUnit 的 `@BeforeEach` 是"过程式"的：方法体里把 `this.db`、`this.service` 一个个初始化好，所有测试方法共享这些字段。问题是：

- 某个测试**不需要** db，照样会执行 `@BeforeEach` 建 db（浪费）。
- 想给不同测试**不同的前置组合**，得拆基类或加 `if`，很别扭。
- 前置之间的**依赖关系**（建 db 才能建 repo）只能靠代码顺序，不显式。

pytest 用**依赖注入**重构这件事：前置逻辑写成 **fixture 函数**，测试函数**通过参数名"点菜"**，要哪个写哪个。

```python
import pytest

@pytest.fixture
def db():                       # 一个 fixture
    conn = connect()
    yield conn                  # yield 之前是 setup，之后是 teardown
    conn.close()

@pytest.fixture
def repo(db):                   # fixture 可以依赖别的 fixture（DI 链）
    return UserRepo(db)

def test_find(repo):            # 测试只声明它要 repo，pytest 自动建 db→repo 注入
    assert repo.find(1).name == "Alice"

def test_pure_math():           # 这个测试不点 db/repo，就完全不会建它们
    assert 2 + 2 == 4
```

这和 Spring 的依赖注入是**同一种思想**：声明你要什么，容器负责装配。pytest 把"前置"从"继承/注解"模型升级成了"DI 容器"模型——更显式、更可组合、按需创建。

### 2.3 约定优于配置：靠命名自动发现

JUnit 5 靠 `@Test` 注解标记。pytest 靠**命名约定**：

- 文件名 `test_*.py` 或 `*_test.py`
- 函数名 `test_*`
- 类名 `Test*`（且不能有 `__init__`）

`pytest` 命令递归扫描，自动收集。没有注解、没有 XML、没有 `@RunWith`。这呼应了 Python 的整体哲学（见 [01 设计哲学与心智模型](01_设计哲学与心智模型.md)）：**约定优于配置，显式优于隐式，但能省的样板尽量省**。

### 2.4 基于属性的测试：让机器找反例

传统"基于样例"的测试（example-based）是你**手写几个输入和期望输出**。问题是你只测了你想得到的情况，想不到的边界（空串、`0`、超大数、Unicode、`NaN`）就漏了。

hypothesis（对应 Java 的 jqwik）代表**基于属性的测试**（property-based testing）思想：你不写具体样例，而是声明一条**对所有输入都该成立的性质**，框架**自动生成成百上千个随机输入**去攻击它，一旦找到反例，还会**自动缩小**（shrink）成最小的失败用例。

```python
from hypothesis import given, strategies as st

@given(st.lists(st.integers()))
def test_sort_idempotent(xs):
    # 性质：排两次和排一次结果一样（对任意整数列表都成立）
    assert sorted(sorted(xs)) == sorted(xs)
```

理念差异：example-based 测"我想到的情况"，property-based 测"我声明的规律"，让机器去找我没想到的反例。这在数据/算法/序列化场景（你做的方向）尤其值钱。

## ③ 实现细节（底层怎么实现的）★本章重点★

### 3.1 pytest 的 assert 重写（assertion rewriting）——本章的"WOW"

这是 pytest 最反直觉、也最该让 Java 老手"知其所以然"的机制。先看现象：

```python
# test_demo.py
def test_it():
    a = [1, 2, 3]
    b = [1, 2, 4]
    assert a == b
```

用纯 Python 跑 `python test_demo.py`，断言失败你只会得到：

```
AssertionError
```

什么信息都没有。但用 `pytest test_demo.py` 跑，你得到的是：

```
    def test_it():
        a = [1, 2, 3]
        b = [1, 2, 4]
>       assert a == b
E       assert [1, 2, 3] == [1, 2, 4]
E         At index 2 diff: 3 != 4
E         Full diff:
E         - [1, 2, 4]
E         + [1, 2, 3]
```

**同样一行裸 `assert`，pytest 凭什么打印出左右值、差异位置、逐元素 diff？** Java 里这需要 AssertJ 的 `assertThat(a).isEqualTo(b)` 链式 API 才能做到。pytest 没让你改代码，秘密在于：

> **pytest 在导入测试模块时，拦截了导入过程，把模块源码解析成 AST，把每一条 `assert` 语句"重写"成一段能捕获子表达式中间值、并在失败时格式化输出的字节码，然后才编译执行。**

这正好呼应 [05 CPython内部](05_CPython内部.md) 讲的"源码 → AST → 字节码"管线——pytest 在 **AST 阶段动了手脚**。机制拆解：

1. **import hook**：pytest 安装了一个自定义的导入器（基于 `importlib` 的 `MetaPathFinder`/`Loader`，见 [18 模块包与导入](18_模块包与导入.md)）。当一个被收集的测试模块**首次导入**时，这个 loader 接管。
2. **解析 AST**：loader 读源码，用 `ast.parse()` 得到抽象语法树。
3. **遍历改写**：一个 `ast.NodeVisitor` 找到所有 `ast.Assert` 节点。对 `assert a == b`，它把表达式拆成子表达式，生成新的 AST：先把 `a`、`b` 分别求值存进临时变量，比较失败时调用 pytest 内部的 `_format_assertmsg` / 比较解释器，拼出那段详细信息。原来的一行 assert 被替换成十几行等价但"会自我解释"的代码。
4. **编译并缓存**：改写后的 AST 用 `compile()` 编成 code object，写进 `__pycache__` 下的 `*.pyc`（带特殊标记，避免和普通 .pyc 混淆）。下次没改源码就直接用缓存。

可以亲眼看到证据：

```python
# 在测试里打印这一帧的 assert 是不是被改写了
def test_show_rewrite():
    import inspect, dis
    # 被 pytest 收集运行时，这个函数的字节码里会出现 pytest 注入的调用
    dis.dis(test_show_rewrite)   # 你会看到对 @py_assert / _format_assertmsg 之类的引用
    assert 1 == 1
```

关键认知：

- **只有"被 pytest 当测试收集"的模块才会被重写**。你的生产代码（被测对象）不会被动手脚。
- 如果你在**非测试模块**里写了想让 pytest 解释的 assert（比如测试用的辅助库），要显式注册：`pytest.register_assert_rewrite("mypkg.testutils")`，否则那里的 assert 失败信息退化成裸 AssertionError。
- 这就是为什么 pytest 文档反复说"用裸 assert 就好"——**重写机制把断言库的活儿在 AST 层干了**，你不需要 assertEquals 家族。

对照 Java：JUnit/AssertJ 在**运行时用方法调用 + 反射/toString** 拼错误信息；pytest 在**导入时用 AST 改写**注入解释逻辑。一个是库层方案，一个是编译期元编程方案——后者对使用者更透明（你只管写 assert）。

### 3.2 fixture 的依赖注入与作用域：一个迷你 DI 容器

fixture 不是普通函数，pytest 把它当成 **DI 容器里的可注入组件**来管理。底层机制：

**注册与发现。** `@pytest.fixture` 装饰器并不立刻执行函数，而是把它登记进 pytest 的 fixture 注册表（按名字索引）。收集到一个测试函数时，pytest 用 `inspect.signature` 读它的**参数名列表**，对每个参数名去注册表里找同名 fixture——**参数名就是注入的 key**（类似 Spring 的 by-name 注入）。找不到就报 `fixture 'xxx' not found`。

**依赖解析。** fixture 之间可以互相依赖（`repo(db)`），pytest 构建一张**依赖图**，做拓扑排序，保证 `db` 在 `repo` 之前建好。这就是 2.2 那条 DI 链的实现。

**作用域（scope）——状态复用的关键。** 每个 fixture 有 `scope`，决定它**多久建一次、被谁共享**：

| scope | 创建/销毁频率 | 类比 JUnit | 典型用途 |
|-------|--------------|-----------|---------|
| `function`（默认） | **每个测试函数**一次 | `@BeforeEach`/`@AfterEach` | 需要全新状态的对象（mock、临时数据） |
| `class` | 每个测试类一次 | — | 类内多个测试共享 |
| `module` | 每个 `.py` 模块一次 | — | 模块内共享只读资源 |
| `package` | 每个包一次 | — | 包级共享 |
| `session` | **整个 pytest 进程**一次 | `@BeforeAll`（但全局） | 昂贵资源：DB 连接池、Docker 容器、HTTP server |

```python
import pytest

@pytest.fixture(scope="session")
def db_engine():                 # 整个测试会话只建一次（昂贵）
    engine = create_engine("sqlite:///:memory:")
    yield engine
    engine.dispose()

@pytest.fixture(scope="function")
def db_session(db_engine):       # 每个测试一个干净事务（便宜、隔离）
    conn = db_engine.connect()
    tx = conn.begin()
    yield conn
    tx.rollback()                # 每个测试结束回滚，互不污染
    conn.close()
```

> 这是 fixture 比 JUnit 注解强的地方：**昂贵的 `db_engine` 全局建一次，便宜的 `db_session` 每个测试建一次**，两者用作用域声明式地组合。JUnit 里 `@BeforeAll`（静态、类级）和 `@BeforeEach` 的组合远没这么灵活。作用域用错会导致状态泄漏，这是 ⑤ 的重点坑。

**yield fixture 做 setup/teardown。** fixture 用 `yield` 而非 `return` 时，`yield` **之前**是 setup、**之后**是 teardown。底层是把 fixture 当**生成器**驱动（见 [16 迭代器与生成器](16_迭代器与生成器.md)）：pytest 调 `next()` 跑到 yield 拿到值注入测试；测试结束后再调一次 `next()`（触发 `StopIteration`）执行 yield 之后的清理。**即使测试抛异常，teardown 也保证执行**——等价于 try-with-resources/`@AfterEach`，但写法更紧凑（setup 和 teardown 在同一个函数里，不会分裂成两个方法）。

**finalizer 顺序。** 多个 fixture 的 teardown 按**逆序**执行（后建的先拆），和资源栈语义一致。

### 3.3 conftest.py 的发现机制：靠文件位置决定作用域

`conftest.py` 是 pytest 的"魔法配置文件"：**放在某目录下的 fixture/hook，自动对该目录及所有子目录的测试可见，无需 import**。

实现机制：pytest 在收集阶段，对每个测试文件**向上回溯目录树**，把沿途每一层的 `conftest.py` 都加载进来，其中的 fixture 注册进对该路径生效的注册表。所以：

```
tests/
├── conftest.py          # 全局 fixture（所有测试可见）
├── api/
│   ├── conftest.py      # 只对 api/ 下测试可见
│   └── test_users.py
└── db/
    └── test_repo.py     # 看不到 api/conftest.py，但看得到顶层 conftest.py
```

这是一种**基于文件系统位置的作用域**——目录层级即可见性范围。对照 Spring：conftest 像"自动扫描的 `@TestConfiguration`"，但作用域由**物理路径**而非注解/包名决定。conftest 里还能放 pytest 的 hook 函数（如 `pytest_collection_modifyitems` 修改收集结果），是 pytest 的插件挂载点。

> 重要：`conftest.py` **不要 import 它**，也几乎不在里面写 `import`（被测代码）。它是 pytest 自动加载的，手动 import 会破坏机制。

### 3.4 mock 的本质：运行时替换属性，结束自动还原

Mockito 靠**动态生成子类/代理**（CGLIB/ByteBuddy）来 mock。Python 因为是动态语言、对象的属性可随时赋值（见 [13 类与对象](13_类与对象.md) 的 `__dict__`），mock 的实现简单得多：**就是把目标对象的某个属性，临时换成一个 `Mock` 对象，测试结束再换回去**。

`unittest.mock.patch` 的机制：

1. 给定一个**字符串路径** `"myapp.service.requests"`，patch 用 `importlib` 找到 `myapp.service` 模块对象，定位它的 `requests` 属性。
2. **保存原值**，把该属性赋成一个 `MagicMock`。
3. 测试运行期间，`myapp.service` 里所有用到 `requests` 的地方，拿到的都是这个 mock。
4. patch 作为上下文管理器/装饰器，退出时（`__exit__`）**把原值赋回去**——自动还原，绝不泄漏到别的测试。

```python
from unittest.mock import patch, MagicMock

# myapp/service.py 里有: import requests; def fetch(url): return requests.get(url).json()
def test_fetch():
    with patch("myapp.service.requests") as mock_requests:   # 替换 service 模块里的 requests
        mock_requests.get.return_value.json.return_value = {"ok": True}
        from myapp.service import fetch
        assert fetch("http://x") == {"ok": True}
        mock_requests.get.assert_called_once_with("http://x")  # 对应 Mockito verify
    # with 块结束，service.requests 自动还原成真的 requests
```

**`MagicMock` 为什么"什么属性/方法都能点"？** 因为它重写了 `__getattr__`：访问任意属性时**惰性地动态创建并缓存一个子 Mock**（见 [19 描述符与元类](19_描述符与元类.md) 的属性查找机制）。`mock.a.b.c()` 一路都不报 `AttributeError`，每一层自动造一个 mock。这也是它的危险：**拼错方法名也不报错**，mock 会假装那个方法存在（坑见 ⑤）。用 `autospec=True` / `spec=` 可让 mock 按真实对象的签名校验，拼错就报错——相当于给 mock 加类型约束。

**patch 位置的铁律（最重要的实践）：patch "使用处"而非"定义处"。** 因为 patch 替换的是"某个模块命名空间里的名字"。如果 `service.py` 写 `from requests import get`，那 `get` 这个名字被绑进了 `service` 模块的命名空间，你必须 patch `"myapp.service.get"`，而不是 `"requests.get"`——patch 后者不影响已经绑好的 `service.get`。这条规则的根因是 Python 的 import 语义（名字绑定，见 [18 模块包与导入](18_模块包与导入.md)），是 Java 转 Python 写 mock 翻车最多的点，⑤ 会用代码演示。

**`monkeypatch` fixture** 是 pytest 对同一思想的封装：在测试内临时改属性/环境变量/字典项/`sys.path`，结束自动还原。比 `patch` 更轻量，适合改简单属性和环境：

```python
def test_env(monkeypatch):
    monkeypatch.setenv("API_KEY", "test-key")        # 测试结束自动还原
    monkeypatch.setattr("myapp.config.TIMEOUT", 1)   # 临时改全局配置
    monkeypatch.delitem(some_dict, "key")
```

### 3.5 coverage.py 的实现：trace 钩子记录命中行

JaCoCo 靠**字节码插桩**（往 class 字节码插探针）。coverage.py 走另一条路：用 CPython 的**追踪钩子**。

机制：`sys.settrace()`（或 3.12+ 更高效的 `sys.monitoring` PEP 669 API）注册一个回调，解释器**每执行一行就回调一次**，coverage 记录下"哪些 (文件, 行号) 被执行过"。跑完和源码的"可执行行集合"求差，没被执行的就是未覆盖行。**分支覆盖**（`--branch`）则额外记录"从哪行跳到哪行"，发现 `if` 只走了 true 分支没走 false 的情况。

```bash
pip install pytest-cov
pytest --cov=myapp --cov-report=term-missing --cov-report=html
# term-missing 直接在终端列出未覆盖的行号；html 生成可点击报告
```

代价：settrace 每行回调有开销，**带 coverage 跑测试会明显变慢**（3.12 的 sys.monitoring 大幅改善了这点）。对照 JaCoCo 的字节码插桩，coverage 的行级 trace 更简单但更慢——这又是 Python "用解释器钩子换实现简单性"的典型取舍。

### 3.6 hypothesis 的策略生成与反例缩小（shrinking）

hypothesis 两大引擎：

**策略（strategy）= 输入生成器。** `st.integers()`、`st.text()`、`st.lists(...)` 是描述"如何生成一类数据"的对象，可组合（`st.lists(st.integers(min_value=0))`）、可映射（`.map()`）、可过滤（`.filter()`）。底层 hypothesis 维护一个**字节流随机源**，策略从这个流"画"出具体值——这个设计让生成是**可重放、可缩小**的。

**缩小（shrinking）= 反例最小化，hypothesis 的灵魂。** 当随机生成的 `[847, -3, 99281, 0, -5]` 让测试挂了，原始反例往往很大、难读。hypothesis 不会直接把这个丢给你，而是**自动反复尝试更小的输入**（去掉元素、把数字往 0 靠、把字符串截短），只要仍然触发失败就继续缩，最终给你一个**最小反例**，比如 `[0]` 或 `1`。机制上它是在那个"字节流随机源"上做搜索——把生成反例的字节序列朝"字典序更小"的方向调整并重跑。

```python
from hypothesis import given, strategies as st

def encode(n: int) -> str:
    return str(n)

def decode(s: str) -> int:
    return int(s)

@given(st.integers())
def test_roundtrip(n):
    # 性质：编码再解码应还原（往返性质，序列化测试的黄金模式）
    assert decode(encode(n)) == n
```

如果你的 `encode` 对负数有 bug，hypothesis 不会报 `-723491` 这种，而是缩小成 `-1` 给你——**最小可复现反例**。它还会把失败的样例存进本地数据库（`.hypothesis/`），下次运行**优先重跑**，保证回归。对照 jqwik：思想一致（generation + shrinking），hypothesis 的策略组合 API 和缩小质量在 Python 生态里是标杆。

### 4.1 第一个 pytest 测试与 AAA 模式

```python
# tests/test_cart.py
def total_price(items: dict[str, float], quantities: dict[str, int]) -> float:
    return sum(items[k] * quantities.get(k, 0) for k in items)

def test_total_price():
    # Arrange（准备）
    prices = {"apple": 1.5, "pear": 2.0}
    qty = {"apple": 2, "pear": 1}
    # Act（执行）
    result = total_price(prices, qty)
    # Assert（断言）—— 裸 assert，失败自动打印中间值
    assert result == 5.0
```

```bash
pytest                      # 自动发现 tests/ 下所有 test_*.py
pytest -v                   # 每个用例一行，显示名字
pytest -q                   # 安静模式
pytest tests/test_cart.py::test_total_price   # 只跑某一个
pytest -k "total and not slow"                # 按名字关键字筛
pytest -x                   # 第一个失败就停（fail fast）
pytest --lf                 # 只重跑上次失败的（last-failed）
```

**AAA 模式**（Arrange-Act-Assert）是 Python 测试的通用骨架，等价于你熟悉的 Given-When-Then。一个测试**只测一件事**，断言聚焦。

### 4.2 parametrize：表驱动测试（取代 JUnit 的 @ParameterizedTest）

```python
import pytest

def is_leap(year: int) -> bool:
    return year % 4 == 0 and (year % 100 != 0 or year % 400 == 0)

@pytest.mark.parametrize("year, expected", [
    (2000, True),    # 能被 400 整除
    (1900, False),   # 被 100 整除但不被 400
    (2024, True),    # 普通闰年
    (2023, False),   # 平年
])
def test_is_leap(year, expected):
    assert is_leap(year) == expected
# 这会生成 4 个独立用例，某个挂了其它照跑，报告里能看到具体是哪组参数
```

对照 JUnit：`@ParameterizedTest` + `@CsvSource`/`@MethodSource` 一堆样板，pytest 一个装饰器搞定。**可叠加**实现笛卡尔积：

```python
@pytest.mark.parametrize("x", [1, 2])
```

### 4.4 测异常：pytest.raises（取代 assertThrows）

```python
import pytest

def withdraw(balance: int, amount: int) -> int:

def test_withdraw_overdraft():

    assert "余额不足" in str(exc_info.value)     # 还能断言异常消息

def test_withdraw_ok():
```

### 4.5 mock / patch 外部依赖（数据库、HTTP）

```python
# myapp/weather.py
import requests
def get_temp(city: str) -> float:
    resp = requests.get(f"https://api.example.com/{city}")
    resp.raise_for_status()
    return resp.json()["temp"]

# tests/test_weather.py
from unittest.mock import patch
from myapp.weather import get_temp

def test_get_temp():
    fake = {"temp": 25.5}
    # 注意 patch 的是 "myapp.weather.requests"（使用处！不是 "requests"）
    with patch("myapp.weather.requests") as m:
        m.get.return_value.json.return_value = fake
        m.get.return_value.raise_for_status.return_value = None
        assert get_temp("beijing") == 25.5
        m.get.assert_called_once_with("https://api.example.com/beijing")  # verify
```

也能用 `@patch` 装饰器形式（mock 作为参数注入，**从下往上对应参数顺序**）：

```python
@patch("myapp.weather.requests")
def test_get_temp_decorated(mock_requests):    # 装饰器离函数最近的 patch 对应第一个参数
    mock_requests.get.return_value.json.return_value = {"temp": 10.0}
    assert get_temp("paris") == 10.0
```

**实战建议**：mock **外部边界**（HTTP、第三方 SDK、时间 `datetime.now`、随机），**不要 mock 你自己的核心逻辑**。对 HTTP，社区更推荐 `responses` 或 `respx`（按 URL 配假响应），比手搓 mock 更真实。

### 4.7 hypothesis：属性测试

```python
from hypothesis import given, strategies as st, settings, example

@given(st.lists(st.integers()))
@example([])                              # 手动钉一个一定要测的样例（边界）
@settings(max_examples=200)               # 每个测试跑 200 个随机样例
def test_reverse_twice(xs):
    assert list(reversed(list(reversed(xs)))) == xs

# 组合策略：测一个解析函数对任意合法输入都不崩
@given(st.text(min_size=1))
def test_parser_no_crash(s):
    try:
        parse(s)
    except ValueError:
        pass            # 业务异常允许；只要不抛 KeyError/TypeError 这类 bug
```

**最适合属性测试的场景**（贴你的数据/AI 方向）：序列化往返（`decode(encode(x)) == x`）、编解码、排序/去重的不变量、数学函数性质（交换律、幂等）、`pydantic` 模型的 round-trip（见 [26 日期时间序列化pydantic](26_日期时间序列化pydantic.md)）。

### 4.8 unittest（标准库，何时用）

`unittest` 是标准库自带、像 JUnit 3/4 的 xUnit 框架。**新项目优先 pytest**，但你会在老代码、标准库测试、或不能装第三方包的环境里遇到它：

```python
import unittest

class TestCart(unittest.TestCase):
    def setUp(self):                       # 对应 @BeforeEach
        self.prices = {"apple": 1.5}

    def test_total(self):
        self.assertEqual(self.prices["apple"] * 2, 3.0)   # assertEquals 风格

    def tearDown(self):                    # 对应 @AfterEach
        pass

if __name__ == "__main__":
    unittest.main()
```

好消息：**pytest 能直接跑 unittest.TestCase**，所以即便有老的 unittest 用例，也用 `pytest` 一把跑全部。`unittest.mock` 则是标准库的 mock，pytest 里照样用（`unittest.mock.patch`）。

推荐结构（`src` 布局，见 [18 模块包与导入](18_模块包与导入.md)）：

```
myproject/
├── pyproject.toml
├── src/
│   └── myapp/
│       ├── __init__.py
│       └── service.py
└── tests/
    ├── conftest.py            # 共享 fixture
```

**测试金字塔**（和 Java 一致）：大量**单元测试**（快、无 I/O、mock 外部）打底，少量**集成测试**（真 DB/HTTP）居中，极少**端到端**测试封顶。Python 里用 marker 区分快慢，CI 快速通道 `pytest -m "not integration"`，夜间全量跑。`pytest -p no:randomly` 等插件可控顺序；**好的测试不该依赖执行顺序**（见 ⑤）。

## ⑤ ⚠️ 易错点（Java 思维翻车）

| Java 习惯 | 翻车现场 | 正确做法/认知 |
|----------|---------|--------------|
| 满屏 `assertEquals`/`assertThat` | 在 pytest 里还写 `self.assertEqual(a, b)`、`assertTrue(x)` 的 unittest 样板 | 用**裸 `assert a == b`**，AST 重写自动打印中间值；不需要断言方法家族 |
| Mockito patch "类/方法定义处" | `from myapp.svc import requests` 后 patch `"requests.get"`，**mock 没生效**，真打了网络 | patch **使用处的名字**：`patch("myapp.svc.requests")`。根因是 import 把名字绑进了使用方模块的命名空间 |
| `@BeforeAll` 静态共享 = `session` fixture 随便用 | 把可变对象设成 `scope="session"`，测试 A 改了它，测试 B 拿到脏状态，**还偶发**（取决于顺序） | 可变/需隔离的状态用默认 `function` scope；只读昂贵资源才用 `session` |
| 漏测异常路径 | 只测正常返回，忘了"非法输入该抛异常"，异常分支零覆盖 | 用 `with pytest.raises(ValueError, match=...)`，异常路径是必测项 |
| Mockito 万物皆 mock | mock 自己的核心逻辑、mock 一长串 `m.a.b.c.d`，测试只验证了 mock 自己，**改了实现测试还绿** | 只 mock 外部边界（HTTP/DB/时间）；mock 越多越脆。考虑 `autospec=True` 防拼错方法名 |
| mock 拼错方法名以为会报错 | `MagicMock` 的 `__getattr__` 让 `mock.serValue()`（拼错 set）也"成功"返回 mock，**假绿** | 用 `spec=RealClass` / `autospec=True`，调用不存在的方法立即报 AttributeError |
| 依赖测试执行顺序（JUnit 默认按名/`@Order`） | 测试 A 往全局/DB 写数据，测试 B 依赖它先跑，单独跑 B 就挂 | 每个测试自洽：用 fixture 准备数据、事务回滚隔离；测试间零耦合 |
| 共享可变 fixture 当常量传 | fixture 返回一个 list，多个测试往里 append，互相污染 | 默认 function scope 每次新建；或在 fixture 里返回副本/不可变结构 |
| 在 `conftest.py` 里手动 import 它 | `import conftest` 或把它当普通模块，机制错乱 | conftest 由 pytest **自动加载**，绝不手动 import |
| 以为 coverage 高 = 测得好 | 100% 行覆盖但全是无断言的"调用一下"，bug 照样漏 | 覆盖率是**下限指标**；配分支覆盖 + 有意义断言 + 属性测试才靠谱 |
| `assert` 写在生产代码里做校验 | 生产代码靠 `assert` 校验入参，上线用 `python -O` 跑，**assert 被全部移除**，校验失效 | 生产校验用 `if ... raise`；`assert` 只用于测试和不变量自检 |

## ⑥ 练习题

### 基础理解

**1.** 用纯 `python xx.py` 跑 `assert [1,2,3] == [1,2,4]` 只得到一行 `AssertionError`，而 `pytest` 跑同样的代码却能打印出左右值和逐元素 diff。从底层机制解释 pytest 是怎么做到的，发生在哪个阶段，为什么你的生产代码不受影响。

**2.** 解释 fixture 的 `scope="function"` 和 `scope="session"` 的区别，各举一个**恰当**的使用场景。为什么把一个"每个测试都要修改的可变对象"设成 `session` scope 是危险的？

### 找坑改错

**3.（patch 位置错）** 下面的测试想 mock 掉网络调用，但运行时**真的发起了 HTTP 请求**，测试还偶尔因网络失败。指出根因并改对。

```python
# myapp/notify.py
from requests import post
def send_alert(msg: str) -> int:
    resp = post("https://hooks.example.com/x", json={"text": msg})
    return resp.status_code

# tests/test_notify.py
from unittest.mock import patch
from myapp.notify import send_alert

def test_send_alert():
    with patch("requests.post") as mock_post:      # ← 这里有问题
        mock_post.return_value.status_code = 200
        assert send_alert("hi") == 200
        mock_post.assert_called_once()
```

**4.（fixture 作用域泄漏）** 下面两个测试单独跑都过，一起跑时 `test_count_after_add` **偶发失败**。指出根因并修复。

```python
import pytest

@pytest.fixture(scope="module")
def cart():
    return []                       # 一个购物车

def test_add_item(cart):
    cart.append("apple")
    assert "apple" in cart

def test_count_after_add(cart):
    cart.append("pear")
    assert len(cart) == 1           # 期望购物车里只有自己加的那 1 件
```

### 综合实战

**5.** 给定下面的"用户折扣"函数，它依赖一个外部汇率服务。请为它写一套 pytest 测试，要求：(a) 用 `@pytest.mark.parametrize` 表驱动覆盖至少 3 组折扣档位；(b) 用 `patch` mock 掉外部 `get_rate` 调用（注意 patch 位置）；(c) 用 `pytest.raises` 覆盖非法输入；(d) 额外写一个 hypothesis 属性测试，验证"折扣后价格永远 ≥ 0 且 ≤ 原价"。

```python
# myapp/pricing.py
from myapp.fx import get_rate          # get_rate(currency) -> float，会发 HTTP

def final_price(amount: float, vip_level: int, currency: str) -> float:
    if amount < 0:
        raise ValueError("amount 不能为负")
    discount = {0: 0.0, 1: 0.05, 2: 0.10, 3: 0.20}.get(vip_level)
    if discount is None:
        raise ValueError(f"未知 vip_level: {vip_level}")
    rate = get_rate(currency)          # 外部依赖
    return round(amount * (1 - discount) * rate, 2)
```

---

### 参考答案

**1.** pytest 安装了一个自定义 import hook（基于 `importlib` 的 finder/loader）。当一个**被收集为测试**的模块首次导入时，loader 拦截：读源码 → `ast.parse()` 得到 AST → 遍历找到所有 `assert` 节点 → 把每条 assert **重写**成一段会先把子表达式求值存入临时变量、失败时调用 pytest 内部格式化逻辑拼出详细信息的 AST → `compile()` 成字节码并缓存进 `__pycache__`。所以详细信息是在**导入阶段的 AST 改写**注入的（呼应 [05 CPython内部](05_CPython内部.md) 的"源码→AST→字节码"管线）。生产代码不受影响，因为**只有 pytest 收集的测试模块**才走这个重写 loader；其余模块正常导入，裸 assert 失败仍只抛 `AssertionError`（这也是为什么要用 `pytest.register_assert_rewrite` 显式登记测试辅助库）。

**2.** `function`（默认）每个测试函数建一次、用完即拆，保证**测试间状态隔离**；`session` 整个 pytest 进程只建一次、所有测试共享，适合**昂贵且只读/可重置**的资源。恰当场景：`function` → 每个测试要全新的 mock、临时数据、DB 事务；`session` → 启动一个测试用 HTTP server、建一个内存 DB engine、拉起一个 Docker 容器。把"每个测试都要改的可变对象"设成 session 危险，因为所有测试共享同一个实例，前一个测试的修改会泄漏给后一个，测试结果**依赖执行顺序、偶发失败**，违背隔离原则。

**3.** 根因：`notify.py` 用 `from requests import post` 把 `post` 这个名字**绑进了 `myapp.notify` 模块的命名空间**。`send_alert` 调用的是 `myapp.notify.post`，和 `requests.post` 是两个独立的名字绑定。`patch("requests.post")` 只改了 `requests` 模块里的 `post`，**没碰** `myapp.notify.post`，所以真请求照发。正确做法是 patch **使用处**：

```python
def test_send_alert():
    with patch("myapp.notify.post") as mock_post:   # ← 改这里
        mock_post.return_value.status_code = 200
        assert send_alert("hi") == 200
        mock_post.assert_called_once()
```

（铁律：patch 你**要替换的那个名字所在的模块**，即"被测代码实际引用它的地方"，而非它的定义处。)

**4.** 根因：`cart` 是 `scope="module"`，整个测试模块**共享同一个 list**。`test_add_item` 先 append `"apple"`，到 `test_count_after_add` 时 list 里已经有 `"apple"`，再 append `"pear"` 后 `len` 是 2 不是 1 → 失败；单独跑各自只有 1 件才通过。这是典型的**作用域泄漏 + 顺序依赖**。修复：改成默认（function）scope，每个测试拿到全新的空 list：

```python
@pytest.fixture                  # 去掉 scope="module"，默认 function
def cart():
    return []
```

（若确实需要共享昂贵资源，就别让测试去**修改**它，或在 fixture 里用 yield 在 teardown 重置状态。)

**5.** 参考实现：

```python
# tests/test_pricing.py
import pytest
from unittest.mock import patch
from hypothesis import given, strategies as st

from myapp.pricing import final_price

# (a)+(b)：表驱动 + mock 外部 get_rate（patch 使用处 myapp.pricing.get_rate）
@pytest.mark.parametrize("amount, vip, expected", [
    (100.0, 0, 100.0),     # 无折扣
    (100.0, 1, 95.0),      # 5% off
    (100.0, 2, 90.0),      # 10% off
    (100.0, 3, 80.0),      # 20% off
])
def test_final_price_levels(amount, vip, expected):
    with patch("myapp.pricing.get_rate", return_value=1.0) as mock_rate:
        assert final_price(amount, vip, "USD") == expected
        mock_rate.assert_called_once_with("USD")

# 验证汇率确实参与计算
def test_final_price_applies_rate():
    with patch("myapp.pricing.get_rate", return_value=7.0):
        assert final_price(100.0, 1, "CNY") == round(100 * 0.95 * 7.0, 2)  # 665.0

# (c)：非法输入用 pytest.raises 覆盖
def test_negative_amount():
    with patch("myapp.pricing.get_rate", return_value=1.0):
        with pytest.raises(ValueError, match="不能为负"):
            final_price(-1.0, 0, "USD")

def test_unknown_vip():
    with patch("myapp.pricing.get_rate", return_value=1.0):
        with pytest.raises(ValueError, match="未知 vip_level"):
            final_price(100.0, 99, "USD")

# (d)：hypothesis 属性测试——折扣后价格永远在 [0, amount*rate] 内
@given(
    amount=st.floats(min_value=0, max_value=1e6, allow_nan=False, allow_infinity=False),
    vip=st.sampled_from([0, 1, 2, 3]),
)
def test_price_bounds(amount, vip):
    with patch("myapp.pricing.get_rate", return_value=1.0):  # rate=1 便于对比原价
        price = final_price(amount, vip, "USD")
        assert price >= 0                       # 永不为负
        assert price <= round(amount, 2) + 0.01 # 不超过原价（留浮点/round 容差）
```

要点回顾：patch 的是 `"myapp.pricing.get_rate"`（使用处，不是 `"myapp.fx.get_rate"`）；parametrize 把 4 个档位变成 4 个独立用例；异常路径用 `pytest.raises(match=...)` 同时断言类型和消息；hypothesis 用 `st.floats` 限定范围（排除 NaN/Inf）+ `st.sampled_from` 选合法档位，声明"价格有界"这条对所有输入都成立的性质，让框架去找反例。
