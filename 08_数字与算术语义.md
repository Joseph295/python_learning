# 08 · 数字与算术语义 ⭐

> 你在 Java 里对数字的肌肉记忆——`int` 32 位会溢出回绕、`/` 默认整除、`%` 跟被除数同号、`==` 比 `double`、`boolean` 和 `int` 老死不相往来——在 Python 里**几乎每一条都不成立**。本章逐条拆掉这些假设：Python 的 `int` 是任意精度永不溢出，`/` 永远返回 `float`，`%` 的符号跟**除数**走，`bool` 是 `int` 的**子类**，算钱必须用 `Decimal`。这是 Java 老手最密集的翻车区，每个坑都附 CPython 底层解释。

## ① Java 对照表

| 概念 | Java | Python | 一句话差异 |
|------|------|--------|-----------|
| 整数 | `int`/`long`（32/64 位，**溢出回绕**） | `int`（**任意精度，永不溢出**） | Python 没有 `MAX_VALUE`，`2**1000` 直接算 |
| 大整数 | `BigInteger`（要显式用） | 就是 `int`（无缝） | Python 不区分 `int` 和 `BigInteger` |
| 浮点 | `double`/`float`（IEEE 754） | `float`（= C `double`，IEEE 754 双精度） | 语义几乎一致，但 Python 只有一种 `float`（双精度） |
| 布尔 | `boolean`（独立类型，**不是数**） | `bool`（**是 `int` 的子类**！） | `True == 1`、`True + True == 2` 在 Python 成立 |
| 高精度十进制 | `BigDecimal` | `decimal.Decimal` | 都是十进制定点、算钱用，API 神似 |
| 分数 | 无（自己实现或第三方） | `fractions.Fraction` | Python 标准库自带精确有理数 |
| 复数 | 无（自己实现或第三方） | `complex`（语言内建，`3+4j`） | Python 把复数做进了语言 |
| 整除 | `7 / 2 == 3`（`int` 除默认截断） | `7 / 2 == 3.5`（`/` 永远是真除法） | Python 整除要用 `//` |
| 取模符号 | 结果符号跟**被除数** | 结果符号跟**除数** | `-7 % 3` 在 Java 是 `-1`，Python 是 `2` |
| 字面量分隔 | `1_000_000`（Java 7+ 下划线） | `1_000_000`（同样支持） | 这条一致 |
| 舍入 | `Math.round` 四舍五入到 `+∞` | `round` **银行家舍入**（四舍六入五成双） | `round(2.5)` 在 Java 是 3，Python 是 2 |

## ② 设计理念（为什么 Python 这么设计）

### 2.1 为什么 `int` 任意精度、永不溢出

Java 的 `int` 是对硬件的**直接映射**：32 位补码，超出 `Integer.MAX_VALUE`（2147483647）就**静默回绕**成负数。这是 C 时代的遗产——为了性能，把溢出风险甩给程序员。无数安全漏洞（缓冲区大小整数溢出）和算法 bug（`(low+high)/2` 溢出）由此而来。

```java
// Java：溢出回绕，静默出错
int x = Integer.MAX_VALUE;
System.out.println(x + 1);   // -2147483648  （回绕，不报错）
```

Python 的取舍相反：**正确性优先于性能**。Python 的 `int` 一旦超过机器字长，自动切换成多精度表示（见 ③），你永远不会遇到溢出。

```python
# Python：永不溢出，想多大算多大
print(2 ** 1000)
# → 10715086071862673209484250490600018105614048117055336074437503883703510511249361224931983788156958581275946729175531468251871452856923140435984577574698574803934567774824230985421074605062371141877954182153046474983581941267398767559165543946077062914571196477686542167660429831652624386837205668069376

import sys
big = sys.maxsize           # 当前平台机器整数上限（如 9223372036854775807）
print(big + 1)              # → 9223372036854775808  （Java 这里会溢出，Python 无感切换）
```

这背后是 Python 的核心哲学：**整数就是数学意义上的整数**，不是"某个固定位宽的二进制寄存器"。代价是慢（每次运算要判断是否溢出、是否要进位到下一个 digit），但对 Python 的定位（胶水语言、热点下沉到 C，见 [05 CPython内部](05_CPython内部.md)）来说，这个代价完全可接受。

### 2.2 为什么 `/` 永远返回 `float`、`//` 才是整除

Java（和 C）里 `/` 是**重载**的：两个 `int` 相除截断取整，有一个是 `double` 就真除法。这导致经典陷阱 `1/2 == 0`。

Python 3 做了一个有争议但正确的决定（PEP 238）：**让运算符的含义不依赖操作数类型**。

- `/` **真除法（true division）**：永远返回 `float`，`6 / 2 == 3.0`（注意是 `3.0` 不是 `3`）。
- `//` **地板除（floor division）**：向下取整，`7 // 2 == 3`，`-7 // 2 == -4`。

```python
print(7 / 2)      # → 3.5    真除法，结果是 float
print(7 // 2)     # → 3      地板除
print(6 / 2)      # → 3.0    注意：是 float 3.0，不是 int 3
print(6 / 3)      # → 2.0
print(type(6 / 3))# → <class 'float'>
```

理念：`/` 这个符号在所有人的数学直觉里就是"除法"，结果该是 `3.5` 而不是 `3`。把"我要整除"这个**特殊**需求交给一个**专门的**符号 `//`。这样 `a / b` 不管 `a`、`b` 是 `int` 还是 `float`，语义都一致——消除了 Java 里"结果类型取决于操作数类型"的隐式陷阱。

> 注意 `//` 不是"截断"而是"地板"（向 −∞ 取整）：`-7 // 2 == -4`（不是 −3）。这点和取模符号规则是一体的，见 ④。

### 2.3 为什么 `bool` 设计成 `int` 的子类

这是最反直觉的一条。在 Java 里 `boolean` 是独立类型，`true + 1` 编译都不过。Python 里：

```python
print(isinstance(True, int))   # → True   bool 是 int 的子类！
print(True + True)             # → 2
print(True == 1)               # → True
print(sum([True, False, True]))# → 2      统计有几个 True，很常用
```

历史原因：Python 0.x/1.x 根本**没有** `bool` 类型，用 `1`/`0` 表示真假（像 C）。`bool` 直到 Python 2.3 才加入（PEP 285）。为了**向后兼容**所有"把 1/0 当真假用"的老代码，`bool` 被设计成 `int` 的子类，`True` 就是值为 1 的特殊 int，`False` 是值为 0 的。

这个设计有实用价值（`sum(条件列表)` 数 `True` 个数、`["偶","奇"][n % 2]` 这类索引技巧），但也埋了坑（见 ⑤）。Java 老手要立刻记住：**Python 的 `True`/`False` 在算术和比较里就是 `1`/`0`**。

## ③ 实现细节（底层怎么实现的）★本章重点★

### 3.1 `PyLongObject`：变长 digit 数组实现的大整数

回忆 [05 CPython内部](05_CPython内部.md)：每个对象都是堆上的 `PyObject`，带 `ob_refcnt` + `ob_type`。`int` 的具体结构是 `PyLongObject`，它是一个**变长对象**——核心是一个 digit 数组：

```c
// 概念结构（CPython 3.12，简化）
typedef struct {
    PyObject_VAR_HEAD          // 含 refcount、类型指针、以及 ob_size（编码位数+符号）
    digit ob_digit[1];         // 变长数组：大整数被切成若干个 "digit"
} PyLongObject;
```

关键设计：

- **每个 `digit` 装 30 位**（在 64 位平台上 `digit` 是 `uint32_t`，但只用其中 30 位，留位给进位运算）。一个数被拆成 base-2³⁰ 的"大位"存进数组。
- **`ob_size` 字段同时编码长度和符号**：绝对值需要几个 digit 用 `|ob_size|` 表示，负数用 `ob_size < 0`（3.12 起内部实现略有调整，但思路不变）。
- 所以一个 Python `int` 的内存随数值大小增长：

```python
import sys
print(sys.getsizeof(0))        # → 24/28   （空 digit）
print(sys.getsizeof(2**30))    # → 32       多用一个 digit
print(sys.getsizeof(2**100))   # → 40       digit 数组更长
print(sys.getsizeof(2**1000))  # → 160      继续增长
```

这就是"永不溢出"的实现：数大到一个 digit 装不下，就**多分配一个 digit**，像 `BigInteger` 一样按需扩容。加法是逐 digit 带进位（复杂度 O(n)，n 是 digit 数），乘法用教科书算法，超大数（>70 digit）切换到 Karatsuba 分治。

**对照 Java**：Java 的 `int` 是栈上 4 字节裸值，CPU 一条 `iadd` 指令搞定。Python 的 `int 5` 是堆上至少 24~28 字节的对象，加法要：查类型 → 判断是否都是小整数 → 逐 digit 相加 → 分配结果对象。这就是 [05 CPython内部](05_CPython内部.md) 讲的"Python 数值运算慢"的根因。想要 Java 那样的连续紧凑整数数组，得用 `array` 模块或 `numpy`。

### 3.2 小整数缓存 `[-5, 256]`（回顾第 05 章的 `is` 陷阱）

CPython 在解释器启动时**预先创建**了 `[-5, 256]` 这 262 个小整数对象，全局共享、永不销毁。任何地方用到这个范围的整数，都指向同一个缓存对象：

```python
a = 256
b = 256
print(a is b)      # → True   都指向缓存的同一个对象
print(a == b)      # → True

x = 257
y = 257
print(x is y)      # → False  超出缓存范围，是两个独立对象（取决于编译单元）
print(x == y)      # → True   值仍然相等
```

为什么是这个范围？小整数（循环计数、字节值 0~255、常见小数字）出现频率极高，缓存它们能省掉海量的对象分配/回收。这是用空间换时间的经典权衡。

**实战教训**：永远用 `==` 比较数值相等，**绝不用 `is`**。`is` 比的是对象身份（同一个对象吗），恰好在 `[-5, 256]` 内"碰巧"为 `True`，超出就 `False`——这是初学者最隐蔽的 bug 来源。`is` 只用于和 `None`、单例比较。

### 3.3 `float`：C `double` 装进对象

Python 的 `float` 实现极简单——就是一个 C `double`（IEEE 754 双精度，64 位）包进 `PyObject`：

```c
typedef struct {
    PyObject_HEAD
    double ob_fval;            // 就是一个 C double
} PyFloatObject;
```

所以：

- Python **只有一种浮点**：双精度。没有 Java 的 32 位 `float`。需要单精度（节省显存）得用 `numpy.float32`。
- 浮点语义和 Java 的 `double` **完全一致**（同一个 IEEE 754 标准）：同样有 `inf`、`-inf`、`nan`，同样的精度和舍入。

```python
print(float('inf'), float('-inf'), float('nan'))   # → inf -inf nan
print(1.5e308 * 10)        # → inf   （溢出到无穷，不报错——和整数不同！）
nan = float('nan')
print(nan == nan)          # → False  NaN 不等于自己（IEEE 754 规定，Java 同理）
```

> 注意一个不对称：**整数永不溢出，但浮点会溢出到 `inf`**。因为 `float` 死守 IEEE 754 双精度的固定 64 位，没有"自动扩容"机制。

### 3.4 为什么 `0.1 + 0.2 != 0.3`（IEEE 754 二进制浮点）

这是全宇宙程序员的共同噩梦，Java 和 Python **完全一样**（同一个 IEEE 754）。根因：**十进制小数 0.1、0.2、0.3 在二进制里都是无限循环小数**，双精度只有 52 位尾数，必须截断。

```python
print(0.1 + 0.2)              # → 0.30000000000000004
print(0.1 + 0.2 == 0.3)       # → False

# 看清真相：0.1 存进 double 后的真实值
print(format(0.1, '.17f'))    # → 0.10000000000000001
print(format(0.3, '.17f'))    # → 0.29999999999999999
```

`0.1` 在内存里其实是 `0.1000000000000000055511151231257827021181583404541015625`（最接近 0.1 的可表示双精度值）。`0.1 + 0.2` 的舍入误差刚好让结果略大于最接近 0.3 的那个双精度值，于是 `==` 失败。

```python
from decimal import Decimal
print(Decimal(0.1))   # 用 Decimal 看 float 0.1 的精确二进制值
# → 0.1000000000000000055511151231257827021181583404541015625
```

**这不是 bug，是 IEEE 754 的固有特性**，Java 里 `0.1 + 0.2` 同样等于 `0.30000000000000004`。区别只是：Java 程序员被 `BigDecimal` 教育过要小心，而 Python 新手常被 REPL 直接打脸。解决办法见 ④（`math.isclose` 比较、`Decimal` 算钱）。

### 3.5 `Decimal`：十进制定点 + 精度上下文

`decimal.Decimal` 对应 Java 的 `BigDecimal`——**用十进制而非二进制**存储，所以 `0.1` 就是精确的 0.1，没有 3.4 的误差。它内部存的是"系数 + 指数 + 符号"（如 `1.23` = 系数 123、指数 −2），运算在十进制下进行。

```python
from decimal import Decimal
print(Decimal('0.1') + Decimal('0.2'))     # → 0.3      精确！
print(Decimal('0.1') + Decimal('0.2') == Decimal('0.3'))  # → True

# 关键：从字符串构造，别从 float 构造（否则把 float 的误差带进来）
print(Decimal(0.1))      # → 0.1000000000000000055511151231257827021181583404541015625  ❌
print(Decimal('0.1'))    # → 0.1   ✅
```

**精度上下文（context）**：`Decimal` 的精度不是无限的，由一个全局/局部"上下文"控制有效数字位数（默认 28 位）和舍入模式：

```python
from decimal import Decimal, getcontext, localcontext, ROUND_HALF_UP

print(getcontext().prec)            # → 28   默认 28 位有效数字
print(Decimal('1') / Decimal('3')) # → 0.3333333333333333333333333333  （28 位）

# 用 localcontext 临时改精度/舍入，不污染全局
with localcontext() as ctx:
    ctx.prec = 6
    print(Decimal('1') / Decimal('3'))   # → 0.333333  （6 位）

# 指定舍入模式做金额舍入（四舍五入到分）
amount = Decimal('19.985')
print(amount.quantize(Decimal('0.01'), rounding=ROUND_HALF_UP))  # → 19.99
```

**对照 `Fraction`**：`Decimal` 是定点十进制（`1/3` 仍是近似的 `0.3333...`），而 `Fraction` 用分子/分母存**精确有理数**（`1/3` 永远精确），但无法表示无理数且运算可能让分母爆炸。算钱用 `Decimal`，需要精确比例运算用 `Fraction`。

### 3.6 类型提升规则（数值塔 numeric tower）

混合类型运算时，Python 按"数值塔"自动提升到更"宽"的类型，规则清晰：

```
int → float → complex          （从窄到宽，自动向右提升）
int 与 Decimal 可混算；Decimal 与 float 不能直接混（会报错，防止误差污染）
int 与 Fraction 可混算
```

```python
print(2 + 3.0)               # → 5.0           int + float → float
print(2 + 3j)                # → (2+3j)        int + complex → complex
print(2.0 + 3j)              # → (2+3j)        float + complex → complex

from fractions import Fraction
print(Fraction(1, 2) + 3)    # → 7/2           Fraction + int → Fraction
print(Fraction(1, 2) + 0.5)  # → 1.0           Fraction + float → float（降级为近似）

from decimal import Decimal
print(Decimal('1.5') + 2)    # → 3.5           Decimal + int 可以
try:
    Decimal('1.5') + 0.5     # Decimal + float ❌
except TypeError as e:
    print("报错:", e)        # → unsupported operand type(s)
```

`Decimal + float` **故意禁止**：因为 `float` 自带二进制误差，让它悄悄混入精确的 `Decimal` 运算会污染结果。Python 宁可报错逼你显式 `Decimal(str(x))` 转换，这是"精确性不容妥协"的设计体现。

## ④ 使用方法与最佳实践

### 4.1 除法三件套：`/`、`//`、`%`、`divmod`

```python
print(17 / 5)        # → 3.4    真除法（float）
print(17 // 5)       # → 3      地板除（向 −∞ 取整）
print(17 % 5)        # → 2      取模（余数）
print(divmod(17, 5)) # → (3, 2) 一次拿到商和余数，比分开算快

# 恒等式（对任意整数成立）：a == (a // b) * b + (a % b)
a, b = 17, 5
print((a // b) * b + (a % b) == a)   # → True
```

`divmod` 在分页、进制转换、时间换算里很顺手：

```python
total_seconds = 3725
minutes, seconds = divmod(total_seconds, 60)
hours, minutes = divmod(minutes, 60)
print(f"{hours:02d}:{minutes:02d}:{seconds:02d}")   # → 01:02:05
```

### 4.2 取模符号规则：结果跟**除数**走（与 Java 相反！）

这是 Java 老手 100% 会踩的坑。Python 的 `%` 结果符号**永远和除数相同**；Java 的 `%` 结果符号和**被除数**相同。

```python
print(-7 % 3)    # → 2    （Python：跟除数 3，正）  Java 这里是 -1
print(7 % -3)    # → -2   （Python：跟除数 -3，负）  Java 这里是 1
print(-7 % -3)   # → -1
print(7 % 3)     # → 1    （都正时两语言一致）

print(-7 // 3)   # → -3   地板除向 −∞ 取整（不是 Java 的截断 -2）
print(divmod(-7, 3))     # → (-3, 2)
```

为什么？Python 选择让 `//` 是**真正的地板除**（向 −∞ 取整），并保持恒等式 `a == (a//b)*b + a%b`。代入 `-7, 3`：`-7 // 3 == -3`，于是 `-7 % 3 == -7 - (-3)*3 == 2`。这套规则的好处：**取模结果可以稳定地映射到 `[0, b)` 区间**，写循环索引、哈希分桶、时钟运算时不用处理负数：

```python
# 实战：把任意索引（含负）安全映射到环形缓冲区 [0, n)
n = 5
for i in [-7, -1, 0, 3, 8]:
    print(i, "→", i % n)     # → 全部落在 0..4，Java 的 % 对负数会给负索引
# -7 → 3,  -1 → 4,  0 → 0,  3 → 3,  8 → 3
```

若你**要** Java 那种"截断除 + 余数跟被除数"的语义，用 `math.fmod` 和 `math.trunc`：

```python
import math
print(math.fmod(-7, 3))   # → -1.0   （C/Java 风格，符号跟被除数）
print(math.trunc(-7 / 3)) # → -2      截断取整（向 0）
```

### 4.3 位运算

和 Java 几乎一致，但作用在任意精度整数上：

```python
print(5 & 3)     # → 1     按位与
print(5 | 2)     # → 7     按位或
print(5 ^ 1)     # → 4     按位异或
print(~5)        # → -6    按位取反（= -(x+1)）
print(1 << 4)    # → 16    左移
print(256 >> 2)  # → 64    右移（Python 无 Java 的 >>> 无符号右移——因为 int 任意精度，没有"符号位"概念）

# 任意精度整数的位操作不会像 Java 那样在 32/64 位截断
print((1 << 100) | 1)    # → 1267650600228229401496703205377

# 实用内省方法
print((255).bit_length())  # → 8     表示该数需要几个二进制位
print((255).bit_count())   # → 8     有几个 1（popcount，3.10+）
```

> Python 没有 `>>>`（无符号右移）。Java 需要它是因为 `int` 定长有符号位；Python 整数是任意精度，负数概念上是"无限长的符号扩展"，无符号右移没有良定义，故不提供。

### 4.4 `math` / `cmath` 模块

纯函数式的数学库，对应 `java.lang.Math`：

```python
import math
print(math.sqrt(2))        # → 1.4142135623730951
print(math.floor(2.7), math.ceil(2.1), math.trunc(-2.7))  # → 2 3 -2
print(math.gcd(12, 18), math.lcm(4, 6))   # → 6 12   （lcm 是 3.9+）
print(math.factorial(20))  # → 2432902008176640000
print(math.pi, math.e)     # → 3.141592653589793 2.718281828459045
print(math.isnan(float('nan')), math.isinf(float('inf')))  # → True True

import cmath                # 复数版数学函数
print(cmath.sqrt(-1))      # → 1j      math.sqrt(-1) 会抛错，cmath 给复数解
```

### 4.5 金额用 `Decimal`（绝不用 `float`）

后端处理钱的铁律。下面是一个典型的下单金额计算：

```python
from decimal import Decimal, ROUND_HALF_UP

def calc_order(unit_price: str, qty: int, tax_rate: str) -> Decimal:
    """单价 × 数量，再加税，结果四舍五入到分。入参用字符串避免 float 误差。"""
    price = Decimal(unit_price)
    subtotal = price * qty
    tax = subtotal * Decimal(tax_rate)
    total = subtotal + tax
    return total.quantize(Decimal('0.01'), rounding=ROUND_HALF_UP)

print(calc_order('19.99', 3, '0.08'))   # → 64.77
# 用 float 算: 19.99*3*1.08 = 64.7676000000001 → 舍入可能出错且不可审计
```

要点：① **从字符串构造 `Decimal`**，不从 `float`；② 用 `quantize` 显式控制到分的舍入；③ 金额舍入业务上通常要 `ROUND_HALF_UP`（四舍五入），而不是 Python `round` 默认的银行家舍入。

### 4.6 分数用 `Fraction`

需要精确有理数运算（避免任何浮点误差）时：

```python
from fractions import Fraction
print(Fraction(1, 3) + Fraction(1, 6))   # → 1/2      精确，自动约分
print(Fraction(2, 4))                    # → 1/2      构造时即约分
print(Fraction('3.14'))                  # → 157/50   从字符串精确解析
print(Fraction(1, 3) * 3)                # → 1        精确，无 0.999... 误差
print(float(Fraction(1, 3)))             # → 0.3333333333333333  需要时转 float

# 注意：从 float 构造会暴露二进制真值
print(Fraction(0.1))   # → 3602879701896397/36028797018963968  （float 0.1 的精确分数）
print(Fraction(0.5))   # → 1/2   （0.5 在二进制里精确，所以干净）
```

### 4.7 `round` 的银行家舍入（四舍六入五成双）

Python 的 `round` **不是**学校教的四舍五入，而是 IEEE 754 推荐的**银行家舍入**（round half to even）：恰好是 `.5` 时舍入到**最近的偶数**，以消除大批量舍入的系统性偏差。

```python
print(round(0.5))   # → 0    （舍向偶数 0，不是 1）
print(round(1.5))   # → 2
print(round(2.5))   # → 2    （舍向偶数 2，不是 3！Java Math.round 给 3）
print(round(3.5))   # → 4
print(round(2.675, 2))  # → 2.67   （注意：2.675 的 float 真值略小于 2.675，所以舍下去）
```

`round(2.5) == 2` 是 Java 老手最爱报的"bug"，其实是**特性**。若业务要传统四舍五入，用 `Decimal.quantize(..., ROUND_HALF_UP)`（见 4.5）。

### 4.8 进制、科学计数法与下划线分隔

```python
print(0xFF, 0o17, 0b1010)        # → 255 15 10    十六/八/二进制字面量
print(1e3, 1.5e-3)               # → 1000.0 0.0015  科学计数法（结果是 float）
print(1_000_000)                 # → 1000000     下划线分隔，纯可读性（Java 7+ 同款）
print(0x1_00, 0b1010_1010)       # → 256 170     下划线可用于任意进制
print(int('101', 2))             # → 5           按指定进制解析字符串
print(int('ff', 16))             # → 255
print(bin(10), hex(255), oct(8)) # → 0b1010 0xff 0o10   转进制字符串
```

> 注意 `1e3` 是 **float**（`1000.0`），不是 int。要 int 写 `1000` 或 `10**3`。字符串到数字的解析细节（异常、`locale`、千分位）见 [09 字符串与文本处理](09_字符串与文本处理.md)。

### 4.9 `int()` / `float()` 转换

```python
print(int(3.9))      # → 3      截断（向 0），不是四舍五入！
print(int(-3.9))     # → -3     向 0 截断
print(int('42'))     # → 42     字符串转 int
print(float('3.14')) # → 3.14
print(float('inf'))  # → inf
print(int(True))     # → 1      bool 转 int
print(bool(0), bool(0.0), bool(''), bool([]))   # → False False False False  （假值）
print(bool(-1), bool('0'))                       # → True True  （非零/非空即真）
```

> `int(3.9) == 3`（截断）≠ `round(3.9) == 4`（舍入）。两者别混。

### 4.10 用 `math.isclose` 比较浮点（不要用 `==`）

```python
import math
print(0.1 + 0.2 == 0.3)                  # → False  永远别这样比
print(math.isclose(0.1 + 0.2, 0.3))      # → True   按相对/绝对容差比较

# 自定义容差
print(math.isclose(1000.0, 1000.1, rel_tol=1e-3))  # → True
print(math.isclose(0.0, 1e-12, abs_tol=1e-9))      # → True  （比较接近 0 时需要 abs_tol）
```

`math.isclose(a, b)` 默认用相对容差 `rel_tol=1e-9`。比较接近 0 的值要额外给 `abs_tol`（相对容差对 0 无意义）。`numpy` 里对应 `np.isclose` / `np.allclose`。

## ⑤ ⚠️ 易错点（Java 思维翻车）

| Java 习惯 | 翻车现场 | 正确做法/认知 |
|----------|---------|--------------|
| 期待 `int` 溢出回绕 | 写溢出检测/用 `long` 防溢出，或假设 `x+1` 会变负——逻辑全错 | Python `int` 任意精度永不溢出，无需防溢出；要定长用 `numpy` / `ctypes` |
| 以为 `/` 是整除 | `total / count` 本想取整，得到 `float`，或拿去做 list 索引报 `TypeError` | 整除用 `//`；`/` 永远返回 `float` |
| `%` 符号跟被除数（Java 习惯） | `-7 % 3` 期待 `-1`，实际 `2`；移植 Java 哈希/环形索引算法出错 | Python `%` 跟**除数**符号；要 Java 语义用 `math.fmod` |
| 用 `float` 算钱 | `0.1 + 0.2 != 0.3`，对账差一分，财务报表对不上 | 金额一律 `Decimal`，从字符串构造，`quantize` 舍入 |
| 用 `==` 比浮点 | `0.1 + 0.2 == 0.3` 为 `False`，测试莫名失败 | 用 `math.isclose`；或转 `Decimal`/`Fraction` |
| 以为 `bool` 不是数 | 困惑 `True + True == 2`、`sum(bool_list)`、`{True: 'a', 1: 'b'}` 只剩一个键 | `bool` 是 `int` 子类，`True is 1` 在算术/字典键上等价于 1 |
| `True == 1` 当成偶然 | 误以为类型不同就不相等 | `True == 1` 且 `True == 1.0` 都为 `True`（值相等） |
| 期待 `round(2.5) == 3` | Java `Math.round` 习惯，Python 给 `2`（银行家舍入） | `round` 是四舍六入五成双；要传统舍入用 `Decimal.quantize(ROUND_HALF_UP)` |
| 用 `is` 比较数值 | `x is 257` 时灵时不灵（小整数缓存 `[-5,256]` 边界） | 数值相等永远用 `==`，`is` 只比身份（和 `None`） |
| `int(2.9)` 当成四舍五入 | 期待 `3`，得到 `2`（截断） | `int()` 向 0 截断；要舍入用 `round()` |

### 综合实战

5. 实现一个发票金额计算函数 `invoice(items, tax_rate, discount_rate)`，要求：
   - `items` 是 `[(单价字符串, 数量), ...]`，如 `[('19.99', 3), ('5.50', 2)]`；
   - 先算小计，再打折（`discount_rate` 如 `'0.1'` 表示 9 折后的优惠 10%），再加税（`tax_rate` 如 `'0.08'`）；
   - 所有中间金额和最终结果都精确，最终结果**四舍五入到分**（传统四舍五入，不是银行家舍入）；
   - 返回 `(小计, 折扣额, 税额, 应付总额)` 四个 `Decimal`。
   写出实现并给出 `invoice([('19.99', 3), ('5.50', 2)], '0.08', '0.1')` 的结果。

---

### 参考答案

**1.**
- `7 / 2` → `3.5`，`float`。`/` 是真除法，永远返回 `float`。
- `7 // 2` → `3`，`int`。地板除取整。
- `-7 // 2` → `-4`，`int`。地板除向 −∞ 取整（不是截断的 −3）。
- `-7 % 2` → `1`，`int`。取模符号跟除数（2 为正），且满足 `-7 == (-4)*2 + 1`。
- `2 ** 100` → `1267650600228229401496703205377`，`int`。任意精度，不溢出。
- `type(6 / 3)` → `<class 'float'>`。即便整除得尽，`/` 仍返回 `float`（`2.0`）。

**2.** `round(0.5) == 0`、`round(1.5) == 2`、`round(2.5) == 2`。Python 用**银行家舍入**（round half to even）：恰为 `.5` 时舍向最近的偶数，以消除批量舍入的累计偏差。Java 的 `Math.round` 是"四舍五入到 +∞"（`.5` 一律进位），所以 `Math.round(2.5) == 3`。业务上要传统四舍五入到分，用 `Decimal`：`Decimal('2.5').quantize(Decimal('1'), rounding=ROUND_HALF_UP)`。

**3.** 至少三个 bug：
- **用 `float` 算钱**：`total` 累加浮点会有误差（`19.99+29.99+49.99+0.03` 未必精确等于 100.0）。应全程用 `Decimal`，从字符串构造。
- **`total == 100.0` 比浮点相等**：即便数学上等于 100，浮点误差也可能让它为 `False`。改用 `Decimal` 后可精确比较，或用 `>=` 判断门槛（包邮通常是"满 100"即 `>= 100`，不是"正好 100"——逻辑也错了）。
- **`avg = total / count` 注释说"整数分"但 `/` 给 float**：若真要整数应 `//`，但金额平均更该保留 `Decimal` 精度，注释与实现矛盾。
- （附加）`discount = total % 3` 注释提到"件数优惠/位运算"但 `total` 是金额，对金额取模无业务意义，且该变量未被使用——属于从 Java 硬搬的无效代码。

修正版：
```python
from decimal import Decimal, ROUND_HALF_UP

def cart_total(prices, count):
    total = sum((Decimal(p) for p in prices), Decimal('0'))
    avg = (total / count).quantize(Decimal('0.01'), rounding=ROUND_HALF_UP)
    print(f"平均: {avg}")
    free_shipping = total >= Decimal('100')      # 满 100 包邮
    return total, free_shipping

prices = ['19.99', '29.99', '49.99', '0.03']     # 金额用字符串传入
print(cart_total(prices, len(prices)))
# → 平均: 25.00
# → (Decimal('100.00'), True)
```

**4.** 因为 `bool` 是 `int` 的子类且 `True == 1 == 1.0`、`False == 0`，它们在 `set` 里被视为**同一个元素**（`set` 用 `==` + `hash`，而 `hash(True) == hash(1) == hash(1.0)`）。所以 `{True, 1, 1.0}` 去重后只剩一个元素，`{False, 0}` 只剩一个，`len(unique_pass)` 是 `2`——这次"碰巧"对，但若全是通过项，集合会塌缩成 1 个，掩盖真实数量。要可靠统计"通过了几个"，应先归一化为真正的布尔再数：
```python
# results = [True, 1, True, 1.0, False, 0]：前 4 个是真值，后 2 个是假值
pass_count = sum(bool(x) for x in results)    # → 4   把通过项数出来
print("通过人数:", pass_count)
# 若要数"种类"，先统一类型：len({bool(x) for x in results}) → 2
```

**5.**
```python
from decimal import Decimal, ROUND_HALF_UP

def invoice(items, tax_rate, discount_rate):
    subtotal = sum(
        (Decimal(price) * qty for price, qty in items),
        Decimal('0'),
    )
    discount = subtotal * Decimal(discount_rate)
    taxable = subtotal - discount
    tax = taxable * Decimal(tax_rate)
    total = (taxable + tax).quantize(Decimal('0.01'), rounding=ROUND_HALF_UP)
    return (
        subtotal.quantize(Decimal('0.01'), rounding=ROUND_HALF_UP),
        discount.quantize(Decimal('0.01'), rounding=ROUND_HALF_UP),
        tax.quantize(Decimal('0.01'), rounding=ROUND_HALF_UP),
        total,
    )

print(invoice([('19.99', 3), ('5.50', 2)], '0.08', '0.1'))
# 小计 = 19.99*3 + 5.50*2 = 59.97 + 11.00 = 70.97
# 折扣 = 70.97 * 0.1 = 7.097 → 7.10
# 应税 = 70.97 - 7.097 = 63.873
# 税额 = 63.873 * 0.08 = 5.10984 → 5.11
# 应付 = 63.873 + 5.10984 = 68.98284 → 68.98
# → (Decimal('70.97'), Decimal('7.10'), Decimal('5.11'), Decimal('68.98'))
```
要点：① 全程 `Decimal`、从字符串构造；② 中间值保持完整精度，只在**展示/最终**时 `quantize`；③ 用 `ROUND_HALF_UP` 满足"传统四舍五入"的业务要求，而非默认银行家舍入。
