# 16 · 迭代器与生成器 ⭐

> 这是 Python 的灵魂特性。在 Java 里你用 `Iterator` 遍历集合、用 `Stream` 做惰性管道，两者是分开的设计。Python 把它们统一进**一套迭代器协议**，再用 `yield` 这个"可暂停的函数"把写迭代器的成本降到几乎为零——一个 `yield` 就是一个惰性序列。理解本章，你才懂 `for` 循环背后真正发生了什么、生成器为什么能表达"无限序列"而不爆内存、以及 `send()` 如何在 2008 年就埋下了 [21 异步编程asyncio](21_异步编程asyncio.md) 的种子。

---

## ① Java 对照表

| 概念 | Java | Python | 关键差异 |
|------|------|--------|---------|
| 可迭代 | `Iterable<T>` / `iterator()` | `__iter__()` | Python 任何带 `__iter__` 的对象都能进 `for` |
| 迭代器 | `Iterator<T>` | `__next__()` + `StopIteration` | Java 用 `hasNext()` 预判，Python 用**异常**结束 |
| 取下一个 | `it.next()` | `next(it)` | Java 内置方法，Python 是内置函数转调 `__next__` |
| 遍历结束信号 | `hasNext()` 返回 `false` | 抛 `StopIteration` | Python 用异常做控制流（见 [15 异常与上下文管理器](15_异常与上下文管理器.md)） |
| 惰性流 | `Stream<T>`（惰性、一次性） | 生成器 / 生成器表达式 | 生成器更轻量，语言级支持 |
| 流式 map | `stream.map(f)` | `map(f, it)` / `(f(x) for x in it)` | Python 的 `map` 本身就是惰性迭代器 |
| 流式 filter | `stream.filter(p)` | `filter(p, it)` / `(x for x in it if p(x))` | 同上 |
| 归约 | `Stream.reduce` / `Collectors` | `functools.reduce` | Python 更鼓励用 `sum`/`any`/推导式而非 `reduce` |
| 无限序列 | `Stream.iterate` / `generate` | `itertools.count` / 生成器 `while True: yield` | Python 生成器写无限序列极自然 |
| 生成器 | **无对应**（Java 没有 `yield`） | 生成器函数 `def ... yield` | Java 要靠手写 `Iterator` 类或反应式库模拟 |
| 协程雏形 | **无对应** | `gen.send()` / `throw()` / `close()` | 生成器双向通信，asyncio 的历史源头 |
| 缓存/偏函数 | Guava `Suppliers.memoize` / 手写 | `functools.lru_cache` / `partial` | 标准库自带，一行搞定 |

> 一句话：Java 的 `Iterator`、`Stream`、无 `yield`——这三件事在 Python 里被**一套迭代器协议 + 生成器**统一了。

## ② 设计理念（为什么 Python 这么设计）

### 2.1 惰性求值：把"无限/超大序列"变得可表达

Java 老手对 `Stream` 的惰性不陌生：`stream.filter().map()` 不会立刻遍历，直到遇到终端操作（`collect`/`forEach`）才"拉"数据。Python 把这个思想做得更彻底、更底层——**惰性是迭代器协议的默认行为**，不是某个 API 的特性。

惰性带来两个 Java 里要费劲才能拿到的能力：

1. **省内存**：处理一个 100GB 的日志文件，你不需要把它读进内存。一个生成器逐行产出，内存占用是 O(1) 而不是 O(文件大小)。
2. **可表达无限**：`itertools.count()` 是一个"所有自然数"的序列。在急切求值（eager）的世界里这会立刻 OOM，在惰性世界里它只是"按需产出下一个"。

```python
import itertools

# "所有自然数的平方" —— 一个无限序列，但不占内存
squares = (n * n for n in itertools.count(0))
print(list(itertools.islice(squares, 5)))   # 输出: [0, 1, 4, 9, 16]
```

### 2.2 迭代器协议：一个抽象，统一一切"可遍历"

Python 的设计哲学是**协议优先于继承**（鸭子类型，见 [01 设计哲学与心智模型](01_设计哲学与心智模型.md)）。"能不能被 `for` 遍历"不取决于你继承了什么接口，而取决于你**有没有实现 `__iter__`**。

这带来惊人的统一性：list、dict、set、文件对象、`range`、`map`、`zip`、生成器、CSV reader、数据库游标——它们行为各异，但**对 `for` 而言完全一样**。你写一个处理"可迭代"的函数，它能吃下以上所有东西，包括你五分钟前自定义的类。Java 也有 `Iterable`，但 Python 的版本不需要你显式 `implements`，且和语言的每个角落（解包、推导式、`*args`、`in` 运算符）深度咬合。

### 2.3 生成器：把"写迭代器"的成本降到零

在 Java 里手写一个 `Iterator` 很啰嗦：你要维护一个状态机——当前位置、`hasNext` 的逻辑、`next` 推进状态。一个稍复杂的遍历（比如树的中序遍历）写成 `Iterator` 类痛苦不堪，因为你得**手动把递归/循环的局部状态外置成字段**。

生成器是对这件事的釜底抽薪。`yield` 让你**用最自然的顺序代码（普通循环、递归）描述"产出什么"**，而 Python 自动帮你把"暂停点、局部变量、执行位置"保存成一个迭代器对象。

```python
# 树的中序遍历：用生成器写就是普通递归 + yield from
def inorder(node):
    if node is None:
        return
    yield from inorder(node.left)
    yield node.value
    yield from inorder(node.right)
```

同样的逻辑用 Java `Iterator` 实现，你需要一个显式栈和一堆状态字段。生成器的本质是**编译器替你把顺序代码自动改写成了状态机**——这正是 ③ 要深挖的实现细节。

### 2.4 send：从"产出"到"双向通信"，协程的雏形

最初生成器只能单向"吐"数据。Python 2.5（PEP 342）给生成器加了 `send()`，让外部能往暂停的生成器里"喂"数据——`x = yield` 这个表达式的值就是 `send` 传进来的。这一步把生成器从"惰性迭代器"升级成了"可暂停、可恢复、可双向通信的协程"。

后来 asyncio 最早的实现（`@asyncio.coroutine` + `yield from`）就是直接建立在生成器的 `send`/`throw` 机制上的——`await` 在底层就是"暂停当前协程、把控制权交还事件循环、等结果回来再 `send` 回来"。直到 3.5 才用 `async`/`await` 关键字把这层语法独立出来。所以本章的 `send` 不是冷僻特性，它是理解 [21 异步编程asyncio](21_异步编程asyncio.md) 的历史钥匙。

## ③ 实现细节（底层怎么实现的）★本章重点★

### 3.1 可迭代 vs 迭代器：两个协议，别混为一谈

这是 Java 老手最容易糊涂的地方，先把定义钉死：

- **可迭代对象（Iterable）**：实现了 `__iter__()`，调用它返回一个**迭代器**。list、dict、str 都是可迭代对象。
- **迭代器（Iterator）**：实现了 `__next__()`（每次返回下一个，耗尽时抛 `StopIteration`）**且**实现了 `__iter__()`（返回自己）。

关键点：**迭代器也是可迭代的**（它的 `__iter__` 返回自身），但**可迭代对象不一定是迭代器**。list 是可迭代的，但 list 本身不是迭代器——它没有 `__next__`，你不能对 list 直接 `next()`。

```python
nums = [10, 20, 30]
# next(nums)                      # TypeError: 'list' object is not an iterator

it = iter(nums)                   # __iter__ 造出一个迭代器（list_iterator）
print(type(it))                   # 输出: <class 'list_iterator'>
print(next(it))                   # 输出: 10
print(next(it))                   # 输出: 20
print(iter(it) is it)             # 输出: True  —— 迭代器的 __iter__ 返回自己
```

这个"可迭代 → 迭代器"的分离不是冗余，而是为了**可重复遍历**：list 是可迭代的，每次 `for` 都通过 `__iter__` 拿一个**全新的迭代器**，所以能遍历无数次；而迭代器自身只有一份状态，耗尽即空（这是 ⑤ 的头号陷阱）。

```
       iter(x)                 next(it)
  可迭代对象  ──────►  迭代器  ──────►  元素 / StopIteration
  (Iterable)         (Iterator)
  有 __iter__        有 __next__ + __iter__(返回self)
  可造多个迭代器      只有一份游标状态，一次性
```

对照 Java：`Iterable.iterator()` ↔ `__iter__()`，`Iterator.next()` ↔ `__next__()`。但 Java 用 `hasNext()` **预判**，Python 用**抛 `StopIteration` 异常**表示结束——这是设计取向差异：Python 把"结束"当成异常控制流（异常在 CPython 里很廉价，见 [15 异常与上下文管理器](15_异常与上下文管理器.md)）。

### 3.2 `for` 循环 desugar：它其实是 `while + next + try`

Java 的 for-each `for (T x : coll)` 编译期会被 desugar 成 `Iterator it = coll.iterator(); while (it.hasNext()) { T x = it.next(); ... }`。Python 的 `for` 也是语法糖，等价改写如下：

```python
# 你写的：
for x in some_iterable:
    process(x)

# 解释器实际执行的（等价手写）：
_it = iter(some_iterable)          # 调 __iter__，拿迭代器
while True:
    try:
        x = next(_it)              # 调 __next__
    except StopIteration:          # 用异常作为结束信号
        break
    process(x)
```

用 `dis` 看字节码（CPython 3.12），能看到这套机制的指令：

```python
import dis

def loop(seq):
    for x in seq:
        pass

dis.dis(loop)
```

输出（节选，3.12）：
```
  GET_ITER                       # 调用 iter(seq)，把迭代器压栈
  FOR_ITER     ... (to L1)       # 调 next()：有值则压栈继续，StopIteration 则跳到 L1
  STORE_FAST   x                 # x = 取到的值
  ...
  JUMP_BACKWARD                  # 回到 FOR_ITER
L1: END_FOR                      # 循环正常结束的收尾
```

`FOR_ITER` 是整套机制的核心：它在 C 层调用迭代器的 `tp_iternext`（即 `__next__`），**内部捕获 `StopIteration` 并把它转成一次跳转**——所以你在 `for` 里感觉不到异常，但它确实在底层发生。这也解释了为什么 `__next__` 必须用 `StopIteration` 而不是别的方式表示结束。

> 引申：凡是接受"可迭代"的地方——`for`、`list(x)`、`sum(x)`、`*x` 解包、`in` 成员判断、推导式——背后都是这套 `iter()` + `next()` 循环。理解了它，你就理解了 Python 一半的语法糖。

### 3.3 生成器是"保存了栈帧的对象"——yield 冻结，next 解冻

这是本章最重要的实现认知，请把它和 [05 CPython内部](05_CPython内部.md) 的 `PyFrameObject`（堆上分配的栈帧）联系起来。

**普通函数调用**：进入时在堆上创建一个帧对象（局部变量、求值栈、指令指针 `f_lasti`），`return` 时帧被销毁、局部变量随之消失。函数没有"中途暂停"的概念。

**生成器函数**（函数体里有 `yield`）：调用它**根本不执行函数体**，而是立刻返回一个**生成器对象**。这个生成器对象内部**持有一个帧对象**——但这个帧不像普通调用那样用完即焚，而是**被生成器长期持有、可反复冻结/解冻**：

- 每次 `next(gen)`：解释器**恢复**那个帧（从上次 `f_lasti` 指令指针继续，局部变量原样还在），一路执行到下一个 `yield`，把 `yield` 的值返回给调用者，然后**冻结**帧（指令指针停在 `yield` 处，所有局部变量保留在帧里）。
- 下一次 `next`：从冻结点继续。
- 执行到函数结尾（或 `return`）：帧被销毁，向调用者抛 `StopIteration`。

换句话说，**生成器 = 一个把自己的栈帧打包带走的迭代器**。Java 没有这个能力，因为 JVM 的栈帧是真栈帧、调用结束即弹出；要模拟就得手动把所有局部状态搬到对象字段里（这正是手写 `Iterator` 的痛苦来源）。

用 `gi_frame` / `gi_running` / `inspect.getgeneratorstate` 直接观察这个被保存的帧：

```python
import inspect

def counter():
    print("  [启动]")
    n = 0
    while n < 2:
        yield n          # ← 暂停点
        n += 1
    print("  [结束]")

g = counter()
print(type(g))                                  # 输出: <class 'generator'>
print(inspect.getgeneratorstate(g))             # 输出: GEN_CREATED  —— 还没开始跑
print(g.gi_frame is not None)                   # 输出: True  —— 帧已存在但未执行

print(next(g))                                  # 打印 [启动]，输出: 0
print(inspect.getgeneratorstate(g))             # 输出: GEN_SUSPENDED  —— 冻结在 yield 处
print(g.gi_frame.f_locals)                      # 输出: {'n': 0}  —— 局部变量被帧保存着！
print(g.gi_frame.f_lasti)                       # 输出: 一个字节码偏移（指令指针停在 yield）

print(next(g))                                  # 从冻结点继续，输出: 1
print(g.gi_frame.f_locals)                      # 输出: {'n': 1}  —— 状态被保留并推进

try:
    next(g)                                      # 打印 [结束]，函数返回
except StopIteration:
    print("StopIteration")                      # 输出: StopIteration
print(inspect.getgeneratorstate(g))             # 输出: GEN_CLOSED
print(g.gi_frame)                               # 输出: None  —— 帧已销毁
```

注意 `g.gi_frame.f_locals` 在两次 `next` 之间从 `{'n': 0}` 变成 `{'n': 1}`：**局部变量 `n` 活在生成器持有的帧里，跨越了 `yield` 暂停而存活**。这就是"可暂停的函数"在 CPython 里的物理实现。

> 字节码层面：生成器函数编译后带有 `CO_GENERATOR` 标志，函数体里 `yield n` 编译成 `YIELD_VALUE` 指令。`YIELD_VALUE` 的语义就是"把栈顶值交出去并暂停"，恢复时从这条指令之后继续。这与普通 `RETURN_VALUE`（交出值并销毁帧）的差别，正是"暂停"与"结束"的差别。

### 3.4 生成器表达式 vs 列表推导式：内存差异的本质

语法上只差一对括号，语义上天壤之别：

```python
import sys

list_comp = [n * n for n in range(1_000_000)]      # 列表推导式：立刻算完，全装进内存
gen_expr  = (n * n for n in range(1_000_000))      # 生成器表达式：只造一个生成器对象

print(type(list_comp), sys.getsizeof(list_comp))   # 输出: <class 'list'> 8448728  （约 8MB）
print(type(gen_expr),  sys.getsizeof(gen_expr))    # 输出: <class 'generator'> 200  （约 200 字节，与数据量无关）
```

列表推导式 `[...]` 是**急切求值**：执行那一刻就把一百万个 int 对象算出来塞进一个 list（约 8MB，且每个 int 还是堆对象）。生成器表达式 `(...)` 是**惰性求值**：只创建一个生成器对象（几百字节，恒定），元素在被 `next` 拉取时才逐个计算、用完即弃。

底层机制：生成器表达式在编译时被**改写成一个匿名生成器函数并立即调用**——`(n*n for n in it)` 大致等价于：

```python
def _genexpr(it):
    for n in it:
        yield n * n
gen_expr = _genexpr(iter(range(1_000_000)))
```

所以它享有 3.3 描述的全部机制：一个保存帧的迭代器。**经验法则**：如果你只是要遍历一次（喂给 `sum`、`for`、`any`、`max`、`"".join`），用生成器表达式；如果你要多次遍历、随机索引、或要 `len()`，才用列表推导式。

```python
total = sum(n * n for n in range(1_000_000))       # 惰性管道，峰值内存 O(1)，无需先建 list
print(total)                                        # 输出: 333332833333500000
```
（注意：当生成器表达式是函数唯一参数时，外层括号可省略，`sum(... for ...)` 即可。）

### 3.5 `yield from`：委托与"打通管道"

`yield from iterable` 把"产出"的职责委托给另一个可迭代对象/生成器。最朴素的理解：

```python
def chain_two(a, b):
    yield from a          # 等价于 for x in a: yield x
    yield from b
print(list(chain_two([1, 2], [3, 4])))     # 输出: [1, 2, 3, 4]
```

但 `yield from` 远不止"省一个 for 循环"。当委托给一个**子生成器**时，它建立了一条**透明的双向通道**：

1. 子生成器 `yield` 的值直接传给最外层调用者；
2. 外层 `send()` 进来的值直接转发给子生成器；
3. 子生成器 `return` 的值成为 `yield from` 表达式的值（普通 for 拿不到 return 值，`yield from` 能）；
4. `throw()` 进来的异常也正确转发。

```python
def sub():
    yield 1
    yield 2
    return "子生成器的返回值"      # 普通迭代拿不到，yield from 能捕获

def main():
    result = yield from sub()      # result 接住 sub 的 return 值
    print("拿到:", result)         # 输出: 拿到: 子生成器的返回值
    yield 3

print(list(main()))                # 输出: [1, 2, 3]，并打印上面那行
```

这第 1~4 点的"透明转发"正是 asyncio 早期 `yield from` 写法能工作的关键：`await coro` 在 3.5 之前就写作 `yield from coro`，靠的就是这条贯通的双向管道把底层 I/O 事件一路传递。

### 3.6 send / throw / close：生成器作为协程

`yield` 不只是语句，还是**表达式**——`x = yield value` 中，`value` 被吐出去，而下一次 `send(v)` 传进来的 `v` 成为 `x` 的值。这让生成器能**接收**数据，而不只是产出。

经典例子：一个流式计算移动平均的协程（持续接收数值、吐出当前均值），常见于实时指标聚合：

```python
def running_average():
    total = 0.0
    count = 0
    average = None
    while True:
        x = yield average          # 吐出当前均值，等待下一个 send 喂入的值
        total += x
        count += 1
        average = total / count

avg = running_average()
next(avg)                          # "预激"(prime)：跑到第一个 yield 处暂停，必须先做
print(avg.send(10))                # 输出: 10.0
print(avg.send(20))                # 输出: 15.0
print(avg.send(30))                # 输出: 20.0
```

三个方法的语义：

| 方法 | 作用 | 在生成器内部表现为 |
|------|------|-------------------|
| `gen.send(v)` | 恢复执行，把 `v` 作为当前 `yield` 表达式的值 | `x = yield ...` 中 `x` 拿到 `v` |
| `gen.throw(exc)` | 恢复执行，但在暂停点**抛出**一个异常 | 在 `yield` 处 `raise exc`，可被内部 `try` 捕获 |
| `gen.close()` | 在暂停点抛 `GeneratorExit`，请求生成器收尾 | `finally` 块得以执行清理 |

```python
def worker():
    print("启动")
    try:
        while True:
            item = yield
            print("处理:", item)
    except ValueError as e:
        print("内部捕获到:", e)      # throw 进来的异常被这里抓住
    finally:
        print("清理资源")            # close() 触发 GeneratorExit，finally 保证收尾

w = worker()
next(w)                            # 输出: 启动
w.send("任务A")                    # 输出: 处理: 任务A
w.throw(ValueError("坏数据"))      # 输出: 内部捕获到: 坏数据 -> 然后继续 / 或结束
```

`close()` 触发 `GeneratorExit` 让 `finally` 跑——这正是生成器能安全地"流式读大文件 + 退出时关文件句柄"的机制保证。把这套 `send/throw/close` 记牢：当你在 [21 异步编程asyncio](21_异步编程asyncio.md) 看到事件循环"驱动协程"时，它驱动的方式本质上就是反复 `send` 和在取消时 `throw`。

### 3.7 `functools.lru_cache`：一个 dict + 双向链表的记忆化

`lru_cache` 把"函数调用结果"缓存起来，相同入参直接命中，是记忆化（memoization）的标准库实现。其底层就是 [06 内置数据结构](06_内置数据结构.md) 讲的 **dict（O(1) 查找）+ 双向循环链表（维护 LRU 顺序）**：

- **键** = 用函数的位置参数和关键字参数构造的一个可哈希的 key（`functools._make_key`，本质是参数打包成 tuple）。
- **值** = 函数返回结果。
- 命中：dict 查到 key，并把对应节点移到链表头部（标记"最近使用"）。
- 未命中且满了（`maxsize`）：淘汰链表尾部（最久未用）那个节点。

```python
from functools import lru_cache

@lru_cache(maxsize=None)           # None = 无上限，纯记忆化
def fib(n):
    return n if n < 2 else fib(n - 1) + fib(n - 2)

print(fib(30))                     # 输出: 832040  （瞬间，否则指数级递归会很慢）
print(fib.cache_info())            # 输出: CacheInfo(hits=28, misses=31, maxsize=None, currsize=31)
fib.cache_clear()                  # 清空缓存
```

两个致命前提（也是 ⑤ 的陷阱）：

1. **参数必须可哈希**——因为要做 dict 的 key。传 list/dict/set 进去会 `TypeError: unhashable type`。
2. **缓存就是强引用，会一直占内存**——`maxsize=None` 等于一个永不收缩的全局 dict。对实例方法用 `lru_cache` 还会让 `self` 被缓存常驻，**阻止实例被回收**（内存泄漏，见 [04 运行时与内存模型](04_运行时与内存模型.md) 的引用计数）。

> 3.9+ 提供了语义更清晰的 `@functools.cache`（等价于 `lru_cache(maxsize=None)`）。对"基于实例的、只算一次的属性"，用 `@functools.cached_property` 更合适——它把结果存进**实例自己的 `__dict__`**，实例被回收时缓存随之释放，不泄漏。

## ④ 使用方法与最佳实践

### 4.1 自定义迭代器类（什么时候手写，什么时候用生成器）

手写迭代器要分清"可迭代"和"迭代器"两个角色。下面是把"可迭代对象"和"迭代器"拆开的标准写法（支持可重复遍历）：

```python
class DateRange:
    """一个左闭右开的日期区间，可迭代且可重复遍历（后端报表常用）。"""
    def __init__(self, start, end):
        self.start = start
        self.end = end

    def __iter__(self):                       # 每次 for 都返回一个全新迭代器
        return _DateRangeIterator(self.start, self.end)

from datetime import date, timedelta

class _DateRangeIterator:
    def __init__(self, current, end):
        self.current = current
        self.end = end

    def __iter__(self):                       # 迭代器的 __iter__ 返回自己
        return self

    def __next__(self):
        if self.current >= self.end:
            raise StopIteration               # 结束信号
        value = self.current
        self.current += timedelta(days=1)
        return value

dr = DateRange(date(2026, 1, 1), date(2026, 1, 4))
print(list(dr))      # 输出: [date(2026,1,1), date(2026,1,2), date(2026,1,3)]
print(list(dr))      # 输出: 同上 —— 可重复遍历，因为每次 __iter__ 造新迭代器
```

**但 90% 的情况你不该这么写**——用生成器一个函数就够了，且天然只遍历一次的语义往往正合适：

```python
def date_range(start, end):
    current = start
    while current < end:
        yield current
        current += timedelta(days=1)
```

经验：**需要"可重复遍历"或要附加方法/状态时，写可迭代类（`__iter__` 返回新迭代器）；只需一次性产出时，直接写生成器函数。**

### 4.2 生成器处理大文件 / 流式数据（核心实战场景）

这是生成器在后端/数据工程里最重要的用法：**逐行/逐块处理超大数据，内存恒定**。

```python
def read_large_log(path):
    """逐行产出，无论文件多大，内存占用都是 O(单行)。"""
    with open(path, encoding="utf-8") as f:
        for line in f:                # 文件对象本身就是迭代器，逐行惰性读取
            yield line.rstrip("\n")
```

`with` 配 `yield` 还保证了**异常/提前退出时文件能正确关闭**（靠 3.6 的 `close()` → `GeneratorExit` → `finally`/`with` 收尾机制）。对照 Java：你会用 `BufferedReader` + try-with-resources 循环 `readLine()`，思路一致，但 Python 把"流"和"迭代"统一了，代码更短。

### 4.3 惰性管道：把多个生成器串成流水线

生成器最优雅的用法是**组合**——每一级都是惰性的，数据像水流一样一次一个流过整条管道，全程不落地：

```python
def parse(lines):
    for line in lines:
        parts = line.split()
        if len(parts) >= 9:
            yield {"ip": parts[0], "status": parts[8]}

def only_errors(records):
    for r in records:
        if r["status"].startswith("5"):     # 5xx 服务端错误
            yield r

# 组装管道：每一级都惰性，整条链不占内存
lines    = read_large_log("access.log")
records  = parse(lines)
errors   = only_errors(records)
for err in errors:                          # 直到这里才真正开始"拉"数据
    print(err["ip"], err["status"])
```

这就是 Java `Stream` 的 `lines().map().filter().forEach()`，但用生成器手搓，更透明、更可控。**关键认知**：在 `for err in errors` 这一行之前，**一行日志都还没读**——惰性意味着只有终端消费才驱动整条链。

### 4.4 `itertools`：迭代器瑞士军刀

标准库 `itertools` 提供一组 C 实现的高效迭代器工具，全部惰性。后端/数据高频用到的：

```python
import itertools as it

# count / cycle / repeat：无限迭代器
print(list(it.islice(it.count(10, 2), 3)))      # 输出: [10, 12, 14]  从10起步长2，取前3个
# islice：对（可能无限的）迭代器做切片（不能用 [a:b]，迭代器不支持索引）
# chain：把多个可迭代"接龙"成一条
print(list(it.chain([1, 2], [3, 4])))           # 输出: [1, 2, 3, 4]
# groupby：按 key 分组（注意：只对相邻元素分组，通常先排序）
data = [("a", 1), ("a", 2), ("b", 3)]
for k, grp in it.groupby(data, key=lambda x: x[0]):
    print(k, [v for _, v in grp])               # 输出: a [1, 2] / b [3]
# accumulate：累积（前缀和等）
print(list(it.accumulate([1, 2, 3, 4])))        # 输出: [1, 3, 6, 10]
# product / combinations / permutations：组合数学
print(list(it.product([1, 2], ["x", "y"])))     # 输出: [(1,'x'),(1,'y'),(2,'x'),(2,'y')]
print(list(it.combinations([1, 2, 3], 2)))      # 输出: [(1,2),(1,3),(2,3)]
# batched（3.12+）：把流切成定长批次，喂数据库/API 批量写极好用
print(list(it.batched(range(7), 3)))            # 输出: [(0,1,2),(3,4,5),(6,)]
```

> `batched` 是 3.12 新增，正好解决"流式数据攒够一批再批量入库"的常见需求——以前要手写攒批逻辑。

### 4.5 `functools` / `map` / `filter`：函数式工具

```python
from functools import reduce, partial

# reduce：归约。但 Pythonic 优先用 sum/any/all/max；reduce 留给"无内置函数"的累积
print(reduce(lambda acc, x: acc + x, [1, 2, 3, 4], 0))   # 输出: 10（其实 sum() 更好）
print(reduce(lambda acc, x: acc * x, [1, 2, 3, 4], 1))   # 输出: 24（无内置，reduce 合适）

# partial：固定函数的部分参数，得到新函数（对照 Java 里手写 lambda 包一层）
import json
dump_compact = partial(json.dumps, separators=(",", ":"), ensure_ascii=False)
print(dump_compact({"name": "张三", "age": 30}))         # 输出: {"name":"张三","age":30}

# map / filter：惰性，返回迭代器（不是 list！Java Stream 老手注意）
nums = range(10)
pipeline = map(lambda x: x * x, filter(lambda x: x % 2 == 0, nums))
print(list(pipeline))                                    # 输出: [0, 4, 16, 36, 64]
```

> Pythonic 取向：`map`/`filter` 能用，但**生成器表达式通常更易读**——`(x*x for x in nums if x%2==0)` 比嵌套的 `map(filter(...))` 清晰。`reduce` 在 Python 里地位不高（Guido 曾想移除它），优先用 `sum`/`min`/`max`/`any`/`all` 等专用归约。

### 4.6 `wraps`：写装饰器时保留元信息（本章只点到）

`functools.wraps` 用于装饰器中，把被包装函数的 `__name__`/`__doc__` 等拷到 wrapper 上，避免元信息丢失。装饰器的完整机制见 [17 装饰器](17_装饰器.md)，这里只需知道：**手写装饰器时几乎总要 `@wraps(func)`**。

## ⑤ ⚠️ 易错点（Java 思维翻车）

| Java 习惯 | 翻车现场 | 正确做法 / 认知 |
|----------|---------|----------------|
| 以为迭代器像集合能反复遍历 | 生成器/迭代器**耗尽后为空**，第二次 `for` 拿不到任何数据，还不报错 | 需多次遍历就 `list()` 物化，或每次重新调生成器函数造新的 |
| 拿生成器当集合传给多个消费者 | `data = gen(); list(data); list(data)` 第二个返回 `[]` | 一个迭代器只能被消费一次；要么物化，要么用 `itertools.tee` 复制 |
| 对生成器 / `map` / `filter` 用 `len()` | `len(map(...))` → `TypeError: object of type 'map' has no len()` | 迭代器不知道长度（可能无限）；先 `list()` 或用计数器 |
| 把 `map()`/`filter()` 当 Java `Stream` 的 list 结果 | `result = map(f, xs)` 后以为是列表，打印出 `<map object>` | Python3 的 `map`/`filter` 是**惰性迭代器**，要 `list()` 才落地 |
| 混淆"可迭代"与"迭代器" | 对 list 直接 `next(my_list)` → `TypeError: not an iterator` | list 是可迭代但非迭代器；先 `it = iter(my_list)` |
| 把 `lru_cache` 当万能缓存，传 list/dict 入参 | `TypeError: unhashable type: 'list'` | 缓存键需可哈希；可改传 tuple，或换缓存方案 |
| 给实例方法套 `@lru_cache` | `self` 进了缓存键，实例被全局缓存常驻**永不回收**（内存泄漏） | 用 `@cached_property`（存进实例 `__dict__`，随实例释放） |
| 过早 `list()` 物化以求"安全" | 把 100GB 文件 `list(read_log())` 直接 OOM | 保持惰性，只在管道终端消费；中途别 `list()` |
| `groupby` 前不排序（用 SQL `GROUP BY` 的直觉） | `groupby` 只合并**相邻**相同 key，乱序数据分组结果支离破碎 | 先 `sorted(data, key=...)` 再 `groupby`，或确保数据已按 key 排好 |
| 以为生成器函数调用就执行了 | `g = gen()` 后以为副作用（打印/建连）已发生，其实没跑一行 | 生成器函数调用只造对象，函数体到第一次 `next` 才执行 |

## ⑥ 练习题

### 基础理解

1. 用自己的话说清"可迭代对象（iterable）"和"迭代器（iterator）"的区别，并解释：为什么 list 能被 `for` 遍历无数次，而一个生成器只能遍历一次？从 `__iter__` 的返回值角度回答。

2. 把 `for x in seq: print(x)` 这个 `for` 循环手动 desugar 成等价的 `while + next + try/except` 代码，并说明 `StopIteration` 在其中扮演什么角色、为什么用户在 `for` 里感觉不到这个异常。

### 找坑改错

3. 下面这段"统计 5xx 错误并打印前 3 条"的代码有两个坑（一个让结果恒为 0/空，一个性能/内存问题），找出来并修正：

```python
def gen_records(path):
    with open(path) as f:
        for line in f:
            yield line.strip()

records = gen_records("access.log")

total_5xx = sum(1 for r in records if " 5" in r)   # 先数一遍 5xx 总数
print("5xx 总数:", total_5xx)

first_three = list(records)[:3]                    # 再取前 3 条
for r in first_three:
    print(r)
```

4. 下面这个给实例方法加缓存的写法，在服务长期运行后会导致内存缓慢上涨，指出根因并给出修复：

```python
from functools import lru_cache

class UserService:
    def __init__(self, db):
        self.db = db

    @lru_cache(maxsize=None)
    def get_profile(self, uid):
        return self.db.query(uid)
```

### 综合实战

5. 实现一个**惰性日志处理管道**，要求：从一个可能上百 GB 的 Nginx 访问日志文件中，流式地（内存 O(1)）找出所有响应状态码为 5xx 的请求，按状态码统计数量，并打印**耗时最长的前 5 条**请求（每行格式假设为 `ip - - [time] "GET /path" status duration_ms`，空格分隔，`status` 是第 6 字段、`duration_ms` 是末字段）。用生成器组合管道实现解析、过滤；用 `itertools` 或 `heapq` 取 Top-5；全程不得把整个文件读入内存。

---

### 参考答案

**1.** 可迭代对象实现 `__iter__()`，每次调用**返回一个新的迭代器**；迭代器实现 `__next__()`（产出下一个/抛 `StopIteration`）且其 `__iter__()` **返回自身**。list 是可迭代对象但不是迭代器，每次 `for list` 都通过 `__iter__` 拿到一个全新的、状态独立的 `list_iterator`，所以能遍历无数次。生成器**本身就是迭代器**，它的 `__iter__` 返回自己、只有一份游标状态，`for gen` 两次拿到的是同一个已被推进的迭代器，第一次耗尽后第二次无元素可产。

**2.** 等价代码：
```python
_it = iter(seq)
while True:
    try:
        x = next(_it)
    except StopIteration:
        break
    print(x)
```
`StopIteration` 是迭代器"已耗尽"的结束信号，由 `__next__` 抛出。`for` 语句（底层是 `FOR_ITER` 字节码）**在解释器内部捕获了这个异常并转成一次跳出循环的跳转**，所以用户在 `for` 里感知不到它——异常被框架吞掉，只表现为循环正常结束。

**3.** 两个坑：
- **坑一（结果恒错）**：`records` 是生成器，`sum(...)` 那行已经把它**完全耗尽**了；后面 `list(records)` 拿到的是空列表 `[]`，`first_three` 为空，什么都打不出。迭代器一次性，不能被消费两遍。
- **坑二（内存）**：即使修好坑一，`list(records)[:3]` 也会把**整个文件**物化成 list 再切片，大文件直接 OOM——只为取 3 条却读全量。
修正：要么对同一份数据只遍历一次（在一次遍历里同时统计和收集前 3 条），要么各自重新创建生成器。且用 `itertools.islice` 惰性取前 3 条，不物化全量：
```python
import itertools

# 各自新建生成器，且用 islice 惰性取前 3 条
total_5xx = sum(1 for r in gen_records("access.log") if " 5" in r)
print("5xx 总数:", total_5xx)

first_three = list(itertools.islice(gen_records("access.log"), 3))
for r in first_three:
    print(r)
```
（更优：若只想遍历一次，在单个 `for` 循环里同时累加计数并用一个长度为 3 的列表收集前几条。）

**4.** 根因：`@lru_cache` 装在实例方法上时，缓存键包含 `self`，于是**缓存对每个 `UserService` 实例的 `self` 持有强引用**。这个缓存是类级别、全局存在且 `maxsize=None` 永不淘汰，导致**所有曾调用过 `get_profile` 的实例都无法被垃圾回收**（引用计数永不归零，见 [04 运行时与内存模型](04_运行时与内存模型.md)），内存随实例创建持续上涨。修复方案：
- 若想"每实例缓存一次、随实例释放"，对**无参（除 self）的属性**用 `@functools.cached_property`；对**带参方法**，在 `__init__` 里给每个实例建独立缓存，或用 `functools.lru_cache` 包一个不含 self 的内部函数；
- 或显式管理一个 `WeakValueDictionary` 缓存，让值可被回收。
最简洁的修法（带参）：在实例上挂独立缓存——
```python
from functools import lru_cache

class UserService:
    def __init__(self, db):
        self.db = db
        self.get_profile = lru_cache(maxsize=1024)(self._get_profile)  # 每实例独立、有上限

    def _get_profile(self, uid):
        return self.db.query(uid)
```
这样缓存生命周期绑定到实例，实例被回收时缓存随之释放，且 `maxsize` 限制了上限。

**5.** 参考实现（管道全惰性，Top-5 用 `heapq.nlargest` 只维护堆，内存 O(1) 相对文件大小）：
```python
import heapq

def read_lines(path):
    with open(path, encoding="utf-8") as f:
        for line in f:                      # 文件对象惰性逐行
            yield line.rstrip("\n")

def parse(lines):
    for line in lines:
        parts = line.split()
        if len(parts) < 6:
            continue
        try:
            status = int(parts[5])          # 第 6 字段
            duration = int(parts[-1])       # 末字段
        except ValueError:
            continue
        yield {"status": status, "duration": duration, "line": line}

def only_5xx(records):
    for r in records:
        if 500 <= r["status"] < 600:        # 5xx
            yield r

from collections import Counter

def report(path):
    # 注意：要同时做"统计"和"Top-5"，必须在一次遍历里完成（迭代器一次性）
    counter = Counter()
    heap = []                                # 维护一个最多 5 元素的最小堆
    for r in only_5xx(parse(read_lines(path))):
        counter[r["status"]] += 1
        # heapq 是最小堆：保持堆里始终是"目前最大的 5 个 duration"
        if len(heap) < 5:
            heapq.heappush(heap, (r["duration"], r["line"]))
        elif r["duration"] > heap[0][0]:
            heapq.heapreplace(heap, (r["duration"], r["line"]))

    print("各 5xx 状态码计数:", dict(counter))
    print("耗时最长 5 条:")
    for duration, line in sorted(heap, reverse=True):
        print(f"  {duration}ms  {line}")

# report("access.log")
```
要点：① `read_lines → parse → only_5xx` 是三级惰性生成器管道，数据逐行流过不落地；② 因为迭代器一次性，统计和 Top-5 必须在**同一次遍历**里完成，不能遍历两遍；③ Top-5 用 `heapq` 维护一个大小恒为 5 的堆，内存与文件大小无关——这是流式找 Top-K 的标准手法，等价于 Java 里用 `PriorityQueue` 维护固定大小的小顶堆。
