# 12 · 函数 ⭐

> 在 Java 里函数从来不是"一等公民"——方法依附于类，lambda 只是 `@FunctionalInterface` 的语法糖。Python 反过来：**函数是堆上一个普通对象（`PyObject`）**，能赋值、能传参、能当返回值、能挂属性。本章把 `def` 拆到 `__code__`/`__defaults__`/`__closure__` 这些字段层面，让你看清"可变默认参数陷阱"和"循环闭包晚绑定"两大经典坑的根因，并把参数体系（位置/关键字/默认/`*args`/`**kwargs`/keyword-only/positional-only）一次讲透。看懂这章，你才理解为什么 Python 不需要、也没有方法重载。

---

## ① Java 对照表

| 概念 | Java | Python | 一句话差异 |
|------|------|--------|-----------|
| 定义函数 | `修饰符 返回类型 方法名(...)`，必依附类 | `def 名字(...):`，可在任意位置（模块/函数内/类内） | Python 函数可独立存在，是顶级对象 |
| 方法重载 | 同名不同签名，编译期按静态类型分派 | **不存在**，后定义的覆盖前一个 | 用默认参数 / `*args` / 关键字参数替代 |
| lambda | `x -> x + 1`，可多语句块 `{...}` | `lambda x: x + 1`，**只能单个表达式** | Python lambda 不能含语句、赋值、循环 |
| 闭包 | lambda 只能捕获 **effectively final** 的变量，且只读 | 闭包可读外层变量，`nonlocal` 后可写 | Python 闭包通过 cell 对象引用变量，非拷贝值 |
| 一等函数 | 方法引用 `Foo::bar`、函数式接口包装 | 函数本身就是值，直接传递 | Python 无需接口包装，函数即对象 |
| 可变参数 | `void f(int... xs)`，编译成数组 | `def f(*xs)`，收成 tuple | 思路一致 |
| 关键字可变参 | **无对应** | `def f(**kw)`，收成 dict | Java 只能用 `Map<String,Object>` 模拟 |
| 仅关键字传参 | 无（靠 Builder 模式） | `def f(*, timeout)` 强制按名传 | Python 语言级支持 |
| 仅位置传参 | 无 | `def f(x, /)` 强制按位置传 | Python 3.8+ 语言级支持 |
| 默认参数 | **无**（靠重载模拟） | `def f(x, y=10)` | Python 一个签名顶 Java 一堆重载 |
| 部分应用 | 无（手写包装 lambda） | `functools.partial(f, ...)` | 标准库直接支持柯里化风格 |
| 类型注解 | 强制、编译期检查 | `def f(x: int) -> str`，**运行时不检查** | 注解是给工具/人看的，解释器忽略 |

## ② 设计理念（为什么 Python 这么设计）

### 2.1 为什么没有方法重载

Java 的重载（overloading）是**编译期**根据实参的**静态类型**选择目标方法：`println(int)` 和 `println(String)` 在 `javac` 阶段就定死调哪个。这套机制的前提是**变量有静态类型**。

Python 是动态类型——变量只是个绑到对象的名字，没有静态类型。解释器在调用 `f(x)` 时，运行期才知道 `x` 是什么。即便允许你写两个同名 `def f`，解释器也无从在调用点按"参数类型"分派。于是 Python 索性规定：**`def` 就是给某个名字绑定一个函数对象，后定义的同名 `def` 直接覆盖前一个**（和 `x = 1; x = 2` 一个道理）。

```python
def greet(name):
    return f"Hello {name}"

def greet(name, lang):          # 不是重载，是覆盖
    return f"{lang}: {name}"

greet("Alice")                  # TypeError: greet() missing 1 required positional argument: 'lang'
```

> 在 Java 里你靠重载实现"参数可选 / 多种入参形态"；在 Python 里这些需求由**默认参数 + 可变参数 + 关键字参数**一并满足，一个签名就够。这是更经济的设计：与其写 5 个重载，不如写 1 个 `def connect(host, port=5432, *, timeout=30, **opts)`。需要按类型分派时，用 `functools.singledispatch`（基于第一个参数的运行时类型分派），那才是 Python 版的"重载"。

### 2.2 一等函数的彻底性

在 Java 里"把行为当数据传递"要绕一圈：定义函数式接口 → 用 lambda 实现它 → 传 `Function<T,R>`。lambda 本质是匿名类实例，编译期被脱糖成 `invokedynamic`。函数本身不是值。

Python 里函数从诞生起就是 `PyObject`（参见 [05 CPython 内部](05_CPython内部.md) 的 PyObject 模型）。`def f(): ...` 干的事就是：编译出一个 code object，包成 function object，把它绑到名字 `f`。所以：

```python
def square(x): return x * x

f = square              # 直接赋值，函数是值
print(type(square))     # <class 'function'>
print(square.__name__)  # square
square.tag = "math"     # 函数能挂任意属性（它是对象）
print(square.tag)       # math
funcs = [square, abs, len]   # 装进容器
print([g(-3) if g is not len else g([1,2,3]) for g in funcs])  # [9, 3, 3]
```

这种彻底性带来了高阶函数（`map`/`filter`/`sorted(key=...)`）、装饰器（[17 章](17_装饰器.md)）、回调、策略模式无需接口等一整套地道写法。

### 2.3 显式 self、默认参数取代重载

Java 的 `this` 隐式存在；Python 坚持 `def method(self, ...)` **显式写出** self（Guido 的名言："Explicit is better than implicit"）。这让"方法就是第一个参数为实例的普通函数"这一事实暴露无遗——也正因如此，函数和方法在底层是同一种对象，绑定机制统一（[13 章](13_类与对象.md) 细讲描述符）。

默认参数取代重载，本质是用"一个灵活签名"换"多个僵硬签名"，把可选性表达在调用点而非定义处。代价是引入了"默认值何时求值"的语义问题——这正是 2.x 节实现细节里那个著名陷阱的来源。

### 2.4 闭包与作用域的取舍

Python 选择了**词法作用域 + LEGB 查找**，并默认让闭包**引用**（而非拷贝）外层变量。这带来灵活性（闭包能反映外层变量后续变化），但也埋了"循环里建闭包共享同一变量"的坑（late binding，见 3.6）。Java 的 lambda 走了保守路线——只允许捕获 effectively final 变量、且是值捕获，从语言层面堵死了这个坑，代价是表达力受限（lambda 内不能改外层变量）。两种哲学：Python 给你绳子也给你打结的风险，Java 直接没收绳子。

## ③ 实现细节（底层怎么实现的）★本章重点★

### 3.1 函数是对象：解剖一个 function object

`def` 执行后产生的 function object 携带一组关键属性。把它们摊开看，函数的全部"状态"一目了然：

| 属性 | 含义 | 对照 Java |
|------|------|-----------|
| `__code__` | code object：字节码、常量、变量名表 | 类似方法的 `Code` 属性（字节码） |
| `__defaults__` | 位置参数的默认值（tuple） | 无（Java 无默认参数） |
| `__kwdefaults__` | keyword-only 参数的默认值（dict） | 无 |
| `__closure__` | 自由变量的 cell 元组（无则为 None） | lambda 捕获的变量（但实现不同） |
| `__globals__` | 函数所属模块的全局命名空间（dict） | 无直接对应 |
| `__name__` / `__qualname__` | 函数名 / 限定名 | `Method.getName()` |
| `__annotations__` | 类型注解（dict） | 注解 + 泛型签名 |
| `__dict__` | 函数自己的属性字典 | 无（方法不能挂属性） |

```python
def connect(host, port=5432, *, timeout=30, **opts):
    return (host, port, timeout, opts)

print(connect.__defaults__)        # (5432,)            位置默认值
print(connect.__kwdefaults__)      # {'timeout': 30}    keyword-only 默认值
print(connect.__code__.co_varnames)  # ('host', 'port', 'timeout', 'opts')
print(connect.__code__.co_argcount)        # 2  普通位置参数个数 (host, port)
print(connect.__code__.co_kwonlyargcount)  # 1  keyword-only 个数 (timeout)
```

> 关键认知：函数对象 = code object（不可变、可共享的"指令模板"）+ 一层可变的运行期状态（默认值、闭包、全局引用）。一个 code object 可以被多个 function object 共享（比如同一个 `def` 在循环里反复执行会复用 code object，但每次产生新 function object 并捕获不同的闭包）。

### 3.2 默认参数在 def 执行时求值一次——可变默认参数陷阱的根因

**这是本章最重要的实现事实：默认参数的值，在 `def` 语句执行时就被求值，并永久存进 `__defaults__`/`__kwdefaults__`，之后每次调用都复用同一个对象。**

`def` 不是声明，而是一条**可执行语句**。执行它时，解释器：① 求值所有默认值表达式 → ② 把结果打包 → ③ 用 `MAKE_FUNCTION` 造出函数对象。用 `dis` 实证 `def g(x=[])` 的编译结果（外层视角）：

```python
import dis
dis.dis(compile("def g(x=[]):\n    return x", "<demo>", "exec"))
```

输出（节选，Python 3.12）：

```
  1   BUILD_LIST               0      # 在 def 执行时就构造这个 [] —— 只此一次
      BUILD_TUPLE              1
      LOAD_CONST    (<code object g>) # g 的字节码模板
      MAKE_FUNCTION            1 (defaults)  # 把那个 [] 绑成默认值，存进 __defaults__
      STORE_NAME               0 (g)
```

注意 `BUILD_LIST 0` 出现在**外层**（def 所在层），不在 `g` 的函数体里——所以那个 `[]` 全程只构造一次，被 `__defaults__` 牢牢攥住。每次 `g()` 不传参，拿到的都是**同一个 list**：

```python
def add_item(item, basket=[]):     # basket 的默认值是一个固定 list
    basket.append(item)
    return basket

print(add_item("a"))               # ['a']
print(add_item("b"))               # ['a', 'b']  —— 上次的 'a' 还在！
print(add_item.__defaults__)       # (['a', 'b'],)  默认值被就地改了
```

对照 Java：Java 没有默认参数，自然没这个坑；最接近的是"静态字段被多次方法调用共享"。这里的本质是 `__defaults__[0]` 就是一个长寿的共享可变对象。正确姿势见 4.4。

> 反过来，如果默认值是**不可变对象**（`None`/`int`/`str`/tuple），复用同一个对象毫无问题（不可变，改不了），所以坑只在默认值可变（list/dict/set/自定义实例）时出现。

### 3.3 闭包通过 cell 对象实现——`__closure__` 与自由变量

当内层函数引用了外层函数的局部变量，这个变量就成了**自由变量（free variable）**。CPython 不会把它的值拷给内层函数，而是把它装进一个 **cell 对象**——一个带 `cell_contents` 字段的小盒子，外层和内层函数共享这个盒子的引用。

```python
def make_adder(n):
    def adder(x):
        return x + n        # n 是自由变量，来自外层
    return adder

add5 = make_adder(5)
print(add5.__closure__)                    # (<cell at 0x...: int object at 0x...>,)
print(add5.__closure__[0].cell_contents)   # 5   盒子里装的值
print(add5.__code__.co_freevars)           # ('n',)  本函数引用的自由变量名
```

字节码层面（`dis.dis(make_adder)`，3.12）证实了 cell 机制：

```
make_adder:
  MAKE_CELL                0 (n)        # 把参数 n 升级成一个 cell（因为内层要用）
  ...
  LOAD_CLOSURE             0 (n)        # 取 n 的 cell
  BUILD_TUPLE              1
  LOAD_CONST  (<code object adder>)
  MAKE_FUNCTION            8 (closure)  # 把 cell 元组绑进 adder 的 __closure__

adder:
  COPY_FREE_VARS           1            # 从 __closure__ 接收自由变量的 cell
  LOAD_FAST                0 (x)
  LOAD_DEREF               1 (n)        # 通过 cell 解引用读 n（不是 LOAD_FAST！）
  BINARY_OP                0 (+)
  RETURN_VALUE
```

关键点：内层读 `n` 用的是 **`LOAD_DEREF`**（穿过 cell 解引用），而不是读普通局部变量的 `LOAD_FAST`。这正是 cell 间接层的体现。因为是**共享引用**而非拷贝，外层后续改变量，内层闭包能看到最新值——这灵活性也是下面这个坑的来源。

### 3.4 LEGB 查找规则与对应字节码（呼应 05 章）

名字解析遵循 **LEGB** 四层，由内到外：

| 层 | 含义 | 典型加载字节码 |
|----|------|----------------|
| **L**ocal | 当前函数局部变量 | `LOAD_FAST`（数组索引，最快） |
| **E**nclosing | 外层函数的变量（闭包自由变量） | `LOAD_DEREF`（穿 cell） |
| **G**lobal | 模块级全局 / 模块命名空间 | `LOAD_GLOBAL`（查 dict，较慢） |
| **B**uiltin | 内置名（`len`/`print`/`range`...） | `LOAD_GLOBAL`（全局没找到再查 builtins） |

**编译期就定死走哪条指令**：CPython 在编译函数时，根据"这个名字在本函数内有没有被赋值"等规则，把每个名字的访问指令静态选好。也就是说，一个名字是 local 还是 global，编译期就决定了，不是运行时动态找——这一点对理解 3.5 的 `UnboundLocalError` 至关重要。

```python
import dis

GLOBAL_TIMEOUT = 30

def handler(req):
    retries = 3                  # local
    def parse():
        return req.body          # req 是 enclosing（自由变量）
    return retries, GLOBAL_TIMEOUT, len(req)  # GLOBAL_TIMEOUT→global, len→builtin

dis.dis(handler)
# 你会看到：retries 用 LOAD_FAST / STORE_FAST，
#           GLOBAL_TIMEOUT 和 len 都是 LOAD_GLOBAL，
#           req（被内层引用）被 MAKE_CELL 成 cell。
```

回顾 [05 章 2.4](05_CPython内部.md)：`LOAD_FAST` 是数组索引访问，`LOAD_GLOBAL` 要查模块 dict + builtins dict，慢得多。热点循环里把全局名（如 `len`）提成局部变量，就是利用 LEGB——把 `LOAD_GLOBAL` 降级成 `LOAD_FAST`。

### 3.5 赋值即声明：UnboundLocalError 的字节码真相

**Python 的规则：函数体内只要对某个名字有赋值动作（哪怕在很后面、哪怕没执行到），该名字在整个函数内就被编译成 local，访问走 `LOAD_FAST`。** 于是"先读后赋值"会炸：

```python
counter = 0                   # 全局

def inc():
    counter += 1              # 等价 counter = counter + 1
    return counter            # 读 counter 时它已是 local，但还没赋值 → 炸

inc()
# UnboundLocalError: cannot access local variable 'counter' where it is not associated with a value
```

根因：因为函数里有 `counter = ...`（`+=` 含赋值），编译器判定 `counter` 是 local，读它用 `LOAD_FAST`；而 `LOAD_FAST` 对应的局部槽位此刻是空的（还没 `STORE_FAST` 过），于是抛 `UnboundLocalError`。这不是运行时"没找到全局变量"，而是编译期就把它定成 local 了。修复用 `global`（见 4.6）。

### 3.6 为什么循环里建闭包会共享同一变量（late binding）

接 3.3：闭包**引用**变量的 cell，而不是在定义时拷贝值。循环里创建的多个闭包，若都引用同一个循环变量，它们共享**同一个 cell**，读到的是循环结束后该变量的**最终值**——这叫 **late binding（延迟绑定）**。

```python
funcs = [lambda: i for i in range(3)]   # 三个 lambda 都引用同一个 i
print([f() for f in funcs])             # [2, 2, 2]  —— 不是 [0, 1, 2]！
```

为什么全是 2？三个 lambda 的 `__closure__` 指向**同一个** `i` 的 cell；列表推导跑完后 `i` 停在 2；事后调用 `f()` 时才去 cell 里读，读到的当然都是 2。"调用时才解析自由变量的值"——这就是 late binding。

> Java 把这个坑从语言层堵死了：lambda 只能捕获 effectively final 变量，`for (int i...)` 的 `i` 在循环里会变，根本不允许被 lambda 捕获，编译直接报错。Python 给了你自由，也给了你这个经典翻车点。修复（默认参数立即绑定 / 工厂函数）见 4.7。

```python
funcs2 = [lambda i=i: i for i in range(3)]  # 默认参数在 def 执行时立即求值（见 3.2）
print([f() for f in funcs2])                # [0, 1, 2]  ✅ 每个绑定当时的 i
```

## ④ 使用方法与最佳实践

### 4.1 参数顺序规则（一图记牢）

完整签名的参数顺序是固定的，记成一条横线：

```
def f(pos_only, /, normal, *args, kw_only, **kwargs):
#     │          │   │       │      │         └ 收集剩余关键字参数 → dict
#     │          │   │       │      └ keyword-only：只能按名传
#     │          │   │       └ 收集剩余位置参数 → tuple
#     │          │   └ 普通参数：可位置可关键字
#     │          └ "/" 之前的：positional-only，只能按位置传
#     └ 仅位置参数
```

带默认值的参数必须排在同组无默认值参数之后（否则 `SyntaxError`）。`*args` 之后的所有命名参数自动成为 keyword-only。

### 4.2 keyword-only（`*`）：强制按名传，提升可读性

裸 `*` 表示"它后面的参数必须用关键字传"。适合布尔开关、可选配置——避免调用点出现一串看不懂的位置实参。

```python
def create_user(name, *, is_admin=False, send_email=True):
    return f"{name} admin={is_admin} mail={send_email}"

create_user("Alice", is_admin=True)        # ✅ 清晰
# create_user("Alice", True)               # ❌ TypeError：is_admin 必须按名传
```

> 经验法则：**布尔参数和可选配置一律做成 keyword-only**。`connect(host, ssl=True)` 远比 `connect(host, True)` 可读——后者在调用点完全看不出 `True` 是什么意思。

### 4.3 positional-only（`/`）：库设计者的工具

`/` 之前的参数只能按位置传，不能按名传。意义：① 让你以后能自由重命名这些参数而不破坏调用方；② 匹配 C 实现的内置函数行为（如 `len(obj)` 不接受 `len(obj=...)`）。

```python
def pow_mod(base, exp, /, mod=None):
    r = base ** exp
    return r % mod if mod else r

pow_mod(2, 10)              # ✅
pow_mod(2, 10, mod=1000)    # ✅ mod 在 / 右边，可按名
# pow_mod(base=2, exp=10)  # ❌ TypeError：base/exp 是 positional-only
```

日常业务代码很少需要 `/`；它主要是库作者保持 API 灵活性的工具（Python 3.8+）。

### 4.4 用 None 作可变默认值的正确姿势

针对 3.2 的陷阱，铁律：**默认值需要是可变对象时，默认写 `None`，在函数体内按需新建**。

```python
def add_item(item, basket=None):
    if basket is None:
        basket = []            # 每次调用新建，互不干扰
    basket.append(item)
    return basket

print(add_item("a"))           # ['a']
print(add_item("b"))           # ['b']  ✅ 独立
```

同理 dict、set、自定义可变对象、以及"求值有副作用 / 依赖当前时间"的默认值（如 `datetime.now()`）都该用这招——否则默认值会被冻结在 def 执行的那一刻。

### 4.5 `*args`/`**kwargs` 转发：写包装器的标配

`*args`/`**kwargs` 在**定义处**是收集，在**调用处**是解包。两者配合就是"透明转发"——装饰器（[17 章](17_装饰器.md)）、中间件、日志包装全靠它：

```python
def logged(fn, *args, **kwargs):
    print(f"→ {fn.__name__} args={args} kwargs={kwargs}")
    result = fn(*args, **kwargs)        # 原样解包转发
    print(f"← {result}")
    return result

logged(int, "ff", base=16)
# → int args=('ff',) kwargs={'base': 16}
# ← 255
```

解包也能用在普通调用：`f(*my_list)` 把列表摊成位置实参，`f(**my_dict)` 把字典摊成关键字实参——对应 Java 的 `f(arr...)` 但更通用（关键字解包 Java 没有）。

### 4.6 global / nonlocal：显式声明要改外层

- `global x`：声明 `x` 指向**模块全局**，函数内对它的赋值改的是全局（解 3.5 的 `UnboundLocalError`）。
- `nonlocal x`：声明 `x` 指向**最近一层外层函数**的变量，让闭包能**写**外层变量。

```python
def make_counter():
    count = 0
    def inc():
        nonlocal count          # 不写这行，count += 1 会 UnboundLocalError
        count += 1
        return count
    return inc

c = make_counter()
print(c(), c(), c())            # 1 2 3   闭包持有可变状态
```

> 谨慎用 `global`——全局可变状态在并发和测试里都是麻烦（和 Java 滥用 static 可变字段一样糟）。`nonlocal` 做小型闭包状态机（计数器、累加器、缓存）很合适，但状态一复杂就该升级成类。

### 4.7 闭包做工厂

闭包最地道的用途是"参数化地造函数"——把配置固化进去，返回定制函数。这是策略模式/配置注入的轻量写法，无需 Java 那样定义接口 + 实现类：

```python
def make_validator(min_len, max_len):
    def validate(s):
        return min_len <= len(s) <= max_len
    return validate

valid_username = make_validator(3, 20)
valid_password = make_validator(8, 64)
print(valid_username("ab"))     # False
print(valid_password("hunter2!"))  # True
```

注意循环里批量造函数时，警惕 3.6 的 late binding——用默认参数立即绑定（`lambda i=i: ...`）或把循环体抽成工厂函数。

### 4.8 lambda 的克制使用

lambda 只能是**单个表达式**，定位是"用完即弃的微型函数"。最常见的合法用途是当 `key`/回调：

```python
records = [{"name": "Bob", "age": 30}, {"name": "Amy", "age": 25}]
records.sort(key=lambda r: r["age"])         # 按 age 排序，地道
print([r["name"] for r in records])          # ['Amy', 'Bob']
```

但**别把逻辑塞进 lambda**。`map(lambda x: x*2, xs)` 不如列表推导 `[x*2 for x in xs]` 清楚；需要名字、文档、复用、多语句时——用 `def`。把 lambda 赋值给变量（`f = lambda x: ...`）更是反模式（PEP 8 明确反对），直接 `def f(x): ...` 即可，还能得到正确的 `__name__`。

### 4.9 functools.partial 与 lru_cache

`functools.partial` 把部分参数预先固定，得到一个"少几个参数"的新可调用——比 lambda 包装更清晰，且保留可读的元信息：

```python
from functools import partial

parse_bin = partial(int, base=2)     # 固定 base=2
print(parse_bin("1010"))             # 10

# 实战：给回调预设参数
def log(level, msg): print(f"[{level}] {msg}")
log_error = partial(log, "ERROR")
log_error("disk full")               # [ERROR] disk full
```

`functools.lru_cache` 给纯函数加记忆化缓存（对照 Spring 的 `@Cacheable`，但零依赖、进程内）：

```python
from functools import lru_cache

@lru_cache(maxsize=None)
def fib(n):
    return n if n < 2 else fib(n-1) + fib(n-2)

print(fib(50))             # 12586269025  （无缓存会指数级爆炸）
print(fib.cache_info())    # CacheInfo(hits=48, misses=51, maxsize=None, currsize=51)
```

> 注意：`lru_cache` 要求参数可哈希（不能用 list/dict 当参数），且缓存按进程驻留——长生命周期服务里 `maxsize=None` 可能内存只增不减，参考 [04 章](04_运行时与内存模型.md) 的缓存泄漏讨论，按需设上限。`@lru_cache` 是装饰器，机制见 [17 章](17_装饰器.md)。

### 4.10 类型注解简述（运行时不检查）

注解是给**人和工具**（mypy/pyright/IDE/pydantic）看的，**解释器运行时完全忽略**，只是存进 `__annotations__`：

```python
def fetch(url: str, retries: int = 3, *, timeout: float = 5.0) -> dict:
    return {"url": url, "retries": retries, "timeout": timeout}

print(fetch.__annotations__)
# {'url': <class 'str'>, 'retries': <class 'int'>, 'timeout': <class 'float'>, 'return': <class 'dict'>}

fetch(123, retries="oops")   # 照常运行，不报错！注解不是约束
```

和 Java 的根本差异：Java 类型是编译期强制的；Python 注解是"可选的、外挂的"。但在工程实践里强烈建议给公共函数加注解——配合静态检查器能在 CI 拦住大量 Bug，是大型 Python 项目可维护性的关键。复杂类型见专门章节，这里只需知道"注解不改变运行时行为"。

### 4.11 文档字符串（docstring）

函数体第一个字符串字面量是 docstring，存进 `__doc__`，`help()` 和 IDE 会读它——比 Java 的 Javadoc 更"活"（运行时可读）：

```python
def connect(host, port=5432):
    """建立数据库连接。

    Args:
        host: 主机地址
        port: 端口，默认 5432
    """
    ...

print(connect.__doc__.splitlines()[0])   # 建立数据库连接。
```

## ⑤ ⚠️ 易错点（Java 思维翻车）

| Java 习惯 | 翻车现场 | 正确做法 / 认知 |
|----------|---------|----------------|
| 没有默认参数，没遇过这坑 | `def f(x, acc=[])` 多次调用，`acc` 累积上次结果 | 默认值在 def 时求值一次并共享；可变默认用 `None` 哨兵，体内新建 |
| lambda 值捕获、循环变量随便用 | `[lambda: i for i in range(3)]` 全返回 2（late binding） | 闭包共享 cell、调用时才读；用 `lambda i=i: i` 立即绑定 |
| 改外层变量天经地义 | 函数里 `count += 1` 改外层/全局 → `UnboundLocalError` | 赋值即声明 local；要改外层用 `nonlocal`，改全局用 `global` |
| 靠重载提供多种签名 | 写两个同名 `def`，以为能并存，结果后者覆盖前者 | Python 无重载；用默认参数 / `*args` / `**kwargs` / `singledispatch` |
| lambda 当多语句代码块用 | `lambda x: (print(x), x+1)` 塞逻辑，难读难调试 | lambda 只放单表达式；复杂逻辑用 `def` |
| 以为类型注解会被强制 | `def f(x: int)` 传 str 照跑，线上才炸 | 注解运行时不检查，要靠 mypy/pyright + CI |
| 布尔参数按位置传 | `create(name, True, False)` 调用点不知所云 | 布尔/可选配置做成 keyword-only（`*` 后） |
| 把方法当只能依附类 | 想不到函数能赋值、进容器、当返回值 | 函数是一等对象，自由传递组合 |
| 默认值用 `datetime.now()` | 所有调用都拿到 def 执行时那一刻的时间 | 默认 `None`，体内 `now = now or datetime.now()` |

## ⑥ 练习题

### 基础理解

**1.** 不跑代码，说出下面三处名字 `x` 各属于 LEGB 的哪一层、分别用什么字节码加载（`LOAD_FAST`/`LOAD_DEREF`/`LOAD_GLOBAL`）：

```python
x = 100                    # (A)
def outer():
    x = 10                 # (B)
    def inner():
        return x + len(str(x))   # inner 里的 x 与 len
    return inner
```

**2.** 用一句话解释：为什么 Python 不需要方法重载？给出至少两种"在 Java 里你会用重载、在 Python 里改用什么特性"的对应。

### 找坑改错

**3.**（经典坑一）下面的批量任务收集器，每注册一个 handler 就把它追加到同一份默认日志里，结果不同实例的日志串台了。指出根因（结合 `__defaults__`）并修复：

```python
def register(name, handlers=[]):
    handlers.append(name)
    return handlers

a = register("auth")
b = register("billing")
print(a)          # 期望 ['auth']，实际？
print(b)          # 期望 ['billing']，实际？
```

**4.**（经典坑二）下面想为 3 个路由各生成一个返回自身编号的处理函数，结果全部返回 2。指出根因（结合 cell / late binding）并给出两种修复：

```python
handlers = []
for route_id in range(3):
    handlers.append(lambda: f"handling route {route_id}")

print([h() for h in handlers])   # 期望 [...0, ...1, ...2]，实际？
```

### 综合实战

**5.** 实现一个轻量级"带超时与重试配置的请求函数工厂" `make_requester`，要求：
- 工厂接收 `base_url`、`default_timeout`，返回一个 `request(path, *, timeout=None, **params)` 函数；
- `request` 调用时若不传 `timeout`，用工厂的 `default_timeout`；
- `timeout` 必须 keyword-only；
- 用闭包保存配置（不用类、不用全局变量）；
- 演示从同一工厂造出两个互不影响的 requester。

（不需真发网络请求，返回一个描述字符串即可，重点是参数体系 + 闭包。）

---

### 参考答案

**1.**
- (A) 模块顶层的 `x = 100` 是 **Global**。
- (B) `outer` 内的 `x = 10` 是 **Local**（对 outer 而言），在 outer 里访问它用 `LOAD_FAST`。
- `inner` 里的 `x`：它没在 inner 内被赋值，但在外层 outer 里有定义 → 是 **Enclosing**（自由变量），用 **`LOAD_DEREF`**（穿 cell）。
- `len`：内置名，是 **Builtin**，用 `LOAD_GLOBAL`（先查模块全局 dict 未果，再查 builtins）。
- 注意：inner 里读到的 `x` 是 10（来自 outer 的 cell），不是顶层的 100——Enclosing 在 Global 之前命中。

**2.** 因为 Python 是动态类型，无法在编译期按参数静态类型分派，重载没有意义；可选性由一个灵活签名表达。对应关系示例：
- Java 多个重载提供"参数可选" → Python 用**默认参数**（`def f(x, y=10)`）。
- Java 重载提供"任意个同类参数" → Python 用 **`*args`**。
- Java 重载按参数类型走不同逻辑 → Python 用 **`functools.singledispatch`**（按第一个参数运行时类型分派）。
- Java 重载提供"多种命名入参组合" → Python 用 **关键字参数 + `**kwargs`**。

**3.** 根因：`handlers=[]` 的默认值在 `def register` 执行时求值一次，存进 `register.__defaults__[0]`，是一个被所有"不传 handlers"的调用共享的同一个 list。所以 `a` 和 `b` 其实是同一个 list：实际输出 `a` 和 `b` 都是 `['auth', 'billing']`。修复用 `None` 哨兵：

```python
def register(name, handlers=None):
    if handlers is None:
        handlers = []          # 每次调用独立新建
    handlers.append(name)
    return handlers
```

**4.** 根因：三个 lambda 都把 `route_id` 当自由变量，共享**同一个 cell**；循环结束后 `route_id` 停在 2，事后调用 lambda 才去 cell 读值（late binding），所以全是 `handling route 2`。两种修复：

```python
# 方案一：默认参数在 def 执行时立即绑定当前 route_id（见 3.2）
for route_id in range(3):
    handlers.append(lambda rid=route_id: f"handling route {rid}")

# 方案二：用工厂函数，每次调用产生独立的 cell
def make_handler(rid):
    return lambda: f"handling route {rid}"
for route_id in range(3):
    handlers.append(make_handler(route_id))
# 两种都得到 ['handling route 0', 'handling route 1', 'handling route 2']
```

**5.** 参考实现：

```python
def make_requester(base_url, default_timeout):
    def request(path, *, timeout=None, **params):
        # timeout 是 keyword-only；不传则回退到工厂的 default_timeout
        effective = default_timeout if timeout is None else timeout
        query = "&".join(f"{k}={v}" for k, v in params.items())
        url = f"{base_url}{path}" + (f"?{query}" if query else "")
        return f"GET {url} (timeout={effective}s)"
    return request

# 同一工厂造两个互不影响的 requester（各自闭包保存自己的配置）
api = make_requester("https://api.example.com", default_timeout=5)
cdn = make_requester("https://cdn.example.com", default_timeout=30)

print(api("/users", page=2))                  # GET https://api.example.com/users?page=2 (timeout=5s)
print(api("/slow", timeout=60))               # GET https://api.example.com/slow (timeout=60s)
print(cdn("/asset.png"))                      # GET https://cdn.example.com/asset.png (timeout=30s)

# 验证闭包隔离：两个 request 的 __closure__ 装着各自的 base_url / default_timeout
print(api.__closure__[1].cell_contents)       # https://api.example.com  （或 5，取决于 cell 顺序）
```

要点：`base_url`/`default_timeout` 通过闭包的 cell 各自独立保存（参见 3.3），两个 requester 互不干扰；`timeout` 用裸 `*` 强制 keyword-only（4.2）；`timeout=None` 哨兵实现"不传则用默认"（4.4）；`**params` 收集任意查询参数再转发（4.5）。
