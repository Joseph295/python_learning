### 坑 1 · 可变默认参数：函数有记忆了

**❌ Java 直觉写法**
```python
def add_item(item, cart=[]):       # 想给个"空购物车"默认值
    cart.append(item)
    return cart
```
Java 里方法没有默认参数这回事，你第一次见这语法会觉得很自然——"不传就给个空 list"。

**💥 翻车现象**
```python
print(add_item("apple"))   # → ['apple']
print(add_item("banana"))  # → ['apple', 'banana']  ❌ 上次的苹果还在！
```
默认的 `cart` 在所有调用间**共享同一个 list**。在 Web 服务里，这意味着 A 用户的购物车混进了 B 用户的商品——一个能上新闻的数据串味事故。

**✅ 正确写法**
```python
def add_item(item, cart=None):
    if cart is None:
        cart = []          # 每次调用新建
    cart.append(item)
    return cart
```

**🔑 一句根因**：默认值在 `def` 执行时**只求值一次**，绑定到函数对象上永久存活；可变对象（list/dict/set）会跨调用累积。用 `None` 哨兵 + 体内新建。

**📖 详见** [12 函数](12_函数.md)

### 坑 2 · `b = a` 不是拷贝：你只是多贴了个标签

**❌ Java 直觉写法**
```python
config = {"retries": 3, "hosts": ["a", "b"]}
backup = config            # 想留个"备份"
backup["retries"] = 0      # 改备份
```
Java 里 `=` 对引用类型也是传引用，但你通常会下意识 `new` 一个或 `clone()`。Python 里 `=` **永远只是绑名字**。

**💥 翻车现象**
```python
print(config["retries"])   # → 0  ❌ 原配置也被改了
```
`backup` 和 `config` 是同一个 dict 的两个名字。更隐蔽的是嵌套：就算你 `backup = dict(config)`（浅拷贝），`backup["hosts"].append("c")` 仍会污染 `config["hosts"]`，因为内层 list 还是共享的。

**✅ 正确写法**
```python
import copy
backup = copy.deepcopy(config)   # 嵌套结构要深拷贝
# 单层不可变值: dict(config) / config.copy() / list(a) 就够
```

**🔑 一句根因**：`=` 从不拷贝对象，只增加一个指向同一对象的引用。要副本用 `copy()`（浅）/ `copy.deepcopy()`（深）；分不清深浅时，嵌套结构一律 deepcopy。

**📖 详见** [03 变量与对象模型](03_变量与对象模型.md)、[06 内置数据结构](06_内置数据结构.md)

### 坑 3 · `is` 不是 `==`：身份 vs 相等

**❌ Java 直觉写法**
```python
if user_input is "quit":      # Java 里 == 比内容不靠谱，所以你格外小心地选了 is？
    ...
```
Java 程序员被 `String` 的 `==` vs `.equals()` 教育过，到 Python 反而容易**矫枉过正**，把 `is` 当成"正确的相等比较"。恰恰相反。

**💥 翻车现象**
```python
a = "qu" + "it"
print(a is "quit")   # → False（CPython 不保证拼接串驻留）
print(a == "quit")   # → True ✅
```
更阴险的是数值（见坑 4）。`is` 比的是**对象身份**（内存地址），`==` 比的是**值**。靠 `is` 比内容，会写出"有时灵有时不灵、随数据和 Python 版本变化"的幽灵 bug。

**✅ 正确写法**
```python
if user_input == "quit":   # 内容相等永远用 ==
    ...
if x is None:              # is 只用于单例: None / True / False / 哨兵对象
    ...
```

**🔑 一句根因**：`is` 比身份，`==` 比值。`is` **只**该用于和 `None`/`True`/`False`/自定义哨兵这类单例比较；其余一切相等判断用 `==`。

**📖 详见** [03 变量与对象模型](03_变量与对象模型.md)、[11 控制流与推导式](11_控制流与推导式.md)

### 坑 4 · 用 `is` 比数值：256 是朋友，257 是陷阱

**❌ Java 直觉写法**
```python
count = compute()
if count is 256:    # 接着坑 3 的误用，专门拎出来因为它太能骗人
    ...
```

**💥 翻车现象**
```python
a = 256; b = 256
print(a is b)   # → True   （小整数缓存命中）
a = 257; b = 257
print(a is b)   # → False  ❌ 同样的代码，换个数就翻脸
```
CPython 预缓存了 `[-5, 256]` 的小整数对象，这个区间内 `is` 碰巧为真；越界就是两个不同对象。于是你在测试里用 `is 200` 一切正常，上线遇到 `300` 就崩——最难复现的那种 bug。

**✅ 正确写法**
```python
if count == 256:    # 数值相等永远用 ==
    ...
```

**🔑 一句根因**：小整数缓存（`-5..256`）是 CPython 实现细节，不是语言保证。数值相等**永远** `==`，别让"碰巧能跑"骗了你。

**📖 详见** [08 数字与算术语义](08_数字与算术语义.md)、[03 变量与对象模型](03_变量与对象模型.md)

### 坑 5 · 用多线程跑 CPU 密集任务：GIL 让你白忙

**❌ Java 直觉写法**
```python
from threading import Thread
# Java 里 4 线程跑 4 核，CPU 密集任务近 4 倍加速
threads = [Thread(target=crunch_numbers, args=(chunk,)) for chunk in chunks]
for t in threads: t.start()
for t in threads: t.join()
```

**💥 翻车现象**：8 核机器，CPU 跑满……**一个核**。多线程版本甚至比单线程**更慢**——因为 GIL 让字节码串行执行，多线程只徒增上下文切换和锁竞争开销。

**✅ 正确写法**
```python
from concurrent.futures import ProcessPoolExecutor
with ProcessPoolExecutor() as ex:        # 多进程，每进程独立 GIL
    results = list(ex.map(crunch_numbers, chunks))
# 或: numpy 向量化把计算下沉到释放了 GIL 的 C 代码
```

**🔑 一句根因**：CPython 的 GIL 保证同一时刻只有一个线程执行 Python 字节码。**多线程只对 I/O 密集有效**（等网络/磁盘时会释放 GIL）；CPU 密集要用**多进程**或向量化。注意 `time.sleep()` 不算 I/O 等待场景里的"释放点"误区——真正释放 GIL 的是阻塞 I/O 系统调用。

**📖 详见** [20 并发与GIL](20_并发与GIL.md)、[30 部署与可观测性](30_部署与可观测性.md)（多 worker 进程绕 GIL）

### 坑 6 · 循环里的闭包：晚绑定让函数全长一个样

**❌ Java 直觉写法**
```python
handlers = []
for i in range(3):
    handlers.append(lambda: i)      # 想造 3 个分别返回 0/1/2 的函数
```
Java 的 lambda 捕获 `final` 变量是**值捕获**，这段在 Java 里会如你所愿。

**💥 翻车现象**
```python
print([h() for h in handlers])   # → [2, 2, 2]  ❌ 全是最后的值
```
所有 lambda 共享同一个变量 `i`，闭包记的是**变量名**不是当时的值；循环结束后 `i` 停在 2，调用时才去读，于是全返回 2。

**✅ 正确写法**
```python
handlers = [lambda i=i: i for i in range(3)]   # 默认参数立即绑定当前值
print([h() for h in handlers])                 # → [0, 1, 2] ✅
```

**🔑 一句根因**：Python 闭包是**晚绑定**——捕获的是变量（cell），不是值，调用时才求值。用默认参数 `lambda i=i:` 在定义时把当前值"冻"进去。

**📖 详见** [12 函数](12_函数.md)、[17 装饰器](17_装饰器.md)

### 坑 7 · 可变类属性被所有实例共享

**❌ Java 直觉写法**
```python
class ShoppingCart:
    items = []          # 想当"实例字段的默认值"，像 Java 的 List<Item> items = new ArrayList<>();
    def add(self, x):
        self.items.append(x)
```
在 Java 里字段写在类体里、每个实例一份；Python 里写在类体里的是**类属性**，全类共享。

**💥 翻车现象**
```python
a = ShoppingCart(); a.add("apple")
b = ShoppingCart(); print(b.items)   # → ['apple']  ❌ 新车里有别人的苹果
```

**✅ 正确写法**
```python
class ShoppingCart:
    def __init__(self):
        self.items = []      # 每个实例独立
```

**🔑 一句根因**：类体里直接赋值的可变对象是**类属性**，被所有实例共享。可变状态一律在 `__init__` 里 `self.x = ...` 初始化；类属性只放不可变常量。这是坑 1 的"类版本"。

**📖 详见** [13 类与对象](13_类与对象.md)

### 坑 8 · `__eq__` 不配 `__hash__`：对象进不了 set

**❌ Java 直觉写法**
```python
class Point:
    def __init__(self, x, y):
        self.x, self.y = x, y
    def __eq__(self, other):     # 只重写了"equals"
        return (self.x, self.y) == (other.x, other.y)
```
Java IDE 会提醒你"重写 equals 要一起重写 hashCode"。Python 没人提醒，而且后果更狠。

**💥 翻车现象**
```python
p = Point(1, 2)
seen = {p}                       # ❌ TypeError: unhashable type: 'Point'
```
一旦你定义了 `__eq__`，Python 会**自动把 `__hash__` 置为 None**，对象直接不可哈希，进不了 `set`、当不了 dict key。

**✅ 正确写法**
```python
class Point:
    def __init__(self, x, y):
        self.x, self.y = x, y
    def __eq__(self, other):
        return (self.x, self.y) == (other.x, other.y)
    def __hash__(self):
        return hash((self.x, self.y))   # 用相同字段，和 __eq__ 一致

# 更省事:
from dataclasses import dataclass
@dataclass(frozen=True)              # 自动生成 __eq__ + __hash__
class Point:
    x: int
    y: int
```

**🔑 一句根因**：定义 `__eq__` 会清掉默认 `__hash__`。两者必须**用相同字段成对实现**（相等的对象必须同 hash），否则用 `@dataclass(frozen=True)`。

**📖 详见** [13 类与对象](13_类与对象.md)、[03 变量与对象模型](03_变量与对象模型.md)

### 坑 9 · 裸 `except` 吞掉一切，包括 Ctrl-C

**❌ Java 直觉写法**
```python
try:
    process(data)
except:                  # 像 catch (Throwable t) 那样兜底
    log("failed")
```

**💥 翻车现象**：裸 `except`（或 `except BaseException`）连 `KeyboardInterrupt`（Ctrl-C）和 `SystemExit` 都吞了——你的程序按不灭、退不出。更糟的是把本该让程序崩掉的 bug（拼写错的属性名、`None` 解引用）也静默吃掉，错误无声消失，数据悄悄写错，几周后才发现。

**✅ 正确写法**
```python
try:
    process(data)
except (ValueError, KeyError) as e:    # 只捕获你能处理的具体异常
    log.warning("bad record: %s", e)
    # 其余异常让它冒泡
```

**🔑 一句根因**：裸 `except` / `except Exception` 兜得太宽，吞掉本该崩的 bug 和中断信号。**只捕获你能处理的具体异常类型**，其余让它冒泡。要兜底也用 `except Exception`（不含信号），并务必记日志 + `raise`。

**📖 详见** [15 异常与上下文管理器](15_异常与上下文管理器.md)

### 坑 10 · async 里调阻塞代码：冻住整个事件循环

**❌ Java 直觉写法**
```python
@app.get("/price")
async def get_price(symbol: str):
    resp = requests.get(f"https://api/{symbol}")   # 同步 requests
    time.sleep(0.5)                                  # 同步 sleep
    return resp.json()
```
Java 的线程池里一个线程阻塞，其他线程照跑。async 不是这样。

**💥 翻车现象**：`requests.get` / `time.sleep` 是**同步阻塞**调用，它们冻结的不是一个请求，而是**整个事件循环**——这个 worker 上所有并发请求一起卡死。你以为 async 能扛高并发，结果比同步的 Spring MVC 还糟。

**✅ 正确写法**
```python
import httpx, asyncio
@app.get("/price")
async def get_price(symbol: str):
    async with httpx.AsyncClient() as client:
        resp = await client.get(f"https://api/{symbol}")   # 异步全程 await
    await asyncio.sleep(0.5)
    # 不得不调阻塞库时:
    # result = await asyncio.to_thread(blocking_call, arg)
    return resp.json()
```

**🔑 一句根因**：async 是**单线程协作式**调度，任何同步阻塞调用都会霸占线程、饿死所有协程。async 端点必须全程 `await` 异步库；非用阻塞库不可时丢进 `asyncio.to_thread()`。

**📖 详见** [21 异步编程asyncio](21_异步编程asyncio.md)、[24 HTTP与Web框架](24_HTTP与Web框架.md)、[30 部署与可观测性](30_部署与可观测性.md)

### 坑 11 · 忘了 `await`：协程根本没跑

**❌ Java 直觉写法**
```python
async def save_order(order):
    db.insert(order)        # 这是个 async 函数，但...

async def handle():
    save_order(order)       # ❌ 忘了 await，像调普通方法一样调它
    return {"ok": True}
```

**💥 翻车现象**：`save_order(order)` 只是**创建了一个协程对象**，函数体一行没执行就被丢弃。订单没存、还可能甩出 `RuntimeWarning: coroutine 'save_order' was never awaited`。代码"成功返回"，数据却凭空消失。

**✅ 正确写法**
```python
async def handle():
    await save_order(order)              # 等它真正执行
    # 后台执行不阻塞当前: task = asyncio.create_task(save_order(order))
    return {"ok": True}
```

**🔑 一句根因**：调用 async 函数只是造一个协程，**必须 `await`（或 `create_task`/`gather`）才会执行**。养成"看到 async 函数就配 await"的肌肉记忆，并打开 `-W error::RuntimeWarning` 让漏 await 直接报错。

**📖 详见** [21 异步编程asyncio](21_异步编程asyncio.md)

### 坑 12 · 不指定 encoding：跨平台乱码定时炸弹

**❌ Java 直觉写法**
```python
with open("report.csv") as f:        # 不写 encoding
    data = f.read()
```
Java 的 `new String(bytes)` 不带 charset 也能跑，你早被这教育过，但到 Python 又忘了。

**💥 翻车现象**：`open()` 在文本模式下**用平台默认编码**——Linux 是 UTF-8，旧版 Windows 是 GBK/CP1252。开发机好好的，部署到另一平台就 `UnicodeDecodeError` 或乱码。这类 bug 在 CI 里测不出来（CI 也是 Linux），专挑生产环境炸。

**✅ 正确写法**
```python
with open("report.csv", encoding="utf-8") as f:
    data = f.read()
# 来源是 Excel 导出的带 BOM 文件: encoding="utf-8-sig"
```

**🔑 一句根因**：文本 I/O 的默认编码依赖平台 locale，不可移植。**所有 `open()`/`.decode()`/`.encode()` 显式写 `encoding="utf-8"`**。Python 3.15 起 `open()` 不指定编码会发警告，提前改掉。

**📖 详见** [09 字符串字节与编码](09_字符串字节与编码.md)、[22 文件IO与系统交互](22_文件IO与系统交互.md)

### 坑 13 · N+1 查询：ORM 把你的循环变成 DDoS

**❌ Java 直觉写法**
```python
authors = session.query(Author).all()
for a in authors:
    print(a.name, len(a.books))    # 像 JPA 一样随手点关系属性
```
Hibernate 的 LAZY 加载也有这毛病，但 Spring 生态里 `@EntityGraph`/`fetch join` 已成肌肉记忆；到 SQLAlchemy 你又裸奔了。

**💥 翻车现象**：1 条查 authors 的 SQL + 每个 author 各 1 条查 books 的 SQL = **1 + N 条查询**。100 个作者就是 101 次往返数据库，列表接口 P99 飙到几秒。在 async ORM 下还会直接抛 `MissingGreenlet`/lazy-load 报错。

**✅ 正确写法**
```python
from sqlalchemy.orm import selectinload
authors = session.query(Author).options(selectinload(Author.books)).all()
# 一对多用 selectinload，多对一用 joinedload；对应 JPA 的 fetch join
```

**🔑 一句根因**：ORM 的惰性加载让"访问关系属性"偷偷变成一次查询，循环里就是 N+1。**预加载**（`selectinload`/`joinedload`）把 N 条合并成 1~2 条。

**📖 详见** [25 数据库与ORM](25_数据库与ORM.md)、[24 HTTP与Web框架](24_HTTP与Web框架.md)

### 坑 14 · SQL 注入：f-string 拼 SQL 是开门揖盗

**❌ Java 直觉写法**
```python
cur.execute(f"SELECT * FROM users WHERE name = '{name}'")   # f-string 拼 SQL
```
Java 里你早知道要用 `PreparedStatement`，但 Python 的 f-string 太顺手，一不留神就拼了。

**💥 翻车现象**：`name = "'; DROP TABLE users; --"` 直接删库。即便没人攻击，名字里一个单引号 `O'Brien` 也会让 SQL 语法崩掉。这是 OWASP Top 1 级别的漏洞。

**✅ 正确写法**
```python
cur.execute("SELECT * FROM users WHERE name = %s", (name,))   # 参数化
# 占位符随驱动: sqlite3 用 ?，psycopg 用 %s，SQLAlchemy 用 :name
# SQLAlchemy: session.execute(text("... WHERE name = :n"), {"n": name})
```

**🔑 一句根因**：永远不要把外部数据拼进 SQL 文本。**参数走 `execute` 的第二个实参**，让驱动负责转义。占位符是参数化的标志，不是字符串格式化。

**📖 详见** [25 数据库与ORM](25_数据库与ORM.md)、[29 工程化最佳实践](29_工程化最佳实践.md)

### 坑 15 · 整数不溢出，但 numpy 会静默回绕

**❌ Java 直觉写法**
```python
import numpy as np
prices = np.array([2_000_000_000, 1_000_000_000], dtype=np.int64)
total = prices.sum()          # 以为 Python int 永不溢出，放心相加
```
你刚学会"Python `int` 任意精度、永不溢出"，于是放下了所有防溢出的戒心——然后 numpy 给你上一课。

**💥 翻车现象**
```python
print(total)    # → 3000000000 这次没事，但...
big = np.array([2**63 - 1], dtype=np.int64)
print(big + 1)  # → [-9223372036854775808]  ❌ 静默回绕成负数，无任何警告
```
纯 Python `int` 确实不溢出，但 **numpy 用定宽 C 整数**（int64/int32），溢出会像 Java 一样**静默回绕**——而且默认不报错。金额、计数器在这里悄悄变负，对账时才发现。

**✅ 正确写法**
```python
total = prices.sum(dtype=np.int64)   # 至少明确位宽
# 可能溢出就用更大类型或 Python 对象数组
big = np.array([2**63 - 1], dtype=object)
print(big + 1)    # → [9223372036854775808] ✅ 退化为 Python 大整数
# 或用 float64（牺牲精度换范围），或上界检查
```

**🔑 一句根因**：Python `int` 不溢出是**纯 Python 层**的保证；一进 numpy/pandas/`ctypes` 就是定宽整数，**会静默溢出回绕**。涉及 numpy 算术，要主动选 `dtype` 和检查范围。

**📖 详见** [08 数字与算术语义](08_数字与算术语义.md)、[27 标准库与数据AI生态](27_标准库与数据AI生态.md)

## 三、Java→Python 习惯迁移清单

把这份当成"出 Java 师门"的体检表。每一条都是要**主动改掉的肌肉记忆**——不是 Python 更优越，而是底层模型不同，旧习惯会翻车。打勾自查：

**判断与比较**
- [ ] 判空用 `if x is None` / `if x is not None`，不用 `== None`
- [ ] 判空集合直接 `if not lst` / `if lst`，不用 `len(lst) == 0`（但 `x == 0` 这种"0 是有效值"的场景要显式比，别被假值坑）
- [ ] 一切内容相等用 `==`；`is` 只留给 `None`/`True`/`False`/哨兵
- [ ] 浮点相等用 `math.isclose`，不用 `==`

**循环与集合**
- [ ] 遍历用 `for x in seq`，要下标用 `enumerate`，**永不** `for i in range(len(seq))`
- [ ] 反复成员检测用 `set`/`dict`（O(1)），不在大 `list` 上 `in`
- [ ] 不边遍历边增删容器；要改就遍历 `list(d)` 副本或推导式重建
- [ ] 拼字符串用 `"".join(parts)`，不在循环里 `+=`
- [ ] 推导式只用来"造集合"，副作用走普通 `for`

**函数与类**
- [ ] 可变默认参数一律 `None` 哨兵 + 体内新建
- [ ] 可变实例状态在 `__init__` 里 `self.x=...`，类体只放不可变常量
- [ ] 闭包在循环里捕获循环变量时用 `lambda i=i:` 立即绑定
- [ ] 自定义类要进 set/dict 时 `__eq__`+`__hash__` 成对实现，或 `@dataclass(frozen=True)`
- [ ] 不指望方法重载；用默认参数 / `*args/**kwargs` / `singledispatch`
- [ ] 装饰器记得 `@functools.wraps`

**资源与异常**
- [ ] 文件/连接/锁一律 `with`，不靠 GC、不手写 `finally close`
- [ ] `open()` 永远显式 `encoding="utf-8"`
- [ ] 只 `except` 你能处理的具体异常；不裸 `except`、慎用 `except Exception`
- [ ] 转换异常用 `raise NewError(...) from e` 保留根因
- [ ] `finally` 只放清理，不写 `return`/`break`/`continue`

**并发与异步**
- [ ] CPU 密集用多进程/向量化，不指望多线程加速（GIL）
- [ ] 即便有 GIL，共享可变状态也要加 `Lock`
- [ ] async 端点全程 `await` 异步库；阻塞调用丢 `asyncio.to_thread`
- [ ] 调 async 函数必配 `await`；并发用 `gather`/`TaskGroup`，不串行 await
- [ ] async 上下文数据用 `contextvars`，不用 `threading.local`

**数据库与 Web**
- [ ] SQL 参数化（`execute(sql, params)`），绝不 f-string 拼 SQL
- [ ] 写操作记得 `commit`；Session 每请求一个、用完即弃
- [ ] 列表查询预加载关系（`selectinload`），消灭 N+1
- [ ] 每个 HTTP 请求显式 `timeout`，复用 `Session`/`Client`，`raise_for_status`

**工程与安全**
- [ ] 用 `logging.getLogger(__name__)`，不用 `print` 当日志；日志用 `%s` 惰性占位
- [ ] 密钥走环境变量 + `pydantic-settings`，`.env` 进 `.gitignore`，绝不硬编码
- [ ] token/盐用 `secrets`，口令用 `bcrypt`/`argon2`，密文比对用 `hmac.compare_digest`
- [ ] 对外序列化用 `json`，绝不 `pickle.loads` 不可信数据
- [ ] 时间一律 aware UTC（`datetime.now(timezone.utc)`），不用 `now()`/`utcnow()`
- [ ] 加类型注解 + mypy/pyright 进 CI；边界输入用 pydantic 运行时校验
- [ ] 锁依赖版本（lock 文件），用 `python -m pkg` 跑包

### 参考答案

#### 练习 1（5 个坑）

1. **可变默认参数 `cache={}`**（坑 1）：跨调用共享，不同批次用户的缓存串味。→ `cache=None` + 体内 `if cache is None: cache = {}`（注意：若本意就是"跨调用持久缓存"，应改用显式模块级缓存或 `functools.lru_cache`，而不是默认参数这种隐式写法）。
2. **`for i in range(len(users))`**：应直接 `for u in users`。
3. **`u.status == None` / `cache.get(u.id) == None`**：用 `is None`。
4. **`u.status is "active"`**（坑 3）：字符串内容比较用 `==`，`is` 会随驻留时灵时不灵。
5. **`print(...)` 当日志 + 字符串拼接**：用 `logging.getLogger(__name__)`，`log.info("found %s active users", len(result))`。

> 详见 [12](12_函数.md)、[11](11_控制流与推导式.md)、[03](03_变量与对象模型.md)、[29](29_工程化最佳实践.md)。

#### 练习 2（4 个坑）

1. **SQL 注入**（坑 14）：f-string 拼 symbol 进 SQL。→ `cur.execute("INSERT INTO query_log(symbol) VALUES(%s)", (symbol,))`。
2. **async 里用同步 `requests`**（坑 10）：冻结事件循环。→ `async with httpx.AsyncClient() as c: resp = await c.get(...)`。
3. **async 里 `time.sleep`**（坑 10）：同样阻塞。→ `await asyncio.sleep(0.2)`。
4. **不查状态码 + 无超时**：对端 4xx/5xx 时 `data["price"]` 直接 KeyError；无 timeout 会永久挂起。→ 加 `timeout`、`resp.raise_for_status()`。
5. （附加）**写库忘 commit**：若连接非 autocommit，日志根本没落库。→ `conn.commit()`。

> 详见 [25](25_数据库与ORM.md)、[21](21_异步编程asyncio.md)、[24](24_HTTP与Web框架.md)。

#### 练习 3（4 个坑）

1. **`total = 0.0` + float 累加算钱**（坑「浮点算钱」）：金额用 `Decimal`，从字符串构造，避免二进制误差。
2. **`total / len(orders)` 可能除零**：`orders` 为空时 `ZeroDivisionError`。→ 先判空。
3. **`round(...)` 是银行家舍入**：`round(2.5)==2`，做金额折扣会和业务预期（四舍五入）不符。→ `Decimal(...).quantize(Decimal("1"), ROUND_HALF_UP)`。
4. **返回的 `discount` 类型不一致 / 精度丢失**：混用 float 和 round 结果，序列化成 JSON 还会带浮点尾巴。→ 全程 `Decimal`，序列化时转 `str`。

> 详见 [08](08_数字与算术语义.md)、[26](26_日期时间序列化pydantic.md)。

#### 练习 4（2 个坑）

1. **可变类属性 `handlers = []`**（坑 7）：所有 `Dispatcher` 实例共享同一个 list，一个实例 `register_all` 覆盖/污染另一个。→ 在 `__init__` 里 `self.handlers = []`。
   - （注：本例 `register_all` 里是整体重新赋值 `self.handlers = callbacks`，赋值会在实例上新建属性、不再碰类属性；但只要有任何一处用 `self.handlers.append(...)`，共享就会爆发。类属性是可变 list 本身就是定时炸弹，必须改。）
2. **闭包晚绑定 `lambda: self.handle(name)`**（坑 6）：循环造的所有 lambda 共享 `name`，最终全部捕获最后一个名字。→ `lambda name=name: self.handle(name)`。

> 详见 [13](13_类与对象.md)、[12](12_函数.md)。

#### 练习 5（5 个坑）

1. **硬编码密钥 `PASSWORD = "..."`**（荣誉提名）：→ 走环境变量 + `pydantic-settings`，进 `.gitignore`。
2. **`pickle.loads` 不可信数据**（坑「pickle RCE」）：配置文件若被篡改可执行任意代码。→ 用 `json`/`tomllib` 解析配置。
3. **`open(path)` 不指定 encoding**（坑 12）：→ `encoding="utf-8"`。
4. **不用 `with` 关文件**（两处 `open` 都没关）：`load_config` 的 `f` 和 `save_report` 的 `out` 都靠 GC，`out` 还可能内容没刷盘。→ 全部 `with open(...) as f:`。
5. **写 CSV 用字符串拼接 + 无转义**：`r + "\n"` 遇到含逗号/引号/换行的字段会破坏 CSV 结构。→ 用 `csv.writer`。

> 详见 [29](29_工程化最佳实践.md)、[26](26_日期时间序列化pydantic.md)、[22](22_文件IO与系统交互.md)、[09](09_字符串字节与编码.md)。
