# 09 · 字符串、字节与编码 ⭐

> 你在 Java 里被 `new String(bytes)` 不指定 charset 坑过、被 ISO-8859-1 和 UTF-8 的乱码折磨过，但 Java 至少把 `String`（文本）和 `byte[]`（字节）分得清清楚楚。Python 2 没分清，酿成史上最大的迁移惨案；Python 3 痛定思痛，把 `str`（Unicode 文本）和 `bytes`（原始字节）**彻底分家、严禁隐式混用**。本章讲清这条边界、PEP 393 的三套内部表示（对照 Java 的 `char[]` 和 Compact Strings）、以及 f-string / 编解码 / 拼接的全部实战坑——这是后端处理 HTTP、文件、数据库的**头号翻车区**。

---

## ① Java 对照表

| 概念 | Java | Python | 关键差异 |
|------|------|--------|---------|
| 不可变文本 | `String` | `str` | 都不可变；但 Python `str` 是码点序列，内部表示三选一（PEP 393），Java 固定 UTF-16 |
| 字节序列（只读） | `byte[]`（可变！） | `bytes` | **Python `bytes` 不可变**，Java `byte[]` 可变；Java 没有"不可变字节序列"内置类型 |
| 可变字节缓冲 | `byte[]` / `ByteBuffer` | `bytearray` | 对应 Java 可变 `byte[]`，可原地改 |
| 字节→文本 | `new String(bytes, charset)` | `bytes.decode("utf-8")` | Python **必须显式**给编码（或吃默认 UTF-8）；Java 不给就用平台默认（坑） |
| 文本→字节 | `str.getBytes(charset)` | `str.encode("utf-8")` | 同上，方向相反 |
| 插值/模板 | 无（用 `+` 或 `String.format`），JDK 21 文本块/模板 | **f-string** `f"{x}"` | f-string 是编译期语法，直接嵌表达式，比 `String.format` 强大得多 |
| 占位格式化 | `String.format("%.2f", x)` / `MessageFormat` | `str.format()` / `f"{x:.2f}"` | `{}` 占位对照 `MessageFormat` 的 `{0}` |
| 高效拼接 | `StringBuilder` | `"".join(iterable)` / `io.StringIO` | Python 没有 StringBuilder；循环拼接用 `join`，别用 `+=` |
| 字符 | `char`（16 位基本类型） | 无 `char`，单字符就是长度 1 的 `str` | Python 没有字符类型；`ord()`/`chr()` 在码点与字符间转换 |
| 长度 | `s.length()`（UTF-16 **代码单元**数） | `len(s)`（**码点**数） | 对 emoji/生僻字，Java 长度可能比 Python 大（代理对算 2） |
| 原始字符串 | 无（`\\` 转义反斜杠） | `r"..."` raw string | 写正则/Windows 路径神器 |

## ② 设计理念（为什么 Python 3 把 str 和 bytes 彻底分开）

### 2.1 Python 2 的"混乱原罪"

理解 Python 3 的设计，必须先看它在逃避什么。Python 2 里有两个类型：

- `str`：其实是**字节串**（等价于今天的 `bytes`），但名字叫 `str`，还能当文本用。
- `unicode`：真正的 Unicode 文本。

致命的是，Python 2 允许 `str`（字节）和 `unicode`（文本）**隐式混用**：拼接时自动用 ASCII 解码字节串去凑 `unicode`。结果是：

```python
# Python 2 的经典惨剧（这是 Py2 行为，不要在 Py3 运行）
"café" + u"bar"
# 字节串 "café" 含非 ASCII 字节 → 自动 ASCII 解码失败
# → UnicodeDecodeError: 'ascii' codec can't decode byte 0xc3 ...
```

最恶心的是：这个错误**只在数据里出现非 ASCII 字符时才爆**。你的代码在测试环境（全英文数据）跑得好好的，一上线遇到中文/emoji/法语就崩——这就是著名的"`UnicodeDecodeError` 上线惊魂"。Java 工程师可以类比成：`byte[]` 和 `String` 之间能隐式互转、且转换悄悄使用平台默认编码，错误延迟到运行时特定数据才暴露。

### 2.2 Python 3 的"显式边界"哲学

Python 3 做了破釜沉舟的决定，也是它和 Python 2 不兼容的最大原因：

1. **`str` 永远是 Unicode 文本**（码点序列），不再是字节。
2. **`bytes` 是原始字节**，和 `str` 是**两个无法隐式转换**的类型。
3. **`str` 和 `bytes` 之间只能通过显式 `encode`/`decode` 转换**，且必须（隐含地）指定编码。

```python
b"abc" + "def"
# TypeError: can't concat str to bytes  ← 直接拒绝，编译不了你的"想当然"
```

这背后的哲学是 Python 之禅的那句 **"Explicit is better than implicit"**：编解码是有损、有歧义、可能失败的操作，必须由程序员**在明确的边界上显式声明**，绝不让解释器替你猜。

> **心智模型（Java 老手最该建立的）**：把 `str` 想成 JVM 里的 `String`，把 `bytes` 想成 `byte[]`，并且**强制规定 Java 永远不许有 `new String(bytes)` 不带 charset 的重载**。所有文本 ↔ 字节的转换点都是显式的 `encode`/`decode`，编码参数不可省（省了就用 UTF-8，而不是某个平台默认）。这就是 Python 3 的世界。

### 2.3 "网络与磁盘是字节，程序内部是文本"

由此引出一个贯穿后端开发的铁律，画成图：

```
   外部世界（字节）              你的程序（文本）            外部世界（字节）
  ┌──────────────┐    decode    ┌──────────────┐   encode   ┌──────────────┐
  │ HTTP body    │ ───────────► │              │ ─────────► │ HTTP response│
  │ 文件 (磁盘)   │   bytes→str  │   str 处理    │  str→bytes │ 写文件        │
  │ socket / DB  │              │  (业务逻辑)   │            │ socket / DB  │
  └──────────────┘              └──────────────┘            └──────────────┘
        bytes                        str                         bytes
```

- **入口**：从 socket / 文件 / HTTP 读到的永远是 `bytes`，**第一时间 `decode` 成 `str`**。
- **中间**：业务逻辑全程操作 `str`，不碰编码。
- **出口**：要写回网络/磁盘前，**最后一刻 `encode` 成 `bytes`**。

把 `decode`/`encode` 锁死在系统边界，中间层永远只见文本——这就是不乱码的根本原则。Java 的最佳实践其实一模一样（边界处处理编码），只是 Java 没强制你，Python 强制你。

## ③ 实现细节（底层怎么实现的）★本章重点★

### 3.1 PEP 393：灵活字符串表示（Flexible String Representation）

这是 Python 3.3 起 `str` 的核心实现，也是和 Java 字符串模型差异最大的地方。

**问题背景**：早期 Python 3 的 `str` 内部固定用 UCS-2 或 UCS-4（每字符 2 或 4 字节）。UCS-4 版本内存翻倍浪费（绝大多数字符串是 ASCII，用 4 字节存一个 `'a'` 太奢侈）；UCS-2 版本又有"窄构建"在 emoji 上数错长度的 bug。

**PEP 393 的解法**：根据字符串里**最大码点（max code point）**，动态选择三种内部表示之一，每个字符串用最省的那种：

| 内部种类 | 每字符字节 | 能表示的码点范围 | 典型内容 |
|---------|-----------|----------------|---------|
| **Latin-1 (1 字节)** | 1 | U+0000 – U+00FF | 纯 ASCII、西欧文本 |
| **UCS-2 (2 字节)** | 2 | U+0000 – U+FFFF（BMP） | 含中文、日文等基本多文种平面字符 |
| **UCS-4 (4 字节)** | 4 | U+0000 – U+10FFFF | 含 emoji、生僻字（增补平面） |

关键点：**一个字符串只用一种表示，由它含的"最大"字符决定**。只要混进一个 emoji，整个字符串就升级成 UCS-4。

```python
import sys

print(sys.getsizeof("hello"))      # 54   纯 ASCII → 1 字节/字符 + 头部开销
print(sys.getsizeof("héllo"))      # 也是 1 字节/字符（é 是 U+00E9，仍在 Latin-1 内）
print(sys.getsizeof("中文你好"))     # 含中文 → 2 字节/字符（UCS-2）
print(sys.getsizeof("hi😀"))        # 含 emoji（U+1F600）→ 4 字节/字符（UCS-4）
# 同样 5 个字符，UCS-4 版本占的内存是 Latin-1 版本的近 4 倍（不含头部）

# 验证"一个字符就能拉高整串的表示"
print(sys.getsizeof("a"))          # 50（1 字节表示，含较小头部）
print(sys.getsizeof("😀"))          # 80（4 字节表示，头部也更大）
```

> 你会发现 `getsizeof` 不只是"字符数 × 每字符字节"，还有一段固定的 `PyASCIIObject` / `PyCompactUnicodeObject` 头部（存长度、哈希缓存、表示种类标志等）。纯 ASCII 用最精简的头部，所以 `"hello"` 比含非 ASCII 的同长字符串还省。

### 3.2 对照 Java 的 `char[]`（UTF-16 + 代理对）和 Compact Strings

Java 的 `String` 模型走了不同的路：

- **JDK 8 及以前**：`String` 内部固定是 `char[]`，每个 `char` 是 **16 位 UTF-16 代码单元**。
- **超出 BMP 的字符**（如 emoji U+1F600）在 UTF-16 里用**代理对（surrogate pair）**——两个 `char` 表示一个字符。这导致 `"😀".length() == 2`（两个代码单元），而 Python 的 `len("😀") == 1`（一个码点）。
- **JDK 9+ 的 Compact Strings（JEP 254）**：和 PEP 393 异曲同工——`String` 内部改成 `byte[]` + 一个 `coder` 标志，纯 Latin-1 内容用 1 字节/字符存，否则用 UTF-16（2 字节）。

把两者并排看：

| 维度 | Java `String`（JDK 9+） | Python `str`（PEP 393） |
|------|------------------------|------------------------|
| 内部表示档位 | **2 档**：Latin-1(1B) 或 UTF-16(2B) | **3 档**：Latin-1(1B) / UCS-2(2B) / UCS-4(4B) |
| 超 BMP 字符（emoji） | UTF-16 **代理对**，占 2 个代码单元 | UCS-4，**1 个码点**，长度算 1 |
| `length()` / `len()` 语义 | UTF-16 **代码单元数** | **码点数** |
| emoji 长度 | `"😀".length()` → **2** | `len("😀")` → **1** |
| 索引语义 | 按代码单元（可能切坏代理对） | 按码点（永远切不坏字符） |

**这是最实际的差异**：Python 的 `str` 索引、切片、`len` 全部以**码点**为单位，你永远不会像 Java 那样在 emoji 中间切出半个代理字符。Python 选了 UCS-4 兜底换来语义干净，代价是含 emoji 的串内存翻倍。

```python
s = "a😀b"
print(len(s))        # 3   （Python：3 个码点）
print(s[1])          # 😀  （索引 1 正好是完整的 emoji）
# Java 里 "a😀b".length() == 4，charAt(1) 拿到的是半个代理字符（乱码）
print(ord(s[1]))     # 128512  即 U+1F600
```

### 3.3 为什么 `len(str)` 是码点数而不是字节数

承上：Python 的 `str` 是**码点序列的抽象**，内部用哪种字节表示是实现细节，对你透明。`len()` 返回码点数，符合"一个字符算一个"的直觉。

但务必区分清楚——**`str` 的 `len` 是码点数，`bytes` 的 `len` 是字节数**：

```python
s = "中文"
print(len(s))                 # 2    两个码点（两个汉字）
print(len(s.encode("utf-8"))) # 6    UTF-8 下每个汉字 3 字节，2×3=6
print(len(s.encode("gbk")))   # 4    GBK 下每个汉字 2 字节
```

这是后端"字段长度限制"最经典的坑：数据库 `VARCHAR(10)` 在 MySQL `utf8mb4` 下按**字符**算（能存 10 个汉字），但如果你在 Python 里用 `len(s)` 校验、底层存储却按字节限长，就会对不上。永远想清楚你在数它的**码点还是字节**。

### 3.4 字符串驻留（interning）

和 Java 的 `String.intern()` / 字符串常量池高度类似：CPython 把一些字符串放进一个全局的 **interned 字典**，相同内容只存一份，让 `is`（身份比较）也能命中、并省内存。

```python
import sys

a = "hello"
b = "hello"
print(a is b)              # True   编译期常量、且是合法标识符 → 自动驻留，同一对象

s1 = "hello world!"
s2 = "hello world!"
print(s1 is s2)            # 可能 False（含空格/标点的字面量不一定自动驻留）

# 手动驻留：高频重复字符串（如解析出的大量相同 key）可省内存、加速比较
x = sys.intern("some_repeated_key")
y = sys.intern("some_repeated_key")
print(x is y)              # True   显式驻留后必然同一对象
```

驻留规则（CPython 实现细节，**不要依赖它写逻辑**）：

- 编译期出现的、**形似标识符**（字母数字下划线）的字符串字面量通常自动驻留。
- 短字符串、空字符串、单字符 Latin-1 字符往往被缓存。
- 运行时**拼接/格式化产生**的字符串一般**不**自动驻留。

> **铁律（和 Java 完全一致）**：判断字符串**内容相等永远用 `==`，绝不用 `is`**。`is` 比的是对象身份（C 层指针），驻留命中是优化巧合，不是语义保证。详见 [05 CPython 内部](05_CPython内部.md) 对 `PyObject` 身份的讨论。

`sys.intern` 的实战价值：解析大文件/JSON 时若有海量重复的字段名，手动 intern 这些 key 能显著省内存、并让 dict 查找的 key 比较走指针快路径。

### 3.5 `bytes` 不可变 vs `bytearray` 可变

- **`bytes`**：不可变的字节序列。底层是 `PyBytesObject`，存了长度 + 一段连续字节 + 缓存的哈希值。因为不可变，所以**可哈希**（能做 dict key / set 元素），且可安全共享。
- **`bytearray`**：可变的字节序列，底层 `PyByteArrayObject` 带一个可增长的缓冲区（类似 `list` 的超额分配策略），支持原地 `append`/`extend`/切片赋值。**不可哈希**。

```python
b = b"abc"
print(b[0])              # 97    索引返回的是 int（单个字节值 0-255），不是 b"a"！
print(b[0:1])            # b'a'  切片才返回 bytes
# b[0] = 100             # TypeError: 'bytes' object does not support item assignment

ba = bytearray(b"abc")
ba[0] = 100              # OK，原地改
ba.append(33)            # 追加一个字节
print(ba)                # bytearray(b'dbc!')
print(bytes(ba))         # b'dbc!'   转回不可变 bytes
```

> **易踩**：`bytes` / `bytearray` 用**整数索引**取单个元素时，返回的是 **`int`（0–255）而非长度 1 的 `bytes`**——这和 `str` 索引返回长度 1 的 `str` 不对称。要拿"一个字节的 bytes"得用切片 `b[i:i+1]`。Java 里 `byte[]` 索引返回 `byte`（有符号 -128~127），Python 的字节是无符号 0~255，这点也要留意。

`bytearray` 的典型用途：高性能拼装二进制协议帧、流式读取时的可变缓冲（对照 Java 的 `ByteArrayOutputStream` / 可变 `byte[]`）。

### 3.6 编解码在 C 层的样子

`str.encode(enc)` / `bytes.decode(enc)` 底层走 CPython 的 **codec 注册表**：编码名（"utf-8"/"gbk"/...）映射到一对 C 函数（encoder/decoder）。UTF-8 这种常用编码有高度优化的 C 快路径（甚至针对 PEP 393 的 1 字节 ASCII 表示直接 `memcpy`——ASCII 字符串编成 UTF-8 字节几乎零成本，因为内容一致）。

`errors` 参数决定遇到无法编解码的字符/字节时的策略，这是 C 层 codec 的回调钩子（见 ④ 4.6）。

## ④ 使用方法与最佳实践

### 4.1 f-string（首选，Python 3.6+）

f-string 是**编译期语法**（编译时直接生成拼接/格式化字节码，不是运行时解析模板），所以比 `str.format` 和 `%` 都快，也最易读。Java 里没有对应物——最接近的是 JDK 21 的字符串模板（预览特性），但 f-string 强大得多：**大括号里能放任意表达式**。

```python
name, qty, price = "牛奶", 3, 12.5

# 基本插值：直接放变量、表达式、函数调用
print(f"{name} x{qty} = {qty * price} 元")
# → 牛奶 x3 = 37.5 元

print(f"含税: {price * 1.13:.2f}")           # → 含税: 14.12   :.2f 保留两位小数

# 格式规约（: 之后，对照 printf 的格式串），后端打日志/对齐报表必备
print(f"{price:>10.2f}")     # →      12.50   右对齐宽 10
print(f"{name:<8}|")         # → 牛奶      |     左对齐宽 8（注意中文宽度按 1 算）
print(f"{name:^8}|")         # →   牛奶    |     居中
print(f"{1234567:,}")        # → 1,234,567     千分位逗号
print(f"{0.1567:.1%}")       # → 15.7%         百分比
print(f"{255:#x}  {255:08b}")# → 0xff  11111111  十六进制 / 补零二进制
print(f"{42:+}")             # → +42           强制带符号

# = 调试语法（Python 3.8+）：自动打印"表达式=值"，调试神器
total = qty * price
print(f"{total=}")           # → total=37.5      省去重复写变量名
print(f"{qty * price=}")     # → qty * price=37.5

# 嵌套：格式规约本身也能用变量
width = 12
print(f"{name:>{width}}")    # 用变量 width 动态指定宽度

# 多行用括号隐式拼接或三引号
msg = (
    f"订单: {name}\n"
    f"金额: {total:.2f}"
)
```

> f-string 里要打**字面量大括号**用 `{{` / `}}`；3.12 起 f-string 内部可以嵌套同种引号、放反斜杠（早期版本受限），写复杂表达式更自由。

### 4.2 `str.format()`（动态模板时用，对照 MessageFormat）

当模板字符串本身是变量（如从配置/i18n 文件读来的），不能用 f-string（它要求编译期字面量），改用 `.format()`：

```python
template = "{user} 你有 {count} 条新消息"     # 假设来自配置文件
print(template.format(user="张三", count=5))
# → 张三 你有 5 条新消息

# 位置参数 + 索引（对照 Java MessageFormat 的 {0} {1}）
print("{0}-{1}-{0}".format("A", "B"))         # → A-B-A
# 格式规约和 f-string 完全一致
print("{:>8.2f}".format(3.14159))             # →     3.14
```

### 4.3 `%` 旧式格式化（遗留代码、logging 里会见到）

类似 C/Java 的 `printf`。新代码别用，但要看得懂；`logging` 模块仍推荐用它（延迟格式化）：

```python
print("%s=%d, %.1f%%" % ("x", 10, 95.5))      # → x=10, 95.5%

import logging
# logging 推荐这种写法：只有日志真要输出时才格式化（惰性，省开销）
logging.warning("user %s failed %d times", "bob", 3)   # 不要用 f-string 拼进去
```

### 4.4 拼接：用 `join`，别用 `+=` 循环（性能关键）

这是从 Java `StringBuilder` 习惯过来最该记住的一条。Python 的 `str` 不可变，循环里 `s += x` 每次都**新建整个字符串并拷贝已有内容**，N 次拼接是 **O(N²)**——和 Java 里循环 `s = s + x`（不用 StringBuilder）的灾难一模一样。

```python
# ❌ Java 萌新味：O(N²)，数据量大时极慢
result = ""
for row in rows:
    result += format_row(row)        # 每次都拷贝整个 result

# ✅ Pythonic：先收集再 join，O(N)。join 是 Python 的 StringBuilder
parts = [format_row(row) for row in rows]
result = "".join(parts)              # 一次性算总长、分配、拷贝

# ✅ 流式/超大量文本：io.StringIO（更接近 StringBuilder 的可写缓冲）
import io
buf = io.StringIO()
for row in rows:
    buf.write(format_row(row))
result = buf.getvalue()
```

> 注意：CPython 对 `s += x` **有一个特判优化**（当 `s` 引用计数为 1 时尝试原地扩展），有时没那么糟，但**这是实现细节、不可移植、对 `join` 仍是普遍劣势**。地道代码一律 `join`。`bytes` 同理用 `b"".join(...)`，`bytearray` 用 `.extend()`。

### 4.5 常用 `str` 方法速查（后端高频）

```python
s = "  Hello, World  "
s.strip()                  # 'Hello, World'      去首尾空白（lstrip/rstrip 单边）
s.strip().lower()          # 'hello, world'       大小写：lower/upper/casefold/title
"a,b,c".split(",")         # ['a', 'b', 'c']      分割（对照 Java split，但不是正则！）
"a,b,c".split(",", 1)      # ['a', 'b,c']         限定分割次数
"line1\nline2".splitlines()# ['line1', 'line2']   按行分割（处理 \n \r\n）
"abc".startswith("ab")     # True                 startswith/endswith（可传元组多选）
"hello".replace("l", "L")  # 'heLLo'              全部替换（不是正则）
"a-b-c".rsplit("-", 1)     # ['a-b', 'c']         从右分割（取扩展名/最后一段常用）
"  x".removeprefix("  ")   # 'x'   (3.9+)         精确去前缀（比 strip 安全，不误删字符）
"file.txt".removesuffix(".txt")  # 'file' (3.9+)
"42".zfill(5)              # '00042'              补零
"abcabc".find("c")         # 2                    找不到返回 -1（index 找不到抛异常）
"Hello".encode("utf-8")    # b'Hello'             转字节
"3".isdigit(), "a".isalpha()  # 类型判断族
",".join(["a", "b"])       # 'a,b'                拼接
```

> **`split` 不是正则**！Python 的 `str.split` 按**字面子串**分割，和 Java `String.split`（接收正则）不同。要按正则分割用 `re.split`，见 [10 正则表达式](10_正则表达式.md)。无参 `s.split()` 是特例：按**任意连续空白**分割并丢弃空串，处理用户输入很顺手。

### 4.6 `encode` / `decode` 与 `errors` 策略

```python
text = "café 北京"

# 编码：str → bytes（必须想清楚目标编码）
text.encode("utf-8")       # b'caf\xc3\xa9 \xe5\x8c\x97\xe4\xba\xac'
text.encode()              # 同上，encode 默认就是 utf-8（Python 3 全面 UTF-8 化）

# 解码：bytes → str
b = b'caf\xc3\xa9'
b.decode("utf-8")          # 'café'

# errors 策略：遇到无法处理的字符/字节怎么办
"café".encode("ascii")                       # ❌ UnicodeEncodeError（é 不在 ASCII）
"café".encode("ascii", errors="ignore")      # b'caf'        丢弃
"café".encode("ascii", errors="replace")     # b'caf?'       替换为 ?
"café".encode("ascii", errors="xmlcharrefreplace")  # b'caf&#233;'  转 XML 实体
"café".encode("ascii", errors="backslashreplace")   # b'caf\\xe9'

# 解码容错：处理脏数据/未知编码字节流
bad = b'caf\xe9'                             # 这是 Latin-1 的 é，不是合法 UTF-8
bad.decode("utf-8")                          # ❌ UnicodeDecodeError
bad.decode("utf-8", errors="replace")        # 'caf�'   替换为 U+FFFD（）
bad.decode("latin-1")                        # 'café'        latin-1 永不报错（坑见 ⑤）
```

> `errors="replace"` 的 U+FFFD（）就是你在网页/日志里见到的"乱码方块/问号"——它意味着**解码时编码猜错了**。看到它别只压制，要回头查真实编码。

### 4.7 文件 I/O：永远显式 `encoding=`

这是后端最高频的乱码源头。`open()` 的文本模式默认编码**依赖操作系统**（Linux/Mac 常是 UTF-8，Windows 可能是 GBK/cp1252）——靠默认就是把炸弹埋进跨平台部署。

```python
# ❌ 危险：默认编码随平台变，Windows 上读 UTF-8 文件就乱码
with open("data.csv") as f:
    text = f.read()

# ✅ 永远显式指定 encoding
with open("data.csv", encoding="utf-8") as f:
    text = f.read()

# 写文件同理
with open("out.json", "w", encoding="utf-8") as f:
    f.write(payload)

# 处理外部脏数据：可加 errors 策略避免整批崩溃
with open("legacy.log", encoding="utf-8", errors="replace") as f:
    for line in f:
        ...

# 二进制模式 "rb"/"wb" 拿到的是 bytes，不经过编解码（图片、protobuf、原始 HTTP body）
with open("photo.jpg", "rb") as f:
    raw: bytes = f.read()
```

> Python 3.15 起 `open()` 的默认文本编码将统一为 UTF-8（PEP 686），但**现在就该养成永远写 `encoding=` 的习惯**，别等默认值变。想立刻锁定可设环境变量 `PYTHONUTF8=1` 或运行时 `-X utf8`（PEP 540 UTF-8 模式），让全进程默认 UTF-8。

### 4.8 处理 BOM

某些 Windows 工具（如旧 Excel 导出的 CSV）会在 UTF-8 文件开头塞一个 **BOM**（字节 `EF BB BF`）。用 `"utf-8"` 解码会在字符串开头多出一个 `'﻿'`，破坏首列字段名匹配：

```python
raw = b'\xef\xbb\xbfname,age\n'
raw.decode("utf-8")        # '﻿name,age\n'   ← 首列变成 '﻿name'，匹配不上！
raw.decode("utf-8-sig")    # 'name,age\n'         ← utf-8-sig 自动吃掉 BOM

# 读文件时
with open("excel_export.csv", encoding="utf-8-sig") as f:   # 兼容有/无 BOM
    ...
```

> `utf-8-sig` 解码时若有 BOM 就去掉、没有也不报错，是**读取来源不明 UTF-8 文件的安全选择**。反过来，写文件用 `"utf-8"`（不写 BOM）是跨平台首选，除非对方明确要 BOM 才用 `"utf-8-sig"` 编码。

### 4.9 raw string：路径与正则的转义救星

`str` 里 `\` 是转义引导符（`\n` 换行、`\t` 制表、`中` 码点等）。写 Windows 路径或正则模式时，满屏 `\\` 既丑又易错。`r"..."` 前缀让反斜杠保持字面：

```python
# ❌ 普通字符串：\n 被当换行，\t 被当制表，路径全废
path = "C:\Users\nick\temp"          # \U 还可能直接 SyntaxError（被当 Unicode 转义）

# ✅ raw string：反斜杠原样保留
path = r"C:\Users\nick\temp"         # 'C:\\Users\\nick\\temp'（真实内容）

# 正则更是离不开 raw string（详见第 10 章）
import re
re.findall(r"\d+\.\d+", "v1.2 v3.45")    # ['1.2', '3.45']   r"" 让 \d 直达正则引擎
```

> 注意 raw string 只是**关闭转义解析**，不是新类型——`type(r"x")` 还是 `str`。一个小坑：raw string **不能以单个反斜杠结尾**（`r"abc\"` 是语法错误，因为 `\"` 被当转义引号）。Java 没有 raw string，写正则得 `"\\d+"`，Python 的 `r"\d+"` 更清爽。

### 4.10 bytes ↔ str 转换与十六进制

```python
"héllo".encode("utf-8")          # b'h\xc3\xa9llo'    文本→字节
b'h\xc3\xa9llo'.decode("utf-8")  # 'héllo'            字节→文本

# 十六进制（看协议/哈希很有用）
b"\xde\xad\xbe\xef".hex()        # 'deadbeef'
bytes.fromhex("deadbeef")        # b'\xde\xad\xbe\xef'

# Base64（HTTP 传二进制、JWT 等）
import base64
base64.b64encode(b"binary data") # b'YmluYXJ5IGRhdGE='   注意返回的还是 bytes！
base64.b64decode(b'YmluYXJ5IGRhdGE=')  # b'binary data'
```

## ⑤ ⚠️ 易错点（Java 思维翻车）

| Java 习惯 | 翻车现场 | 正确做法 / 认知 |
|----------|---------|----------------|
| `byte[]` 和 `String` 能隐式凑合 | `b"data" + "txt"` 或 `f"{some_bytes}"` 直接 `TypeError` | `str` 和 `bytes` 严禁混拼；显式 `.decode()`/`.encode()` 跨界 |
| `new String(bytes)` 不带 charset | Python 里 `open()` / `decode()` 不写编码，靠平台默认，跨机器换平台就乱码 | 所有边界**显式 `encoding="utf-8"`**；上线机和开发机默认可能不同 |
| 循环 `s = s + x` 拼接 | 大列表拼成 O(N²)，量大时卡死 | 用 `"".join(list)` 或 `io.StringIO`，这是 Python 的 StringBuilder |
| 以为 `len(s)` 是字节数 | 按 `len` 做 `VARCHAR`/网络包长度限制，中文/emoji 全算错 | `len(str)` 是**码点数**；要字节数得 `len(s.encode("utf-8"))` |
| 以为 `"😀".length()==2` 那套（代理对） | 把 Java 代理对的心智搬来，对 emoji 索引/切片绕弯路 | Python `str` 按**码点**索引，`len("😀")==1`，切片永不切坏字符 |
| `bytes[i]` 当成"一个字节串" | `b"abc"[0]` 得到 `97`（int）不是 `b"a"`，后续比较/拼接报错 | 单字节是 `int`；要 bytes 用切片 `b[i:i+1]` |
| 路径/正则直接写 `\` | `"C:\new\table"` 里 `\n`/`\t` 被转义，路径全错，`\U` 还可能直接报错 | 用 raw string `r"C:\new\table"` |
| 用 `is` 比字符串内容（靠驻留侥幸） | `s1 is s2` 在拼接产生的串上变 `False`，逻辑随机出错 | 内容比较永远用 `==`；`is` 只比身份 |
| 解码遇错就 `errors="ignore"`/换 `latin-1` 压住 | `latin-1` 对任意字节都不报错 → 错误被吞，乱码潜伏到下游才爆 | 先查**真实编码**；`latin-1`/`ignore` 只在确知场景用，别拿它"消灭报错" |
| 读 Excel 导出的 UTF-8 CSV 用 `"utf-8"` | 开头 BOM 变成 `﻿`，首列字段名匹配不上 | 来源不明的 UTF-8 文件用 `encoding="utf-8-sig"` |

> **中文乱码的根因，一句话总结**：乱码 = **编码用的字符集 ≠ 解码用的字符集**。写文件时用 A 编码、读时用 B 解码，必乱。排查三步：①确认数据**写入时**是什么编码（看产出方）；②确认你**读取时**指定了同一编码；③中间链路（HTTP `Content-Type` 的 charset、数据库连接 charset、终端编码）有没有第三方悄悄转码。把编解码锁死在系统边界、全链路统一 UTF-8，是治本之道。

## ⑥ 练习题

### 基础理解

1. 解释为什么 `len("中文")` 是 2 而 `len("中文".encode("utf-8"))` 是 6。这两个数字分别在什么后端场景下是你真正关心的？

2. `bytes` 和 `bytearray` 有什么区别？为什么 `bytes` 可以做 dict 的 key 而 `bytearray` 不行？另外，`b"abc"[0]` 返回什么类型、值是多少？

### 找坑改错

3. 下面这段"Java 味"的日志聚合代码有两个问题（一个性能、一个正确性），指出并改写成 Pythonic 版本：

```python
def build_report(records):              # records: 可能上万条
    report = ""
    for r in records:
        report += r.name + ": " + str(r.value) + "\n"
    # 写到文件给运营看
    with open("report.txt", "w") as f:
        f.write(report)
    return report
```

4. 一个同事写了个"通用解码"工具函数，号称"任何字节都能解码、永不报错"，线上却频繁出现乱码方块且无人报警。指出根本问题：

```python
def safe_decode(data: bytes) -> str:
    try:
        return data.decode("utf-8")
    except UnicodeDecodeError:
        return data.decode("latin-1")   # 兜底：latin-1 对任何字节都不报错
```

### 综合实战

5. 你在写一个接收上传文件的后端接口。客户端 POST 上来一段 `request.body`（类型是 `bytes`），里面是一个 CSV，但**编码未知**（可能是 UTF-8、可能带 BOM、可能是 GBK 的中文老系统导出）。要求：把它稳健地解码成 `str`，按行解析，统计行数；解码失败时不要让整个请求 500，而是降级处理并记录告警。写出处理函数并说明每步的取舍。

---

### 参考答案

**1.** `len("中文")` 数的是**码点数**——两个汉字就是 2 个码点（`str` 是码点序列的抽象，PEP 393 内部用 2 字节/字符存，但这对 `len` 透明）。`"中文".encode("utf-8")` 把它编成 UTF-8 字节序列，每个汉字占 3 字节，共 6 字节，`len` 数的是字节数。**码点数**关心的场景：给用户看的"字数"、`VARCHAR(n)` 按字符计的数据库限长、UI 显示截断；**字节数**关心的场景：网络包/HTTP `Content-Length`、按字节限长的存储、磁盘占用、二进制协议字段。

**2.** `bytes` **不可变**、`bytearray` **可变**（可原地 `append`/索引赋值/切片赋值）。`bytes` 不可变所以**可哈希**（哈希值在生命周期内不变，能安全做 dict key / set 元素），`bytearray` 可变、哈希会随内容变所以**不可哈希**，不能做 key——这和 `list` 不能做 key、`tuple` 可以是同一个道理。`b"abc"[0]` 返回 **`int`**，值 **97**（字符 `'a'` 的字节值）；要拿长度 1 的 bytes 得用切片 `b"abc"[0:1]` → `b'a'`。

**3.** 两个问题：
- **性能**：`report += ...` 在循环里拼不可变 `str`，O(N²)，上万条会明显变慢。改用 `list` + `"".join`（或 `io.StringIO`）。
- **正确性**：`open(..., "w")` 没指定 `encoding`，依赖平台默认；含中文 `r.name` 时在 Windows（默认 GBK）上写出的文件，别人按 UTF-8 读就乱码。必须显式 `encoding="utf-8"`。

```python
def build_report(records):
    lines = [f"{r.name}: {r.value}" for r in records]   # O(N) 收集
    report = "\n".join(lines) + "\n"                    # 一次 join
    with open("report.txt", "w", encoding="utf-8") as f:  # 显式编码
        f.write(report)
    return report
```

**4.** 根本问题：**`latin-1` 能把任意字节序列"成功"解码成字符串、永不抛异常，所以它把错误吞掉了**——本该是"编码猜错"的告警信号（`UnicodeDecodeError`）被静默吞下，解出来的是错误文本（乱码），且**没有任何异常/日志**，问题潜伏到下游（写库、展示给用户）才暴露，且难以定位。它不是"安全解码"，是"把错误藏起来"。正确做法：要么真正探测编码（如 `charset-normalizer`/`chardet`），要么解码失败时用 `errors="replace"` 并**主动记录告警日志**（带上来源/前若干字节），让乱码可观测，而不是假装成功。

**5.** 参考实现与取舍：

```python
import logging
import csv
import io

logger = logging.getLogger(__name__)

def parse_uploaded_csv(body: bytes) -> list[list[str]]:
    # ① 优先尝试 utf-8-sig：覆盖"纯 UTF-8"和"带 BOM 的 UTF-8"两种最常见情况，
    #    且它对无 BOM 的 UTF-8 也正常工作，是最安全的首选。
    text: str | None = None
    for enc in ("utf-8-sig", "gbk"):     # ② 再退到 GBK，兼容中文老系统导出
        try:
            text = body.decode(enc)
            if enc != "utf-8-sig":
                logger.warning("CSV 非 UTF-8，按 %s 解码成功", enc)
            break
        except UnicodeDecodeError:
            continue

    if text is None:
        # ③ 所有已知编码都失败 → 降级：用 utf-8 + replace 强行解出（不 500），
        #    但必须告警，让乱码可观测、可追查来源。
        logger.error("CSV 解码失败，降级为 utf-8/replace，前 32 字节=%r", body[:32])
        text = body.decode("utf-8", errors="replace")

    # ④ 用 splitlines 而非 split("\n")：自动兼容 \n / \r\n / \r 各种换行
    #    用标准库 csv 解析而非手动 split(",")：正确处理引号包裹的逗号、转义。
    reader = csv.reader(io.StringIO(text))
    rows = list(reader)
    logger.info("CSV 解析完成，共 %d 行", len(rows))
    return rows
```

取舍说明：
- **为什么先 `utf-8-sig`**：UTF-8 是事实标准，且 `utf-8-sig` 顺手处理 Excel 导出的 BOM，一举两得；对无 BOM 的 UTF-8 也正确。
- **为什么探测顺序是"严格→宽松"**：UTF-8 解码对非法字节会抛异常（容错性低，正好用来判别"不是 UTF-8"），GBK 作为中文老系统的常见编码垫后。生产中更稳的做法是用 `charset-normalizer`（`requests` 的依赖）做统计探测，而非硬编码顺序。
- **为什么降级用 `errors="replace"` 而非 `latin-1`**：见第 4 题——`replace` 会留下可见的  并配合 `logger.error` 告警，乱码可观测；`latin-1` 会静默吞错。
- **为什么用 `csv` 模块 + `splitlines` 语义**：手动 `split("\n")`/`split(",")` 会被字段内的逗号、引号、`\r\n` 换行坑掉；标准库 `csv` 正确处理这些边界。
- **绝不 500**：解析阶段任何编码问题都降级+告警，把"脏数据"挡在边界处理掉，符合 ②2.3 "网络是字节，边界处显式解码"的原则。
