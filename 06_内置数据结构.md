# 06 · 内置数据结构 ⭐

> 在 Java 里你要先决定用 `ArrayList` 还是 `LinkedList`、`HashMap` 还是 `TreeMap`，再 `import java.util.*`。Python 把这一切收敛成四个**语法内置**的容器——`list` / `dict` / `set` / `tuple`，连 `import` 都不用。它们覆盖了你 90% 的需求，而且 `dict` 还是整个语言的基石（对象属性、命名空间、`kwargs` 全是 dict）。本章对照 Java 集合框架，把这四个容器的底层实现（动态数组扩容、compact dict、开放寻址）和复杂度挖到底，让你既会用、又知道每个操作背后的代价。

---

## ① Java 对照表

| Python 类型 | 最接近的 Java 对应物 | 关键差异（一句话） |
|------------|---------------------|------------------|
| `list` | `ArrayList<Object>` | 动态数组，但**元素无类型**、存的是指针；扩容约 1.125x（ArrayList 1.5x） |
| `dict` | `LinkedHashMap`（保序）+ `HashMap` 性能 | 3.7 起**保证插入顺序**；开放寻址而非拉链；是语言核心数据结构 |
| `set` | `HashSet` | 同样基于哈希表，但**不保证顺序**（不像 dict） |
| `frozenset` | `Set.of(...)` / 不可变 Set | 不可变、可哈希，能当 dict key / set 元素 |
| `tuple` | 不可变数组 / `record` / `List.of(...)` | 定长不可变；常用于多返回值、可当 dict key |
| `dict` 当对象 | `Map<String,Object>` 或 POJO 字段 | Python 对象的属性本身就存在一个 `__dict__` 里 |
| 推导式 `[x for x in ...]` | Stream `.map().collect(toList())` | 语法内置、更短，无需流水线 |
| `in` 运算符 | `.contains()` | 对 list 是 O(n)，对 set/dict 是 O(1)——**选错容器是经典翻车** |

> 一个根本性差异：Java 容器是**泛型**的（`List<String>`，编译期类型安全）；Python 容器是**异构**的（一个 list 里可以同时装 int、str、对象），类型检查靠运行时和可选的类型注解（`list[str]`，第 19 章）。Java 用类型系统约束容器，Python 用约定和注解。

## ② 设计理念（为什么 Python 这么设计）

### 2.1 四个容器覆盖一切：少即是多

Java 集合框架庞大：`List`/`Set`/`Map`/`Queue` 四大接口，每个又有 `ArrayList`/`LinkedList`/`HashSet`/`TreeSet`/`HashMap`/`TreeMap`/`ArrayDeque`…… 十几个实现类，选型本身就是一门学问。

Python 的取舍相反：**语言内置极少数、极通用的容器，把选型成本降到接近零**。

- **顺序集合** → `list`（几乎不需要 `LinkedList`，因为尾部操作已经够快，中间插入少见；真要双端队列才用 `collections.deque`）。
- **键值映射** → `dict`（不需要 `TreeMap`，因为有序遍历需求远比"插入顺序"少见；真要排序临时 `sorted()`）。
- **去重集合** → `set`。
- **不可变记录 / 多值** → `tuple`。

这背后是 Python 的实用主义哲学：**"应该有一种——最好只有一种——显而易见的做法"**（The Zen of Python）。与其给你 10 个 List 实现让你纠结，不如给一个调校得足够好的 `list`，覆盖绝大多数场景。需要特化结构时，才去 [07 collections模块](07_collections模块.md) 拿 `deque`/`defaultdict`/`Counter`，或第三方库。

### 2.2 为什么 dict 是语言的核心

在 Java 里 `HashMap` 只是一个工具类。在 Python 里 **dict 是语言运行时的基础设施**——几乎所有东西底层都是 dict：

```python
class User:
    def __init__(self, name):
        self.name = name

u = User("alice")
print(u.__dict__)          # 输出: {'name': 'alice'}  —— 对象属性就是一个 dict

print(type(globals()))     # 输出: <class 'dict'>     —— 模块的全局命名空间是 dict

def f(**kwargs):
    print(kwargs)
f(a=1, b=2)                # 输出: {'a': 1, 'b': 2}    —— kwargs 就是 dict
```

- **对象属性**：`u.name` 本质是 `u.__dict__['name']`（除非用了 `__slots__`，见 [04 运行时与内存模型](04_运行时与内存模型.md)）。
- **命名空间**：模块的全局变量、函数的局部变量（部分）、类的属性表，底层都是 dict。
- **关键字参数**：`**kwargs` 收集成 dict。
- **import 系统**：`sys.modules` 是 dict。

正因为 dict 无处不在，CPython 对它做了**极致优化**（2.3 节和第 ③ 段细讲），它的性能直接决定整个解释器的性能。理解 dict，就理解了 Python 对象模型的一半。

### 2.3 可变 vs 不可变：为什么需要 tuple 和 frozenset

Java 里你用 `final` 修饰引用、用 `Collections.unmodifiableList` 或 `List.of` 得到不可变视图。Python 把"可变 / 不可变"做成了**类型层面的根本区分**：

- `list` ↔ `tuple`：可变列表 vs 不可变序列。
- `set` ↔ `frozenset`：可变集合 vs 不可变集合。
- `dict`：可变（没有内置的不可变 dict，3.3+ 有 `types.MappingProxyType` 做只读视图）。

为什么要有不可变版本？**因为只有不可变（可哈希）对象才能当 dict 的 key 或 set 的元素。** 哈希表要求 key 的哈希值在生命周期内不变——如果允许可变对象当 key，改了它的内容后哈希值变了，就再也找不到它了。所以：

```python
point = (3, 4)
d = {point: "origin-ish"}     # OK，tuple 可哈希
# d = {[3, 4]: "x"}           # TypeError: unhashable type: 'list'

cache_key = frozenset({"read", "write"})
perms = {cache_key: "rw"}     # OK，frozenset 可哈希，可当 key
```

这就是 tuple 和 frozenset 存在的核心理由：**为哈希结构提供可哈希的复合 key**。顺带它们还表达了"这组数据不该被改"的语义意图。

## ③ 实现细节（底层怎么实现的）★本章重点★

这一段是和普通教程拉开差距的地方。我们逐个拆开四个容器的 C 层结构。

### 3.1 list = 动态数组（不是链表！）

尽管叫 "list"，CPython 的 `list` **不是链表，而是动态数组**——和 Java 的 `ArrayList` 同一思路。C 层结构（简化）：

```c
typedef struct {
    PyObject_VAR_HEAD          // 含 refcount、类型指针、ob_size（当前元素个数）
    PyObject **ob_item;        // 指向一块指针数组的指针
    Py_ssize_t allocated;      // 已分配的容量（>= ob_size）
} PyListObject;
```

关键点：**`ob_item` 是一个指针数组**。`list` 里存的不是值本身，而是指向各个 `PyObject` 的指针。

```
my_list = [10, "hi", obj]

         ob_item ──► [ ptr0 ][ ptr1 ][ ptr2 ][ 空 ][ 空 ]   ← allocated=5, ob_size=3
                       │        │       │
                       ▼        ▼       ▼
                   PyLong(10) PyStr("hi") <obj>            ← 散落在堆上
```

对照 Java：`ArrayList<Integer>` 内部 `Object[] elementData` 也是存引用（指针），这点一致。但 Java 的 `int[]` 是**连续的裸值**，而 Python 的 list **永远是指针数组**，没有"原始类型数组"的概念。这就是为什么数值计算必须用 numpy（连续 C 数组），见 [05 CPython 内部](05_CPython内部.md) 2.3 节。

**扩容策略（over-allocation）**：当 `append` 时 `ob_size == allocated`，list 会重新分配一块更大的内存。CPython 的增长公式（`listobject.c` 的 `list_resize`）：

```c
new_allocated = ((size_t)newsize + (newsize >> 3) + 6) & ~(size_t)3;
```

即 **新容量 ≈ newsize × 1.125 + 6**，再向上取到 4 的倍数。增长序列大致是：

```
0, 4, 8, 16, 25, 35, 46, 58, 72, 88, ...
```

对照 **Java ArrayList 的扩容是 1.5x**（`newCapacity = oldCapacity + (oldCapacity >> 1)`）。Python 的 ~1.125x 更保守——增长更平缓、内存更省，代价是扩容次数略多。但因为均摊（amortized）下来 `append` 仍是 O(1)，且 Python 对象本身就大（每个 int 28 字节），数组指针的开销占比小，所以选了更省内存的低增长因子。

可以亲眼看到扩容：

```python
import sys

lst = []
prev = -1
for i in range(20):
    size = sys.getsizeof(lst)
    if size != prev:
        print(f"len={len(lst):2d}  bytes={size}")   # bytes 跳变处就是扩容点
        prev = size
    lst.append(i)
# 输出（64 位）类似:
# len= 0  bytes=56
# len= 1  bytes=88     ← 一次性分配了 4 个槽
# len= 5  bytes=120    ← 扩到 8
# len= 9  bytes=184    ← 扩到 16
# ...
```

> `sys.getsizeof([])` 是 56 字节（对象头 + 元数据），每个槽位 8 字节（一个指针）。注意这只算 list 本身的指针数组，**不含元素对象**的大小。

**预分配优化**：如果你知道最终大小，`[None] * n` 一次性分配，比循环 append 省下多次扩容拷贝。或者用 `list(range(n))`、推导式（CPython 对推导式有大小提示优化）。

### 3.2 dict = 开放寻址哈希表 + compact dict（3.6+）

这是本章最精彩的部分。Python 的 dict 和 Java 的 HashMap 都是哈希表，但**解决冲突的方式和内存布局完全不同**。

**Java HashMap = 拉链法（separate chaining）**：一个桶数组，每个桶挂一条链表（冲突多时转红黑树）。

**Python dict = 开放寻址（open addressing）+ 紧凑布局（compact dict，3.6 起）**。

3.6 之前 Python dict 就是普通的开放寻址哈希表，每个槽存 `(hash, key, value)`，稀疏数组浪费内存且不保序。**3.6 起引入 compact dict**（Raymond Hettinger 设计），把结构拆成两部分：

```
┌─────────────────────────────────────────────────┐
│ indices（稀疏的索引数组，存的是"第几个 entry"）      │
│   [ -1 ][ 0 ][ -1 ][ -1 ][ 1 ][ -1 ][ 2 ][ -1 ]  │  ← 哈希定位到这里
└─────────────────────────────────────────────────┘
                 │ 值是下面 entries 的下标
                 ▼
┌─────────────────────────────────────────────────┐
│ entries（紧凑、按插入顺序排列的真实数据）            │
│   [0] (hash_a, key_a, val_a)                     │
│   [1] (hash_b, key_b, val_b)   ← 紧凑，无空洞      │
│   [2] (hash_c, key_c, val_c)                     │
└─────────────────────────────────────────────────┘
```

工作原理：

1. **查找**：对 key 算哈希 → 映射到 `indices` 数组的某个位置 → 取出里面的下标 `i`（如果是 -1 表示空槽）→ 去 `entries[i]` 拿真实的 `(hash, key, value)`，比对 key 确认。
2. **冲突**：`indices` 槽位被占了，用**伪随机探测序列**（probing）找下一个空槽（不是简单 +1 的线性探测，而是 `perturb` 扰动算法，减少聚集）。
3. **插入**：新数据**永远追加到 `entries` 数组末尾**，并在 `indices` 里记下它的下标。

**这就是 dict 保持插入顺序的原理**：`entries` 数组本身就是按插入先后紧凑排列的，遍历 dict 就是顺序遍历 `entries`。Java 要单独的 `LinkedHashMap`（额外维护一条双向链表）才能保序，而 Python 的 compact dict **顺带就保了序，还更省内存**（稀疏的只是小小的 `indices` 索引数组，真实数据紧凑存放）。

**3.7 起，"保持插入顺序"从实现细节升级为语言规范**（之前 3.6 是"恰好如此但不保证"）。所以现在你可以放心依赖 dict 的顺序。

内存对比：compact dict 比旧版省约 20~25% 内存。`indices` 数组的元素宽度还会自适应——dict 小（<256 项）时每个索引只用 1 字节（int8），大了才用 2/4/8 字节。

对照 Java：HashMap 默认负载因子 0.75，超了扩容（容量翻倍 + rehash）。Python dict 在装填到 **2/3（约 0.66）** 时扩容，新容量约为已用项数的 3 倍（取 2 的幂）。冲突处理上 Java 用链表/红黑树（最坏 O(log n)），Python 用开放寻址探测（依赖好的哈希函数和扰动避免聚集）。

### 3.3 set = 没有 value 的 dict

`set` 和 `frozenset` 的底层就是**哈希表，只存 key 不存 value**——可以粗略理解为"value 永远是占位符的 dict"。但实现上 set 是独立的 `setobject.c`，**没有采用 compact 布局**，所以：

- set 操作（`in`、`add`、`&`、`|`）是 O(1) / O(n)，和 dict 同级。
- **set 不保证顺序**（不像 3.7+ 的 dict）——因为它没用 compact 的 entries 数组，遍历顺序取决于哈希分布。这是个常被忽略的差异。

```python
print(list({3, 1, 2, 100, 5}))   # 顺序不可预测，别依赖它
d = {3: 0, 1: 0, 2: 0}
print(list(d))                    # [3, 1, 2]  dict 保插入顺序
```

`frozenset` 就是不可变的 set，构造后不能增删，因此可哈希、能当 key/元素。

### 3.4 tuple = 定长数组 + 小元组空闲列表复用

`tuple` 的 C 结构是**定长数组**，元素指针直接内联在对象里（不像 list 还要一层 `ob_item` 间接指向外部数组）：

```c
typedef struct {
    PyObject_VAR_HEAD
    PyObject *ob_item[1];     // 变长内联数组，元素指针就紧跟在对象后面
} PyTupleObject;
```

因为定长不可变，tuple 比 list 更省内存、构造更快、访问略快（少一层间接）：

```python
import sys
print(sys.getsizeof((1, 2, 3)))   # 64  元素指针内联，无额外间接
print(sys.getsizeof([1, 2, 3]))   # 80（3.12 精确分配）list 多了 allocated 字段和外部数组开销
```

**小元组空闲列表（free list）复用**：创建/销毁元组非常频繁（每个多返回值、每次函数调用打包参数都在造 tuple）。为避免反复 malloc/free，CPython 为长度 0~20 的小元组维护**空闲列表**：销毁一个小元组时不真正释放内存，而是缓存起来，下次创建同长度元组直接复用。这类似 [04 运行时与内存模型](04_运行时与内存模型.md) 提到的小整数对象池思路，是高频小对象的标准优化。

```python
a = (1, 2)
b = (3, 4)
# CPython 内部：销毁 a 后造 b 很可能复用了 a 的内存块（同长度）
```

> 此外空元组 `()` 是**单例**——整个解释器里所有 `()` 都是同一个对象，因为不可变空元组没有区分的必要。

### 3.5 各操作时间复杂度表（务必记住）

| 操作 | list | dict | set | tuple | 说明 |
|------|------|------|-----|-------|------|
| 索引 `x[i]` | O(1) | — | — | O(1) | 数组随机访问 |
| 按 key 取 `d[k]` | — | O(1) 均摊 | — | — | 哈希查找 |
| `x[i] = v`（改） | O(1) | O(1) 均摊 | — | ✗ 不可变 | |
| `append` / `add` | **O(1) 均摊** | O(1) 均摊 | O(1) 均摊 | ✗ | 偶尔扩容 O(n) |
| `insert(i, v)` | **O(n)** | — | — | ✗ | 要搬移后面所有元素 |
| `x in c`（成员检测） | **O(n)** ⚠️ | **O(1)** | **O(1)** | O(n) | **选错容器的重灾区** |
| `pop()`（尾部） | O(1) | — | — | ✗ | |
| `pop(0)`（头部） | **O(n)** ⚠️ | — | — | ✗ | 要整体前移；需要双端用 deque |
| `pop(k)`（按 key） | — | O(1) 均摊 | — | — | |
| `remove(v)` | O(n) | — | O(1) | ✗ | list 要先线性查找 |
| `del x[i]` | O(n) | O(1) | — | ✗ | list 删中间要搬移 |
| `len(c)` | O(1) | O(1) | O(1) | O(1) | 都存了 size 字段 |
| 集合运算 `&` `\|` `-` | — | — | O(len) | — | set 专属 |
| 遍历 | O(n) | O(n) | O(n) | O(n) | |

**记两条铁律：**
1. **成员检测 `in` 用 set/dict（O(1)），不要用 list（O(n)）。** 这是 Java 工程师最常踩的坑——`ArrayList.contains()` 也是 O(n)，但 Python 里 list 太顺手，容易随手用。
2. **头部增删用 [07 collections模块](07_collections模块.md) 的 `deque`（O(1)），不要用 list 的 `pop(0)`/`insert(0)`（O(n)）。**

## ④ 使用方法与最佳实践

### 4.1 list：索引、负索引、切片

```python
nums = [10, 20, 30, 40, 50]

print(nums[0])       # 10    正索引从 0
print(nums[-1])      # 50    负索引从尾部，-1 是最后一个（Java 要写 nums.get(nums.size()-1)）
print(nums[-2])      # 40

# 切片 nums[start:stop:step]，左闭右开，返回新 list（浅拷贝）
print(nums[1:3])     # [20, 30]
print(nums[:2])      # [10, 20]      省略 start
print(nums[2:])      # [30, 40, 50]  省略 stop
print(nums[::-1])    # [50, 40, 30, 20, 10]  步长 -1 = 反转
print(nums[::2])     # [10, 30, 50]  隔一个取

# 切片赋值（Java 没有的能力）
nums[1:3] = [99]     # 替换区间
print(nums)          # [10, 99, 40, 50]
```

> 切片**永不越界**：`nums[1:999]` 不抛异常，自动截断到末尾。这和 Java 的 `subList` 越界抛 `IndexOutOfBoundsException` 不同。

推导式（对照 Java Stream，但更简洁）：

```python
# Java: list.stream().filter(n -> n > 100).map(n -> n*2).collect(toList())
orders = [50, 150, 200, 80]
big = [n * 2 for n in orders if n > 100]
print(big)           # [300, 400]
```

### 4.2 dict：get / setdefault / 推导式 / 合并

```python
user = {"name": "alice", "age": 30}

# 安全取值：key 不存在返回 None 或默认值，不抛异常（对照 Java map.getOrDefault）
print(user.get("email"))            # None
print(user.get("email", "n/a"))     # n/a
# print(user["email"])              # KeyError! 直接下标取不存在的 key 会抛

# setdefault：取值，不存在则先插入默认值再返回（常用于分组）
groups = {}
for name, dept in [("a", "eng"), ("b", "eng"), ("c", "sales")]:
    groups.setdefault(dept, []).append(name)
print(groups)        # {'eng': ['a', 'b'], 'sales': ['c']}
# 注：这个场景更地道的是 collections.defaultdict，见第 07 章

# 遍历
for k, v in user.items():           # 同时拿 key 和 value
    print(k, v)                     # name alice / age 30
print(list(user.keys()))            # ['name', 'age']
print(list(user.values()))          # ['alice', 30]

# 字典推导式
squares = {n: n*n for n in range(4)}
print(squares)       # {0: 0, 1: 1, 2: 4, 3: 9}

# 合并：| 运算符（3.9+），返回新 dict；|= 就地更新
defaults = {"timeout": 30, "retries": 3}
override = {"retries": 5}
config = defaults | override        # 右边覆盖左边
print(config)        # {'timeout': 30, 'retries': 5}
# 3.9 之前要写 {**defaults, **override} 或 dict(defaults, **override)
```

### 4.3 set：去重与集合运算

```python
# 去重（保留无所谓顺序时最快的方式）
ids = [3, 1, 3, 2, 1]
print(set(ids))                     # {1, 2, 3}

# 需要去重又保序 → 用 dict.fromkeys（dict 保序，set 不保序）
print(list(dict.fromkeys(ids)))     # [3, 1, 2]

# 集合运算（对照数学，比 Java 的 retainAll/addAll 直观得多）
a = {1, 2, 3, 4}
b = {3, 4, 5, 6}
print(a & b)         # {3, 4}        交集 intersection
print(a | b)         # {1,2,3,4,5,6} 并集 union
print(a - b)         # {1, 2}        差集 difference
print(a ^ b)         # {1,2,5,6}     对称差 symmetric_difference

# 典型场景：求两批用户 ID 的差异
old_users = {101, 102, 103}
new_users = {102, 103, 104}
print(new_users - old_users)        # {104}  新增的
print(old_users - new_users)        # {101}  流失的

# ⚠️ 空 set 必须用 set()，不能用 {}（那是空 dict！）
empty = set()
print(type({}))      # <class 'dict'>
print(type(set()))   # <class 'set'>
```

### 4.4 tuple：解包与多返回值

tuple 在 Python 里最常见的用途是**多返回值**和**解包**，几乎天天用：

```python
# 多返回值（函数返回 tuple，调用方解包）—— 对照 Java 只能返回一个对象或数组
def divmod_demo(a, b):
    return a // b, a % b            # 实际返回 (a//b, a%b)

q, r = divmod_demo(17, 5)          # 解包
print(q, r)          # 3 2

# 交换变量：一行搞定，无需临时变量（底层是打包成 tuple 再解包）
x, y = 1, 2
x, y = y, x
print(x, y)          # 2 1

# 星号解包：抓取剩余
first, *rest = [1, 2, 3, 4]
print(first, rest)   # 1 [2, 3, 4]
head, *mid, tail = [1, 2, 3, 4, 5]
print(head, mid, tail)  # 1 [2, 3, 4] 5

# 遍历 enumerate / items 时的解包
for i, val in enumerate(["a", "b"]):
    print(i, val)    # 0 a / 1 b

# ⚠️ 单元素 tuple 必须带逗号！
t1 = (1,)            # tuple
t2 = (1)             # 这只是括号里的整数 1，不是 tuple！
print(type(t1), type(t2))   # <class 'tuple'> <class 'int'>
```

### 4.5 何时用哪个容器（决策表）

| 需求 | 选择 | 理由 |
|------|------|------|
| 有序、可增删、允许重复 | `list` | 默认序列容器 |
| 键值映射 | `dict` | O(1) 查找，保插入顺序 |
| 去重、频繁成员检测 | `set` | O(1) 的 `in` |
| 固定的一组值、要当 dict key | `tuple` | 不可变、可哈希、省内存 |
| 不可变的去重集合当 key | `frozenset` | 唯一能当 key 的集合 |
| 频繁头部增删 / 双端队列 | `deque`（第 07 章） | list 的 `pop(0)` 是 O(n) |
| 计数 / 分组 / 默认值 | `Counter`/`defaultdict`（第 07 章） | 比手写 setdefault 地道 |

### 4.6 `in` 对 list vs set 的复杂度差异（亲手验证）

```python
import time

n = 100_000
data_list = list(range(n))
data_set = set(range(n))
target = n - 1                       # 最坏情况：在末尾

t = time.perf_counter()
for _ in range(1000):
    _ = target in data_list          # O(n) 每次扫到末尾
print(f"list in: {time.perf_counter()-t:.4f}s")   # 约 1~2 秒

t = time.perf_counter()
for _ in range(1000):
    _ = target in data_set           # O(1) 哈希一次命中
print(f"set in:  {time.perf_counter()-t:.6f}s")   # 约 0.0001 秒
# 差距上万倍。批量去重 / 查白名单 / 求交集，永远先转 set。
```

> 经验法则：**只要你会对一个容器反复做 `in` 检测，就该用 set 或 dict，而不是 list。** 一次性构建 set 是 O(n)，但之后每次查询 O(1)，总成本远低于反复 O(n) 扫 list。

## ⑤ ⚠️ 易错点（Java 思维翻车）

| Java 习惯 | 翻车现场 | 正确做法 / 认知 |
|----------|---------|----------------|
| 习惯 `list.contains()` 随手查 | 用 `x in big_list` 做成员检测，循环里 O(n²) 性能爆炸 | 反复查就转 `set`/`dict`，`in` 变 O(1) |
| 用可变对象当 map key（Java 也不该但能编译） | `d = {[1,2]: "x"}` 直接 `TypeError: unhashable type: 'list'` | key 用不可变类型：`tuple`/`frozenset`/`str`/数字 |
| 边遍历边改容器（Java 抛 ConcurrentModificationException） | `for k in d: del d[k]` 抛 `RuntimeError: dictionary changed size`；改 list 则**静默跳过元素** | 遍历副本 `for k in list(d):` 或推导式重建 |
| 单元素集合写 `(1)` | 以为造了 tuple，其实是 int，后续解包/索引报错 | 单元素 tuple 必须 `(1,)` 带逗号 |
| 空集合写 `{}` 当 set | `{}` 是空 dict 不是空 set，`.add()` 抛 AttributeError | 空 set 用 `set()`，空 dict 才用 `{}` |
| 以为 set 也保插入顺序（因为 dict 保了） | 依赖 `set` 遍历顺序，结果不可预测、跨运行变化 | 只有 `dict` 保序；set 要有序请 `sorted(s)` |
| `b = a` 以为是拷贝 | `b = a` 只是同一个 list 的别名，改 b 等于改 a | 浅拷贝 `a.copy()`/`a[:]`/`list(a)`；深拷贝 `copy.deepcopy` |
| 浅拷贝以为是深拷贝 | `copy.copy(nested)` 后改内层 list 仍影响原对象 | 嵌套结构要 `copy.deepcopy` |
| 把 `dict.keys()` 当成 Java 的快照 List | `keys()` 是**动态视图**，dict 变了它跟着变；不能下标索引 | 要静态快照 / 索引就 `list(d.keys())` |
| 默认值用可变对象 `def f(x=[])` | 默认 list 在所有调用间共享，越积越多（经典大坑） | 默认值用 `None`，函数内 `if x is None: x = []` |

几个翻车现场的代码：

```python
# 翻车 1：边遍历边删 list —— 静默漏删！
nums = [1, 2, 3, 4, 5, 6]
for n in nums:
    if n % 2 == 0:
        nums.remove(n)       # 删了元素后索引错位
print(nums)                  # [1, 3, 5] 看着对？换 [2,4] 试试就漏删
# 正确：推导式重建
nums = [n for n in [1,2,3,4,5,6] if n % 2 != 0]

# 翻车 2：可变默认参数
def add_item(item, bucket=[]):   # bucket 只在定义时创建一次！
    bucket.append(item)
    return bucket
print(add_item("a"))         # ['a']
print(add_item("b"))         # ['a', 'b']  ← 竟然累积了！
# 正确：
def add_item_ok(item, bucket=None):
    if bucket is None:
        bucket = []
    bucket.append(item)
    return bucket

# 翻车 3：别名 vs 拷贝
a = [1, 2, 3]
b = a                        # 别名，不是拷贝
b.append(4)
print(a)                     # [1, 2, 3, 4] ← a 也变了
c = a.copy()                 # 浅拷贝
c.append(5)
print(a)                     # [1, 2, 3, 4] ← a 不受影响
```

### 综合实战

5. 你在做一个 API 网关的请求去重 / 限流模块。给定一批请求记录 `requests`，每条是 `{"user": str, "path": str, "ts": int}`。要求：
   - (a) 统计每个 `(user, path)` 组合出现了多少次（用于发现重复请求）。
   - (b) 找出同时访问过 `/admin` 和 `/billing` 两个路径的用户集合。
   - (c) 给定一个允许的路径白名单，O(1) 判断每条请求的 path 是否合法。

   用本章的容器实现，并说明每处为什么选这个容器。

---

### 参考答案

**1.** Java 的 `HashMap` 用拉链法，桶按哈希值散布，遍历顺序由哈希决定、与插入无关；要保序必须额外维护一条双向链表（`LinkedHashMap`），有额外内存和指针开销。Python 的 compact dict 把存储拆成两块：稀疏的 `indices` 索引数组负责哈希定位（存的是"第几个 entry"），紧凑的 `entries` 数组**按插入先后顺序追加**真实的 `(hash, key, value)`。遍历 dict 就是顺序扫 `entries` 数组，所以保序是结构自带的副产品，不需要额外链表——反而比旧布局更省内存。

**2.**
- `a = {}` → **dict**（空大括号是空字典，不是空集合）。
- `b = {1}` → **set**（大括号里只有值没有冒号 → 集合）。
- `c = {1: 2}` → **dict**（有冒号 → 键值对）。
- `d = (1)` → **int**（括号被当作普通分组括号；要 tuple 得写 `(1,)`）。

**3.** 两个问题：
- 性能：`u["id"] in blacklist` 中 `blacklist` 是 list，`in` 是 O(n)，外层又遍历所有 users，整体 O(n×m)。把 blacklist 转成 set，`in` 变 O(1)。
- 异常：`result[dept] = result[dept] + 1` 在 dept 第一次出现时 `result[dept]` 不存在，抛 `KeyError`。用 `result.get(dept, 0) + 1`（或第 07 章的 `Counter`/`defaultdict(int)`）。

```python
def process(users, blacklist):
    blacklist = set(blacklist)            # O(1) 成员检测
    result = {}
    for u in users:
        if u["id"] in blacklist:
            continue
        dept = u["dept"]
        result[dept] = result.get(dept, 0) + 1   # 不存在时默认 0
    return result
# 更地道：from collections import Counter / defaultdict（第 07 章）
```

**4.** 错误：`set` 不保证顺序，`list(set(items))` 出来的顺序不可预测，不满足"保持原顺序"。两种正确写法：
```python
# 写法一：dict.fromkeys 利用 dict 保序 + key 去重（最简洁，推荐）
def dedup(items):
    return list(dict.fromkeys(items))

# 写法二：手动用 set 记录已见，list 保序
def dedup(items):
    seen = set()
    out = []
    for x in items:
        if x not in seen:        # O(1) 检测
            seen.add(x)
            out.append(x)
    return out
```

**5.** 参考实现：
```python
from collections import Counter   # 第 07 章，这里用于 (a) 最地道

def gateway_stats(requests, allowed_paths):
    allowed = set(allowed_paths)              # (c) set：O(1) 判断 path 是否合法

    # (a) 统计 (user, path) 组合次数 —— 用 tuple 当 key（不可变、可哈希）
    combo_count = Counter((r["user"], r["path"]) for r in requests)

    # (b) 每个用户访问过的路径集合，再求同时含两者的用户
    user_paths = {}                            # dict: user -> set(paths)
    for r in requests:
        user_paths.setdefault(r["user"], set()).add(r["path"])
    both = {u for u, paths in user_paths.items()
            if {"/admin", "/billing"} <= paths}   # 子集判断

    # (c) 校验每条请求
    invalid = [r for r in requests if r["path"] not in allowed]  # O(1) 每次

    return combo_count, both, invalid
```
选型说明：
- (a) key 用 **tuple** `(user, path)`，因为要把两个字段组合成一个不可变、可哈希的复合 key；用 `Counter`（或普通 dict + `get`）计数。
- (b) 每个用户的路径集合用 **set**（天然去重 + 支持 `<=` 子集判断）；外层 `user -> set` 用 **dict**。`{"/admin","/billing"} <= paths` 判断子集，比两次 `in` 更地道。
- (c) 白名单用 **set**，把每条请求的 O(n) 线性查找降为 O(1)，请求量大时是关键优化。
