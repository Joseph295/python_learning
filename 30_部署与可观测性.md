# 30 · 部署与可观测性 ⭐

> 在 Java 世界，部署是一条你闭着眼睛都能走的路：`mvn package` 出一个 fat jar，`java -jar app.jar` 启动，Tomcat 线程池吃满多核，Actuator 暴露 `/actuator/health` 和 `/actuator/prometheus`，Micrometer 把指标喂给 Prometheus，Sleuth/Micrometer Tracing 自动织入 traceId。这套"打包 → 起进程 → 自带可观测性"的肌肉记忆，到 Python 几乎每一环都要重写一遍——不是因为 Python 缺东西，而是因为**底下的并发模型不同（GIL，见 [20 并发与GIL](20_并发与GIL.md)）、运行载体不同（容器而非 JVM）、可观测性是"自己拼装"而非"框架自带"**。本章把这条链路从头讲透：为什么 Python 生产用**多进程 worker** 而不是多线程、gunicorn 的 **pre-fork 模型**到底 fork 了什么、怎么写一个又小又安全的生产 Dockerfile、以及怎么用 `prometheus_client` + OpenTelemetry 把 logs/metrics/traces 三支柱补齐。看懂这章，你就能把 [24 章](24_HTTP与Web框架.md) 写的 FastAPI 应用真正、安全地送上生产。

## ① Java 对照表

| 概念 / 任务 | Java 世界 | Python 世界 | 关键差异 |
|------------|-----------|-------------|---------|
| 应用服务器 / 容器 | Tomcat / Jetty / Undertow（线程池模型） | **gunicorn**（WSGI）/ **uvicorn**（ASGI） | Java 一进程多线程吃满多核；Python 靠**多进程 worker** 绕过 GIL |
| 内嵌服务器 | Spring Boot 内嵌 Tomcat | uvicorn / gunicorn（独立进程，框架不含 server） | Python 框架与 server 分离（[23 网络编程](23_网络编程.md)） |

| 调优旋钮 | `-Xmx`/`-Xss`/GC/线程池大小 | **worker 数** / worker 类型 / 协程并发 | JVM 调内存与线程；Python 调进程数 |

| 日志 | Logback/Log4j2 → 文件 + ELK | logging → **stdout**（JSON），容器收集 | 容器里日志写 stdout 而非文件（[29 工程化最佳实践](29_工程化最佳实践.md)） |
| 配置 | `application.yml` / `@Value` / env | **环境变量** / pydantic-settings | 12-factor：配置从环境注入（[29 章](29_工程化最佳实践.md)） |

## ② 设计理念（为什么 Python 这么设计）

### 2.1 为什么生产用「多 worker 进程」而非「多线程」

这是整章的地基，直接承接 [20 并发与GIL](20_并发与GIL.md)。

在 Java 里，一个 Tomcat 进程开一个几十~几百大小的线程池，每个请求占一个线程，**多个线程在多个核上真并行**——因为 JVM 没有全局解释器锁。你的 8 核机器，一个 JVM 进程就能榨干。

Python 不行。CPython 有 **GIL**（[05 CPython内部](05_CPython内部.md)、[20 章](20_并发与GIL.md)）：同一时刻只有一个线程在执行 Python 字节码。这意味着——

> **一个 Python 进程，无论你开多少线程，CPU 密集部分（路由匹配、序列化、pydantic 校验、业务计算）永远只能用满一个核。**

要吃满 8 核，Java 开 1 进程 × 8 线程；**Python 必须开 8 个进程**，每个进程有自己独立的解释器、自己独立的 GIL，8 把 GIL 在 8 个核上真并行。这就是 gunicorn `-w 8`、uvicorn `--workers 8` 的全部动机：

```
Java（Tomcat）：                    Python（gunicorn）：
┌─────────────────────┐            ┌────────┐┌────────┐┌────────┐┌────────┐
│  1 JVM 进程          │            │worker 1││worker 2││worker 3││worker 4│
│  ┌──┐┌──┐┌──┐┌──┐   │            │ 1 GIL  ││ 1 GIL  ││ 1 GIL  ││ 1 GIL  │
│  │T1││T2││T3││T4│... │  真并行    │ 1 解释器││ 1 解释器││ 1 解释器││ 1 解释器│
│  └──┘└──┘└──┘└──┘   │  (无 GIL)  └────────┘└────────┘└────────┘└────────┘
│  共享堆              │            各自独立内存，靠 OS 调度到不同核，真并行
└─────────────────────┘
```

代价和 [20 章 3.3](20_并发与GIL.md) 讲的多进程代价一致：进程比线程重（每个 worker 是一份完整的解释器 + 你的应用 + 加载的库，几十到几百 MB），进程间不共享内存（指标聚合因此要特殊处理，见 3.4）。但对无状态的 Web 服务，这恰恰是优点——**worker 之间天然隔离，一个崩了不影响别的，master 把它重启即可**。

### 2.2 ASGI/WSGI 服务器与应用分离（呼应 23 章）

[23 网络编程](23_网络编程.md) 已经讲透 WSGI/ASGI 是 Python 版的 Servlet 规范。这里强调它对**部署**的意义：因为有这层契约，**"框架"和"服务器"是两个独立的东西，分开部署**。

- 你写的 `app`（FastAPI/Flask/Django 实例）只是一个符合 WSGI/ASGI 规范的可调用对象，它**不监听端口、不管并发、不解析 HTTP**。
- gunicorn/uvicorn 才是"服务器"：监听 socket、fork worker、解析 HTTP 字节、按规范调用你的 `app`、管理生命周期。

这和 Spring Boot "内嵌 Tomcat、一个 jar 全包了"的体感不同——在 Python 里服务器是一条**独立的命令**（`gunicorn myapp:app`），你能自由替换（gunicorn ↔ uvicorn ↔ hypercorn）、独立调参（worker 数、超时、keep-alive），而业务代码一行不动。这种解耦正是 [01 设计哲学与心智模型](01_设计哲学与心智模型.md) 里"组合优于内聚大一统"的体现。

### 2.3 为什么用容器镜像而不是「fat jar」

Java 的 fat jar 能成立，是因为它假设**宿主机已经装好了一份兼容的 JVM**——jar 里只有字节码，跨平台靠 JVM 抹平。Python 没有这种奢侈：

- Python 代码要跑，得有**对应版本的解释器**（3.11 和 3.12 字节码不兼容），还常常依赖**带 C 扩展的二进制 wheel**（numpy、pydantic-core、asyncpg），这些 wheel 和操作系统、glibc 版本、CPU 架构强绑定。
- "在我机器上能跑"在 Python 里格外脆弱：本机装了系统级 `libpq`、Python 3.12，换台机器就缺。

所以 Python 工业界的标准答案是**容器**：把解释器、所有依赖（含编译好的 C 扩展）、你的代码一起封进一个镜像，`docker run` 到哪都一样。**容器之于 Python，约等于 fat jar + 一份打包进去的 JVM。** 这也是为什么本章把"写好 Dockerfile"当成核心技能——它是 Python 部署的"打包"环节。

### 2.4 可观测性三支柱：logs / metrics / traces

Spring Boot 把可观测性做成了"开箱即用"：加个 `spring-boot-starter-actuator`，健康端点、指标、追踪自动就有了。Python 没有这种"全家桶 starter"，你要**自己拼**——但拼的是同一套业界标准的三支柱，概念和 Java 完全互通：

| 支柱 | 回答的问题 | Java | Python |
|------|-----------|------|--------|
| **Logs（日志）** | "出事时具体发生了什么"（离散事件） | Logback → ELK | `logging` → JSON → stdout（[29 章](29_工程化最佳实践.md)） |
| **Metrics（指标）** | "系统整体健康吗"（聚合数值，QPS/延迟/错误率） | Micrometer | `prometheus_client` |
| **Traces（追踪）** | "一个请求跨服务走了哪些路、慢在哪" | Sleuth / Micrometer Tracing | OpenTelemetry |

三者互补：指标告诉你"5xx 突然涨了"，追踪定位到"是下游 DB 调用慢"，日志给出"那条慢查询的具体 SQL 和参数"。本章 ③④ 会逐个落地。

### 2.5 12-factor：配置从环境注入

[29 工程化最佳实践](29_工程化最佳实践.md) 会系统讲配置管理，这里只钉住部署相关的一条 **12-factor** 原则：**配置存在环境变量里，不写死在代码/镜像里**。

同一个镜像，开发、预发、生产环境不该重新构建——只是注入不同的 `DATABASE_URL`、`LOG_LEVEL`、`OTEL_EXPORTER_OTLP_ENDPOINT`。这和 Spring 的 `application-{profile}.yml` + 环境变量覆盖是同一个目标，但 Python/容器世界更纯粹地倒向"**全用环境变量**"（K8s 的 ConfigMap/Secret 也是注入成环境变量）。镜像是不可变的二进制，配置是运行时注入的——这条原则直接影响你怎么写 Dockerfile（不要 `COPY .env`）和怎么读配置（`os.environ` / pydantic-settings）。

## ③ 实现细节（底层怎么实现的）★本章重点★

### 3.1 gunicorn 的 pre-fork worker 模型 ★

这是 Python 部署最该理解的底层机制，也是 Java 老手最陌生的部分。gunicorn 用的是经典的 **pre-fork** 模型（和 Nginx、Apache prefork MPM、PostgreSQL 同源的 Unix 服务器范式）。流程是这样的：

```
1. master 进程启动：
   - 解析配置
   - bind() 监听 socket 到 :8000，listen()           ← 关键：master 持有那个 listen fd
   - import 你的应用（取决于 preload 设置，见 3.2）

2. master fork() 出 N 个 worker 子进程：
   ┌──────────────────────────────────────────────┐
   │ master (pid 1)   ── fork ──▶ worker 1 (pid 7) │
   │   持 listen socket           继承同一个 listen fd│
   │                  ── fork ──▶ worker 2 (pid 8) │
   │                  ── fork ──▶ worker 3 (pid 9) │
   │                  ── fork ──▶ worker 4 (pid 10)│
   └──────────────────────────────────────────────┘

3. 所有 worker 共享同一个 listen socket（fork 时 fd 被继承）：
   - 每个 worker 各自 accept() 同一个监听 socket
   - 内核保证一个新连接只被一个 worker accept 到（惊群由内核/gunicorn 调和）
   - worker 拿到连接 → 解析 HTTP → 调你的 app → 回响应

4. master 只做一件事：盯着 worker（不处理任何请求）
   - 通过心跳/信号监控 worker 是否卡死、是否退出
   - worker 崩了/卡了 → master 把它 kill 掉重新 fork 一个
   - 收到 SIGTERM → 优雅地让 worker 处理完手头请求再退出
```

几个对 Java 老手的关键认知点：

**① master 不处理请求，只当"进程保姆"。** 这和 Tomcat 不同——Tomcat 的主线程也参与，而 gunicorn master 纯粹是监督进程，类比 K8s 里 Deployment 控制器对 Pod 的角色，只是缩小到了进程级。

**② N 个 worker 通过 fork 共享 listen socket，靠多进程绕开 GIL。** 这正是 2.1 的落地：每个 worker 是独立解释器、独立 GIL，运行在不同 CPU 核上，**真并行**地处理请求。8 核机器 8 个 worker，就有 8 把 GIL 同时干活——这是 Python Web 服务吃满多核的唯一正路（[20 章](20_并发与GIL.md)）。

**③ fork 的代价与 [20 章 3.4](20_并发与GIL.md) 的坑。** worker 是 fork 出来的，会继承 master 在 fork 那一刻的内存状态。如果你在 import 应用时（master 里）就建立了数据库连接、打开了文件、起了后台线程，这些状态会被**复制**到每个 worker——而连接、锁这类资源跨 fork 复制往往出问题（多个 worker 共用一个被复制的 DB socket → 数据错乱）。所以**资源要在 worker 内初始化**，gunicorn 提供 `post_fork` 钩子正是为此（见 4.2）。

### 3.2 worker 类型：sync / gthread / uvicorn worker ★

gunicorn 的 worker 不只一种，选错了性能天差地别。worker 类型决定**单个 worker 进程内部怎么处理并发**：

| worker 类型 | `-k` 值 | 单 worker 内并发模型 | 适合 | 对照 Java |
|------------|---------|---------------------|------|----------|
| **sync**（默认） | `sync` | **一次一个请求**，处理完才接下一个 | CPU 偏多、请求快、WSGI | 一个线程的 Servlet（但只有 1 个） |
| **gthread** | `gthread` | 进程内开**线程池**，每请求一线程 | I/O 偏多的 WSGI（线程等 I/O 时释放 GIL） | Tomcat 线程池（缩到单进程内） |
| **uvicorn worker** | `uvicorn.workers.UvicornWorker` | 进程内跑 **asyncio 事件循环**，单线程协程并发 | ASGI（FastAPI/Starlette） | Netty/WebFlux 的事件循环 |
| gevent / eventlet | `gevent` | 协程（monkey-patch 协程） | 老式高并发 WSGI | 类似虚拟线程 |

理解三种主力的内部结构：

**sync worker**：最朴素。一个 worker = 一个进程，进程里没有线程池，**串行**处理请求：`accept → 处理完整个请求 → accept 下一个`。并发度 = worker 数。一个请求在等数据库时，这个 worker 整个闲置（但别的 worker 还在跑）。简单、健壮、可预测，是 WSGI（Flask/Django）默认。请求里有慢 I/O 时利用率低。

**gthread worker**：在 sync 基础上，每个 worker 进程内再开一个线程池（`--threads N`）。请求分给线程处理，**线程等 I/O 时释放 GIL**（[20 章 3.1](20_并发与GIL.md)），别的线程能跑——所以对 I/O 密集的同步应用，gthread 比 sync 并发更高。总并发 ≈ `workers × threads`。注意：CPU 部分仍被 GIL 串行化，线程只对 I/O 有用。

**uvicorn worker（ASGI，FastAPI 的标配）**：这是把两层叠起来——

```
gunicorn master
   ├── worker 1  = 一个独立进程，进程里跑一个 asyncio 事件循环（uvicorn）
   │                └── 单线程，靠 async/await 协程并发成千上万个 I/O 中的请求
   ├── worker 2  = 同上，另一个进程，另一个事件循环
   └── ...（worker 数 = CPU 核数，吃满多核）
```

- **gunicorn 负责"横向"**：fork 出 N 个进程吃满 N 个核（绕 GIL，2.1）。
- **uvicorn 负责"纵向"**：每个进程内用单线程事件循环（[21 异步编程asyncio](21_异步编程asyncio.md)），一个进程就能并发处理上万个**在等 I/O** 的请求。

这就是为什么 FastAPI 的生产命令长这样：`gunicorn app:app -k uvicorn.workers.UvicornWorker -w 4`——4 个进程各自一个事件循环，既吃满 4 核，每个核又靠协程扛高并发 I/O。两个维度的并发叠加，这是 Python 异步 Web 服务的最优形态。

> ⚠️ 致命前提（呼应 [24 章 3.5](24_HTTP与Web框架.md)、[21 章 3.5](21_异步编程asyncio.md)）：uvicorn worker 内是**单线程事件循环**，一旦你的 `async` 端点里有同步阻塞调用（同步 `requests`、`time.sleep`、重 CPU 计算），整个事件循环**冻结**，这个 worker 上所有并发请求一起卡死。这比 sync worker 还糟。

> 新版说明：uvicorn 自身也支持 `--workers` 多进程（内部用 supervisor 管理子进程），所以现在很多人直接 `uvicorn app:app --workers 4` 而不再套 gunicorn。gunicorn 套 uvicorn worker 的价值在于 gunicorn 更成熟的进程管理、优雅重启、超时杀卡死 worker 等运维能力。两种都对，按团队习惯选。

### 3.3 worker 数怎么定：经验公式 vs 协程并发 ★

这是 Python 版的"JVM 调优"——但调的不是堆和线程，是**进程数**。

**对 sync / gthread（同步）worker**，gunicorn 官方经验公式：

```
workers = (2 × CPU 核数) + 1
```

为什么是 `2N+1` 而不是 `N`？因为同步 worker 串行处理，当一个 worker 阻塞在 I/O（等 DB、等下游）时它闲置，多备一些 worker 能在这空隙里接别的请求，提高 CPU 利用率。`2N+1` 是"够用又不过分"的经验值——太多 worker 会因内存（每个 worker 几十~几百 MB）和上下文切换反而变慢。这和 [20 章 4.2](20_并发与GIL.md) "CPU 密集 worker 数 = 核数"的差别在于：Web 请求通常是**混合**的（有 I/O 等待），所以略微超配。

```python
# 用代码算 worker 数（常写进 gunicorn 配置或启动脚本）
import os
cpu = os.cpu_count() or 1
workers = 2 * cpu + 1
print(f"建议 sync worker 数: {workers}")   # 例如 8 核 → 17
```

**对 uvicorn worker（异步）**，逻辑完全不同：

- 单个异步 worker 靠**事件循环 + 协程**就能并发处理海量 I/O 请求（[21 章](21_异步编程asyncio.md)），并发量不靠 worker 数堆，靠协程数。
- 所以 worker 数只需 **≈ CPU 核数**（`workers = N` 或 `N+1`），目的纯粹是吃满多核做 CPU 部分（路由、序列化、校验），**不需要 `2N+1`**——因为 I/O 并发已经由协程在单进程内解决了，多开进程只是浪费内存。

```
同步 worker（sync/gthread）：       异步 worker（uvicorn）：
  并发 = worker 数（× 线程数）          并发 = 协程数（单进程可上万）
  worker = 2N+1（补 I/O 空隙）         worker = N（仅为吃满多核）
  每加并发 = 多一个进程/线程（重）       每加并发 = 多一个协程（极轻）
```

> 容器环境的大坑：`os.cpu_count()` 返回的是**宿主机**的核数，不是容器 cgroup 限制的核数。一台 64 核机器上给容器限了 2 核，`os.cpu_count()` 仍返回 64，于是你算出 129 个 worker，内存瞬间爆炸、疯狂上下文切换。生产里 worker 数应**显式用环境变量传入**（`WEB_CONCURRENCY`，gunicorn/uvicorn 都认这个变量），由部署编排（K8s）按实际 CPU limit 设定，而不是让进程自己猜。

### 3.4 prometheus_client 在多进程下的陷阱 ★

这是 Python 可观测性最反直觉、最容易翻车的底层细节，根因还是 2.1 的多进程模型。

`prometheus_client` 默认把指标存在**进程内存里的全局对象**（一个 `Counter`/`Histogram` 就是模块级单例，内部维护计数）。在单进程下完美。但在 gunicorn 多 worker 下：

```
请求被负载到不同 worker：
  worker 1 内存里：http_requests_total = 1500
  worker 2 内存里：http_requests_total = 1480
  worker 3 内存里：http_requests_total = 1520
  worker 4 内存里：http_requests_total = 1490

Prometheus 来抓 /metrics 时，连接被随机分到某一个 worker：
  这次抓到 worker 1 → 报告 1500
  下次抓到 worker 3 → 报告 1520
  → 指标在 worker 之间「跳来跳去」，每次抓到的是某一个 worker 的局部值，
    不是全局总和。Counter 看起来甚至会「倒退」。
```

这是因为**进程间不共享内存**（[20 章 3.3](20_并发与GIL.md)），每个 worker 各自记一份，互相不知道。Java 没这问题——Micrometer 的 registry 是 JVM 内单例，所有线程共享同一份。

`prometheus_client` 的官方解法是 **multiprocess 模式**：让各 worker 把指标写到一个**共享目录的 mmap 文件**里，抓取时由一个收集器把所有 worker 的文件**聚合**起来：

```python
# 多进程模式下的 /metrics 端点（关键是用 MultiProcessCollector 聚合）
import os
from prometheus_client import CollectorRegistry, multiprocess, generate_latest

def metrics_app():
    # 前提：环境变量 PROMETHEUS_MULTIPROC_DIR 指向一个可写空目录
    registry = CollectorRegistry()
    multiprocess.MultiProcessCollector(registry)   # 聚合所有 worker 的 mmap 文件
    return generate_latest(registry)               # Prometheus 文本格式
```

要让它工作，必须：
1. 设环境变量 `PROMETHEUS_MULTIPROC_DIR=/tmp/prom`（一个空目录，每次启动前清空）。
2. worker 退出时调 `multiprocess.mark_process_dead(pid)` 清理（gunicorn 的 `child_exit` 钩子里做）。
3. 注意 `Gauge` 的多进程聚合语义要显式指定（`multiprocess_mode='livesum'` 等），`Histogram` 也是聚合各 worker 的桶。

> 实务上的简化：很多团队**不开 multiprocess 模式**，改为让 uvicorn worker 数 = 适中、每个 worker 暴露自己的 `/metrics`，靠 Prometheus 的服务发现抓每个实例并在 PromQL 里 `sum()`；或者干脆把指标推到 sidecar/statsd-exporter。但只要你在**一个进程里挂多 worker 共享一个端口**的拓扑下采集指标，就必须用 multiprocess 模式，否则数据是错的。这是 Java 转 Python 在监控上最常见的"指标对不上"事故源。

### 3.5 Docker 多阶段构建：分离 build 与 runtime ★

写 Python 的 Dockerfile，最大的两个目标是**镜像小**和**安全**，核心手段是**多阶段构建**（multi-stage build）。理解它要先看一个反例为什么大：

```dockerfile
# ❌ 反面教材：又大又不安全的单阶段镜像
FROM python:3.12                      # 完整镜像约 1GB（带编译器、git、各种工具）
COPY . /app                           # 把整个项目（含 .git、.venv、测试、缓存）拷进去
RUN pip install -r requirements.txt   # 编译时的 gcc、头文件都留在最终镜像里
CMD ["python", "app.py"]
```

问题：`python:3.12` 完整镜像近 1GB；装某些带 C 扩展的包要 `gcc`/`python3-dev` 这类**构建工具链**，它们只在 `pip install` 编译时需要，却永远留在了最终镜像里（增大体积、扩大攻击面）；`COPY .` 把 `.git`、虚拟环境、`__pycache__` 全拷进去了。

**多阶段构建**把"构建"和"运行"切成两个阶段，只把构建产物（装好的依赖）拷到一个干净的运行阶段，丢弃所有构建工具：

```
阶段 1 (builder)：用带编译器的镜像
   pip install → 编译 C 扩展 → 得到装好的包
                    │
                    │ 只 COPY 装好的包（不带 gcc）
                    ▼
阶段 2 (runtime)：用 slim 镜像
   拷入上一阶段的包 + 你的代码 → 最终镜像（小、无编译器）
```

最终镜像里只有解释器 + 装好的依赖 + 你的代码，没有 gcc、没有 build 缓存。体积能从 1GB 降到 150~300MB。完整可用的 Dockerfile 见 4.1。

### 3.6 slim 基础镜像 与 层缓存（呼应 02 章）

**slim 镜像**：`python:3.12-slim` 是 Debian 精简版，去掉了文档、man、一堆开发工具，约 120MB（vs 完整版近 1GB）。够跑大多数纯 Python + 预编译 wheel 的应用。还有更极端的 `python:3.12-alpine`（约 50MB，基于 musl libc）——但 **Python 场景慎用 alpine**：musl 和 glibc 不兼容，很多带 C 扩展的包（numpy/pandas）在 alpine 上没有预编译 wheel，要从源码编译，反而又慢又大又容易出错。**生产推荐 slim，不推荐 alpine。**

**层缓存（layer caching）**：Docker 镜像是分层的，每条指令是一层，**只要某层及其之前的内容没变，就复用缓存不重跑**。这直接呼应 [02 环境与工具链](02_环境与工具链.md) 讲的依赖管理——关键技巧是**先拷依赖清单、装依赖，再拷源代码**：

```dockerfile
# ✓ 正确：依赖清单单独先拷、先装 —— 改代码不会让依赖层失效
COPY requirements.txt .               # 只拷清单
RUN pip install -r requirements.txt   # 装依赖（这层会被缓存）
COPY . .                              # 最后才拷源代码

# ❌ 错误：先拷全部，改一行代码就让依赖层缓存失效，每次都重装所有包
COPY . .
RUN pip install -r requirements.txt
```

原理：你改业务代码的频率远高于改依赖。把"装依赖"放在"拷代码"**之前**，那么改代码时，`COPY requirements.txt` 和 `pip install` 两层因为输入没变而命中缓存，Docker 直接跳过——构建从几分钟变几秒。反过来，先 `COPY .` 的话，任何一行代码变动都改变了这一层的内容，导致后面的 `pip install` 层缓存全部失效，每次都重装上百个包。这是 Python（以及任何依赖管理）Dockerfile 的头号优化，和 Maven 的依赖缓存层一个道理。

### 3.7 为什么不该用 root 跑容器

默认 Docker 容器以 **root（uid 0）** 运行进程。这是个安全隐患：如果应用被攻破（RCE），攻击者在容器里就是 root，配合容器逃逸漏洞可能危及宿主机；即便不逃逸，root 也能改容器内任意文件、读任何挂载卷。

最小权限原则要求**用非 root 用户跑应用**。在 Dockerfile 里建一个普通用户并 `USER` 切过去：

```dockerfile
RUN useradd --create-home --uid 1000 appuser
USER appuser                          # 之后的进程都以 appuser(uid 1000) 运行
```

Java 容器同理（Spring Boot 官方镜像也建议非 root），但 Python 社区因为大量教程图省事用 root，翻车更常见。注意：切了非 root 后，应用要写的目录（如 `PROMETHEUS_MULTIPROC_DIR`、日志临时目录）必须对该用户**可写**，否则启动报权限错。

### 3.8 OpenTelemetry 自动埋点与 trace 传播（呼应 21 章 contextvars）

OpenTelemetry（OTel）是 Sleuth/Micrometer Tracing 的对应物，但它是**跨语言的开放标准**——同一套 traceId 能贯穿你的 FastAPI 服务、Java 下游、Go 下游。两个底层机制值得 Java 老手理解：

**① 自动埋点（auto-instrumentation）靠 monkey-patch。** 你不用改业务代码，用 `opentelemetry-instrument` 命令包住启动：

```bash
opentelemetry-instrument uvicorn app:app
```

它在导入时**猴补丁（monkey-patch）**常见库——给 FastAPI/Starlette、httpx、requests、asyncpg、redis 等的关键方法包一层 span 创建逻辑。于是每个 HTTP 请求自动开一个 span、每次下游 httpx 调用自动是子 span，全程零业务代码改动。这对应 Sleuth 用 AOP/字节码增强自动织入 traceId，只是 Python 用的是运行时方法替换。

**② trace 上下文靠 contextvars 传播（[21 章 3.7](21_异步编程asyncio.md)）。** 一个请求在异步服务里跑在协程上，成千上万个请求复用同一个线程（事件循环）。要让"当前 span"跟着请求走而不串味，**绝不能用 `threading.local`**（[21 章](21_异步编程asyncio.md) 讲过会在协程间串味）——OTel 用的正是 `contextvars.ContextVar` 存当前 span 上下文。事件循环切换协程时自动保存/恢复 context，所以 traceId/spanId 精确地跟着每个请求，跨 `await` 不丢、不串。跨服务时，traceId 通过 HTTP 头 **`traceparent`（W3C Trace Context 标准）**传给下游——httpx 的自动埋点会把它注入出站请求头，下游服务读出来续上同一条 trace。这和 Java 的传播机制（B3/W3C 头）完全互通。

### 4.1 生产级 Dockerfile：多阶段 / slim / 非 root / 层缓存

把 3.5~3.7 的所有要点落到一个完整、可直接用的 Dockerfile：

```dockerfile
# syntax=docker/dockerfile:1
# ===== 阶段 1：builder —— 装依赖、编译 C 扩展 =====
FROM python:3.12-slim AS builder

# 装编译某些 C 扩展可能需要的工具链（仅这一阶段需要，最终镜像不含）
RUN apt-get update && apt-get install -y --no-install-recommends gcc \
    && rm -rf /var/lib/apt/lists/*

WORKDIR /app

# 关键：把依赖装进一个独立的虚拟环境目录，便于整体拷到下一阶段（3.5）
ENV VIRTUAL_ENV=/opt/venv
RUN python -m venv $VIRTUAL_ENV
ENV PATH="$VIRTUAL_ENV/bin:$PATH"

# ★层缓存：先只拷依赖清单再装，改代码不会让这层失效（3.6，呼应 02 章）
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# ===== 阶段 2：runtime —— 干净的运行镜像（不含 gcc / build 缓存）=====
FROM python:3.12-slim AS runtime

# 12-factor & 容器友好的环境变量
ENV PYTHONUNBUFFERED=1 \           # ★日志立刻刷到 stdout，不被缓冲（容器关键，见 4.4/⑤）
    PYTHONDONTWRITEBYTECODE=1 \    # 不写 .pyc，省空间
    PATH="/opt/venv/bin:$PATH"

# 建非 root 用户（3.7）
RUN useradd --create-home --uid 1000 appuser

WORKDIR /app

# 只从 builder 拷「装好的虚拟环境」——这就是多阶段的价值：不带编译器进最终镜像
COPY --from=builder /opt/venv /opt/venv

# 最后才拷源代码（层缓存：改代码只重跑这一层，依赖层命中缓存）
COPY --chown=appuser:appuser . .

USER appuser                       # 切到非 root（3.7）

EXPOSE 8000

# uvicorn worker 跑 FastAPI；worker 数从环境变量 WEB_CONCURRENCY 注入（3.3 容器陷阱）
CMD ["gunicorn", "app:app", \
     "-k", "uvicorn.workers.UvicornWorker", \
     "--bind", "0.0.0.0:8000", \
     "--workers", "4", \
     "--access-logfile", "-", \    # "-" = 写到 stdout（4.4）
     "--error-logfile", "-"]
```

配套一个 `.dockerignore`（避免把 `.git`/`.venv`/缓存拷进镜像，呼应 3.5、⑤ 的"把 .venv 拷进镜像"坑）：

```gitignore
# .dockerignore
.git
.venv
__pycache__
*.pyc
.pytest_cache
.mypy_cache
tests/
.env
```

> 为什么拷 `.venv` 是大坑：本机的 `.venv` 里装的 C 扩展是**为你的 OS/架构编译**的，拷进 Linux 容器（甚至不同架构）直接不能用；而且它体积大、含路径硬编码。永远在镜像内用 `pip install` 重新装，靠 `.dockerignore` 把本机 `.venv` 挡在外面。

### 4.2 gunicorn 配置文件与生命周期钩子

复杂配置写进 `gunicorn.conf.py`（比一长串命令行参数清晰），并用钩子处理 fork 相关的资源初始化（呼应 3.1③）：

```python
# gunicorn.conf.py  —  启动：gunicorn app:app -c gunicorn.conf.py
import os

# ---- 基本配置 ----
bind = "0.0.0.0:8000"
worker_class = "uvicorn.workers.UvicornWorker"     # ASGI（FastAPI）
# worker 数从环境变量注入，由部署平台按容器 CPU limit 设定（3.3 容器陷阱）
workers = int(os.environ.get("WEB_CONCURRENCY", "4"))

# ---- 超时与健壮性 ----
timeout = 30              # worker 处理单请求超过 30s 被 master 杀掉重启（防卡死）
graceful_timeout = 30     # 收到重启信号后，给 worker 最多 30s 处理完手头请求（4.6）
keepalive = 5             # keep-alive 连接保持秒数
max_requests = 1000       # 每个 worker 处理 1000 个请求后重启 —— 防内存泄漏累积
max_requests_jitter = 100 # 加随机抖动，避免所有 worker 同时重启

# ---- 日志到 stdout/stderr（容器环境，4.4）----
accesslog = "-"
errorlog = "-"

# ---- 多进程指标清理（3.4）----
def child_exit(server, worker):
    # worker 退出时清理它的 prometheus mmap 文件，避免聚合到已死 worker 的旧值
    from prometheus_client import multiprocess
    multiprocess.mark_process_dead(worker.pid)

def post_fork(server, worker):
    # ★在 worker 内（而非 master 内）初始化资源：DB 连接池、客户端等（3.1③、20 章 3.4）
    # 这样每个 worker 有自己独立的连接，不会跨 fork 复制同一个 socket 导致错乱
    server.log.info(f"worker {worker.pid} forked, init per-worker resources here")
```

### 4.3 健康检查端点：liveness 与 readiness

对照 Spring Actuator 的 `/actuator/health`，Python 要手写——但概念一样，且 K8s 部署里要区分两种探针：

```python
# health.py —— 挂到 FastAPI 应用上
from fastapi import APIRouter, Response, status

router = APIRouter()

@router.get("/healthz")        # liveness：进程还活着吗？（K8s livenessProbe）
async def liveness():
    # 只检查"进程没死锁"，不查下游——否则下游抖动会让 K8s 误杀你的 Pod
    return {"status": "ok"}

@router.get("/readyz")         # readiness：能接流量吗？（K8s readinessProbe）
async def readiness(response: Response):
    # 检查关键依赖（DB、缓存、下游）是否就绪；没就绪就返回 503，K8s 暂不导流量
    checks = {"db": await check_db(), "cache": await check_cache()}
    if all(checks.values()):
        return {"status": "ready", "checks": checks}
    response.status_code = status.HTTP_503_SERVICE_UNAVAILABLE
    return {"status": "not_ready", "checks": checks}

async def check_db() -> bool:
    try:
        # await db.execute("SELECT 1")   # 真实：执行一条轻量查询
        return True
    except Exception:
        return False

async def check_cache() -> bool:
    return True
```

> **关键区分（Java 老手常合二为一翻车）**：
> - **liveness（`/healthz`）**只回答"进程要不要被重启"，**绝不能依赖下游**——若你在 liveness 里查 DB，DB 抖一下 K8s 就把你所有 Pod 杀光重启，雪上加霜。
> - **readiness（`/readyz`）**回答"要不要给我导流量"，**应该查下游**——下游没好时返回 503，K8s 把流量切走但不杀你，等就绪了自动恢复。
> 这正是 Spring Actuator `livenessState` 与 `readinessState` 的区分，K8s 里务必分开配。

### 4.4 结构化 JSON 日志输出到 stdout（呼应 29 章）

容器环境的日志铁律：**写 stdout/stderr，不写文件**。原因——容器是无状态、可随时销毁的，写进容器内文件的日志会随容器消失；而 Docker/K8s 的日志采集（fluentd/promtail）统一从容器的 stdout 收集。这和 [29 工程化最佳实践](29_工程化最佳实践.md) 的日志原则一致，部署层再强调一遍。结构化成 **JSON**，便于 ELK/Loki 检索：

```python
# logging_setup.py —— 结构化 JSON 日志到 stdout
import logging
import json
import sys
from datetime import datetime, timezone

class JSONFormatter(logging.Formatter):
    def format(self, record: logging.LogRecord) -> str:
        log = {
            "ts": datetime.now(timezone.utc).isoformat(),
            "level": record.levelname,
            "logger": record.name,
            "msg": record.getMessage(),
        }
        # 把 trace 上下文带进日志（与 OTel 关联，见 4.7）——靠 contextvars，不串味
        if trace_id := getattr(record, "trace_id", None):
            log["trace_id"] = trace_id
        if record.exc_info:
            log["exc"] = self.formatException(record.exc_info)
        return json.dumps(log, ensure_ascii=False)

def setup_logging(level: str = "INFO") -> None:
    handler = logging.StreamHandler(sys.stdout)    # ★stdout，不是文件
    handler.setFormatter(JSONFormatter())
    root = logging.getLogger()
    root.handlers.clear()
    root.addHandler(handler)
    root.setLevel(level)

# 用法：
# setup_logging(os.environ.get("LOG_LEVEL", "INFO"))   # 级别从环境变量来（12-factor）
# logging.getLogger("app").info("service started")
# → {"ts": "2026-06-28T12:00:00+00:00", "level": "INFO", "logger": "app", "msg": "service started"}
```

> 配合 Dockerfile 里的 `ENV PYTHONUNBUFFERED=1`（4.1）——否则 Python 的 stdout 是块缓冲的，日志会卡在缓冲区里不及时输出，容器里看起来"没日志"，排障时抓狂。这是 Python 容器化的高频暗坑。

### 4.5 暴露 /metrics 与用 prometheus_client 埋指标

对照 Micrometer 的 `Counter`/`Timer`/`Gauge`，`prometheus_client` 提供 `Counter`/`Histogram`/`Gauge`。三类核心指标埋法：

```python
# metrics.py
import time
from prometheus_client import Counter, Histogram, Gauge
from fastapi import FastAPI, Request

# Counter：只增不减（请求总数、错误总数）—— 对应 Micrometer Counter
REQUESTS = Counter(
    "http_requests_total", "HTTP 请求总数",
    ["method", "path", "status"],          # 标签（维度），对应 Micrometer tags
)
# Histogram：分布（延迟），自带分桶 —— 对应 Micrometer Timer
LATENCY = Histogram(
    "http_request_duration_seconds", "请求延迟",
    ["method", "path"],
    buckets=(0.01, 0.05, 0.1, 0.5, 1.0, 2.5, 5.0),   # 自定义桶，按 SLA 设
)
# Gauge：可增可减的瞬时值（当前并发数、队列长度）—— 对应 Micrometer Gauge
IN_PROGRESS = Gauge("http_requests_in_progress", "处理中的请求数")

def setup_metrics(app: FastAPI) -> None:
    @app.middleware("http")
    async def instrument(request: Request, call_next):
        IN_PROGRESS.inc()
        start = time.perf_counter()
        try:
            response = await call_next(request)
            status = response.status_code
        except Exception:
            status = 500
            raise
        finally:
            elapsed = time.perf_counter() - start
            # 注意：path 用「路由模板」而非原始 URL，否则 /items/1 /items/2 ... 标签爆炸
            path = request.scope.get("route").path if request.scope.get("route") else request.url.path
            REQUESTS.labels(request.method, path, status).inc()
            LATENCY.labels(request.method, path).observe(elapsed)
            IN_PROGRESS.dec()
        return response
```

`/metrics` 端点（兼容单进程与多进程，3.4）：

```python
import os
from fastapi import FastAPI, Response
from prometheus_client import generate_latest, CONTENT_TYPE_LATEST, REGISTRY

app = FastAPI()

@app.get("/metrics")
async def metrics():
    if os.environ.get("PROMETHEUS_MULTIPROC_DIR"):
        # 多进程模式：聚合所有 worker（3.4）
        from prometheus_client import CollectorRegistry, multiprocess
        registry = CollectorRegistry()
        multiprocess.MultiProcessCollector(registry)
        data = generate_latest(registry)
    else:
        data = generate_latest(REGISTRY)            # 单进程
    return Response(data, media_type=CONTENT_TYPE_LATEST)
```

> 标签基数（cardinality）是 Prometheus 头号杀手，和 Micrometer 一模一样的坑：**标签值必须是有限集合**。用路由模板 `/items/{id}` 而非真实路径 `/items/123`，否则每个不同 id 造一条时间序列，几小时就把 Prometheus 撑爆。同理别拿 user_id、trace_id 当标签。

### 4.6 优雅关闭（graceful shutdown）

K8s 滚动更新、缩容时给容器发 **SIGTERM**，进程要"处理完手头请求再退出"，而不是粗暴断连。gunicorn/uvicorn 默认就响应 SIGTERM 做优雅关闭（worker 停止接新请求、等手头请求做完、超过 `graceful_timeout` 才强杀，见 4.2）。应用层若有需要清理的资源（连接池、刷指标），用 FastAPI 的 lifespan：

```python
from contextlib import asynccontextmanager
from fastapi import FastAPI
import httpx

@asynccontextmanager
async def lifespan(app: FastAPI):
    # 启动：初始化资源（在 worker 内，呼应 3.1③）
    app.state.client = httpx.AsyncClient(timeout=10.0)
    yield
    # 关闭：收到 SIGTERM 优雅退出时执行，释放资源
    await app.state.client.aclose()

app = FastAPI(lifespan=lifespan)
```

> K8s 实务：容器 `terminationGracePeriodSeconds` 要 ≥ gunicorn 的 `graceful_timeout`，否则 K8s 在 worker 还没处理完时就 SIGKILL，照样断连。这对应 Spring 的 `server.shutdown=graceful` + `spring.lifecycle.timeout-per-shutdown-phase`。

### 4.7 接入 OpenTelemetry（自动埋点）

最省事的方式是**零代码自动埋点**（3.8），用 `opentelemetry-instrument` 包住启动命令：

```bash
# 1. 安装并自动检测已装的库，下载对应 instrumentation
pip install opentelemetry-distro opentelemetry-exporter-otlp
opentelemetry-bootstrap -a install         # 扫描已装库，装上对应的自动埋点包

# 2. 用 opentelemetry-instrument 包住启动（配置全走环境变量，12-factor）
export OTEL_SERVICE_NAME=item-api
export OTEL_EXPORTER_OTLP_ENDPOINT=http://otel-collector:4317
export OTEL_TRACES_EXPORTER=otlp
opentelemetry-instrument gunicorn app:app -k uvicorn.workers.UvicornWorker -w 4
```

这样 FastAPI 的每个请求、每次 httpx 出站调用、每次 DB 查询都自动成为 span，traceId 通过 `traceparent` 头跨服务传播（3.8）。需要在业务里手动加 span（自定义埋点）时：

```python
from opentelemetry import trace

tracer = trace.get_tracer(__name__)

async def compute_score(raw: dict) -> float:
    with tracer.start_as_current_span("compute_score") as span:   # 手动 span
        span.set_attribute("item.id", raw["id"])
        result = await do_scoring(raw)
        span.set_attribute("score", result)
        return result
```

把 traceId 关联进日志（4.4 的 `trace_id` 字段），三支柱就打通了——指标发现异常 → trace 定位慢点 → 日志看具体上下文：

```python
from opentelemetry import trace

def current_trace_id() -> str | None:
    span = trace.get_current_span()
    ctx = span.get_span_context()
    if ctx.is_valid:
        return format(ctx.trace_id, "032x")   # W3C 格式的 traceId
    return None
# 在日志 filter 里把它塞进 record.trace_id（4.4），日志和 trace 就能互相跳转
```

## ⑥ 练习题

### 基础理解

**1.** 你的 Java 同事看到 FastAPI 生产命令 `gunicorn app:app -k uvicorn.workers.UvicornWorker -w 4`，问："Tomcat 一个进程开几百个线程就够了，为什么 Python 要开 4 个进程？这 4 个进程和 uvicorn worker 又是什么关系？" 请解释：(a) 为什么是多进程而非多线程；(b) gunicorn 的 4 个进程和每个进程内 uvicorn 事件循环各自负责哪个维度的并发。

**2.** 对"同步 worker（sync/gthread）"和"异步 worker（uvicorn）"，worker 数的经验取值分别是多少（用 CPU 核数 N 表示）？为什么两者不同？另外说明：为什么在容器里**不能**让程序用 `os.cpu_count()` 自己决定 worker 数？

### 找坑改错

**3.** 下面这个 Dockerfile 有**至少四处**会导致镜像巨大、构建慢或不安全的问题，逐一指出并改正：

```dockerfile
FROM python:3.12

WORKDIR /app
COPY . .
RUN pip install -r requirements.txt

CMD ["gunicorn", "app:app", "-w", "1", "--bind", "0.0.0.0:8000"]
```

（补充信息：`app:app` 是一个 FastAPI 应用；运行在 8 核容器上；项目目录里有本机的 `.venv/` 和 `.git/`。）

**4.** 一个团队把 FastAPI 应用用 `gunicorn -w 8 -k uvicorn.workers.UvicornWorker` 部署，挂了 `prometheus_client` 的默认 `Counter` 统计 `http_requests_total`，并在端点用 `generate_latest(REGISTRY)` 暴露 `/metrics`。上线后 Grafana 里这个 Counter 曲线**剧烈抖动、时不时往下掉**，看起来请求数在减少。请解释根因，并说明正确做法。

### 综合实战

**5.** 你要把 [24 章](24_HTTP与Web框架.md) 写的 FastAPI "图书 API" 部署上生产。请给出：
- ① 一条生产启动命令（FastAPI，吃满多核 + 高并发 I/O）；
- ② 一个生产级多阶段 Dockerfile（slim 基础镜像、层缓存、非 root、stdout 日志友好）；
- ③ 说明你会暴露哪些运维端点（健康检查 + 指标），liveness 和 readiness 的检查内容有何不同、为什么。

### 参考答案

**1.**
- **(a) 多进程而非多线程**：CPython 有 GIL（[05](05_CPython内部.md)/[20 章](20_并发与GIL.md)），同一进程内同一时刻只有一个线程能执行 Python 字节码，所以一个进程的 CPU 部分（路由、序列化、pydantic 校验、业务计算）永远只用满一个核。Java 的 JVM 无 GIL，一个进程多线程就能真并行吃满多核；Python 必须开多个**进程**，每个进程有独立解释器和独立 GIL，才能在多核上真并行。`-w 4` = 4 个进程 = 4 把 GIL 跑在 4 个核上。
- **(b) 两个维度**：gunicorn 的 4 个进程负责"**横向**"——fork 出 4 个 worker 吃满 4 个 CPU 核（绕 GIL）；每个进程内的 uvicorn 事件循环负责"**纵向**"——单线程靠 async/await 协程并发处理成千上万个**在等 I/O** 的请求（[21 章](21_异步编程asyncio.md)）。前者解决"用满多核"，后者解决"单核内高并发 I/O"，叠加起来既吃满 CPU 又扛高并发连接。

**2.** 同步 worker：`workers = 2×N + 1`（N=CPU 核数）；异步 uvicorn worker：`workers ≈ N`（或 N+1）。差异原因：同步 worker 串行处理，一个请求阻塞在 I/O 时该 worker 闲置，多备 worker（`2N+1`）能填补 I/O 空隙、提高 CPU 利用率，每增并发要多一个进程/线程（重）；异步 worker 的 I/O 并发已由单进程内的事件循环 + 协程解决，每增并发只是多一个协程（极轻），所以 worker 数只需 ≈ 核数来吃满多核做 CPU 部分，多开纯属浪费内存。容器里不能用 `os.cpu_count()` 自决，因为它返回**宿主机**核数而非容器 cgroup 的 CPU limit——64 核宿主上给容器限 2 核时它仍返回 64，算出的 worker 数会暴涨导致内存爆炸和疯狂上下文切换；正确做法是用 `WEB_CONCURRENCY` 环境变量由部署平台按实际 CPU limit 注入。

**3.** 四处（及以上）问题：
- **用完整 `python:3.12`（近 1GB）+ 单阶段**：带编译器和构建工具留在最终镜像，体积大、攻击面广。改用**多阶段构建** + `python:3.12-slim`，builder 阶段装依赖、runtime 阶段只拷装好的包。
- **先 `COPY .` 再 `pip install`**：改任何代码都让依赖层缓存失效，每次重装所有包。改成**先 `COPY requirements.txt` + `pip install`，最后 `COPY .`**。
- **没有 `.dockerignore`，会把 `.venv/`、`.git/` 拷进镜像**：`.venv` 里的 C 扩展是本机编译的、容器里不能用且巨大；`.git` 泄漏历史。加 `.dockerignore` 排除它们。
- **root 运行**：没建非 root 用户。加 `useradd` + `USER appuser`。
- **`-w 1` 单 worker**：8 核机器只用一核，吞吐严重不足；且 FastAPI 是 ASGI 但用了默认 sync worker。改 `-w` 为合适值（ASGI 异步 ≈ 核数，如 `-w 8` 或用 `WEB_CONCURRENCY`）并加 `-k uvicorn.workers.UvicornWorker`。
- （加分）没设 `ENV PYTHONUNBUFFERED=1`，日志会被缓冲。

改正版要点（结构同 4.1 的 Dockerfile）：多阶段 + slim + 先装依赖后拷码 + 非 root + `-k uvicorn.workers.UvicornWorker` + `WEB_CONCURRENCY` + `.dockerignore` + `PYTHONUNBUFFERED=1`。

**4.** 根因：`prometheus_client` 的默认 `Counter` 把计数存在**进程内存**里，而 `-w 8` 起了 8 个独立 worker 进程，进程间不共享内存（[20 章 3.3](20_并发与GIL.md)），所以每个 worker 各记自己处理的那部分请求数。Prometheus 抓 `/metrics` 时，连接被随机负载到某一个 worker，于是每次抓到的是**某一个 worker 的局部计数**而非全局总和——这次抓到 worker A（值大）、下次抓到 worker B（值小），曲线就剧烈抖动甚至"倒退"（Counter 本不该减小）。正确做法：开启 `prometheus_client` 的 **multiprocess 模式**——设环境变量 `PROMETHEUS_MULTIPROC_DIR` 指向一个共享空目录，各 worker 把指标写进该目录的 mmap 文件，`/metrics` 端点用 `CollectorRegistry()` + `multiprocess.MultiProcessCollector(registry)` 聚合所有 worker 的文件再 `generate_latest`（见 3.4、4.5）；并在 gunicorn 的 `child_exit` 钩子里调 `multiprocess.mark_process_dead(worker.pid)` 清理死掉的 worker。

**5.** 参考方案：

**① 生产启动命令**（FastAPI，吃满多核 + 协程高并发 I/O）：
```bash
WEB_CONCURRENCY=8 gunicorn books:app \
    -k uvicorn.workers.UvicornWorker \
    --bind 0.0.0.0:8000 \
    --access-logfile - --error-logfile - \
    --timeout 30 --graceful-timeout 30 --max-requests 1000
# 8 个 worker 进程吃满 8 核（绕 GIL），每个进程内 uvicorn 事件循环靠协程扛高并发 I/O
```

**② 多阶段 Dockerfile**（与 4.1 同构，要点齐全）：
```dockerfile
# syntax=docker/dockerfile:1
FROM python:3.12-slim AS builder
RUN apt-get update && apt-get install -y --no-install-recommends gcc \
    && rm -rf /var/lib/apt/lists/*
WORKDIR /app
RUN python -m venv /opt/venv
ENV PATH="/opt/venv/bin:$PATH"
COPY requirements.txt .                          # 层缓存：先装依赖
RUN pip install --no-cache-dir -r requirements.txt

FROM python:3.12-slim AS runtime
ENV PYTHONUNBUFFERED=1 PYTHONDONTWRITEBYTECODE=1 PATH="/opt/venv/bin:$PATH"
RUN useradd --create-home --uid 1000 appuser     # 非 root
WORKDIR /app
COPY --from=builder /opt/venv /opt/venv          # 只拷装好的包（不带 gcc）
COPY --chown=appuser:appuser . .                 # 最后拷源码
USER appuser
EXPOSE 8000
CMD ["gunicorn", "books:app", "-k", "uvicorn.workers.UvicornWorker", \
     "--bind", "0.0.0.0:8000", "--access-logfile", "-", "--error-logfile", "-"]
```
配 `.dockerignore`（排除 `.git`/`.venv`/`__pycache__`/`tests`/`.env`）。worker 数靠 `WEB_CONCURRENCY` 注入。

**③ 运维端点**：
- `/metrics`（Prometheus 指标）：`http_requests_total`（Counter，按 method/路由模板/status 打标签）、`http_request_duration_seconds`（Histogram，延迟分布）。因为多 worker，必须开 multiprocess 模式聚合（3.4）。
- `/healthz`（**liveness**）：只回答"进程是否活着、是否死锁"，**不查 DB/下游**——否则下游抖动会触发 liveness 失败，K8s 把所有 Pod 杀光重启，把局部故障放大成全面雪崩。直接返回 `{"status":"ok"}` 即可。
- `/readyz`（**readiness**）：回答"能不能接流量"，**应该查关键下游**（DB `SELECT 1`、缓存、必需的下游服务）；任一未就绪返回 503，K8s 把流量切走但**不杀** Pod，等就绪后自动恢复导流。
- 区别核心：liveness 决定"重启与否"（不能依赖外部），readiness 决定"导流与否"（应反映外部依赖状态）——对应 Spring Actuator 的 `livenessState` 与 `readinessState`。
